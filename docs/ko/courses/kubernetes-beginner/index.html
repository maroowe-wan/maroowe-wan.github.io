01. 쿠버네티스란 무엇인가 - 왜 컨테이너 오케스트레이션이 필요한가

쿠버네티스란 무엇인가 - 왜 컨테이너 오케스트레이션이 필요한가

학습 목표

  • 컨테이너만으로 서비스를 운영할 때 마주치는 한계(확장, 장애 복구, 배포)를 이해한다
  • 쿠버네티스가 해결하는 문제와 핵심 가치를 자신의 말로 설명할 수 있다
  • 오케스트레이션이라는 개념을 일상 비유로 직관적으로 이해한다

본문

왜 이걸 배우나요?

"쿠버네티스로 전환해야 한다", "우리도 컨테이너를 도입하자" 같은 말을 한 번쯤 들어보셨을 겁니다. 그런데 정작 왜 필요한지를 모른 채 따라가면, 복잡한 용어의 늪에 빠지기 쉽습니다. 이번 강의에서는 명령어나 설정 파일은 잠시 미뤄두고, "애초에 무슨 문제를 풀려고 쿠버네티스가 등장했는가" 라는 가장 근본적인 질문부터 함께 풀어보겠습니다. 이 출발점을 제대로 잡으면, 앞으로 배울 모든 개념이 "왜 이게 있는지" 자연스럽게 이해됩니다.

컨테이너부터 빠르게 짚고 가기

먼저 컨테이너(Container) 가 무엇인지 한 줄로 정리해 봅시다. 컨테이너란 애플리케이션과 그것이 돌아가는 데 필요한 모든 것(코드, 라이브러리, 설정)을 하나로 포장한 독립 실행 단위 입니다.

비유하자면 이렇습니다. 한 건물에 임차인을 들이려는데, 입주할 때마다 벽을 세우고 인테리어 공사를 다시 하면 너무 번거롭겠죠. 그래서 이사 갈 사람이 살림이 다 갖춰진 '컨테이너 박스'를 통째로 가지고 와서 건물 안에 쏙 넣는다고 상상해 보세요. 공사가 필요 없고, 나갈 때는 박스만 들어내면 됩니다. 컨테이너 기술이 바로 이렇습니다. 예전의 가상 머신(VM) 방식이 매번 벽을 세우는 무거운 공사였다면, 컨테이너는 가볍고 빠르게 옮길 수 있는 포장 박스입니다.

컨테이너 덕분에 "내 PC에서는 잘 됐는데 서버에서는 안 돼요"라는 고질적인 문제가 크게 줄었습니다. 실행 환경을 통째로 싸서 옮기기 때문입니다.

컨테이너 하나는 쉽다, 그런데 수백 개라면?

문제는 여기서 시작됩니다. 요즘 서비스는 하나의 거대한 프로그램이 아니라, 기능별로 잘게 쪼갠 작은 서비스들의 모음입니다. 이를 마이크로서비스(Microservice) 라고 부릅니다. 예를 들어 쇼핑몰 하나에도 화면을 그리는 프론트엔드, 주문을 처리하는 백엔드, 데이터를 보관하는 데이터베이스가 각각 따로 돌아갑니다. 트래픽이 몰리면 잘 나가는 메뉴(예: 프론트엔드)만 여러 개로 복제해야 하니, 컨테이너 개수는 금세 수십, 수백, 때로는 수천 개로 불어납니다.

이렇게 많은 컨테이너를, 그것도 여러 대의 서버에 걸쳐 사람이 손으로 관리한다고 상상해 보세요. 다음과 같은 일을 직접 챙겨야 합니다.

  • 어느 서버에 어떤 컨테이너를 띄울지 일일이 결정하기
  • 컨테이너가 죽으면 한밤중에라도 깨어나 다시 살리기
  • 트래픽이 몰리면 수동으로 복제본을 늘리고, 빠지면 줄이기
  • 새 버전을 배포할 때 서비스가 끊기지 않도록 조심조심 교체하기
  • 컨테이너끼리 서로를 어떻게 찾아 통신할지 직접 연결해 주기

스크립트나 자체 제작 도구로 어찌어찌 해볼 수는 있지만, 규모가 커지면 곧 한계에 부딪히고 결국 사람이 감당할 수 없는 수준이 됩니다. 바로 이 지점에서 컨테이너 오케스트레이션이 등장합니다.

오케스트레이션 - 지휘자 비유로 이해하기

오케스트레이션(Orchestration) 이라는 단어는 오케스트라에서 왔습니다. 수십 명의 연주자(컨테이너)가 각자 악기를 들고 있어도, 이들을 조화롭게 이끄는 지휘자가 없으면 음악이 아니라 소음이 됩니다. 컨테이너 오케스트레이션 도구는 바로 이 지휘자 역할을 합니다.

또 다른 비유를 들어볼까요. 당신이 여러 채의 건물을 가진 건물주라고 해봅시다. 건물마다 컨테이너를 척척 다뤄주는 전문가(이것이 바로 컨테이너 관리 도구인 도커입니다)를 한 명씩 배치했습니다. 처음엔 괜찮았지만 건물이 늘어나니, 모든 건물의 전문가에게 일일이 전화로 지시하느라 정신이 없습니다. 그래서 유능한 총괄 매니저를 한 명 고용합니다. 그 매니저의 이름이 바로 쿠버네티스입니다.

이제 당신은 매니저에게 큰 그림만 전달하면 됩니다. "프론트엔드는 3개 띄워두고, 백엔드는 2개 유지해줘." 그러면 쿠버네티스가 알아서 어느 서버에 배치할지 정하고, 하나가 죽으면 조용히 다시 살려 항상 그 개수를 맞춰 줍니다. 사소한 것 하나하나 명령하지 않아도 되는 이유는, 쿠버네티스가 원하는 상태(desired state) 를 기억하고 현실을 거기에 끊임없이 맞추기 때문입니다.

아래 흐름도처럼, 쿠버네티스는 우리가 선언한 "원하는 상태"와 실제 현재 상태를 끊임없이 비교하며 둘이 어긋나면 스스로 행동에 나섭니다. 이 반복 루프가 바로 선언형 관리의 핵심입니다.

쿠버네티스가 보장하는 핵심 가치

쿠버네티스는 구글이 만들어 오픈소스로 공개한 컨테이너 오케스트레이션 플랫폼입니다. 도커든 다른 기술이든 컨테이너의 종류를 가리지 않고, 물리 서버·가상 머신·클라우드 등 어떤 환경에서도 컨테이너 무리를 관리해 줍니다. 그 핵심 가치는 다음 네 가지로 요약할 수 있습니다.

1. 자가 치유(Self-healing)와 고가용성 — 컨테이너가 죽으면 자동으로 다시 띄워, 사용자가 느끼는 다운타임을 최소화합니다. 항상 "원하는 개수"를 유지합니다.

2. 확장성(Scalability) — 트래픽이 몰리면 복제본을 늘리고, 한가하면 줄입니다. 쿠버네티스는 각 서버의 남은 자원을 보고 어디에 띄우는 게 가장 효율적인지 판단해 배치합니다. 이 똑똑한 배치 결정을 스케줄링이라고 합니다.

3. 서비스 디스커버리와 로드 밸런싱 — 컨테이너는 자주 죽고 새로 태어나며, 그때마다 주소(IP)가 바뀝니다. 쿠버네티스는 변하지 않는 고정 창구를 앞에 세워, 뒤의 컨테이너가 바뀌어도 다른 서비스가 항상 같은 주소로 접속하고 트래픽을 고르게 나눠 받게 해줍니다.

4. 재해 복구(Disaster Recovery) — 인프라에 문제가 생겨 데이터가 날아가도, 마지막 상태로 되돌릴 수 있는 백업·복구 장치를 갖추고 있습니다.

개발자와 운영자의 시야 차이

마지막으로 한 가지 관점을 더 짚겠습니다. 같은 시스템을 봐도 개발자와 운영자의 시야는 다릅니다. 개발자는 컨테이너 안에 집중합니다. 내 코드, 의존성, 동작이 제대로 도는지에만 관심이 있죠. 반면 운영자는 그 아래 전체 그림을 봅니다. 어느 서버에 무엇을 배포하고, 어떻게 확장하고, 외부에 어떻게 노출하고, 장애를 어떻게 감지할지를 책임집니다.

아래 계층도처럼, 개발자는 위쪽의 "내 컨테이너"에 집중하고, 쿠버네티스가 그 아래 운영 영역을 맡아 여러 서버에 흩어진 컨테이너 무리를 대신 다스립니다.

쿠버네티스는 이 운영자의 무거운 짐을 대부분 자동화해 줍니다. 그래서 개발자는 "내 컨테이너"에만 집중할 수 있고, 운영팀은 수백 개의 컨테이너를 사람의 야근이 아니라 선언된 규칙으로 다스릴 수 있게 됩니다. 최근 SRE(사이트 신뢰성 엔지니어) 같은 역할이 주목받는 것도 이런 자동화된 운영 위에서 안정성을 설계하는 일이 중요해졌기 때문입니다.

핵심 요약

  • 컨테이너는 앱과 실행 환경을 통째로 포장한 가볍고 빠른 단위지만, 컨테이너가 수백 개로 늘고 여러 서버에 흩어지면 사람이 손으로 관리하기란 불가능에 가깝다.
  • 이 문제를 풀기 위한 도구가 컨테이너 오케스트레이션이고, 그 사실상의 표준이 구글이 만든 오픈소스 플랫폼 쿠버네티스다.
  • 쿠버네티스는 오케스트라의 지휘자(또는 건물 총괄 매니저)처럼, 우리가 "원하는 상태"만 알려주면 배치·복제·자가 치유·통신 연결을 알아서 맞춰 준다.
  • 핵심 가치는 자가 치유와 고가용성, 확장성과 스마트 스케줄링, 서비스 디스커버리·로드 밸런싱, 재해 복구의 네 가지다.
  • 이 "왜"를 이해했다면, 다음 강의에서 다룰 쿠버네티스의 내부 구조(아키텍처)가 훨씬 자연스럽게 읽힐 것이다.

Q1. 컨테이너 기술을 도입해 마이크로서비스 환경을 운영한다고 할 때, 수백 개의 컨테이너를 사람이 직접 관리하면 어떤 문제가 발생하는가? 본문에서 언급된 내용으로 가장 적절한 것을 고르시오.

  • 컨테이너 이미지 빌드 속도가 느려져 개발 생산성이 떨어진다
  • 어느 서버에 컨테이너를 배치할지, 장애 시 재시작, 트래픽 증감에 따른 복제본 조절 등을 사람이 일일이 처리해야 한다
  • 컨테이너가 가상 머신보다 무거워서 서버 자원을 과도하게 소모한다
  • 컨테이너 내부 코드의 버그를 자동으로 탐지하기 어렵다
정답은 2번입니다. 본문은 수백 개의 컨테이너를 손으로 관리할 때 발생하는 구체적인 문제로 '어느 서버에 배치할지 결정', '컨테이너 장애 시 수동 재시작', '트래픽 변화에 따른 복제본 수동 조절', '무중단 배포', '컨테이너 간 통신 연결'을 열거합니다. 1번은 빌드 속도 문제로 오케스트레이션과 무관합니다. 3번은 사실과 반대입니다. 컨테이너는 VM보다 훨씬 가볍습니다. 4번은 코드 품질 문제로 오케스트레이션 도구가 다루는 영역이 아닙니다.

Q2. 쿠버네티스의 '원하는 상태(desired state)' 관리 방식을 가장 잘 설명한 것은 무엇인가?

  • 운영자가 컨테이너 하나하나에 명령어를 실행해 정해진 절차대로 직접 제어한다
  • 사용자가 목표 상태를 선언하면 쿠버네티스가 현재 상태와 지속적으로 비교해 스스로 조정한다
  • 컨테이너가 죽을 때마다 알림을 보내 관리자가 수동으로 복구 여부를 결정한다
  • 한 번 배포한 컨테이너 구성은 변경 없이 고정되어 안정성을 보장한다
정답은 2번입니다. 쿠버네티스의 핵심은 '선언형 관리'입니다. 사용자는 '프론트엔드 3개, 백엔드 2개'처럼 원하는 결과만 선언하고, 쿠버네티스는 현재 상태와 끊임없이 비교해 어긋나면 자동으로 맞춥니다. 본문의 조정 루프(flowchart)가 이를 시각화합니다. 1번은 선언형의 반대인 '명령형' 방식으로 쿠버네티스가 해결하려는 바로 그 문제 상황입니다. 3번은 알림 후 수동 복구로, 자가 치유와 정반대입니다. 4번은 쿠버네티스가 동적으로 상태를 조정한다는 사실과 모순됩니다.

Q3. 다음 시나리오를 읽고 쿠버네티스가 수행할 동작으로 가장 적절한 것을 고르시오. [시나리오] 새벽 2시, 쇼핑몰 서비스에 이벤트 트래픽이 급증하면서 프론트엔드 컨테이너 한 대가 과부하로 다운됐다. 현재 명세에는 프론트엔드를 항상 3개 유지하도록 선언되어 있다.

  • 쿠버네티스는 상태 변화를 감지하고 담당 운영자에게 SMS를 발송한 뒤 운영자의 승인을 기다린다
  • 쿠버네티스는 다운된 컨테이너를 감지해 자동으로 새 컨테이너를 생성하고, 트래픽이 더 몰리면 추가 복제본을 띄워 3개 상태를 회복한다
  • 쿠버네티스는 프론트엔드 서비스 전체를 일시 중단하고 안정적인 환경에서 재배포한다
  • 쿠버네티스는 다운된 컨테이너의 로그를 수집해 데이터베이스에 저장하고 대기한다
정답은 2번입니다. 이 문항은 '자가 치유(Self-healing)'와 '확장성(Scalability)'이라는 두 핵심 가치가 실제 장애 상황에서 어떻게 작동하는지를 묻는 판단형 문제입니다. 쿠버네티스는 원하는 상태(프론트엔드 3개)를 기억하고, 현실(1개 다운)과 어긋나면 즉시 새 컨테이너를 자동 생성해 3개를 회복합니다. 새벽이든 주말이든 관계없이 사람의 개입 없이 동작합니다. 1번은 사람 승인을 요구하므로 자동화의 의미가 없습니다. 3번은 불필요한 전체 중단으로 고가용성 목표와 반대됩니다. 4번은 로그 수집만 하고 행동하지 않으므로 자가 치유가 아닙니다.

Q4. 본문에서 쿠버네티스를 '건물 총괄 매니저'에, 도커를 '각 건물의 컨테이너 전문가'에 비유한 이유로 가장 적절한 것은?

  • 도커는 단일 서버에서 컨테이너를 실행하는 역할을 하고, 쿠버네티스는 여러 서버에 흩어진 컨테이너들을 통합적으로 조율하기 때문이다
  • 도커는 오래된 기술이고 쿠버네티스가 도커를 완전히 대체했기 때문이다
  • 쿠버네티스는 도커 없이 독자적으로 컨테이너를 생성·실행할 수 있기 때문이다
  • 도커가 네트워크와 보안을 담당하고 쿠버네티스가 스토리지를 전담하기 때문이다
정답은 1번입니다. 본문의 비유에서 '건물마다 전문가(도커)를 배치했지만 건물이 늘어나자 총괄 매니저(쿠버네티스)가 필요해졌다'는 구조는, 도커가 단일 호스트에서 컨테이너를 관리하는 도구이고 쿠버네티스가 수십 대의 서버(노드)에 분산된 컨테이너 전체를 조율하는 상위 계층임을 나타냅니다. 2번은 사실이 아닙니다. 도커는 여전히 컨테이너 런타임으로 널리 쓰이며, 쿠버네티스는 도커를 포함한 다양한 런타임을 지원합니다. 3번은 부분적으로 맞을 수 있으나, 비유의 핵심 의도(단일 vs. 다중 서버 관리)와 다릅니다. 4번은 본문에서 전혀 언급되지 않은 역할 분담 방식입니다.

Q5. 다음 중 쿠버네티스의 '서비스 디스커버리와 로드 밸런싱' 기능이 필요한 근본적인 이유로 가장 적절한 것은?

  • 컨테이너 이미지의 버전 정보를 중앙에서 일괄 관리하기 위해서
  • 컨테이너는 죽고 새로 생성될 때마다 IP 주소가 바뀌기 때문에, 고정된 접속 창구 없이는 다른 서비스가 정확한 주소를 알 수 없기 때문이다
  • 여러 개발자가 동시에 같은 컨테이너에 코드를 배포할 때 충돌을 방지하기 위해서
  • 컨테이너 내부의 메모리 사용량을 실시간으로 수집해 성능을 분석하기 위해서
정답은 2번입니다. 본문은 '컨테이너는 자주 죽고 새로 태어나며, 그때마다 주소(IP)가 바뀐다'고 설명합니다. 이 동적 특성 때문에 쿠버네티스는 변하지 않는 고정 창구(서비스 오브젝트)를 앞에 두어, 뒤에서 컨테이너가 교체되어도 클라이언트가 항상 같은 주소로 접속할 수 있게 하고 트래픽도 고르게 분산합니다. 1번은 이미지 레지스트리 관리 문제로 서비스 디스커버리와 무관합니다. 3번은 코드 배포 충돌 방지로 별도의 CI/CD 영역입니다. 4번은 모니터링·관측성 도구의 역할로 서비스 디스커버리와 다릅니다.

02. 쿠버네티스 아키텍처 - 컨트롤 플레인과 워커 노드

쿠버네티스 아키텍처 - 컨트롤 플레인과 워커 노드

학습 목표

  • 클러스터, 마스터(컨트롤 플레인), 워커 노드가 각각 무슨 역할을 하는지 구분한다.
  • API Server, Scheduler, etcd, kubelet 등 핵심 구성요소가 무엇을 담당하는지 이해한다.
  • 사용자가 명령을 내렸을 때 그 요청이 클러스터 안에서 어떤 순서로 처리되는지 그릴 수 있다.

본문

왜 아키텍처를 알아야 할까

1강에서 우리는 "쿠버네티스가 컨테이너를 알아서 배치하고, 죽으면 다시 살려준다"고 배웠습니다. 그런데 그 "알아서"의 정체를 모르면, 막상 무언가 잘못됐을 때 어디를 들여다봐야 할지 막막합니다. Pod가 안 뜰 때, 어떤 부품이 일을 안 하고 있는 걸까요?

쿠버네티스는 마법 상자가 아니라, 여러 개의 작은 프로그램(프로세스)이 서로 통신하며 협력하는 시스템입니다. 이 구조를 한 번 머릿속에 그려 두면, 이후 모든 실습이 훨씬 쉬워집니다. 이번 강의는 직접 명령을 치는 실습은 없지만, 앞으로의 실습을 떠받치는 가장 중요한 밑그림입니다.

비유하자면 쿠버네티스 클러스터는 하나의 큰 물류 창고 회사와 같습니다. 실제로 물건을 나르고 쌓는 작업자들이 있는 창고(워커 노드)가 있고, 그 작업자들에게 "무엇을, 어디에, 몇 개 둘지" 지시하는 본사 관제실(컨트롤 플레인)이 있습니다.

클러스터, 컨트롤 플레인, 워커 노드

먼저 가장 큰 그림부터 봅시다.

  • 클러스터(Cluster): 여러 대의 머신(서버)을 하나로 묶어 컨테이너 앱을 실행하는 전체 시스템입니다. 회사 전체라고 보면 됩니다. 한 조직이 운영 환경용, 테스트 환경용처럼 여러 클러스터를 동시에 운영하기도 합니다.
  • 노드(Node): 클러스터를 구성하는 개별 머신 한 대입니다. 클라우드의 가상 머신(EC2 등)일 수도, 물리 서버일 수도, 심지어 여러분의 노트북일 수도 있습니다.

이 노드들은 역할에 따라 두 종류로 나뉩니다.

  • 컨트롤 플레인(Control Plane, 옛 명칭 '마스터'): 클러스터 전체를 관리하고 결정을 내리는 두뇌입니다. 사용자 요청을 받고, 어떤 컨테이너를 어디에 띄울지 정하고, 클러스터의 상태 데이터를 보관하며, 장애에 반응합니다. 직접 앱을 실행하지는 않습니다 — 명령하는 쪽입니다.
  • 워커 노드(Worker Node): 실제로 여러분의 애플리케이션이 돌아가는 작업장입니다. 컨트롤 플레인의 지시를 받아 컨테이너를 실제로 띄우고 유지합니다.

운영 환경에서는 두뇌가 멈추면 클러스터 전체가 위험하므로, 컨트롤 플레인을 여러 노드에 분산 배치해 고가용성을 확보합니다. 반면 테스트용이라면 컨트롤 플레인과 워커를 한 머신에 몰아넣어도 됩니다.

한 가지 중요한 사실: 워커 노드는 컨테이너를 직접 실행하지 않고 "Pod"라는 단위를 실행합니다. 도커에서 최소 단위가 컨테이너였다면, 쿠버네티스의 최소 배포 단위는 Pod입니다. Pod 하나 안에는 한 개 이상의 컨테이너가 들어가며, 앱을 더 늘리고 싶으면 Pod를 더 띄우면 됩니다. (Pod는 3강에서 자세히 다룹니다.)

컨트롤 플레인의 핵심 부품 4가지

관제실(컨트롤 플레인) 안에는 역할이 다른 네 개의 프로그램이 있습니다.

1. API Server (관제실의 안내데스크)

클러스터로 들어오는 모든 요청이 거쳐 가는 유일한 정문입니다. 우리가 kubectl(쿠버네티스 명령줄 도구)이나 대시보드로 명령을 내리면, 실제로는 이 API Server와 대화하는 것입니다. 심지어 워커 노드의 부품들조차 API Server를 통해서만 소통합니다. 정문인 만큼 인증(누구인지 확인)과 인가(권한이 있는지 확인), 요청이 올바른 형식인지 검증하는 일도 여기서 합니다.

2. etcd (회사의 장부)

클러스터의 모든 상태 데이터를 저장하는 키-값 저장소입니다. 어떤 노드가 있고, 어떤 Pod가 어디서 돌고 있으며, 사용자가 무엇을 요청했는지 — 이 모든 "현재 사실"이 etcd에 기록됩니다. 그래서 etcd를 클러스터의 단일 진실 공급원(single source of truth)이라 부릅니다. Pod를 만들면 그 정보가 etcd에 기록되고, 지우면 etcd에서도 지워집니다.

3. Scheduler (배치 담당자)

새로 만들어야 할 Pod를 어느 워커 노드에 올릴지 결정합니다. 단순히 빈자리에 넣는 게 아니라, Pod가 요구하는 CPU·메모리 양, 노드의 여유 자원, "이 Pod는 저 Pod와 가까이 둬라" 같은 제약 조건을 종합해 가장 적합한 노드를 고릅니다. 한정된 서버에 여러 앱을 최적으로 욱여넣는 이른바 '짐 싸기(bin packing) 문제'를 푸는 역할입니다.

4. Controller Manager (현장 감독)

시스템을 항상 사용자가 원하는 상태(desired state)로 되돌리려는 백그라운드 프로세스입니다. 예를 들어 Deployment나 ReplicaSet으로 "Pod 3개가 떠 있어야 한다"고 선언했는데 하나가 죽으면, 컨트롤러가 이를 감지하고 부족한 만큼 새 Pod 객체를 만들어 채웁니다. 노드가 다운되면 감지하는 노드 컨트롤러, 배포를 관리하는 디플로이먼트 컨트롤러처럼, 자원 종류마다 전담 컨트롤러가 돌아갑니다. 쿠버네티스의 '자가 치유'가 바로 여기서 나옵니다.

Controller Manager는 "선언한 개수만큼 Pod 객체가 존재하는가"를 맞추는 역할이지, 개별 Pod를 어느 노드에서 직접 기동시키는 주체가 아닙니다. 이미 노드가 정해진 Pod를 실제로 띄우는 일은 그 노드의 kubelet이 합니다.

클라우드에서 운영한다면 Cloud Controller Manager가 추가로 붙어, 노드용 가상 머신 생성 같은 작업을 클라우드 제공자(AWS·GCP·Azure)에게 위임합니다.

워커 노드의 핵심 부품 3가지

작업장(워커 노드)에는 Pod를 실제로 돌리기 위한 세 가지 프로그램이 있습니다.

1. kubelet (현장 작업반장)

모든 워커 노드에서 도는 에이전트입니다. 컨트롤 플레인(API Server)과 통신해 "이 노드에서 어떤 Pod를 돌려라"는 지시를 받고, 그 Pod가 실제로 살아 건강하게 돌아가는지 책임집니다. 노드와 두뇌를 잇는 연락책이라고 보면 됩니다.

2. Container Runtime (컨테이너 실행 엔진)

실제로 컨테이너를 실행하는 소프트웨어입니다. 이미지를 레지스트리에서 내려받고, 컨테이너를 시작·중지하며 자원을 관리합니다. 흔히 도커를 떠올리지만, 도커는 여러 런타임 중 하나일 뿐입니다. 쿠버네티스는 표준 규격(CRI)만 만족하면 containerd, CRI-O 등 어떤 런타임이든 끼워 쓸 수 있습니다. kubelet이 이 런타임에게 실제 실행을 지시합니다.

3. kube-proxy (네트워크 교통정리)

각 노드의 네트워크 규칙을 관리해, 클러스터 안팎의 트래픽이 올바른 Pod로 전달되도록 라우팅하고 부하를 분산합니다.

지금까지 설명한 클러스터의 전체 구성을 한 장으로 정리하면 아래 구성도와 같습니다. 컨트롤 플레인의 4대 부품과 워커 노드의 3대 부품이 각각 어디에 속하는지 눈으로 확인해 보세요.

사용자 명령이 처리되는 흐름

이제 부품들이 어떻게 협력하는지, "Pod 하나를 만들어줘"라는 요청이 클러스터를 흐르는 과정을 따라가 봅시다. 핵심은 모든 소통이 API Server를 거친다는 점입니다. 흐름은 다음과 같습니다.

1. 사용자가 kubectl로 "이 컨테이너를 Pod로 실행해줘"를 API Server에 요청한다.

2. API Server가 인증·인가와 요청 형식을 검증한 뒤, 이 Pod 정보를 etcd에 기록한다.

3. 아직 어느 노드에 둘지 안 정해진 Pod를 Scheduler가 발견하고, 자원 상황을 분석해 최적의 워커 노드를 골라 "이 Pod는 이 노드에서 돌려라"라고 API Server에 알린다(바인딩). 이 결정도 etcd에 기록된다.

4. 선택된 워커 노드의 kubelet이 API Server를 계속 지켜보다가(watch), 자기 노드에 배정된 Pod를 발견하고 직접 실행에 들어간다.

5. kubelet이 노드의 Container Runtime에게 지시하고, 런타임이 이미지를 내려받아 컨테이너를 실행한다 → Pod가 뜬다.

6. kubelet이 "이 Pod는 Running 상태"라고 API Server에 보고하고, 그 상태 변화가 다시 etcd에 저장된다. 이제 사용자가 kubectl get pod로 확인하면 "Running"이 보인다.

여기서 한 가지 짚을 점. 단일 Pod를 직접 만든 이 흐름에는 Controller Manager가 등장하지 않습니다. Scheduler가 노드를 정하면, 그 노드의 kubelet이 API Server를 지켜보다가 스스로 Pod를 띄우기 때문입니다. Controller Manager가 일하는 때는 따로 있습니다 — Deployment나 ReplicaSet처럼 "Pod를 몇 개 유지하라"고 선언했을 때, 현재 개수가 모자라면 부족한 만큼 Pod 객체를 생성하는 일을 합니다. 그렇게 만들어진 Pod 역시 다시 위 4~6단계(Scheduler 배치 → kubelet 실행)를 거칩니다.

이 6단계를 시간 순서대로 그리면 아래 시퀀스 다이어그램과 같습니다. 모든 화살표가 API Server를 지나간다는 점에 주목하세요.

여기서 한 가지 더. 만약 Deployment 같은 컨트롤러로 띄운 Pod가 갑자기 죽으면 어떻게 될까요? Controller Manager가 "원하는 상태(Pod N개 실행 중)와 현재 상태(N-1개)가 다르다"는 것을 감지하고 부족한 Pod 객체를 새로 만들면, 그 Pod는 다시 위 흐름(Scheduler → kubelet)을 타고 새 노드에 떠오릅니다. 이것이 우리가 손대지 않아도 클러스터가 스스로 복구되는 원리입니다.

핵심 요약

  • 클러스터는 여러 노드의 묶음이며, 노드는 결정을 내리는 컨트롤 플레인과 앱을 실제 실행하는 워커 노드로 나뉜다.
  • 컨트롤 플레인 4대 부품: API Server(유일한 정문·인증), etcd(상태를 보관하는 단일 진실 공급원), Scheduler(Pod를 어느 노드에 둘지 결정), Controller Manager(선언한 개수만큼 Pod 객체를 맞추는 자가 치유).
  • 워커 노드 3대 부품: kubelet(API Server를 watch하며 자기 노드 Pod를 직접 실행·유지), Container Runtime(컨테이너 실제 실행), kube-proxy(네트워크 라우팅).
  • 워커 노드는 컨테이너가 아니라 Pod를 실행한다.
  • 모든 요청과 내부 통신은 반드시 API Server를 거친다. 단일 Pod 생성은 API Server → etcd → Scheduler(노드 결정) → 해당 노드의 kubelet(직접 실행) → Runtime 순으로 흐르며, Controller Manager는 Deployment/ReplicaSet 등으로 replica 수를 맞출 때 Pod 객체를 생성하는 역할이다.

📝 퀴즈

Q1. 컨트롤 플레인과 워커 노드의 역할 구분에 대한 설명으로 가장 옳은 것은?

  • 컨트롤 플레인은 사용자의 애플리케이션 컨테이너를 직접 실행하며, 워커 노드는 어느 노드에 컨테이너를 배치할지 결정한다.
  • 컨트롤 플레인은 클러스터 전체를 관리하고 결정을 내리는 두뇌 역할이며, 실제 애플리케이션 Pod는 워커 노드에서 실행된다.
  • 컨트롤 플레인과 워커 노드는 모두 Pod를 실행할 수 있으며, 역할 차이는 보유한 자원의 크기에 따라 결정된다.
  • 워커 노드 없이 컨트롤 플레인만으로도 컨테이너 앱을 정상적으로 서비스할 수 있다.
정답(B): 컨트롤 플레인은 '지시·관리'를, 워커 노드는 '실행'을 담당한다. 컨트롤 플레인(API Server·etcd·Scheduler·Controller Manager)은 애플리케이션 컨테이너를 직접 실행하지 않고, 어디에 무엇을 띄울지 결정하고 조율한다. 실제 Pod는 오직 워커 노드에서만 실행된다. (A) 역할이 뒤바뀐 오답이다. (C) 컨트롤 플레인은 운영 환경에서 앱 Pod를 실행하지 않으며, 역할은 자원 크기가 아닌 설계에 따라 구분된다. (D) 워커 노드 없이는 실제로 컨테이너를 실행할 노드가 없으므로 앱을 서비스할 수 없다.

Q2. etcd에 대한 설명으로 옳은 것은?

  • etcd는 사용자의 kubectl 명령을 직접 수신하고 인증·인가를 처리하는 클러스터 정문 역할을 한다.
  • etcd는 Pod를 어느 워커 노드에 배치할지 계산하는 스케줄링 엔진이다.
  • etcd는 클러스터의 모든 상태 데이터를 보관하는 키-값 저장소로, 클러스터의 단일 진실 공급원이다.
  • etcd는 각 워커 노드에서 실행되며 컨테이너의 헬스 체크와 실행을 담당하는 에이전트다.
정답(C): etcd는 노드 목록, Pod 배치 정보, 사용자 선언 등 클러스터의 모든 상태 데이터를 저장하는 키-값 저장소이며, '단일 진실 공급원(single source of truth)'이라 불린다. Pod를 만들면 etcd에 기록되고, 삭제하면 etcd에서도 지워진다. (A) 모든 요청을 수신하고 인증·인가를 처리하는 것은 API Server의 역할이다. (B) 배치 노드를 결정하는 것은 Scheduler의 역할이다. (D) 각 워커 노드에서 실행되며 컨테이너 실행을 책임지는 것은 kubelet이다.

Q3. Scheduler가 특정 Pod를 워커 노드 A에 배치하기로 결정했다. 이 결정 이후 실제로 컨테이너를 실행하는 주체는 누구인가? (판단형)

  • Scheduler가 직접 워커 노드 A의 Container Runtime에 컨테이너 실행을 지시한다.
  • Controller Manager가 워커 노드 A를 선택한 뒤 kubelet에게 실행을 명령한다.
  • 워커 노드 A의 kubelet이 API Server를 watch하다가 자기 노드에 배정된 Pod를 발견하고, Container Runtime에게 실행을 지시한다.
  • API Server가 워커 노드 A의 Container Runtime을 직접 호출해 컨테이너를 기동시킨다.
정답(C): Scheduler는 '어느 노드에 둘지 결정(바인딩)'하는 역할까지만 담당하고, 실제 실행은 하지 않는다. 결정 결과는 API Server를 통해 etcd에 기록된다. 이후 해당 노드의 kubelet이 API Server를 지속적으로 watch하다가 자기 노드에 배정된 Pod를 발견하고, 노드의 Container Runtime에게 이미지 다운로드 및 컨테이너 실행을 지시한다. (A) Scheduler는 노드에 직접 접근하지 않는다. (B) Controller Manager는 이 단일 Pod 생성 흐름에는 등장하지 않으며, Deployment 등으로 replica 수를 맞출 때 Pod 객체를 생성하는 역할이다. (D) API Server는 요청을 중계·검증하고 데이터를 etcd에 기록하는 역할이며, Container Runtime을 직접 호출하지 않는다.

Q4. 사용자가 kubectl로 Pod 생성을 요청했을 때, 처리 순서로 올바른 것은?

  • kubectl → Scheduler → API Server → etcd → kubelet → Container Runtime
  • kubectl → API Server → etcd 기록 → Scheduler(노드 결정) → kubelet(실행 지시) → Container Runtime
  • kubectl → kubelet → Container Runtime → API Server → etcd → Scheduler
  • kubectl → etcd → API Server → Scheduler → Controller Manager → kubelet
정답(B): 모든 요청은 반드시 API Server를 첫 번째 관문으로 거친다. 흐름은 다음과 같다. (1) kubectl이 API Server에 요청 전송 → (2) API Server가 인증·인가·형식 검증 후 Pod 정보를 etcd에 기록 → (3) Scheduler가 배치 안 된 Pod를 발견하고 최적 노드를 선정(바인딩)해 API Server에 알림, etcd에도 기록 → (4) 해당 노드의 kubelet이 API Server를 watch하다 자기 노드 Pod를 발견 → (5) kubelet이 Container Runtime에 실행 지시 → Pod 기동. (A) Scheduler가 API Server보다 앞서면 인증·인가를 건너뛰게 되므로 틀렸다. (C) kubelet과 Container Runtime이 etcd·Scheduler보다 먼저 나오는 것은 불가능하다. (D) 단일 Pod 생성 흐름에서 Controller Manager는 관여하지 않는다.

Q5. Deployment로 Pod 3개를 선언해 운영하던 중, 워커 노드 한 대가 갑자기 다운되어 Pod 1개가 사라졌다. 쿠버네티스가 자동으로 이 Pod를 다른 노드에 복구할 때, 가장 먼저 '부족한 Pod 객체를 새로 생성'하는 역할을 맡는 구성요소는? (적용형)

  • Scheduler — 남은 노드 중 최적의 위치를 계산해 Pod를 즉시 재배치한다.
  • kubelet — 죽은 Pod를 감지하고 스스로 새 Pod 객체를 생성한다.
  • Controller Manager — 선언된 replica 수(3개)와 실제 개수(2개)의 차이를 감지하고 부족한 Pod 객체를 생성한다.
  • etcd — 저장된 desired state와 current state의 차이를 자동으로 맞추어 Pod를 재생성한다.
정답(C): Controller Manager(ReplicaSet 컨트롤러)가 '원하는 상태(Pod 3개)'와 '현재 상태(Pod 2개)'의 차이를 지속적으로 감시하다가 불일치를 감지하면, 부족한 수만큼 Pod 객체를 새로 생성한다. 그렇게 만들어진 Pod는 다시 Scheduler가 노드를 결정하고, 해당 노드의 kubelet이 실제 실행하는 정상 흐름을 탄다. (A) Scheduler는 이미 Pod 객체가 존재할 때 노드를 '결정'하는 역할이며, Pod 객체를 새로 만들지는 않는다. (B) kubelet은 자기 노드에 배정된 Pod를 실행·유지하는 역할이지, 새 Pod 객체를 생성하는 주체가 아니다. (D) etcd는 상태를 '저장'하는 저장소일 뿐이며, 스스로 아무 행동을 취하지 않는다.

03. Pod와 컨테이너 - 쿠버네티스의 최소 배포 단위

Pod와 컨테이너 - 쿠버네티스의 최소 배포 단위

학습 목표

  • Pod가 무엇이며 컨테이너와 어떻게 다른지 이해한다
  • Pod의 생명주기와 단일/멀티 컨테이너 패턴을 안다
  • kubectl로 Pod를 생성하고 상태를 확인할 수 있다

본문

왜 컨테이너가 아니라 Pod일까

앞선 강의에서 컨테이너가 "애플리케이션 코드와 실행에 필요한 모든 것을 하나로 묶은 단위"라는 것을 배웠습니다. 그런데 막상 쿠버네티스를 다루다 보면 컨테이너를 직접 만지는 명령은 거의 없고, 늘 Pod(파드) 라는 단어가 등장합니다. 왜 쿠버네티스는 컨테이너를 직접 다루지 않을까요?

쿠버네티스에서 Pod는 배포할 수 있는 가장 작은 단위입니다. 쿠버네티스는 컨테이너를 직접 다루지 않고, 컨테이너를 Pod라는 봉투에 한 번 감싸서 다룹니다. 즉 우리가 클러스터에 무언가를 올린다는 것은 곧 "Pod를 올린다"는 뜻입니다.

핵심: 컨테이너는 "패키징 단위", Pod는 쿠버네티스가 다루는 "실행/배포 단위"입니다. 컨테이너를 직접 클러스터에 던지는 게 아니라, 항상 Pod로 감싸서 올립니다.

Pod란 무엇인가 - 컨테이너를 담는 봉투

Pod는 하나 이상의 컨테이너를 묶은 그룹입니다. 핵심은 같은 Pod 안에 든 컨테이너들이 다음을 공유한다는 점입니다.

  • 네트워크: 같은 Pod의 컨테이너들은 동일한 IP 주소를 가지며, 서로를 localhost로 부를 수 있습니다. 포트도 공유합니다.
  • 스토리지(볼륨): 같은 디렉터리를 공유해 데이터를 주고받을 수 있습니다.

다만 컨테이너는 여전히 각자의 파일시스템을 갖고 있어 격리는 유지됩니다. 비유하자면 Pod는 "한 집(주소·네트워크는 공유)에 사는 룸메이트들"이고, 각자의 방(파일시스템)은 분리되어 있는 셈입니다.

여기서 초보자가 자주 놓치는 중요한 사실이 있습니다. Pod에 컨테이너가 한 개든 여러 개든, 같은 Pod의 컨테이너는 항상 같은 노드(서버)에 함께 배치되고, 함께 생성되고 함께 사라집니다. 따로 떼어 다른 노드로 옮길 수 없습니다.

단일 컨테이너 vs 멀티 컨테이너 패턴

실무에서 대부분의 Pod는 컨테이너 1개짜리(단일 컨테이너 패턴) 입니다. "하나의 애플리케이션 = 하나의 컨테이너 = 하나의 Pod"가 가장 기본이고 권장되는 형태입니다. 처음 배울 때는 이 형태만 생각해도 충분합니다.

그렇다면 멀티 컨테이너는 언제 쓸까요? 서로 아주 밀접하게 협력하며, 떨어지면 의미가 없는 보조 컨테이너가 있을 때입니다. 대표적인 예가 사이드카(sidecar) 패턴입니다.

  • 메인 컨테이너: 웹 콘텐츠를 제공하는 파이썬 애플리케이션
  • 보조 컨테이너: 그 애플리케이션의 로그를 수집해 외부로 전송하는 컨테이너

이 둘은 같은 데이터를 다루고 localhost로 긴밀히 통신해야 하므로, 별도 Pod로 떼지 않고 한 Pod에 함께 둡니다. 그러면 항상 같은 곳에서 함께 실행되고 함께 관리됩니다.

아래 구성도처럼 두 패턴 모두 하나의 Pod 안에서 IP와 볼륨을 공유하되, 컨테이너마다 파일시스템은 분리되어 있습니다.

주의: 멀티 컨테이너는 "한 화면에 보이니까 그냥 같이 넣자"가 아닙니다. 생명주기를 함께해야 할 만큼 결합도가 높을 때만 사용하고, 그렇지 않으면 각각 별도의 Pod로 분리하는 것이 정석입니다.

Pod의 생명주기 - 그리고 "Pod는 일회용이다"

Pod를 만들면 다음과 같은 상태(Phase)를 거칩니다.

  • Pending: 클러스터가 요청을 받았고, 어느 노드에 둘지 정하고 이미지를 내려받는 등 준비 중인 상태
  • Running: 노드에 배치되어 컨테이너가 실제로 실행 중인 상태
  • Succeeded: (일회성 작업의 경우) 컨테이너가 정상 종료되어 끝난 상태
  • Failed: 컨테이너가 오류로 종료된 상태

아래 상태 전이도처럼 Pod는 준비 단계를 거쳐 실행되고, 작업 성격에 따라 정상 종료 또는 실패로 끝납니다.

여기서 반드시 기억할 개념이 있습니다. Pod는 일회용(ephemeral, 임시적) 입니다. Pod가 죽으면 쿠버네티스는 그 Pod를 "되살리지" 않습니다. 대신 똑같은 새 Pod를 새로 만들어 줍니다. 이때 새 Pod는 새 IP를 받을 수도 있습니다.

그러면 Pod가 죽었을 때 누가 새로 만들어 줄까요? 우리가 Pod를 만들 때 정한 재시작 정책(restart policy) 과, 쿠버네티스 내부의 컨트롤러 매니저(controller manager) 안에 있는 레플리케이션 컨트롤러가 그 일을 합니다. 이 컨트롤러는 "원하는 상태(desired state)"와 "실제 상태(actual state)"를 끊임없이 비교하다가, Pod가 사라진 것을 발견하면 새 Pod를 띄워 다시 원하는 상태로 맞춥니다. 이것이 바로 1강에서 말한 쿠버네티스의 "자가 치유" 원리입니다.

그래서 실무에서는 Pod를 직접 만들기보다, 다음 강의에서 배울 Deployment로 Pod를 관리합니다. 이 강의에서는 개념을 익히기 위해 Pod를 직접 다뤄봅니다.

kubectl로 Pod 만들고 확인하기 (실습)

이제 직접 손으로 해봅니다. 로컬에서 쿠버네티스를 연습하려면 minikube(개발 PC에 1노드 클러스터를 띄우는 도구)나 Docker Desktop의 쿠버네티스 기능을 사용하면 됩니다. 클러스터가 준비되어 kubectl이 동작하는 상태라고 가정합니다.

kubectl은 쿠버네티스에 명령을 보내는 커맨드라인 도구입니다. 명령은 API 서버로 전달되고, 그 흐름은 다음과 같습니다.

먼저 nginx 웹서버 컨테이너 하나를 담은 Pod를 만들어 봅니다.


kubectl run my-web --image=nginx


pod/my-web created
  • run: Pod를 하나 생성하는 명령
  • my-web: Pod의 이름
  • --image=nginx: 사용할 컨테이너 이미지

이제 Pod 목록과 상태를 확인합니다.


kubectl get pods


NAME     READY   STATUS    RESTARTS   AGE
my-web   1/1     Running   12s        12s
  • READY 1/1: Pod 안의 컨테이너 1개가 모두 준비됨 (멀티 컨테이너라면 2/2처럼 표시)
  • STATUS Running: 정상 실행 중 (방금 만들었다면 잠깐 ContainerCreating이나 Pending으로 보일 수 있습니다)
  • RESTARTS: 컨테이너가 재시작된 횟수

더 자세한 정보(어느 노드에 배치됐는지, IP, 이벤트 등)를 보려면 describe를 씁니다. 문제가 생겼을 때 가장 먼저 보게 될 명령입니다.

Events 부분을 보면 스케줄러가 노드를 정하고(Scheduled), 이미지를 받고(Pulled), 컨테이너를 시작(Started)하는 2강의 흐름이 실제 로그로 찍혀 있는 것을 확인할 수 있습니다.

컨테이너가 내보내는 로그를 보거나, 컨테이너 안으로 들어가 명령을 실행할 수도 있습니다.

실습이 끝나면 Pod를 삭제합니다.

직접 만든(kubectl run) Pod를 삭제하면 그대로 사라집니다. 하지만 다음 강의의 Deployment로 만든 Pod는, 지우면 컨트롤러가 곧바로 새 Pod를 다시 띄웁니다. 바로 이 차이가 "Pod는 일회용"이라는 말의 실체입니다.

정리하며

Pod는 쿠버네티스가 다루는 최소 단위로, 컨테이너를 감싸 네트워크와 스토리지를 공유하게 해주는 봉투입니다. 대부분은 컨테이너 1개짜리 단일 Pod를 쓰고, 긴밀히 협력하는 보조 컨테이너가 있을 때만 멀티 컨테이너로 묶습니다. 그리고 Pod는 일회용이라 죽으면 되살아나는 게 아니라 새로 태어납니다. 그래서 실무에서는 Pod를 직접 관리하지 않고, 다음 강의에서 배울 Deployment에게 맡기게 됩니다.

핵심 요약

  • Pod는 쿠버네티스의 최소 배포 단위로, 하나 이상의 컨테이너를 감싼 그룹이다. 클러스터에는 컨테이너가 아니라 Pod를 올린다.
  • 같은 Pod의 컨테이너들은 IP·네트워크·스토리지를 공유하고 localhost로 통신하지만, 파일시스템은 격리된다. 항상 같은 노드에서 함께 생성·소멸한다.
  • 대부분은 단일 컨테이너 Pod를 쓰고, 떼어낼 수 없을 만큼 결합도가 높은 보조 컨테이너가 있을 때만 멀티 컨테이너(사이드카) 로 묶는다.
  • Pod 생명주기는 Pending → Running → (Succeeded/Failed)이며, Pod는 일회용이라 죽으면 컨트롤러가 동일한 새 Pod를 만들어 원하는 상태를 유지한다.
  • 실습 명령: kubectl run(생성) → kubectl get pods(목록·상태) → kubectl describe pod(상세·이벤트) → kubectl logs/kubectl exec(로그·접속) → kubectl delete pod(삭제).

YAML 방식은 'kubectl run'과 달리 모든 설정을 파일로 관리할 수 있어 버전 관리(git)에 적합하고 실무 표준이다. 'kubectl apply -f'는 파일을 선언적으로 적용하며, 같은 파일로 수정 적용도 가능하다. 'describe'의 Events 섹션은 Scheduled → Pulled → Started 순서로 기록되며 문제 진단 시 가장 먼저 확인하는 부분이다. 'exec -it'는 실행 중인 컨테이너 안에서 인터랙티브 셸을 여는 명령으로, 파일 구조 확인·환경 변수 점검 등 디버깅에 유용하다. 'delete -f'는 apply와 쌍을 이뤄 같은 파일로 자원을 제거하므로 이름을 따로 기억할 필요가 없다.

실습 3. 의도적으로 존재하지 않는 이미지 'nginx:doesnotexist999'를 사용하는 Pod를 이름 'broken-pod'로 생성하라. 생성 직후 'kubectl get pods'를 반복 실행해 STATUS 변화를 관찰한다. STATUS가 'ErrImagePull' 또는 'ImagePullBackOff'로 바뀌면 'kubectl describe pod broken-pod'의 Events 섹션에서 실패 이유를 찾아 기록한다. 진단이 끝나면 Pod를 삭제하고, 올바른 이미지('nginx:stable')로 새 Pod 'fixed-pod'를 만들어 Running 상태가 되는지 확인한다.

# 오류 Pod 생성
kubectl run broken-pod --image=nginx:doesnotexist999

# 상태 반복 확인 (몇 초 간격으로 2~3회 실행)
kubectl get pods

# 상태가 ErrImagePull 또는 ImagePullBackOff가 되면 상세 확인
kubectl describe pod broken-pod
# Events 섹션에서 'Failed to pull image' 메시지를 찾는다

# 진단 후 삭제

kubectl run fixed-pod --image=___

힌트
  • STATUS가 바뀌지 않고 Pending에 오래 머무른다면 이미지 pull이 아니라 노드 자원 부족일 수 있다. 그 경우에도 describe Events에서 'Insufficient cpu/memory' 메시지로 구분된다.
  • ErrImagePull과 ImagePullBackOff는 같은 문제의 두 단계다. ErrImagePull은 첫 실패 직후이고, ImagePullBackOff는 쿠버네티스가 재시도를 늦추는 대기 상태다. 둘 다 이미지 문제라고 봐도 무방하다.
  • nginx 공식 이미지의 유효한 태그 목록은 Docker Hub(hub.docker.com/_/nginx/tags)에서 확인할 수 있다. 'latest', 'stable', 'alpine'이 가장 많이 쓰인다.

Pod 생성 요청은 성공(pod created)해도 실제 컨테이너 실행은 kubelet이 이미지를 pull할 때 비로소 결정된다. 존재하지 않는 이미지 태그는 레지스트리에서 404를 반환하고 kubelet이 이를 'ErrImagePull'로 기록한다. 쿠버네티스는 pull 실패 시 지수 백오프(exponential backoff) 방식으로 재시도 간격을 늘리며, 이 상태가 'ImagePullBackOff'다. describe의 Events 섹션에는 'Failed to pull image'와 함께 오류 원인이 명시되므로, 운영 중 Pod가 0/1로 멈춰 있을 때 가장 먼저 실행해야 하는 진단 명령이다. 이미지 이름 오타, private registry 인증 실패, 존재하지 않는 태그가 현업에서 가장 흔한 원인이다.

실습 4. 하나의 Pod 안에 두 컨테이너를 함께 구성하는 YAML 파일을 직접 작성하라. - 메인 컨테이너: 이름 'web', 이미지 'nginx', 포트 80 - 사이드카 컨테이너: 이름 'log-watcher', 이미지 'busybox', args로 '/bin/sh -c while true; do echo sidecar-running; sleep 5; done' 을 무한 루프로 실행 파일명은 'sidecar-pod.yaml', Pod 이름은 'sidecar-demo'로 한다. 생성 후 다음을 수행한다. 1. 'kubectl get pods'에서 READY 컬럼이 '2/2'임을 확인한다. 2. '-c web' 옵션으로 web 컨테이너의 로그를, '-c log-watcher'로 사이드카 컨테이너의 로그를 각각 출력한다. 3. 두 컨테이너가 동일한 Pod IP를 공유하는지 'kubectl describe pod sidecar-demo'의 IP 필드와 'kubectl exec -it sidecar-demo -c web -- printenv HOSTNAME'으로 검증한다. 4. 실습 후 Pod를 삭제한다.

# sidecar-pod.yaml 파일 작성 (빈칸을 채운다)
apiVersion: v1
kind: Pod
metadata:
  name: sidecar-demo
spec:
  containers:
    - name: web
      image: nginx
      ports:
        - containerPort: 80
    - name: log-watcher
      image: busybox
      args:
        - /bin/sh
        - -c
        - ___  # 5초마다 'sidecar-running'을 출력하는 무한 루프
힌트
  • YAML args 배열에서 while 루프 명령 전체를 하나의 문자열 원소로 적어야 한다. '-c' 뒤에 오는 원소가 하나의 문자열이어야 /bin/sh가 그것을 스크립트로 실행한다.
  • 컨테이너가 CrashLoopBackOff 상태가 되면 명령이 즉시 종료됐다는 뜻이다. busybox 이미지에서 while 루프 없이 단순히 'echo hello'만 넣으면 컨테이너가 종료되어 쿠버네티스가 재시작을 반복한다. 반드시 무한 루프로 작성해야 한다.
  • 두 컨테이너가 localhost 통신을 실제로 확인하고 싶다면, 'kubectl exec -it sidecar-demo -c log-watcher -- wget -qO- localhost:80'을 실행해 보라. log-watcher에서 web(nginx)의 응답을 받을 수 있다.

04. Deployment와 Service - 앱 배포와 외부 노출

Deployment와 Service - 앱 배포와 외부 노출

학습 목표

  • Deployment로 Pod 복제, 자가 치유(self-healing), 롤링 업데이트를 관리한다.
  • Service(ClusterIP / NodePort)로 Pod에 접근하는 방법을 이해한다.
  • kubectl로 Deployment를 배포하고 Service로 외부에 노출해본다.

본문

왜 Pod를 직접 만들지 않을까

3강에서 우리는 kubectl run으로 Pod를 직접 만들어 보았습니다. 그런데 실무에서는 Pod를 손으로 직접 만드는 일이 거의 없습니다. 이유는 간단합니다. Pod는 언제든 사라질 수 있는 일회용 자원이기 때문입니다. 노드가 죽거나, 누군가 실수로 지우거나, 클러스터가 Pod를 다른 노드로 옮기면 그 Pod는 그대로 없어집니다. 직접 만든 Pod라면 누군가 다시 만들어주기 전까지 서비스는 멈춰 있게 됩니다.

그래서 쿠버네티스는 Pod 위에 한 단계 더 높은 추상화인 Deployment를 제공합니다. 우리는 "Pod 몇 개를 항상 띄워 둬"라고 원하는 상태만 선언하고, 실제로 그 상태를 유지하는 일은 쿠버네티스에게 맡깁니다.

쿠버네티스의 핵심 철학은 "어떻게 하라"가 아니라 "어떤 상태가 되길 원한다"를 선언하는 것입니다. Deployment는 이 선언형 사고방식의 대표적인 예입니다.

Deployment가 하는 세 가지 일

Deployment는 Pod의 복제본(replica) 묶음을 관리합니다. 핵심 기능은 세 가지입니다.

1. 복제(Replication) — 여러 개를 똑같이 띄운다.

Deployment에 replicas: 3이라고 적으면, 동일한 Pod 3개가 항상 떠 있도록 유지됩니다. 트래픽이 늘면 숫자만 키우면 되고, 여러 개가 떠 있으니 한 개가 죽어도 서비스는 계속됩니다.

2. 자가 치유(Self-healing) — 죽으면 알아서 살린다.

Deployment는 자신이 관리하는 Pod 개수를 끊임없이 감시합니다. 만약 Pod 하나가 죽어서 3개에서 2개가 되면, 쿠버네티스는 곧바로 새 Pod 하나를 만들어 다시 3개를 맞춥니다. 사람이 개입하지 않아도 "원하는 상태(3개)"가 자동으로 회복됩니다. 이것이 우리가 Pod를 직접 만들지 않는 가장 큰 이유입니다.

3. 롤링 업데이트(Rolling Update) — 무중단으로 새 버전을 배포한다.

앱을 v1에서 v2로 올릴 때, Deployment는 모든 Pod를 한꺼번에 갈아치우지 않습니다. 기존 Pod를 하나씩 새 버전으로 천천히 교체합니다. 이렇게 하면 업데이트 중에도 서비스가 멈추지 않습니다(zero downtime).

롤링 업데이트의 흐름은 다음과 같습니다.

1. 새 버전(v2) Pod를 하나 추가로 생성한다.

2. 새 Pod가 정상적으로 떠서 요청을 받을 준비가 되면, 트래픽을 그 Pod로 보낸다.

3. 그 다음 기존 버전(v1) Pod를 하나 종료한다.

4. 이 과정을 모든 Pod가 새 버전이 될 때까지 반복한다.

아래 흐름도처럼 새 Pod를 먼저 띄우고 기존 Pod를 하나씩 내리는 과정을 반복합니다.

이때 두 가지 안전장치가 동작합니다. maxUnavailable은 "동시에 몇 개까지 내려가도 되는가"를, maxSurge는 "원하는 개수보다 몇 개까지 더 띄워도 되는가"를 정합니다. 기본값은 둘 다 25%입니다. 즉 Pod가 4개라면 한 번에 1개만 내려가고, 잠깐 1개를 더 띄워(최대 5개) 빈자리 없이 교체합니다.

새 버전에 문제가 생겼다면 kubectl rollout undo로 이전 버전으로 즉시 되돌릴 수 있습니다. Deployment가 버전 이력을 기억하고 있기 때문입니다.

Deployment를 직접 배포해 보자

이제 직접 해봅니다. (Minikube 같은 로컬 클러스터가 떠 있다고 가정합니다.) 아래는 nginx 웹 서버를 3개 복제본으로 배포하는 명령입니다.

복제본 개수를 바꾸는 것도 명령 한 줄입니다.


kubectl scale deployment web --replicas=5

Service - 흩어진 Pod에 안정적으로 접근하기

문제가 하나 남았습니다. Pod는 죽고 다시 태어날 때마다 IP 주소가 바뀝니다. 위에서 봤듯이 Pod가 교체되면 이름도, IP도 달라집니다. 그렇다면 우리 앱에 접속하려는 사용자는 매번 바뀌는 IP를 어떻게 알 수 있을까요?

이 문제를 해결하는 것이 Service입니다. Service는 여러 Pod 앞에 놓인 고정된 출입구이자 부하 분산기입니다. Service는 변하지 않는 하나의 주소를 가지며, 들어온 요청을 자신이 관리하는 Pod들에게 골고루 나눠 보냅니다(load balancing). Pod가 죽고 새로 생겨 IP가 바뀌어도, Service는 자동으로 살아있는 Pod 목록을 갱신하므로 사용자는 변화를 전혀 느끼지 못합니다.

Service가 어떤 Pod에게 요청을 보낼지는 레이블(label) 셀렉터로 결정합니다. "app=web 레이블이 붙은 Pod에게 보내"라고 지정하면, 그 레이블을 가진 Pod가 몇 개든 알아서 찾아 분배합니다.

Service에는 여러 종류가 있는데, 초급 단계에서 꼭 알아야 할 두 가지는 다음과 같습니다.

ClusterIP (기본값) — 클러스터 내부에서만 통하는 주소를 부여합니다. 외부에서는 접근할 수 없고, 같은 클러스터 안의 다른 Pod끼리 통신할 때 씁니다. 예를 들어 웹 서버 Pod가 데이터베이스 Pod에 접속할 때 데이터베이스를 ClusterIP Service로 노출합니다. 외부에 열 필요가 없으니 가장 안전한 기본 선택입니다.

NodePort — 클러스터의 모든 노드에 동일한 포트(30000~32767 범위)를 열어, 그 포트로 들어온 외부 요청을 Service로 연결합니다. 노드의IP:노드포트로 클러스터 밖에서도 접속할 수 있어, 학습이나 간단한 테스트에 적합합니다.

세 가지 접근 경로를 정리하면 이렇습니다.

  • 외부 사용자 → (NodePort) → Service → Pod 여러 개
  • 클러스터 내부 Pod → (ClusterIP) → Service → Pod 여러 개

아래 구성도처럼 외부 요청은 NodePort를 통해, 내부 통신은 ClusterIP를 통해 같은 Service로 들어와 뒤쪽 Pod들로 분배됩니다.

Service로 노출해 보자

앞서 만든 web Deployment를 NodePort Service로 외부에 노출합니다.

kubectl expose deployment web --type=NodePort --port=80

# 생성된 Service 확인
kubectl get service web


NAME   TYPE       CLUSTER-IP      EXTERNAL-IP   PORT(S)        AGE

이 모든 과정 동안 Service의 주소는 그대로이고, 사용자 요청은 끊기지 않습니다. Deployment(앱의 생명 유지)와 Service(안정적인 접근 통로)가 짝을 이뤄 동작하는 것이 쿠버네티스 앱 운영의 가장 기본적인 형태입니다.

핵심 요약

  • Pod는 일회용이라 직접 만들지 않고, Deployment로 관리한다.
  • Deployment는 복제(원하는 개수 유지), 자가 치유(죽으면 자동 재생성), 롤링 업데이트(무중단 배포)를 담당한다.
  • 롤링 업데이트는 새 Pod를 준비 완료까지 띄운 뒤 기존 Pod를 하나씩 교체하며, maxUnavailable·maxSurge로 속도와 안전을 조절한다. 문제가 생기면 kubectl rollout undo로 되돌린다.
  • Pod는 IP가 계속 바뀌므로, Service가 고정 주소와 부하 분산을 제공한다. 어떤 Pod로 보낼지는 레이블 셀렉터로 정한다.
  • ClusterIP는 클러스터 내부 통신용(기본값), NodePort는 외부 노출용(노드 포트로 접근).
  • 실습 명령: kubectl create deployment, kubectl scale, kubectl expose, kubectl set image, kubectl rollout status/undo.

🛠 실습

실습 1. kubectl 명령 한 줄로 'nginx' 이미지를 사용하는 Deployment 'web'을 복제본 3개로 생성하세요. 생성 후 Deployment 상태와 실제로 만들어진 Pod 목록을 각각 조회해 'READY 3/3' 상태를 확인합니다.

# 아직 아무 Deployment도 없는 상태에서 시작합니다.
# 아래 두 명령을 순서대로 실행하세요.

kubectl create deployment web --image=nginx --replicas=___
kubectl get deployments
kubectl get pods
힌트
  • '--replicas=3' 플래그가 빠지면 기본값 1개로 생성됩니다. 반드시 명시하세요.
  • Pod가 'ContainerCreating' 상태로 잠시 머무를 수 있습니다. 이미지를 처음 내려받는 경우 수십 초가 걸릴 수 있으니 'kubectl get pods' 를 반복해서 Running 상태가 되는지 기다리세요.
  • Deployment 세부 내용을 보려면 'kubectl describe deployment web' 을 실행하면 이벤트 로그까지 확인할 수 있습니다.
정답 보기 
# 1. Deployment 생성
kubectl create deployment web --image=nginx --replicas=3

# 2. Deployment 상태 조회
kubectl get deployments
# 예상 출력:
# NAME   READY   UP-TO-DATE   AVAILABLE   AGE
# web    3/3     3            3           15s

# 3. 실제 Pod 목록 확인
kubectl get pods
# 예상 출력 (이름 해시 부분은 환경마다 다름):
# NAME                   READY   STATUS    RESTARTS   AGE
# web-<hash>-aaaaa       1/1     Running   0          20s
# web-<hash>-bbbbb       1/1     Running   0          20s
# web-<hash>-ccccc       1/1     Running   0          20s

'kubectl create deployment web --image=nginx --replicas=3' 은 쿠버네티스 API 서버에 "app=web 레이블을 가진 nginx Pod를 항상 3개 유지하라"는 Deployment 오브젝트를 선언적으로 등록합니다. Deployment 컨트롤러가 이 선언을 읽고 ReplicaSet을 통해 Pod 3개를 실제로 생성합니다. 'READY 3/3'은 원하는 수(3)와 실제 준비된 수(3)가 일치함을 뜻합니다. Pod 이름이 'web-<랜덤해시>-<랜덤해시>' 형태인 것은 Deployment가 자동으로 이름을 붙이기 때문이며, 나중에 Pod가 재생성될 때 이름이 바뀌는 이유이기도 합니다.

실습 2. 과제 1에서 만든 'web' Deployment의 Pod 목록에서 Pod 이름 하나를 복사한 뒤 강제 삭제하세요. 삭제 직후 즉시 Pod 목록을 다시 조회해 쿠버네티스가 새 Pod를 자동 생성하는 자가 치유 동작을 눈으로 확인합니다. 최종적으로 Pod가 다시 3개인지 확인하세요.

# 1단계: 현재 Pod 이름을 확인합니다.
kubectl get pods

# 2단계: 목록에서 Pod 이름 하나를 복사해 아래 명령에 붙여넣고 실행합니다.
kubectl delete pod <여기에_Pod_이름_붙여넣기>

# 3단계: 삭제 직후 Pod 목록을 반복 조회합니다.
kubectl get pods
힌트
  • Pod 이름을 타이핑하다 오타 내기 쉽습니다. 'kubectl get pods' 결과에서 이름을 드래그해 터미널에 붙여넣으세요.
  • 삭제 명령 실행 후 0~2초 안에 'kubectl get pods' 를 실행해야 'ContainerCreating' 상태의 새 Pod를 볼 수 있습니다. 타이밍을 놓쳤다면 이미 Running 상태가 된 것이니 총 개수가 3개인지만 확인하면 됩니다.
  • '-w' 플래그를 붙여 'kubectl get pods -w' 를 실행하면 Pod 상태 변화를 실시간으로 스트리밍해 볼 수 있습니다. Ctrl+C로 종료합니다.
정답 보기 
# 1. Pod 이름 확인
kubectl get pods
# 예: web-5d4f8c7b9c-7k2lp, web-5d4f8c7b9c-q8xvr, web-5d4f8c7b9c-zm4tn

# 2. Pod 하나 강제 삭제 (본인 환경의 실제 이름으로 교체)
kubectl delete pod web-5d4f8c7b9c-7k2lp

# 3. 삭제 직후 Pod 목록 재조회
kubectl get pods
# 예상 출력 — 삭제된 Pod 자리에 이름이 다른 새 Pod가 ContainerCreating 상태로 등장:
# NAME                   READY   STATUS              RESTARTS   AGE
# web-5d4f8c7b9c-q8xvr   1/1     Running             0          3m
# web-5d4f8c7b9c-zm4tn   1/1     Running             0          3m
# web-5d4f8c7b9c-rt9wd   0/1     ContainerCreating   0          2s

# 4. 잠시 후 다시 조회하면 새 Pod가 Running 상태로 총 3개 유지됨을 확인
kubectl get pods

Pod를 직접 삭제하면 Deployment 컨트롤러는 '원하는 상태(replicas=3)'와 '현재 상태(2개 Running)'가 불일치함을 즉시 감지합니다. 컨트롤러는 차이(1개 부족)를 해소하기 위해 새 Pod 생성을 ReplicaSet에 요청하고, ReplicaSet이 새 Pod를 만들어 다시 3개로 맞춥니다. 새 Pod의 이름 해시가 달라진다는 점에서 '재시작'이 아니라 '새로 만든' 것임을 알 수 있습니다. 이것이 Pod를 직접 만드는 대신 Deployment를 쓰는 핵심 이유입니다.

실습 3. 'web' Deployment를 NodePort 타입의 Service로 노출하고, 할당된 NodePort 번호를 확인한 뒤 minikube IP와 해당 포트로 curl 요청을 보내 nginx 기본 페이지 HTML을 수신하세요. 이후 'kubectl scale' 로 복제본을 5개로 늘리고 Service 주소가 여전히 동일한지 확인합니다.

# 1단계: Service 생성
kubectl expose deployment web --type=___ --port=80

# 2단계: 생성된 Service 정보 조회
kubectl get service web

# 3단계: minikube에서 접속 URL 얻기
minikube service web --url

# 4단계: curl로 nginx 페이지 수신
curl <위에서_얻은_URL>

# 5단계: 복제본을 5개로 스케일 아웃
kubectl scale deployment web --replicas=___
kubectl get pods
힌트
  • '--type=NodePort' 를 생략하면 기본값 ClusterIP로 만들어져 외부에서 접속이 안 됩니다. 반드시 명시하세요.
  • curl 결과가 'Connection refused' 로 나온다면 Pod가 아직 ContainerCreating 상태일 수 있습니다. 'kubectl get pods' 로 모두 Running인지 확인한 뒤 다시 시도하세요.
  • minikube가 driver=docker 모드로 실행 중이면 'minikube service web --url' 없이는 NodePort 주소로 직접 접속이 안 될 수 있습니다. 이 명령이 출력하는 URL을 그대로 사용하는 것이 가장 안전합니다.
정답 보기 
# 1. NodePort Service 생성
kubectl expose deployment web --type=NodePort --port=80

# 2. Service 확인 (NodePort 번호는 환경마다 다름)
kubectl get service web
# 예상 출력:
# NAME   TYPE       CLUSTER-IP      EXTERNAL-IP   PORT(S)        AGE
# web    NodePort   10.96.142.31    <none>        80:31845/TCP   5s

# 3. minikube가 열어준 전체 URL 확인
minikube service web --url
# 예: http://192.168.49.2:31845

# 4. curl로 nginx 기본 페이지 수신
curl http://192.168.49.2:31845
# 예상 출력:
# <!DOCTYPE html>
# <html>
# <head><title>Welcome to nginx!</title></head>
# ...

# 5. 복제본 5개로 스케일 아웃
kubectl scale deployment web --replicas=5
kubectl get pods
# 5개의 Pod가 Running 상태로 나타나야 함

# 6. 스케일 후에도 Service URL이 동일한지 재확인
curl http://192.168.49.2:31845
# 동일한 nginx 페이지가 응답됨 → Service 주소 불변 확인

'kubectl expose' 는 Deployment의 레이블 셀렉터(app=web)를 그대로 상속한 Service 오브젝트를 생성합니다. NodePort 타입을 지정하면 쿠버네티스가 30000~32767 범위 안에서 사용하지 않는 포트를 자동 배정해 모든 노드에 엽니다. Service는 고정된 ClusterIP를 가지므로 Pod가 죽고 새로 생겨 Pod IP가 바뀌어도 접속 주소가 변하지 않습니다. 'kubectl scale' 로 복제본을 늘려도 Service는 레이블 셀렉터로 새 Pod를 자동으로 엔드포인트에 추가하기 때문에 추가 설정 없이 부하 분산 대상이 늘어납니다. minikube 전용 'minikube service --url' 명령은 노드 IP를 자동으로 구해 전체 URL을 출력합니다.

실습 4. 'web' Deployment의 nginx 이미지 버전을 'nginx:1.25' 로 업데이트해 롤링 업데이트를 시작하고, 진행 상황을 'kubectl rollout status' 로 관찰합니다. 업데이트가 완료된 뒤 의도적으로 존재하지 않는 이미지 태그('nginx:99.99.99')로 다시 업데이트해 롤링 업데이트가 중간에 멈추는 상황을 만들고, 'kubectl rollout undo' 로 정상 버전으로 복구하세요. 복구 후 Pod가 모두 Running 상태인지 확인합니다.

# 1단계: nginx:1.25 로 정상 롤링 업데이트
kubectl set image deployment/web nginx=nginx:___
kubectl rollout status deployment/web

# 2단계: 존재하지 않는 이미지로 업데이트 (의도적 실패 유발)
kubectl set image deployment/web nginx=nginx:99.99.99

# 3단계: Pod 상태를 반복 조회해 ImagePullBackOff 상태를 확인
kubectl get pods

# 4단계: 이전 버전으로 롤백
kubectl rollout undo deployment/web

# 5단계: 복구 후 Pod 상태 최종 확인
kubectl get pods
kubectl rollout status deployment/web
힌트
  • 'kubectl set image' 에서 컨테이너 이름은 Deployment를 만들 때 기본적으로 이미지 이름과 동일하게 설정됩니다. 'nginx' Deployment라면 컨테이너 이름도 'nginx' 입니다. 확인은 'kubectl describe deployment web | grep -A5 Containers' 로 할 수 있습니다.
  • ImagePullBackOff 가 바로 나타나지 않을 수 있습니다. 처음에는 'ErrImagePull' 로 표시되고 수 초 뒤 'ImagePullBackOff' 로 바뀝니다. 'kubectl get pods -w' 로 실시간 관찰하면 전환 과정을 볼 수 있습니다.
  • 'kubectl rollout history deployment/web' 을 실행하면 Deployment의 개정판(revision) 이력을 볼 수 있습니다. 'kubectl rollout undo deployment/web --to-revision=1' 처럼 특정 버전으로 직접 돌아갈 수도 있습니다.
정답 보기 
# 1. nginx:1.25 로 롤링 업데이트
kubectl set image deployment/web nginx=nginx:1.25
kubectl rollout status deployment/web
# 예상 출력:
# Waiting for deployment "web" rollout to finish: 2 out of 5 new replicas have been updated...
# deployment "web" successfully rolled out

# 2. 존재하지 않는 이미지로 업데이트 (실패 시나리오)
kubectl set image deployment/web nginx=nginx:99.99.99

# 3. 새 Pod가 ImagePullBackOff 로 실패하는 것을 관찰
kubectl get pods
# 예상 출력:
# NAME                   READY   STATUS             RESTARTS   AGE
# web-<new-hash>-aaaaa   0/1     ImagePullBackOff   0          15s
# web-<old-hash>-bbbbb   1/1     Running            0          3m
# web-<old-hash>-ccccc   1/1     Running            0          3m
# ...
# → 롤링 업데이트가 새 Pod를 띄우지 못하므로 기존 Pod는 그대로 살아 있음

# 4. 이전 버전으로 즉시 롤백
kubectl rollout undo deployment/web
# 출력: deployment.apps/web rolled back

# 5. 롤백 완료 후 상태 확인
kubectl rollout status deployment/web
# 출력: deployment "web" successfully rolled out

kubectl get pods
# 모든 Pod가 Running 1/1 상태여야 함

# 6. 현재 이미지 버전 확인 (nginx:1.25 로 돌아갔는지 점검)
kubectl describe deployment web | grep Image
# 출력: Image: nginx:1.25

롤링 업데이트는 새 Pod를 하나씩 띄우고 Ready 상태가 되면 기존 Pod를 하나씩 내리는 방식으로 진행됩니다. 새 이미지 태그가 존재하지 않으면 'ImagePullBackOff' 상태가 발생해 새 Pod가 Ready가 되지 못하고, maxUnavailable 한도 안에서 기존 Pod가 살아있으므로 서비스는 계속됩니다. 'kubectl rollout undo' 는 Deployment가 기억하는 직전 ReplicaSet을 다시 활성화시켜 변경을 취소합니다. 이 과정에서도 동일하게 롤링 방식으로 Pod가 교체되므로 서비스 중단 없이 복구됩니다. 이 시나리오는 실제 운영에서 잘못된 이미지를 배포했을 때의 긴급 복구 절차와 동일합니다.

05. YAML 매니페스트로 직접 배포하기 - 선언형 구성 실습

YAML 매니페스트로 직접 배포하기 - 선언형 구성 실습

학습 목표

  • 명령형(imperative)과 선언형(declarative) 배포 방식의 차이를 이해한다
  • Deployment와 Service의 YAML 매니페스트를 직접 작성한다
  • kubectl apply로 매니페스트를 적용하고 결과를 검증한다

본문

왜 YAML로 배포하는가

3강과 4강에서 우리는 kubectl run이나 kubectl create deployment 같은 명령어로 Pod와 Deployment를 만들었습니다. 명령어 한 줄로 끝나니 빠르고 편리하죠. 그런데 실무 현장에서는 이렇게 명령어로 자원을 만드는 경우가 거의 없습니다. 대신 YAML 파일에 원하는 상태를 적어 두고 그 파일을 클러스터에 적용합니다. 이번 강의에서는 그 이유와 방법을 직접 손으로 따라 하며 익힙니다.

YAML(YAML Ain't Markup Language)은 사람이 읽기 쉬운 설정 파일 형식입니다. 쿠버네티스에서는 "어떤 자원을, 어떤 모습으로 띄울지"를 적는 설계도 역할을 합니다.

명령형 vs 선언형 — 핵심 차이

두 방식의 차이는 한 문장으로 요약됩니다.

  • 명령형(Imperative): "이렇게 해라"라고 행동을 지시합니다. 예) kubectl run, kubectl create, kubectl delete. 내가 어떤 절차를 밟아야 할지 직접 명령합니다.
  • 선언형(Declarative): "최종 모습은 이래야 한다"라고 상태를 선언합니다. 예) YAML 파일 + kubectl apply. 어떻게 거기에 도달할지는 쿠버네티스가 알아서 계산합니다.

식당에 비유하면, 명령형은 주방에 들어가 "불을 켜고, 패티를 굽고, 빵을 데워라"라고 단계마다 지시하는 것이고, 선언형은 "치즈버거 하나 주세요"라고 원하는 결과만 말하는 것입니다. 주방(쿠버네티스)이 현재 상태와 목표 상태를 비교해 부족한 만큼만 알아서 채웁니다.

이 "현재 상태를 목표 상태에 맞춰 끊임없이 조정하는 것"을 조정 루프(reconciliation loop) 라고 부릅니다. 선언형 방식이 강력한 이유가 바로 여기 있습니다. 같은 YAML을 두 번, 세 번 적용해도 결과는 항상 동일하고(멱등성), 누군가 Pod를 지워도 쿠버네티스가 선언된 상태로 되돌립니다.

아래 그림처럼 조정 루프는 "목표 상태와 현재 상태를 비교 → 차이가 있으면 조정 → 다시 비교"를 끝없이 반복합니다.

선언형 방식의 실무 장점을 정리하면 다음과 같습니다.

  • 버전 관리: YAML을 Git에 올려 변경 이력을 추적하고 되돌릴 수 있습니다.
  • 재현성: 같은 파일로 개발/스테이징/운영 환경에 똑같은 구성을 재현합니다.
  • 협업: 팀원에게 파일만 공유하면 동일한 자원을 만들 수 있습니다.
  • 검토 가능: 코드처럼 리뷰(PR)를 거쳐 배포할 수 있습니다.

실험·학습 단계에서는 kubectl run도 충분합니다. 하지만 실제 운영 환경이라면 YAML + kubectl apply가 정답입니다.

모든 매니페스트의 공통 뼈대

자원 종류가 Pod든 Deployment든 Service든, 매니페스트는 항상 4개의 최상위 필드로 시작합니다.

| 필드 | 의미 |

|------|------|

| apiVersion | 사용할 쿠버네티스 API 버전 (자원 종류마다 다름) |

| kind | 만들 자원의 종류 (Pod, Deployment, Service 등) |

| metadata | 이름·라벨 등 자원을 식별하는 정보 |

| spec | 자원이 갖춰야 할 실제 명세(원하는 상태) |

여기서 초보자가 가장 많이 틀리는 지점이 apiVersion입니다. 자원마다 값이 다릅니다. Pod와 Service는 v1, Deployment는 apps/v1을 씁니다. 또 YAML은 들여쓰기(공백 2칸)로 계층을 표현하므로, 탭 대신 스페이스를 쓰고 콜론(:) 뒤에는 반드시 한 칸을 띄워야 합니다.

실습 1 — Deployment 매니페스트 작성

이제 nginx 웹 서버를 2개 복제본으로 띄우는 Deployment를 작성해 봅니다. nginx-deployment.yaml 파일을 만듭니다.


apiVersion: apps/v1          # Deployment는 apps/v1
kind: Deployment
metadata:
  name: nginx-deployment      # 이 Deployment의 이름
  labels:
    app: nginx
spec:
  replicas: 2                 # Pod를 몇 개 띄울지 (원하는 상태)
  selector:
    matchLabels:
      app: nginx              # 아래 라벨과 일치하는 Pod를 내 관리 대상으로 삼음
  template:                   # 찍어낼 Pod의 설계도
    metadata:
      labels:
        app: nginx            # selector와 반드시 일치해야 함
    spec:
      containers:
        - name: nginx         # 컨테이너 목록(리스트)이라 '-'로 시작
          image: nginx:1.27
          ports:
            - containerPort: 80

핵심은 selector의 라벨과 template의 라벨이 반드시 일치해야 한다는 점입니다. Deployment는 이 라벨을 기준으로 "내가 책임질 Pod"를 찾기 때문입니다. 불일치하면 적용 단계에서 에러가 납니다.

실습 2 — Service 매니페스트 작성

Deployment가 만든 Pod들은 클러스터 내부에서만 보이고, IP도 재시작 때마다 바뀝니다. 외부에서 접속하려면 안정적인 진입점인 Service가 필요합니다. nginx-service.yaml을 만듭니다.


apiVersion: v1               # Service는 v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: NodePort             # 노드의 포트로 외부 노출
  selector:
    app: nginx               # 이 라벨을 가진 Pod로 트래픽 전달
  ports:
    - port: 80               # Service가 받는 포트
      targetPort: 80         # Pod(컨테이너)로 전달할 포트
      nodePort: 30080        # 외부에서 접속할 노드 포트(30000~32767)

Service 역시 selector 라벨(app: nginx)로 대상 Pod를 찾습니다. Deployment에서 붙인 라벨과 같기 때문에, Service는 그 2개의 Pod에 트래픽을 자동으로 나눠 보냅니다. Pod IP를 일일이 몰라도 되고, Pod가 죽고 새로 떠도 라벨만 맞으면 알아서 연결됩니다.

아래 구성도처럼 Deployment와 Service는 둘 다 app: nginx 라벨을 매개로 같은 Pod 묶음과 연결됩니다. 이 라벨이 일치해야 모든 연결이 성립합니다.

실습 3 — kubectl apply로 적용하고 검증하기

작성한 매니페스트를 클러스터에 적용합니다. 명령형의 create와 달리 apply를 씁니다.


kubectl apply -f nginx-deployment.yaml
kubectl apply -f nginx-service.yaml


deployment.apps/nginx-deployment created
service/nginx-service created

create는 "새로 만들기"만 하지만, apply는 "이 상태가 되도록 맞춰라"입니다. 파일을 수정한 뒤 다시 apply하면 바뀐 부분만 반영됩니다. 이것이 선언형 운영의 핵심입니다.

이제 결과를 검증합니다.

핵심 요약

  • 명령형은 "행동"을 지시하고(kubectl run/create), 선언형은 "원하는 상태"를 선언한다(YAML + kubectl apply).
  • 선언형은 버전 관리·재현성·협업에 유리하며, 조정 루프 덕분에 멱등성과 자가 치유가 보장된다.
  • 모든 매니페스트는 apiVersion, kind, metadata, spec 4개 필드로 구성된다. Pod/Service는 v1, Deployment는 apps/v1.
  • Deployment·Service는 selector 라벨로 대상 Pod를 찾으므로 라벨 일치가 매우 중요하다.
  • kubectl apply -f로 적용하고 kubectl get으로 검증하며, 파일을 고쳐 다시 apply하면 변경분만 반영된다.

실습 2. 아래 요구사항에 맞는 Deployment 매니페스트(`my-deployment.yaml`)를 처음부터 작성하고 클러스터에 적용하세요. 요구사항: - 이름: `my-nginx` - 복제본 수: 3 - 컨테이너 이름: `nginx-container`, 이미지: `nginx:1.27`, 컨테이너 포트: 80 - Pod 라벨: `app: my-nginx` 적용 후 다음 명령으로 상태를 검증하세요. ```bash kubectl get deployment my-nginx kubectl get pods -l app=my-nginx ``` 예상 결과: READY 칼럼이 `3/3`, Pod 3개가 모두 Running이어야 합니다.

실습 3. 핵심 실습 1에서 만든 `my-nginx` Deployment가 실행 중인 상태에서 NodePort 타입의 Service 매니페스트(`my-service.yaml`)를 작성하고 적용하세요. 요구사항: - 이름: `my-nginx-svc` - 타입: NodePort - 셀렉터 라벨: `app: my-nginx` - port: 80, targetPort: 80, nodePort: 30090 적용 후 다음 명령으로 검증하고, minikube 환경이라면 curl로 응답을 확인하세요. ```bash kubectl get service my-nginx-svc # minikube 환경에서 실제 접속 확인 curl http://$(minikube ip):30090 ``` `Welcome to nginx!` 문자열이 응답에 포함돼 있으면 성공입니다.

# my-service.yaml — 아래부터 직접 작성하세요
# apiVersion, kind, metadata, spec 순으로 작성
# spec.type, spec.selector, spec.ports 필드가 필요합니다
힌트
  • Service의 `apiVersion`은 `v1`입니다. `spec.type: NodePort`를 빠뜨리면 기본값인 ClusterIP 타입이 되어 외부에서 접속할 수 없습니다.
  • `spec.selector`의 라벨값이 핵심 실습 1에서 정한 `app: my-nginx`와 정확히 일치해야 합니다. 오타 한 글자만 달라도 Pod로 트래픽이 전달되지 않습니다.
  • curl 응답이 없거나 `Connection refused`가 뜨면 `kubectl get endpoints my-nginx-svc` 를 실행하세요. ENDPOINTS 열에 Pod IP가 표시되면 Service가 Pod와 연결된 것이고, `<none>`이면 라벨 셀렉터 불일치입니다.
정답 보기 
# my-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-nginx-svc
spec:
  type: NodePort
  selector:
    app: my-nginx
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30090

Service의 `apiVersion`은 `v1`입니다(Deployment와 달리 `apps/v1`이 아님). `spec.selector`의 라벨(`app: my-nginx`)은 Deployment가 Pod에 붙인 라벨과 일치해야 합니다. 이 라벨 매칭이 곧 'Service가 어떤 Pod에 트래픽을 전달할지'를 결정합니다. NodePort 타입에서 세 포트 필드의 역할을 구분하면: - `port: 80` — 클러스터 내부에서 Service의 ClusterIP로 접속할 때 쓰는 포트 - `targetPort: 80` — 실제 컨테이너(Pod)가 수신하는 포트 - `nodePort: 30090` — 클러스터 외부에서 노드 IP로 접속할 때 쓰는 포트 (30000~32767 범위) `kubectl get service my-nginx-svc`의 PORT(S) 컬럼에 `80:30090/TCP`가 보이면 정상입니다. minikube 환경에서는 `minikube ip`가 노드 IP(보통 192.168.x.x 대역)를 반환하고, 그 IP의 30090 포트로 nginx에 접속됩니다. 정리: `kubectl delete -f my-service.yaml -f my-deployment.yaml`

실습 4. 핵심 실습 1에서 작성한 `my-deployment.yaml`의 `replicas`를 3에서 5로 수정한 뒤 `kubectl apply`를 다시 실행하세요. 기존 클러스터 자원을 지우지 말고 그대로 두고 파일만 수정해서 apply하는 것이 핵심입니다. 단계: 1. `my-deployment.yaml`의 `replicas: 3`을 `replicas: 5`로 수정 2. `kubectl apply -f my-deployment.yaml` 3. 아래 명령으로 Pod 수가 5개로 늘었는지 확인 ```bash kubectl get deployment my-nginx kubectl get pods -l app=my-nginx ``` 예상 결과: READY `5/5`, Running Pod 5개. 추가 도전: Pod 하나를 강제로 삭제(`kubectl delete pod <pod-name>`)한 뒤, 잠시 기다렸다가 `kubectl get pods -l app=my-nginx`를 다시 실행해 Pod가 5개로 자동 복구되는지 확인하세요.

# kubectl delete pod <my-nginx-xxx-yyy>
# kubectl get pods -l app=my-nginx   # 잠시 후 재실행 → 5개 유지 확인

이 응용 과제는 선언형 방식의 핵심 장점 두 가지를 직접 체험합니다. 첫째, **멱등성(idempotency)**: `kubectl apply`는 파일의 현재 상태와 클러스터의 현재 상태를 비교해 차이나는 부분만 반영합니다. `replicas: 3 → 5` 로 변경하면 쿠버네티스는 기존 3개 Pod는 그대로 두고 새로운 2개만 추가합니다. 서비스 중단 없이 스케일 아웃이 이루어집니다. 둘째, **자가 치유(self-healing)**: Deployment 컨트롤러는 조정 루프를 통해 항상 '복제본 5개'라는 목표 상태를 유지하려 합니다. Pod 하나를 강제 삭제하면 컨트롤러가 즉시 감지하고 새 Pod를 생성해 5개를 맞춥니다. 이것이 운영 환경에서 Deployment를 쓰는 핵심 이유입니다. `kubectl get deployment my-nginx` 출력에서 `UP-TO-DATE` 값이 5로 바뀌고 `AVAILABLE`이 5에 도달하는 시점을 지켜보세요. 전체 정리: `kubectl delete deployment my-nginx && kubectl delete service my-nginx-svc`
