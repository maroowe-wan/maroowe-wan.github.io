01. CI/CD 파이프라인 전체 그림과 이 강좌의 목표

학습 목표

  • CI/CD가 해결하는 문제를 이해한다: 수동 배포는 느리고, 실수가 잦으며, 재현이 안 된다.
  • CI와 CD가 각각 무엇을 의미하는지, 모든 파이프라인이 공유하는 4단계가 무엇인지 안다.
  • 이 강좌에서 직접 만들 파이프라인의 전체 구조를 미리 파악한다 — GitLab → Jenkins → AWS ECR → EC2.

본문

문제: 수동 배포

자동화 없이 변경 사항을 배포하려면 누군가가 직접 브랜치를 병합하고 충돌을 해결한 뒤 로컬에서 테스트를 돌려야 한다. 그런 다음 서버에 SSH로 접속해 컨테이너를 멈추고, Docker Compose 파일을 수정하고, 올바른 순서로 모든 것을 다시 기동한다. 매 단계마다 사람이 직접 챙겨야 하다 보니, 팀은 금요일 저녁 배포를 두려워한다. 밤 9시에 장애가 나도 아무도 없으니까. 결과는 느리고, 스트레스가 심하며, 무엇보다 재현이 안 된다: 열 번 해도 열 번 결과가 조금씩 다르다.

CI/CD는 반복적이고 실수가 잦은 단계에서 사람을 빼내, 팀이 진짜 판단이 필요한 일에 집중하게 만드는 방법론이다.

CI와 CD의 의미

CI — 지속적 통합(Continuous Integration). 개발자들이 작은 변경 사항을 자주 병합하면, 자동화 시스템이 즉시 빌드하고 테스트한다. 문제가 일찍, 그리고 독립적으로 드러나기 때문에 고치는 비용이 훨씬 적다.

CD — 지속적 제공/배포(Continuous Delivery / Deployment). 코드가 자동화된 검사를 통과하면 자동으로 패키징되어 릴리스 방향으로 이동한다. 지속적 제공(Continuous Delivery)은 모든 변경 사항을 언제든 배포 가능한 상태로 유지하는 것이다(운영 환경 최종 반영은 원클릭 승인으로 남겨둘 수 있다). 지속적 배포(Continuous Deployment)는 한 단계 더 나아가, 통과한 모든 변경 사항을 사람 개입 없이 운영 환경에 자동으로 릴리스한다.

둘을 합치면 CI/CD 파이프라인은 코드를 위한 자동화된 조립 라인이 된다. 푸시 하나가 일련의 단계를 트리거하고, 각 단계는 앞 단계가 성공해야만 다음으로 넘어간다.

모든 파이프라인이 공유하는 4단계

어떤 도구를 쓰든 거의 모든 파이프라인은 같은 구조를 따른다.

1. 소스(Source) — 저장소(여기서는 GitLab)에 코드 변경 사항이 푸시된다. 이것이 트리거다.

2. 빌드(Build) — 코드와 의존성을 배포 가능한 산출물로 조립한다. 여기서 그 산출물은 Docker 이미지다.

3. 테스트(Test) — 빌드된 결과에 자동화 테스트를 실행한다. 실패하면 파이프라인이 멈춘다. 나쁜 코드는 절대 사용자에게 닿지 않는다.

4. 배포(Deploy) — 검증된 산출물을 실행 환경(AWS EC2 서버)에 릴리스한다.

Netflix나 Amazon 같은 회사가 하루에 수십~수천 번 배포할 수 있는 이유가 바로 여기 있다. git push 하나가 매번 정확히 동일한 자동화 경로를 거친다.

우리가 만들 파이프라인

이 강좌는 실습 중심이다. 마치면 자신의 코드를 git push 하나로 EC2 위의 실행 중인 컨테이너까지 보낼 수 있는 파이프라인이 완성된다. 중간에 터미널을 직접 건드릴 필요가 없다. 전체 흐름은 다음과 같다.

1. GitLab 저장소에 코드를 푸시한다.

2. Webhook이 Jenkins에 변경 사항이 생겼음을 알린다.

3. Jenkins가 코드를 체크아웃하고 빌드와 테스트를 실행한다 — 테스트가 실패하면 여기서 멈춘다.

4. 성공하면 Jenkins가 앱의 Docker 이미지를 빌드한다.

5. Jenkins가 AWS ECR(프라이빗 이미지 레지스트리)에 인증하고 태그된 이미지를 푸시한다.

6. Jenkins가 SSH로 EC2에 접속해 docker compose로 새 이미지를 pull하고 실행 중인 컨테이너를 교체한다.

7. 헬스 체크로 새 버전이 정상 동작하는지 확인한다. 실패하면 파이프라인이 이전 이미지로 롤백한다. 시크릿은 코드에 포함되지 않고 안전하게 주입된다.

아래 다이어그램은 이 전체 흐름을 한눈에 보여준다. 테스트 실패 시 파이프라인이 어디서 멈추는지, 헬스 체크 실패 시 어디서 롤백이 발생하는지도 함께 표시했다.

예제로는 언어에 구애받지 않는 간단한 웹앱을 사용한다. Dockerfile, Jenkinsfile, 그리고 배포 로직이 진짜 배울 내용이며, 언제든 자신의 애플리케이션으로 바꿔 쓸 수 있다.

초기 구성에 며칠이 걸릴 수 있지만, 한 번 해두면 빠뜨린 단계로 인한 실수도, 특정 시니어 엔지니어를 기다릴 필요도 없어진다. 릴리스는 빨라지고, 모든 변경 사항이 매번 동일하게 테스트됐다는 확신을 갖게 된다.

핵심 요약

  • 수동 배포는 느리고 스트레스가 심하며 재현이 안 된다. 사람이 매번 같은 단계를 직접 반복해야 하기 때문이다.
  • CI는 작은 변경 사항을 자주 통합하고 테스트하며, CD는 테스트를 통과한 결과를 자동으로 제공(또는 배포)한다.
  • 거의 모든 파이프라인은 소스 → 빌드 → 테스트 → 배포의 4단계를 따르며, 앞 단계가 성공해야만 다음으로 넘어간다.
  • 이 강좌에서는 GitLab에 git push 하나로 Jenkins가 빌드·테스트·이미지 push(AWS ECR)·EC2 배포를 모두 수행하는 파이프라인을 만든다. 헬스 체크, 롤백, 시크릿 관리까지 갖춘 완전한 파이프라인이다.

📝 퀴즈

Q1. 한 개발자가 '지속적 배포(Continuous Deployment)는 프로덕션에 반영되기 전에 반드시 수동 승인이 필요하다'고 말한다. 이 주장에서 잘못된 부분은 무엇이고 왜 그런가?

  • 이 주장은 완전히 맞다 — CD에서는 항상 수동 승인이 필요하다.
  • 지속적 배포(Continuous Deployment)는 자동 검사를 통과한 모든 변경을 프로덕션까지 자동으로 배포한다. 수동 승인 단계를 유지하는 것은 지속적 전달(Continuous Delivery)이다.
  • 지속적 배포는 수동 검사를 거쳐 주 1회만 배포한다는 뜻이다.
  • 지속적 전달과 지속적 배포 모두 프로덕션 반영 전에 수동 승인 단계가 필요하다.

정답은 2번이다. 지속적 전달(Continuous Delivery)은 모든 변경을 항상 배포 가능한 상태로 유지하되, 최종 수동 승인 단계를 남겨 둘 수 있다. 반면 지속적 배포(Continuous Deployment)는 그 단계마저 없애고 자동 검사를 통과한 모든 변경을 프로덕션까지 자동으로 배포한다. 나머지 선택지는 두 용어를 혼동하거나, 강의에서 언급하지 않은 주 1회 배포 주기를 임의로 만들어낸 것이다.

Q2. 소스 → 빌드 → 테스트 → 배포 순서로 파이프라인 단계를 구성하는 이유 중 프로덕션 환경을 보호하는 원리를 가장 잘 설명한 것은?

  • 모든 개발자가 동시에 코드를 머지하도록 강제해 충돌을 막는다.
  • 각 단계가 게이트 역할을 한다. 실패하면 파이프라인이 멈춰 미완성·불안정한 코드가 배포 단계까지 진행될 수 없다.
  • 단계를 순서대로 실행하는 것이 병렬 실행보다 빠르다.
  • 소스 단계가 5분 이상 실행되지 않으면 배포 단계가 시작되지 않는다.

정답은 2번으로, 강의에서 설명한 4단계 파이프라인의 핵심 원리를 담고 있다. 각 단계는 성공했을 때만 다음 단계로 제어권을 넘긴다. 빌드 실패 시 테스트가 실행되지 않고, 테스트 실패 시 이미지 생성과 배포가 차단된다. 이 '빠른 실패(fail fast)' 패턴이 나쁜 코드가 사용자에게 닿지 않도록 막는다. 선택지 1, 3, 4는 관계없거나 실제로 존재하지 않는 규칙을 서술한다.

Q3. 이 강좌의 파이프라인에서 GitLab으로부터 웹훅 알림을 받아 빌드·테스트·배포 단계를 실행하는 구성 요소는 무엇인가?

  • AWS EC2
  • AWS ECR
  • Jenkins
  • Docker Compose

Jenkins가 이 파이프라인의 자동화 서버다. 개발자가 코드를 푸시하면 GitLab이 HTTP POST(웹훅)를 Jenkins에 보내고, Jenkins가 코드를 체크아웃하고 테스트를 실행하고 Docker 이미지를 빌드해 ECR에 푸시한 뒤 EC2에 배포한다. EC2는 배포 대상 서버이고, ECR은 이미지 레지스트리이며, Docker Compose는 EC2에서 컨테이너를 실행하는 도구다. 이 셋 중 어느 것도 GitLab 푸시 이벤트에 직접 반응하지 않는다.

Q4. 한 팀이 매주 수동으로 배포하다가 금요일 밤 9시에 배포 실패를 겪었다. 이 상황이 수동 배포의 어떤 특성을 보여주는가?

  • 수동 배포는 빠르지만 추적성이 없다.
  • 수동 배포는 재현 가능하지만 시니어 엔지니어의 참여가 필요하다.
  • 수동 배포는 느리고 스트레스가 심하며 사람이 자리에 있어야 한다. 그래서 업무 시간 외 장애 복구가 어렵다.
  • 수동 배포는 시니어 엔지니어가 매 단계를 검토하므로 안전하다.

강의에서 이 상황을 명시적으로 다룬다. 밤 9시에 문제가 터졌는데 아무도 보고 있지 않으면 고객은 아침까지 기다려야 한다. 이는 수동 배포가 사람이 자리에 있고, 깨어 있고, 실수하지 않아야 한다는 점, 그리고 그렇지 못할 때 복구가 늦어진다는 점을 보여 준다. 선택지 1은 수동 배포가 빠르지 않으므로 틀리고, 선택지 2는 수동 배포의 최대 약점이 재현 불가능성이므로 틀리며, 선택지 4는 강의가 반박하는 낭만화된 표현이다.

Q5. 이 강좌를 마친 후, 개발자가 'git push'를 실행한 순간부터 EC2에 새 버전이 배포되기까지 어떤 일이 자동으로 일어나는가?

  • 개발자가 EC2에 SSH로 접속해 docker compose up을 수동으로 실행한다.
  • GitLab CI 잡이 Jenkins 없이 이미지를 빌드하고 배포한다.
  • Jenkins가 코드를 체크아웃하고 빌드·테스트한 뒤 Docker 이미지를 만들어 ECR에 푸시하고, EC2에 SSH로 접속해 컨테이너를 교체하고 헬스체크로 결과를 확인한다. 모두 자동으로.
  • 새 이미지가 운영팀에 이메일로 전달되고 팀이 업무 시간에 배포한다.

선택지 3은 강의에서 소개한 자동화 엔드투엔드 흐름을 정확히 설명한다. GitLab 웹훅 → Jenkins → 체크아웃 → 빌드/테스트 → Docker 이미지 → ECR 푸시 → EC2 SSH → docker compose → 헬스체크/롤백. 선택지 1은 파이프라인이 대체하는 수동 과정이다. 선택지 2는 GitLab CI 잡이 아니라 Jenkins가 파이프라인을 조율하므로 틀리다. 선택지 4는 실제로 존재하지 않는 방법이다.

02. 내 앱을 컨테이너로 — Dockerfile 작성과 로컬 실행

학습 목표

  • 왜 컨테이너 이미지가 전체 파이프라인의 배포 단위인지 이해한다.
  • 웹앱을 이미지로 만드는 표준 Dockerfile 패턴을 익힌다.
  • docker build와 docker run으로 이미지를 로컬에서 빌드·실행하고 동작을 확인한다.

본문

왜 컨테이너를 쓰나

컨테이너는 코드와 실행에 필요한 모든 것 — 런타임, 라이브러리, 설정 — 을 하나로 묶는다. 덕분에 내 노트북에서도, EC2에서도 똑같이 실행된다. "내 로컬에서는 되는데요"는 더 이상 통하지 않는다. 세 가지 용어를 먼저 정리해 두자.

  • Dockerfile — 빌드 명령이 담긴 텍스트 레시피.
  • 이미지(Image) — Dockerfile로 빌드한 불변 스냅샷.
  • 컨테이너(Container) — 이미지의 실행 인스턴스.

이 파이프라인에서 이미지는 내 노트북에서 Jenkins를 거쳐 ECR로, 다시 EC2로 이동하는 단 하나의 배포 산출물이다. 이후에 자동화하는 모든 것은 결국 "이 이미지를 빌드하고, 저장하고, 어딘가에서 실행한다"는 흐름이다. Dockerfile이 그 모든 것의 기초다.

1단계 — Dockerfile 작성

프로젝트 루트에 확장자 없이 Dockerfile이라는 이름으로 파일을 만든다. 아래 예제는 Node.js 기준이지만 구조 자체는 어떤 언어에든 적용된다.


# 런타임이 포함된 공식 slim 베이스 이미지를 지정한다
FROM node:20-alpine

# 이미지 안의 작업 디렉터리를 설정한다
WORKDIR /app

# 의존성 목록을 먼저 복사하고 설치한다 — 이 레이어가 캐시된다
COPY package.json package-lock.json ./
RUN npm install

# 나머지 소스 코드를 복사한다
COPY . .

# 앱이 리스닝하는 포트를 명시한다
EXPOSE 8080

# 컨테이너가 시작될 때 실행할 명령을 지정한다
CMD ["node", "index.js"]

핵심 포인트 세 가지:

  • FROM — slim 공식 이미지를 쓴다. node:20-alpine은 Alpine Linux(약 5 MB) 기반이라 이미지를 작고 빠르게 유지한다.
  • 레이어 캐싱이 순서를 결정한다. 각 명령은 캐시되는 레이어다. Docker는 변경된 레이어부터만 재빌드하므로, 의존성은 앞에 두고 소스는 뒤에 둬야 한다. 의존성은 거의 안 바뀌지만 소스는 수시로 바뀐다 — package.json을 복사하고 npm install을 먼저 실행해 두면, 코드만 고쳤을 때 설치 단계를 건너뛸 수 있다. 순서를 뒤집으면 코드 한 줄 바꿀 때마다 전체 재설치가 강제된다.
  • CMD는 배열(exec) 형식을 쓴다 — 문자열 대신 ["node", "index.js"]. shell을 거치지 않고 프로세스를 직접 실행하므로 시그널 처리와 종료 동작이 올바르게 작동한다.

2단계 — .dockerignore 추가

COPY . .은 로컬 node_modules나 혹시 있을 시크릿 파일까지 모든 것을 복사한다. .gitignore처럼 제외 목록을 만든다.


node_modules
.git
*.env

이미지를 작게 유지하고, 이미지를 pull하는 누군가가 레이어를 들여다봤을 때 시크릿이 노출되지 않도록 막아 준다.

3단계 — 이미지 빌드

-t 플래그로 이름을 붙이고, 끝의 .으로 현재 디렉터리를 빌드 컨텍스트로 지정한다.


docker build -t myapp:local .

빌드된 이미지를 확인한다.


docker images

4단계 — 실행 및 확인


docker run -p 5000:8080 myapp:local

-p 5000:8080 플래그가 핵심이다. 컨테이너 포트는 기본적으로 외부에서 접근할 수 없다. 이 플래그가 호스트 포트 5000을 컨테이너 포트 8080(EXPOSE한 포트)에 연결한다. 형식은 호스트:컨테이너다. http://localhost:5000을 열면 앱이 보여야 한다.

백그라운드에서 실행하고 관리하려면 아래 명령을 쓴다.


docker run -d -p 5000:8080 myapp:local   # 백그라운드 실행
docker ps                                # 실행 중인 컨테이너 확인
docker logs <container>                  # 로그 확인
docker stop <container>                  # 중지

이제 재현 가능한 이미지가 만들어졌고 로컬에서 동작도 확인했다. 지금 수동으로 한 모든 단계가 이후 강의에서 Jenkins 스테이지로 바뀐다.

핵심 정리

  • 컨테이너는 앱과 실행 환경 전체를 묶어 어디서든 동일하게 동작하는 단 하나의 배포 산출물을 만든다.
  • Dockerfile → 이미지 → 컨테이너 의 흐름을 기억한다.
  • Dockerfile은 레이어 캐싱을 고려해 의존성을 소스 앞에 두고, slim 베이스 이미지와 .dockerignore로 이미지를 작고 안전하게 유지한다.
  • docker build -t 이름 .으로 빌드하고, docker run -p 호스트:컨테이너 이름으로 실행한다. 포트 매핑 없이는 앱에 접근할 수 없다.

실습 2. 핵심 실습: 올바른 Dockerfile을 갖춘 최소한의 Node.js 웹앱을 만들어 빌드하고, 'http://localhost:5000'에서 응답이 오도록 로컬에서 실행하라.

요구사항:

1. 'index.js' 작성 — 포트 8080에서 'Hello from container!'를 응답하는 Node.js HTTP 서버.

2. 'start' 스크립트가 있는 'package.json' 작성.

3. 다음 조건을 갖춘 Dockerfile 작성: 'node:20-alpine' 사용, WORKDIR을 /app으로 설정, 소스 코드 복사 전에 의존성을 복사·설치(레이어 캐싱), 포트 8080 EXPOSE, CMD는 exec 형식 사용.

4. 'node_modules', '.git', '*.env'를 제외하는 '.dockerignore' 작성.

5. 'docker build -t myapp:local .'로 이미지를 빌드하고 'docker run -p 5000:8080 myapp:local'로 실행.

6. http://localhost:5000을 열어 응답을 확인한다.

const http = require('http');
const server = http.createServer((req, res) => {
  res.writeHead(200);
  res.end('Hello from container!');
});
server.listen(8080, () => console.log('Listening on port 8080'));

# package.json
{
  "name": "myapp",
  "version": "1.0.0",
  "scripts": { "start": "node index.js" }
}

# Dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package.json ./
RUN npm install
COPY . .
EXPOSE 8080
CMD ["node", "index.js"]

# .dockerignore
node_modules
.git
*.env

# 빌드 및 실행:
docker build -t myapp:local .
docker run -p 5000:8080 myapp:local
# http://localhost:5000을 열면 'Hello from container!'가 반환된다

'COPY . .'보다 먼저 'package.json'을 복사하고 'npm install'을 실행하는 것이 레이어 캐싱의 핵심이다. 의존성은 자주 바뀌지 않으므로, 소스 코드만 바뀐 다음 빌드에서 Docker가 설치 레이어를 재사용할 수 있다. 'node:20-alpine'을 쓰면 이미지 크기가 ~1 GB가 아닌 ~150 MB 수준으로 유지된다. 포트 매핑 '-p 5000:8080'은 호스트 포트 5000을 컨테이너 포트 8080(EXPOSE한 포트)에 연결한다. 이 플래그 없이는 컨테이너 네트워크가 격리되어 localhost에 아무것도 표시되지 않는다.

실습 3. 핵심 실습(계속): 'docker ps', 'docker logs', 'docker stop'으로 실행 중인 컨테이너를 관리하라. 그런 다음 index.js의 응답 텍스트만 바꿔 이미지를 재빌드해 Docker가 어떤 레이어를 재사용하는지 확인하라.

# 이전 실습에서 'docker run -p 5000:8080 myapp:local'이 실행 중이라고 가정한다.
# 1단계: 컨테이너 ID / 이름 확인
docker ps
# 2단계: 로그 읽기
docker logs <컨테이너-이름-또는-id>
# 3단계: 중지
docker stop <컨테이너-이름-또는-id>
# 4단계: index.js의 응답 텍스트를 바꾼 뒤 재빌드
docker build -t myapp:local .
# 빌드 출력에서 'CACHED' 줄을 확인한다
힌트
  • 'docker ps'가 빈 목록을 반환하면 컨테이너가 종료됐을 수 있다. 'docker ps -a'(종료된 컨테이너 포함 전체 목록)로 찾는다.
  • 재빌드 시 npm install 단계에 'CACHED'가 뜨지 않으면 Dockerfile에서 'RUN npm install' 전에 'COPY . .'을 먼저 쓴 것이 아닌지 확인한다. 그렇게 하면 소스 변경이 그 레이어를 무효화한다.
정답 보기 
# 'docker ps' 실행 후 다음과 같이 표시된다:
# CONTAINER ID   IMAGE         COMMAND           STATUS          PORTS
# a1b2c3d4e5f6   myapp:local   "node index.js"   Up 2 minutes   0.0.0.0:5000->8080/tcp

# docker logs a1b2c3d4e5f6  →  'Listening on port 8080'

# docker stop a1b2c3d4e5f6  →  컨테이너 종료

# index.js 편집: 'Hello from container!'를 'Hello, CI/CD!'로 변경
# 그런 다음: docker build -t myapp:local .
# 예상 빌드 출력:
#  => CACHED [2/5] WORKDIR /app
#  => CACHED [3/5] COPY package.json ./
#  => CACHED [4/5] RUN npm install       <-- 캐시 재사용, 재실행 안 됨
#  => [5/5] COPY . .                     <-- 소스 변경으로 무효화됨
# npm install 레이어가 재사용되어 재빌드가 빠르다.

Docker는 변경된 레이어와 그 이후 레이어만 재빌드한다. 소스 변경이 'COPY . .' 레이어(5단계)만 건드리기 때문에, 비용이 큰 'npm install'을 포함한 그 이전 레이어는 모두 캐시에서 제공된다. Dockerfile에서 의존성 복사·설치가 'COPY . .' 앞에 와야 하는 이유가 바로 이것이다. 순서를 바꾸면 소스가 바뀔 때마다 설치 캐시가 무효화되어 빌드가 불필요하게 느려진다.

실습 4. 응용 도전: 동료의 Dockerfile에 세 가지 문제가 있다. 빌드가 느려지거나, 이미지가 커지거나, 동작이 잘못될 수 있는 문제들이다. 각 문제를 찾아내고 수정된 Dockerfile을 작성하라.

문제 있는 Dockerfile:


FROM node:20
WORKDIR /app
COPY . .
RUN npm install
EXPOSE 8080
CMD node index.js

# 위 Dockerfile을 검토하고 답하라:
# 1. 어떤 베이스 이미지 선택이 불필요하게 큰 이미지를 만드는가?
# 2. COPY와 RUN의 순서 중 레이어 캐싱을 망치는 부분은 어디인가?
# 3. CMD 명령의 문법에 어떤 문제가 있는가?
# 그런 다음 수정된 Dockerfile을 작성하라.
힌트
  • 이미지 크기 확인: 'FROM node:20' 버전과 'FROM node:20-alpine' 버전 둘 다 빌드한 뒤 'docker images'를 실행해 SIZE 열을 비교한다.
  • 캐싱 차이 확인: 잘못된 버전으로 빌드하고, index.js를 수정한 뒤 재빌드하여 몇 개 레이어가 'CACHED'로 표시되는지 본다. 그런 다음 수정된 버전으로 같은 작업을 해 비교한다.
  • CMD shell 대 exec 형식은 쉽게 구분된다. exec 형식은 대괄호와 따옴표가 있는 JSON 배열을 쓰고, shell 형식은 대괄호 없는 일반 문자열이다.
정답 보기 
# 문제 1: 'FROM node:20'은 전체 Debian 기반 이미지(~1 GB)를 사용한다.
#   수정: 'FROM node:20-alpine' (~150 MB)

# 문제 2: 'RUN npm install' 전에 'COPY . .'이 온다.
#   소스 코드가 바뀔 때마다 COPY 레이어가 무효화되어 매번 npm install이 강제된다.
#   수정: package.json만 먼저 복사하고 설치한 뒤 소스를 복사한다.

# 문제 3: 'CMD node index.js'는 shell 형식이다(/bin/sh -c로 감쌈).
#   Docker가 shell에 시그널을 보내므로 Node 프로세스가 제대로 종료되지 않는다.
#   수정: 'CMD ["node", "index.js"]' (exec 형식)

# 수정된 Dockerfile:
FROM node:20-alpine
WORKDIR /app
COPY package.json ./
RUN npm install
COPY . .
EXPOSE 8080
CMD ["node", "index.js"]

이 세 가지는 실무에서 가장 흔한 Dockerfile 실수다. 베이스 이미지 선택은 CI 빌드마다 이미지 push/pull 시간에 직접 영향을 준다. 레이어 캐시 순서는 Dockerfile에서 성능에 가장 큰 영향을 주는 결정이다 — 잘못 작성하면 코드를 한 줄 바꿀 때마다 전체 의존성 재설치가 발생한다. CMD exec 형식 대 shell 형식의 차이는 시그널 전달에 영향을 미친다. 시그널을 제대로 받지 못한 프로세스는 SIGTERM 대신 SIGKILL로 강제 종료되어 데이터 손실이나 비정상 종료로 이어질 수 있다.

03. GitLab 저장소 구성과 Push 트리거(Webhook)

학습 목표

  • dev/main 브랜치 전략으로 GitLab 저장소에 코드를 푸시한다.
  • push 시 Jenkins 빌드가 자동으로 시작되도록 GitLab과 Jenkins를 연결한다.
  • 공유 시크릿 토큰으로 보호되는 GitLab 웹훅을 설정하고 동작을 확인한다.

본문

파이프라인의 목적은 명령어를 손으로 실행하는 것이 아니라 코드 푸시 하나로 모든 작업이 시작되게 하는 것이다. 이를 위해 코드를 올릴 장소(GitLab)와 Jenkins에 "뭔가 바뀌었다"고 알리는 수단(웹훅)이 필요하다. 이번 강의에서 이 둘을 연결한다.

Task 1 — 앱을 GitLab에 푸시하기

무엇을, 왜: Jenkins가 감시할 저장소가 있어야 한다. 이전 강의에서 만든 Dockerfile이 포함된 앱을 새 GitLab 프로젝트에 푸시한다.


git init
git remote add origin https://gitlab.com/<your-namespace>/<your-project>.git
git add .
git commit -m "Initial commit with Dockerfile"
git push -u origin main

# 일상 작업용 dev 브랜치 생성
git checkout -b dev
git push -u origin dev

브랜치 전략: 일상 작업은 dev에서 하고 main은 "배포 대상"으로 유지한다. 트리거를 dev에만 걸어두면 배포 브랜치를 건드리지 않고 마음껏 실험할 수 있다.

Task 2 — GitLab 개인 액세스 토큰 만들기

무엇을, 왜: Jenkins가 GitLab API와 통신하려면 토큰이 필요하다.

1. GitLab → 아바타(좌상단) → Preferences(구 UI: Edit profile) → Access Tokens.

2. Add new token을 클릭해 이름과 만료일을 설정하고 범위를 api로 지정한다.

3. 토큰을 즉시 복사한다 — GitLab은 단 한 번만 보여 준다. 잃어버리면 새 토큰을 만들어야 한다.

Task 3 — Jenkins에 GitLab 플러그인 설치하기

무엇을, 왜: 이 플러그인이 있어야 GitLab이 빌드를 트리거할 수 있고, Jenkins가 빌드 결과를 GitLab에 보고할 수 있다.

  • Manage Jenkins → Plugins → Available → GitLab 검색 → 설치.
  • GitLab API Plugin이 자동으로 함께 설치된다.

Task 4 — Jenkins에 GitLab 연결 생성하기

무엇을, 왜: GitLab 서버와 API 토큰을 등록해 Jenkins가 인증할 수 있도록 한다.

Manage Jenkins → System → GitLab 섹션:

  • Connection name — 원하는 이름, 예: gitlab-saas.
  • GitLab host URL — https://gitlab.com(SaaS) 또는 자체 호스팅 URL.
  • Credentials → Add → 종류: GitLab API token → Task 2에서 복사한 토큰 붙여넣기.

Test Connection을 클릭해 성공 메시지를 확인하고 Save한다.

Task 5 — Jenkins 잡이 푸시를 감지하도록 설정하기

무엇을, 왜: 어떤 이벤트가 빌드를 시작할지 잡에 알려준다.

파이프라인 잡 설정 → Build Triggers → "Build when a change is pushed to GitLab" 활성화. Push Events(필요 시 Merge Request Events도) 선택.

이어서 Advanced를 펼치고 Generate를 클릭해 시크릿 토큰을 생성한다. 토큰을 복사해 두고, 이 화면에 표시된 웹훅 URL도 함께 메모한다. 다음 단계에서 둘 다 사용한다. Save.

Task 6 — GitLab에 웹훅 만들기

무엇을, 왜: 웹훅은 이벤트가 발생했을 때 GitLab이 실제로 호출하는 대상이다. Jenkins에서 가져온 URL과 시크릿 토큰을 그대로 입력해야 양쪽이 맞아떨어진다.

GitLab 프로젝트 → Settings → Webhooks:

  • URL — Jenkins에서 복사한 웹훅 URL(Task 5).
  • Secret token — Jenkins에서 생성한 토큰(Task 5).
  • 트리거: Push events 선택 후 브랜치 지정.
  • Add webhook.

연결의 핵심은 단 두 가지다. Jenkins 잡을 가리키는 URL과, 요청이 진짜 GitLab 프로젝트에서 왔음을 증명하는 공유 시크릿 토큰. 이 둘이 양쪽에서 일치하면 된다.

GitLab의 Test → Push events로 샘플 이벤트를 보내 연결이 정상인지 확인한다.

Task 7 — 엔드투엔드 검증

무엇을, 왜: 실제 푸시 한 번으로 빌드가 자동으로 시작되는지 확인한다.


git checkout dev
echo "trigger test" >> README.md
git commit -am "Test webhook"
git push

Jenkins를 지켜보자. 1~2초 안에 새 빌드가 나타나고 콘솔 출력에 "Started by GitLab push"가 표시된다. 이제 명령어가 아니라 코드 푸시가 파이프라인을 움직인다.

아래 시퀀스 다이어그램은 git push부터 Jenkins가 자동으로 빌드를 시작하기까지의 흐름을 전체적으로 보여 준다.

핵심 정리

  • GitLab에 코드를 푸시하는 것 자체가 파이프라인을 시작한다. 더 이상 명령어를 직접 실행할 필요가 없다.
  • 웹훅은 Jenkins가 반복 폴링하는 대신 GitLab이 이벤트 발생 즉시 알림을 보내는 방식이다.
  • 연결은 두 가지 사실이 양쪽에서 일치해야 성립한다. Jenkins 잡을 가리키는 URL과 요청을 인증하는 공유 시크릿 토큰.
  • 잡에서 "GitLab에 변경이 푸시되면 빌드"를 활성화하고, GitLab에 매칭 웹훅을 만든 뒤, 실제 푸시 후 "Started by GitLab push"가 나타나는지로 동작을 확인한다.

실습 2. 핵심 실습: GitLab과 Jenkins를 연결해 코드 푸시가 파이프라인 빌드를 자동으로 트리거하도록 설정하라.

아래 5단계를 모두 완료한다.

1. 범위가 'api'인 GitLab 개인 액세스 토큰을 만든다.

2. Jenkins에 GitLab 플러그인을 설치한다.

3. Jenkins에 GitLab 연결을 생성한다(Manage Jenkins → System → GitLab 섹션). 'Test Connection'이 성공하면 된다.

4. Jenkins 파이프라인 잡의 Build Triggers에서 'Build when a change is pushed to GitLab'을 활성화하고 Push Events를 선택한다.

5. GitLab 프로젝트(Settings → Webhooks)에서 Jenkins의 Build Triggers 섹션에 표시된 URL과 시크릿 토큰을 사용해 웹훅을 만든다.

GitLab의 'Test → Push events' 버튼을 사용해 빌드가 Jenkins에 나타나는지 확인하라.

# 이 실습은 설정 중심이다. 아래 체크리스트로 진행 상황을 추적한다:
# [ ] GitLab 개인 액세스 토큰 생성 (범위: api)
# [ ] Jenkins에 GitLab 플러그인 설치
# [ ] Jenkins에 GitLab 연결 생성 및 'Test Connection' 성공
# [ ] Jenkins 파이프라인 잡에 'Build when pushed to GitLab' 활성화
# [ ] Jenkins에서 복사한 Webhook URL과 시크릿 토큰을 GitLab에 입력
# [ ] GitLab 'Test > Push events' 실행
# [ ] Jenkins 잡 빌드 기록에 새 빌드 등장
힌트
  • GitLab 'Test → Push events'가 비 200 상태 코드를 반환하거나 타임아웃된다면, Jenkins 호스트가 인터넷에서 접근 가능한지 확인한다. GitLab SaaS는 Jenkins에 POST를 보낼 수 있어야 한다. Jenkins를 로컬에서 실행 중이라면 ngrok 같은 터널링 도구가 필요하다. 'ngrok http 8080'을 실행하면 공개 URL을 받을 수 있다.
  • 'Test Connection'이 실패하면 GitLab host URL에 슬래시가 없는지, Credential 종류가 'GitLab API token'인지 확인한다('Username with password'가 아니어야 함).
  • 테스트 이벤트 후 빌드가 나타나지 않으면 GitLab에서 Settings → Webhooks → Edit으로 이동해 'Recent deliveries'를 확인한다. Jenkins가 반환한 응답이 403이면 시크릿 토큰이 일치하지 않는다는 의미다.

연결은 양쪽에서 일치해야 하는 두 가지 사실로 이루어진다. GitLab에 HTTP POST를 어디로 보낼지 알려주는 URL, 그리고 요청이 신뢰할 수 있는 곳에서 왔음을 증명하는 공유 시크릿 토큰. Jenkins가 push 이벤트를 받으면 요청 헤더의 토큰을 검증하고 일치하면 빌드를 시작한다. 따라서 Jenkins에서 GitLab으로 토큰을 정확히 복사하는 것이 중요하다. 불일치하면 403 오류가 반환되고 빌드가 실행되지 않는다. GitLab 플러그인은 빌드 결과를 GitLab 머지 리퀘스트에 다시 보고하는 기능도 제공해 피드백 루프를 닫아 준다.

실습 3. 응용 실습: 실제 코드 푸시로 웹훅 트리거가 엔드투엔드로 작동하는지 증명하라. 'dev' 브랜치에 사소한 변경을 만들어 푸시하고, Jenkins가 빌드를 자동으로 시작하며 콘솔 출력에 'Started by GitLab push'가 표시되는지 확인하라.

git add README.md
git commit -m "Test webhook trigger"
git push
# 그 다음 Jenkins를 열어 잡의 빌드 기록을 확인한다.

  • 푸시 후 빌드가 나타나지 않으면 GitLab → Settings → Webhooks → Recent deliveries에서 POST 시도 여부와 Jenkins의 응답을 확인한다.
  • 'dev' 대신 'main'에 푸시했다면 Build Trigger에 설정된 브랜치가 무엇인지 확인한다. 푸시한 브랜치와 일치해야 트리거가 실행된다.

04. Jenkins 설치와 첫 파이프라인(Jenkinsfile)

학습 목표

  • Jenkins를 기동하고 파이프라인 잡을 생성한다.
  • Jenkinsfile과 stage/step 구조를 이해한다.
  • GitLab에서 소스를 가져오는 checkout 스테이지를 작성한다.

본문

이전 강의에서 GitLab이 push마다 Jenkins에 알림을 보내도록 설정했다. 하지만 Jenkins는 그 알림을 받아도 아직 아무것도 하지 않는다. 이번에는 Jenkins에 지시사항을 전달한다 — 저장소 안에 Jenkinsfile이라는 파일로 코드를 작성하는 방식으로.

1. Jenkins 실행하기

무엇을·왜: Jenkins 자체를 Docker 컨테이너로 실행해 설정을 깔끔하고 재현 가능하게 유지한다.


docker run -d --name jenkins \
  -p 8080:8080 -p 50000:50000 \
  -v jenkins_home:/var/jenkins_home \
  jenkins/jenkins:lts
  • 8080 → 웹 UI(브라우저로 접속), 50000 → 빌드 에이전트(JNLP) 포트. 두 포트는 용도가 다르므로 반드시 구분한다.
  • -v jenkins_home:/var/jenkins_home 볼륨은 잡과 설정을 컨테이너 재시작 후에도 보존한다 — 빠뜨리면 모든 것이 사라진다.

확인: 로그에서 초기 관리자 비밀번호를 복사해 http://<호스트>:8080의 잠금 해제 화면에 입력한다.


docker logs jenkins   # 잠금 해제 비밀번호 복사

권장 플러그인을 설치하고 관리자 계정을 만들면 준비 완료다.

jenkins_home 볼륨은 반드시 유지한다. 없으면 컨테이너를 재시작할 때마다 설정한 잡이 모두 날아간다.

2. Pipeline 잡 만들기 (Freestyle 말고)

무엇을·왜: Jenkins에는 Freestyle 잡(웹 폼으로 설정, 서버 안에만 저장)과 Pipeline 잡(코드로 정의)이 있다. Pipeline을 쓴다 — 버전 관리가 되고 서버가 날아가도 남아 있으며 조건문·반복문·병렬 스테이지를 지원한다.

Jenkins에서 New Item → 이름 입력 → Pipeline → OK 순으로 선택한다.

3. 동작하는 가장 작은 파이프라인

무엇을·왜: 선언적 파이프라인은 pipeline → agent → stages → stage → steps 형태로 중첩된다. agent any는 사용 가능한 워커라면 어디서든 실행하겠다는 뜻이고, 각 stage는 스테이지 뷰의 열(column)이 된다.


pipeline {
    agent any
    stages {
        stage('Hello') {
            steps {
                echo 'Pipeline is alive!'
            }
        }
    }
}

확인: 인라인 Pipeline 스크립트 박스에 붙여 넣고 Build Now를 클릭한다. 스테이지 뷰에 초록색 "Hello" 열이 보이면 성공이다. 실패하면 Console Output에서 어느 줄이 문제인지 바로 확인할 수 있다.

4. GitLab에서 Jenkinsfile 읽어오기

무엇을·왜: 스크립트를 인라인으로 붙여 넣으면 Pipeline as Code의 목적이 무너진다. Jenkins가 저장소에서 직접 Jenkinsfile을 읽도록 설정하면 빌드 프로세스 변경이 곧 커밋이 된다.

잡 설정의 Pipeline 섹션에서 다음과 같이 지정한다.

  • Definition: Pipeline script from SCM
  • SCM: Git
  • Repository URL + credentials: GitLab 저장소 주소와 자격증명
  • Branch: 예) main
  • Script Path: Jenkinsfile

5. Checkout 스테이지 작성하기

무엇을·왜: 파이프라인이 맨 처음 해야 할 일은 최신 코드를 가져오는 것이다 — 바로 checkout 스테이지다. SCM에서 읽어오는 방식으로 설정했으면 Jenkins는 이미 소스 위치를 알고 있으므로, 내장 스텝 하나로 트리거된 정확한 커밋을 클론할 수 있다. 이 파일을 Jenkinsfile이라는 이름으로 저장소 루트에 커밋한다.


pipeline {
    agent any
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
    }
}

다른 저장소를 명시적으로 체크아웃해야 할 때는 다음처럼 직접 지정한다.


git branch: 'main',
    url: 'https://gitlab.com/<namespace>/<project>.git',
    credentialsId: 'gitlab-creds'

확인: Jenkinsfile을 푸시한다. 이전 강의에서 설정한 웹훅이 빌드를 트리거하고, 이번에는 실제로 코드를 클론하는 파이프라인이 실행된다. 스테이지 뷰에 초록색 Checkout 열이 보이면 루프가 완성된 것이다. push → 트리거 → checkout. 이후 스테이지는 여기에 이어 붙이기만 하면 된다.

핵심 정리

  • 영구 jenkins_home 볼륨을 사용해 Docker 컨테이너로 Jenkins를 실행한다. 8080은 웹 UI, 50000은 에이전트 포트다.
  • Freestyle 대신 Pipeline을 선택한다 — 코드로 관리되고, 버전이 추적되며, 서버에 종속되지 않는다.
  • 선언적 문법은 pipeline → agent → stages → stage → steps 형태로 중첩된다.
  • Pipeline script from SCM으로 설정하고 checkout scm을 첫 번째 스테이지로 만들어 트리거된 커밋을 가져온다.

실습 2. 핵심 실습: 'Hello' 스테이지 하나가 있는 최소한의 선언적 Jenkinsfile을 작성하고, 인라인 스크립트를 사용하는 Pipeline 잡을 Jenkins에 만들어 빌드를 실행한 뒤, Jenkins 스테이지 뷰에서 스테이지가 초록색으로 표시되는지 확인하라.

            }

  • 'Started'는 보이지만 UI에 스테이지 박스가 나타나지 않으면 브라우저를 새로 고쳐야 할 수 있다. Stage View는 Pipeline Stage View 플러그인이 필요하다 — 설치 여부를 확인한다.

'checkout scm'은 잡에 이미 설정된 동일 저장소와 자격증명을 사용해 빌드를 트리거한 정확한 커밋을 클론하는 단축 스텝이다. Jenkinsfile 안에 저장소 URL을 다시 적지 않아도 되므로 유지 보수 부담이 줄어든다. SCM에서 Jenkinsfile을 읽음으로써 빌드 프로세스가 소스 기록의 일부가 된다. 파이프라인에 대한 모든 변경이 버전 관리되고, 리뷰 가능하고, 애플리케이션 코드처럼 롤백할 수 있다. 이것이 'Pipeline as Code'의 철학이다.

실습 4. 응용 실습: Jenkinsfile에 'Verify Workspace'라는 두 번째 스테이지를 추가하라. 이 스테이지에서 'sh' 스텝으로 Checkout 스테이지가 체크아웃한 파일 목록을 출력하고('ls -la'), 현재 Git 커밋 SHA도 출력한다('git rev-parse --short HEAD'). 업데이트된 Jenkinsfile을 푸시하고 두 스테이지가 모두 초록색으로 표시되며 출력이 콘솔에 보이는지 확인하라.

pipeline {
    agent any
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        stage('Verify Workspace') {
            steps {
                // 파일 목록을 출력하는 sh 명령:
                sh '___'
                // 짧은 커밋 SHA를 출력하는 sh 명령:
                sh '___'
            }
        }
    }
}
힌트
  • 'sh' 명령이 'sh: command not found'를 반환하면 Jenkins 에이전트가 Windows 실행기에서 실행 중일 수 있다. Windows에서는 'sh' 대신 'bat'을 사용한다(bat 'dir' 및 bat 'git rev-parse --short HEAD').
  • 'git rev-parse --short HEAD'가 오류를 반환하면 'checkout scm'이 먼저 실행됐는지 확인한다. git 명령은 워크스페이스에 클론된 저장소가 필요하다.
정답 보기 
pipeline {
    agent any
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        stage('Verify Workspace') {
            steps {
                sh 'ls -la'
                sh 'git rev-parse --short HEAD'
            }
        }
    }
}

// git commit -m "Add Verify Workspace stage"
// git push

// [Verify Workspace] $ ls -la
// total 32
// drwxr-xr-x ... Dockerfile index.js package.json ...
// [Verify Workspace] $ git rev-parse --short HEAD
// abc1234

// Stage View에 두 개의 초록색 박스가 나타난다: 'Checkout'과 'Verify Workspace'.

이 실습은 두 가지 핵심 메커니즘을 확인한다. 첫째, 'checkout scm'이 실제로 워크스페이스에 파일을 기록했음을 'ls -la'로 눈으로 확인한다. 둘째, 'git rev-parse --short HEAD'로 커밋 SHA를 캡처하는 것은 강의 6에서 사용할 태깅 전략의 미리보기다. 이 SHA가 Docker 이미지에 태그로 붙어 실행 중인 컨테이너와 소스 코드 사이의 추적 가능한 연결을 만든다. Jenkinsfile에 스테이지를 추가할 때마다 Stage View에 새 열(column)이 생긴다 — 파이프라인의 시각적 진행 보드다.

05. CI 스테이지: 빌드·테스트·이미지 생성

학습 목표

  • Jenkinsfile에 build·test 스테이지를 추가해, 문제가 생기는 즉시 파이프라인이 멈추도록 한다.
  • 테스트를 통과한 경우에만 Docker 이미지를 빌드한다.
  • CI가 자동 품질 게이트 역할을 한다는 개념을 이해한다.

본문

파이프라인은 이미 push를 감지해 코드를 체크아웃한다. 이제 CI를 실질적으로 가치 있게 만드는 스테이지를 추가할 차례다: build, test, build image. 핵심 원칙은 빠른 실패(fail fast)다. 각 스테이지는 게이트 역할을 하며, 선언형 Jenkinsfile에서는 어떤 단계든 0이 아닌 종료 코드를 반환하면 해당 스테이지가 실패하고 이후 모든 단계가 취소된다. 스테이지는 위에서 아래로 순서대로 실행되므로, 순서만 올바르게 배치하면 테스트를 거치지 않은 코드가 이미지가 되는 상황을 원천 차단할 수 있다.

파이프라인을 빨간색으로 만드는 실패한 테스트는 문제가 아니다 — 파이프라인이 가장 중요한 역할을 제대로 수행하고 있다는 신호다.

1. 빌드 스테이지 추가

목적: 빌드 스테이지는 코드와 의존성을 조립한다. '빌드'의 구체적인 내용은 언어마다 다르지만, 형태는 언제나 같다 — 명령을 실행하고, 0이 아닌 종료 코드가 나오면 스테이지를 실패 처리한다.


stage('Build') {
    steps {
        sh 'npm ci'
        sh 'npm run build'
    }
}

sh 단계는 에이전트에서 셸 명령을 실행한다(Windows 에이전트에서는 bat 사용). 둘 중 하나라도 실패하면 파이프라인은 여기서 멈춘다 — 빌드조차 안 된 코드를 테스트할 이유가 없기 때문이다.

확인: 파이프라인을 실행해 Jenkins 스테이지 뷰에서 Build 스테이지가 녹색으로 표시되는지 확인한다.

2. 테스트 스테이지 추가

목적: 버그를 잡아내는 게이트다. 자동화 테스트를 스테이지로 실행하면, 테스트 하나라도 실패할 경우 해당 스테이지가 실패하고 이후 모든 단계가 취소된다.


stage('Test') {
    steps {
        sh 'npm test'
    }
}

단위 테스트, 통합 테스트, 린트/포맷 검사 모두 여기에 넣을 수 있다. 테스트가 실패하더라도 리포트는 항상 수집되도록 post { always { ... } } 블록을 활용하면, 어떤 테스트가 깨졌는지 언제든 확인할 수 있다:


stage('Test') {
    steps {
        sh 'npm test'
    }
    post {
        always {
            junit 'reports/**/*.xml'
        }
    }
}

확인: 의도적으로 실패를 만들어본다(예: 테스트가 단언하는 값을 바꿔본다). 파이프라인이 빨간색으로 바뀌고 이미지 빌드 전에 멈추는지 확인한 뒤, 원래 코드로 되돌린다.

3. 이미지 빌드 — 테스트 통과 후에만

목적: 실패한 스테이지가 파이프라인을 멈추는 구조 덕분에, 이미지 빌드를 테스트 스테이지 다음에 배치하는 것만으로 초록 테스트를 통과한 코드에서만 이미지가 만들어진다는 것을 보장할 수 있다. 여기서 생성된 Docker 이미지는 다음 강의에서 레지스트리에 push하고 배포하게 된다.


stage('Build Image') {
    steps {
        sh 'docker build -t myapp:${BUILD_NUMBER} .'
    }
}

꼭 지켜야 할 습관이 두 가지 있다. 첫째, 파이프라인 이미지에는 절대 latest 태그를 쓰지 않는다 — 유동적인 태그는 어떤 빌드가 어디서 실행 중인지 알 수 없게 만들고, 깔끔한 롤백도 불가능하게 한다. ${BUILD_NUMBER}처럼 고유하고 추적 가능한 값을 사용한다(Jenkins 내장 변수이며, 다음 강의에서 커밋 SHA로 전환한다). 둘째, 에이전트가 Docker 데몬에 접근할 수 있어야 한다 — 일반적으로 호스트의 Docker 소켓을 Jenkins 컨테이너에 마운트하는 방식을 쓴다.

확인: 파이프라인이 녹색으로 완료된 뒤, 호스트에서 docker images를 실행해 빌드 번호가 태그로 붙은 myapp 이미지가 보이는지 확인한다.

지금까지의 전체 파이프라인


pipeline {
    agent any
    stages {
        stage('Checkout')    { steps { checkout scm } }
        stage('Build')       { steps { sh 'npm ci'; sh 'npm run build' } }
        stage('Test')        { steps { sh 'npm test' } }
        stage('Build Image') { steps { sh 'docker build -t myapp:${BUILD_NUMBER} .' } }
    }
}

위에서 아래로 읽으면 CI의 본질이 그대로 보인다: 모든 push는 체크아웃되고, 빌드되고, 테스트되며 — 이 모든 과정을 통과해야만 — 배포 가능한 이미지로 만들어진다. 이 이미지는 이제 레지스트리에 올릴 준비가 됐으며, 바로 다음 강의에서 그 과정을 다룬다.

핵심 정리

  • 스테이지 순서는 Build → Test → Build Image로 배치한다. 선언형 파이프라인에서는 0이 아닌 종료 코드가 기본적으로 스테이지를 실패 처리하므로, 이 순서만으로 테스트를 통과한 코드에서만 이미지가 나온다는 것을 보장할 수 있다.
  • 빠른 실패(fail fast) — 스테이지가 실패하면 파이프라인이 멈추고 팀에 알림이 간다.
  • 실제 자동화 테스트를 실행하고 리포트를 발행한다(예: junit). 모든 빌드에서 결과를 바로 확인할 수 있어야 한다.
  • 이미지 태그에는 빌드 번호나 커밋 SHA처럼 고유하고 추적 가능한 값을 사용한다 — 유동적인 latest는 절대 쓰지 않는다.

실습 1. 준비 실습: Jenkinsfile에 Node.js 앱에서 'npm ci'를 실행하는 'Build' 스테이지를 추가하라. 'package.json'을 일시적으로 삭제해 의도적으로 실패를 유발하고, Jenkins가 빌드를 FAILED로 표시하고 이후 스테이지로 진행하지 않는지 확인하라. 그런 다음 파일을 복원하고 스테이지가 초록색이 되는지 확인하라.

// 현재 Jenkinsfile (강의 4 버전):
pipeline {
    agent any
    stages {
        stage('Checkout') { steps { checkout scm } }
        // 아래에 Build 스테이지를 추가한다:
        stage('___') {
            steps {
                sh '___'
            }
        }
    }
}
힌트
  • 'npm ci'가 'npm: command not found'를 반환하면 Jenkins 에이전트에 Node.js가 설치되지 않은 것이다. 에이전트에 Node.js를 설치하거나 Docker 기반 에이전트를 사용한다. 'agent any'를 'agent { docker { image "node:20-alpine" } }'로 바꾸면 Node 컨테이너 안에서 스테이지를 실행할 수 있다.
  • package.json을 삭제했는데 푸시 후 빌드가 트리거되지 않는다면 GitLab 웹훅이 여전히 활성 상태인지 확인한다(Settings → Webhooks). 브랜치도 일치해야 한다.
정답 보기 
// 1단계: Build 스테이지 추가
pipeline {
    agent any
    stages {
        stage('Checkout') { steps { checkout scm } }
        stage('Build') {
            steps {
                sh 'npm ci'
            }
        }
    }
}

// 2단계: 의도적으로 실패 유발
// package.json을 _package.json으로 이름 바꾸기(또는 삭제)하고 푸시한다.
// Jenkins가 웹훅을 받아 'npm ci'를 실행하면
// package.json이 없어 0이 아닌 종료 코드로 종료된다.
// Stage View에 빨간색 'Build' 박스가 나타난다.
// Console Output이 'npm error enoent'와 'Build FAILURE'로 끝난다.
// Build 이후의 스테이지는 실행되지 않는다 — 빠른 실패 게이트가 작동한 것이다.

// 3단계: 복원
// _package.json을 다시 package.json으로 이름 바꾸고 다시 푸시한다.
// 'Checkout'과 'Build' 박스가 모두 초록색이 된다.

'sh' 스텝의 0이 아닌 종료 코드는 해당 스테이지를 즉시 실패시키고 파이프라인 전체를 중단한다 — 이후 스테이지는 실행되지 않는다. 이것이 '빠른 실패' 원칙이 실제로 작동하는 모습이다. 의존성조차 설치하지 못한 코드를 테스트하거나 패키징할 의미가 없다. 의도적으로 실패를 유발함으로써 프로덕션에서 그 게이트에 의존하기 전에 게이트가 실제로 존재한다는 것을 확인한다.

실습 2. 핵심 실습: Build 스테이지 다음에 'Test' 스테이지를 추가하라. 앱에 의도적으로 실패하는 테스트를 작성하고(예상값이 틀린 단언문), 푸시해서 Jenkins가 테스트 스테이지를 빨갛게 표시하고 'Build Image' 스테이지 앞에서 파이프라인을 중단하는지 확인한다. 그 다음 테스트를 수정하고 다시 푸시해 세 스테이지가 모두 초록색이 되는지 확인하라.

// package.json에 테스트 스크립트 추가:
// "scripts": { "test": "node test.js" }

// test.js 파일을 만들어 간단한 단언문 작성:
const assert = require('assert');
function greet(name) { return 'Hello, ' + name + '!'; }
// 하나는 통과, 하나는 의도적으로 실패하도록 작성:
assert.strictEqual(greet('World'), 'Hello, World!');   // 통과
assert.strictEqual(greet('World'), '___');             // 실패하게 만들기
console.log('All tests passed.');

// Jenkinsfile에 추가할 Test 스테이지:
stage('Test') {
    steps {
        sh '___'
    }
}
힌트
  • 실패하는 단언문이 있어도 'npm test'가 종료 코드 0으로 끝난다면, 'node test.js'가 AssertionError 종료 코드를 실제로 전파하는지 확인한다. Node는 처리되지 않은 AssertionError에서 기본적으로 코드 1로 종료한다 — try/catch로 감싸고 다시 던지지 않으면 종료 코드가 0으로 유지된다.
  • Jenkins에서 테스트 결과를 그래픽으로 보려면 Mocha 같은 JUnit XML 리포트를 생성하는 프레임워크로 교체하고 Test 스테이지에 'post { always { junit "reports/*.xml" } }'를 추가한다.
정답 보기 
// test.js — 실패 버전:
const assert = require('assert');
function greet(name) { return 'Hello, ' + name + '!'; }
assert.strictEqual(greet('World'), 'Hello, World!');
assert.strictEqual(greet('World'), 'Hello, CI/CD!')  // 실패: 실제 값은 'Hello, World!'
console.log('All tests passed.');

// package.json scripts 섹션:
"scripts": {
  "start": "node index.js",
  "test": "node test.js"
}

// Test 스테이지가 포함된 Jenkinsfile:
pipeline {
    agent any
    stages {
        stage('Checkout')    { steps { checkout scm } }
        stage('Build')       { steps { sh 'npm ci' } }
        stage('Test')        { steps { sh 'npm test' } }
        stage('Build Image') { steps { sh 'echo Image build placeholder' } }
    }
}

// 실패 테스트로 푸시: Stage View에서 Build=초록, Test=빨강, Build Image=건너뜀.
// Console Output에 AssertionError와 'Build FAILURE'가 표시된다.

// 수정: 두 번째 단언문의 기댓값을 올바른 값으로 변경한다:
assert.strictEqual(greet('World'), 'Hello, World!');
// 다시 푸시: 네 스테이지가 모두 초록색이 된다.

Test 스테이지는 CI 품질 게이트를 실제로 보여 준다. 파이프라인은 첫 번째 실패 스테이지에서 멈추므로, 테스트가 실패하면 'Build Image'가 절대 실행되지 않는다. 나쁜 이미지가 레지스트리에 들어갈 수 없다. 강의에서 정확히 설명한 시나리오다. 개발자가 인사말 문자열을 바꿨지만 단언문 업데이트를 잊은 것을 파이프라인이 즉시 잡아냈다. 수정은 테스트(또는 코드)에 있는 것이지 파이프라인을 약화시키는 것이 아니다.

실습 3. 핵심 실습: Jenkinsfile에 'Build Image' 스테이지를 추가하라. Jenkins BUILD_NUMBER 환경변수로 이미지를 태그한다. 전체 파이프라인(Checkout → Build → Test → Build Image)을 실행하고 Jenkins 호스트의 'docker images'에 빌드 번호가 태그된 이미지가 나타나는지 확인하라.

// Jenkinsfile의 Test 스테이지 다음에 추가:
stage('Build Image') {
    steps {
        sh 'docker build -t myapp:___ .'
    }
}
힌트
  • 'docker build'가 'permission denied while trying to connect to Docker daemon socket' 오류로 실패하면 Jenkins 프로세스가 Docker 소켓에 접근하지 못하는 것이다. '-v /var/run/docker.sock:/var/run/docker.sock' 플래그를 추가해 Jenkins 컨테이너를 다시 만들고 그 안에 Docker CLI를 설치한다.
  • 이미지는 빌드되지만 'docker images'가 로컬 머신에서만 보인다면, Jenkins를 로컬에서 실행 중인 경우 이미지도 같은 머신에 있는 것이다. 원격 에이전트를 사용하는 경우 해당 에이전트에 접속해 확인한다.
정답 보기 
stage('Build Image') {
    steps {
        sh 'docker build -t myapp:${BUILD_NUMBER} .'
    }
}

// 참고: 'docker build'를 실행하려면 Jenkins가 Docker 데몬에 접근할 수 있어야 한다.
// Jenkins를 Docker 컨테이너로 실행 중이라면 호스트 소켓 접근 권한을 부여한다:
// docker run -d --name jenkins \
//   -p 8080:8080 -p 50000:50000 \
//   -v jenkins_home:/var/jenkins_home \
//   -v /var/run/docker.sock:/var/run/docker.sock \
//   jenkins/jenkins:lts
// 그 다음 Jenkins 컨테이너 안에 Docker CLI를 설치한다:
// docker exec -u root jenkins apt-get update
// docker exec -u root jenkins apt-get install -y docker.io

// 파이프라인 실행 후:
// docker images | grep myapp
// 예상 출력:
// myapp   5   a1b2c3d4e5f6   2 minutes ago   145MB
// (여기서 5는 해당 실행의 BUILD_NUMBER)

BUILD_NUMBER는 빌드마다 자동으로 증가하는 Jenkins 내장 환경변수다. 'latest' 대신 이것을 태그로 사용하면 각 이미지에 고유하고 순서가 있는 식별자가 부여된다. 어느 빌드가 어떤 이미지를 만들었는지 항상 알 수 있다. 다음 강의에서는 더 강력한 추적 가능성을 위해 Git 커밋 SHA로 전환한다. '/var/run/docker.sock'을 Jenkins 컨테이너에 마운트하면 Jenkins 프로세스가 호스트의 Docker 데몬을 사용할 수 있다 — 이미지를 빌드해야 하는 CI 파이프라인에서 편리하고 흔한 설정이다.

실습 4. 응용 실습: 동료의 Jenkinsfile에 심각한 순서 문제가 있다. 아래 파이프라인을 분석하고 왜 잘못됐는지 설명한 뒤 올바르게 재작성하라.

문제 있는 Jenkinsfile:


pipeline {
    agent any
    stages {
        stage('Checkout')    { steps { checkout scm } }
        stage('Build Image') { steps { sh 'docker build -t myapp:latest .' } }
        stage('Test')        { steps { sh 'npm test' } }
        stage('Build')       { steps { sh 'npm ci' } }
    }
}

// 수정된 버전을 작성하기 전에 다음을 먼저 답한다:
// 1. 이 스테이지들이 어떤 순서로 실행되고 무엇이 잘못되는가?
// 2. 'latest' 태그가 나중에 무엇을 불가능하게 만드는가?
// 3. 올바른 스테이지 순서와 더 나은 이미지 태그를 적용한 수정된 Jenkinsfile을 작성하라.
힌트
  • 문제를 시각화하려면: 스테이지 순서를 흐름도로 그린다. 'Test가 실패하면 이미 무엇이 만들어졌는가?'를 묻는다. 잘못된 버전에서 답은 'Docker 이미지'이고 그것이 문제다.
  • 'latest' 태그 위험을 실제로 보려면: 두 개의 커밋을 빠르게 연속으로 푸시하고 두 빌드가 모두 'myapp:latest'를 태그하려 할 때 두 번째가 첫 번째를 덮어쓰는 것을 지켜본다.
정답 보기 
// 문제점:
// 1. 스테이지 순서: Checkout → Build Image → Test → Build
//    'docker build'가 'npm ci'보다 먼저, 'npm test'보다도 먼저 실행된다.
//    결과: 설치되지 않고 테스트도 거치지 않은 코드에서 이미지가 만들어진다.
//    테스트가 실패하면 'Test' 스테이지에서 파이프라인이 멈추지만
//    'Build Image' 스테이지에서 이미 나쁜 이미지가 만들어진 뒤다.
//    테스트 게이트가 이미지를 전혀 보호하지 못한다.
//
// 2. 'latest' 태그는 변경 가능하다. 동시에 또는 빠르게 연속으로 두 빌드가 실행되면
//    각각이 'latest' 태그를 덮어쓴다.
//    롤백이 불가능해진다. 실행 중인 컨테이너가 어느 빌드에서 왔는지 알 수 없다.
//
// 수정된 Jenkinsfile:
pipeline {
    agent any
    stages {
        stage('Checkout')    { steps { checkout scm } }
        stage('Build')       { steps { sh 'npm ci' } }
        stage('Test')        { steps { sh 'npm test' } }
        stage('Build Image') { steps { sh 'docker build -t myapp:${BUILD_NUMBER} .' } }
    }
}
// 이제: npm ci와 npm test가 모두 성공했을 때만 이미지가 만들어진다.
// 태그에 BUILD_NUMBER(빌드별 고유값)를 사용해 각 이미지를 구분할 수 있다.

CI 파이프라인에서 스테이지 순서는 단순한 형식이 아니다. 품질 게이트가 무엇을 보호하는지를 정의하는 것이다. 테스트 전에 이미지를 빌드하는 것은 CI의 목적 전체를 무효화한다. 테스트를 거치지 않고 잠재적으로 나쁜 코드가 레지스트리 산출물을 만들 수 있다는 뜻이기 때문이다. '빠른 실패' 보장은 위험한 산출물(이미지)이 검증 검사(테스트) 뒤에 올 때만 유효하다. 'latest' 태그 문제도 마찬가지로 실용적이다. 고유하고 불변하는 태그 없이는 롤백이 추측 게임이 되고, 다중 빌드 환경에서 동시에 진행되는 두 빌드가 서로의 태그를 덮어쓸 수 있다.

06. 이미지 레지스트리에 푸시(AWS ECR)

학습 목표

  • 컨테이너 레지스트리(ECR)가 필요한 이유를 이해한다.
  • Jenkins에서 ECR 인증을 처리하고 이미지를 푸시한다.
  • Git 커밋 SHA로 이미지를 태깅해 추적성을 확보한다.

본문

이전 강의에서 Jenkins가 빌드한 이미지는 Jenkins 에이전트 위에만 존재한다. EC2가 Jenkins 안으로 들어가 이미지를 가져올 수는 없으니, 양쪽이 모두 접근할 수 있는 공유 저장소가 필요하다. Amazon ECR(Elastic Container Registry)은 AWS의 프라이빗 Docker 레지스트리다. Jenkins가 이미지를 push하면 EC2가 나중에 pull해서 실행한다. ECR은 파이프라인의 CI 절반(빌드)과 CD 절반(배포) 사이의 중간 지점이다.

레지스트리는 이미지를 빌드하는 것과 실행하는 것을 분리한다. Jenkins는 서버에 대해 아무것도 알 필요가 없고, 서버는 빌드 도구가 필요 없다. 완성된 불변 이미지를 태그로 pull하기만 하면 된다.

1단계 — ECR 저장소 만들기

AWS 콘솔에서 ECR을 열고 앱 이름(예: myapp)으로 저장소를 만든다. ECR이 다음 형태의 저장소 URI를 제공한다.


<aws_account_id>.dkr.ecr.<region>.amazonaws.com/myapp

이 전체 URI는 태그가 아니다. 레지스트리 호스트(<aws_account_id>.dkr.ecr.<region>.amazonaws.com)와 저장소 경로/이미지명(/myapp)을 합친 것이다. 태그는 콜론(:) 뒤에 붙는 식별자다(:latest, :<커밋-sha>). Docker는 이미지명에 포함된 레지스트리 호스트를 보고 push 대상을 결정한다. 따라서 전체 참조는 호스트/이미지:태그 형태, 즉 저장소 URI 뒤에 태그를 붙여 구성한다.


<aws_account_id>.dkr.ecr.<region>.amazonaws.com/myapp:<태그>

2단계 — Jenkins에서 ECR 인증하기

프라이빗 레지스트리에 push하려면 신원을 증명해야 한다. ECR 인증은 두 단계다. AWS 자격증명으로 단기 비밀번호를 얻은 뒤 그것을 docker login에 전달한다. AWS 자격증명은 반드시 Jenkins에 저장하고(코드에 하드코딩 금지), 파이프라인에서 아래처럼 사용한다.


aws ecr get-login-password --region <region> \
  | docker login --username AWS --password-stdin \
    <aws_account_id>.dkr.ecr.<region>.amazonaws.com

get-login-password는 임시 토큰을 반환하고, --password-stdin으로 전달하면 빌드 로그에 토큰이 노출되지 않는다. ECR의 사용자 이름은 항상 문자 그대로 AWS다.

프로덕션에서는 Jenkins 인스턴스에 IAM 역할을 연결해 ECR 권한을 자동으로 부여하는 것이 좋다. 장기 키를 저장하거나 교체할 필요가 없어진다. 학습 목적이라면 정적 키도 괜찮지만 권한 범위를 최소화한다.

3단계 — 커밋 SHA로 태깅하고 푸시하기

latest 태그는 이미지에 어떤 코드가 담겨 있는지 아무것도 알려 주지 않는다. 업계 표준은 Git 커밋 SHA로 태깅하는 것이다. 실행 중인 컨테이너에서 정확히 어떤 소스 커밋으로 빌드됐는지 역추적할 수 있는 영구적인 연결 고리가 생긴다. Jenkinsfile에서 짧은 SHA를 캡처해 태그로 쓴다.


stage('Push to ECR') {
    steps {
        script {
            def registry = "<aws_account_id>.dkr.ecr.<region>.amazonaws.com"
            def commit = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
            sh """
                aws ecr get-login-password --region <region> \
                  | docker login --username AWS --password-stdin ${registry}
                docker tag myapp:${BUILD_NUMBER} ${registry}/myapp:${commit}
                docker push ${registry}/myapp:${commit}
            """
        }
    }
}

docker tag 줄을 주목하자. 로컬 이미지(myapp:${BUILD_NUMBER})를 ECR 전체 참조로 재지정한다. 레지스트리 경로를 이미지명에 붙이고(${registry}/myapp), 커밋 SHA를 콜론 뒤 태그로 붙이는 것이다. 실전에서는 같은 이미지에 태그 두 개를 함께 push하는 경우가 많다. 추적성과 롤백을 위한 불변 태그 ${commit}과, 배포 스크립트가 참조할 "이 브랜치의 최신 이미지" 포인터 역할의 유동 태그(예: 브랜치명 main)를 함께 쓰는 것이다.

4단계 — 푸시 확인하기

ECR 콘솔에서 저장소를 열면 커밋 SHA 태그와 "pushed" 타임스탬프가 달린 이미지를 확인할 수 있다. 방금 빌드·테스트한 산출물이 내구성 있는 프라이빗 레지스트리에 저장돼, 권한 있는 서버라면 어디서든 pull할 준비가 됐다는 증거다.

이제 파이프라인은 서버에서 실행하는 단계만 빼고 모든 것을 처리한다. GitLab 푸시 → 체크아웃 → 빌드 → 테스트 → 이미지 → ECR. 그 이미지를 EC2에 올려 실행하는 것이 다음 강의에서 완성하는 내용이다.

핵심 정리

  • ECR은 Jenkins가 push하고 EC2가 pull하는 공유 저장소로, 이미지 빌드와 실행을 분리한다.
  • ECR 인증은 두 단계다. aws ecr get-login-password로 임시 토큰을 얻은 뒤 docker login에 전달한다. 프로덕션에서는 정적 키보다 IAM 역할을 선호한다.
  • Docker는 이미지명에 포함된 레지스트리 호스트를 보고 push 대상을 결정한다. 따라서 이미지 태그는 <저장소-URI>:<태그> 형태로 구성한다(URI는 호스트+이미지명, 태그는 콜론 뒤).
  • Git 커밋 SHA로 태깅해 추적성을 확보하고, 필요에 따라 브랜치명 유동 태그를 추가한다.

🛠 실습

실습 1. 준비 실습: AWS 콘솔이나 AWS CLI로 AWS 계정에 ECR 저장소를 만들어라. 저장소 URI를 확인하고 형식이 '<account-id>.dkr.ecr.<region>.amazonaws.com/<repo-name>'과 일치하는지 검증하라.

# 방법 A — AWS 콘솔:
# 1. AWS 콘솔에서 ECR 서비스 열기
# 2. Create repository → 이름 'myapp' 입력 → Create
# 3. 표시된 저장소 URI 복사

# 방법 B — AWS CLI:
aws ecr create-repository \
  --repository-name myapp \
  --region <your-region>
# 출력에 'repositoryUri'가 포함된다 — 복사한다.
힌트
  • 'aws ecr create-repository'가 'Unable to locate credentials'를 반환하면 AWS CLI가 설정되지 않은 것이다. 'aws configure'를 실행하고 액세스 키 ID, 시크릿 액세스 키, 기본 리전을 입력한다.
  • 'RepositoryAlreadyExistsException'이 나타나면 저장소가 이미 있는 것이다. 'aws ecr describe-repositories'를 실행해 기존 URI를 찾아 복사한다.

# AWS CLI 예시 (리전: us-east-1, 계정: 123456789012):
aws ecr create-repository \
  --repository-name myapp \
  --region us-east-1

# 예상 JSON 출력 (관련 부분):

# 저장소 목록으로 생성 확인:

ECR 저장소 URI에는 Docker 라우팅에 필요한 모든 정보가 담겨 있다. AWS 계정 ID와 리전으로 시작하는 URI로 'docker push'를 실행하면 Docker 클라이언트가 어떤 ECR 엔드포인트로 인증하고 push해야 하는지 안다. 이미지 이름만이 아닌 전체 URI를 태그에 포함해야 하는 이유다 — Docker는 태그를 주소로 사용한다.

실습 2. 핵심 실습: Jenkins 에이전트의 Docker 클라이언트를 ECR에 인증하고, 로컬에서 빌드한 이미지를 ECR 저장소에 수동으로 push하라. ECR 콘솔에 올바른 태그로 이미지가 나타나는지 확인하라.

# 플레이스홀더를 실제 값으로 교체한다:
REGION=<your-region>
ACCOUNT=<your-account-id>
REGISTRY=${ACCOUNT}.dkr.ecr.${REGION}.amazonaws.com

# 1단계: Docker를 ECR에 인증
aws ecr get-login-password --region ${REGION} \
  | docker login --username AWS --password-stdin ${REGISTRY}

# 2단계: 강의 5에서 빌드한 이미지 태그 변경
docker tag myapp:1 ${REGISTRY}/myapp:<tag>

# 3단계: Push
docker push ${REGISTRY}/myapp:<tag>
힌트
  • 'docker login'이 'no basic auth credentials'로 실패하면 'aws ecr get-login-password' 단계가 자동으로 실패했을 수 있다(예: 리전 오류 또는 IAM 권한 부족). 먼저 'aws ecr get-login-password --region <region>'을 단독으로 실행해 토큰 문자열이 출력되는지 확인한다.
  • push가 'denied: User ... is not authorized to perform: ecr:InitiateLayerUpload'로 실패하면 IAM 사용자/역할에 'AmazonEC2ContainerRegistryPowerUser' 정책(또는 최소한 ecr:GetAuthorizationToken, ecr:BatchCheckLayerAvailability, ecr:PutImage, ecr:InitiateLayerUpload, ecr:UploadLayerPart, ecr:CompleteLayerUpload)이 필요하다.
  • push 중 'no space left on device'가 나타나면 Jenkins 에이전트의 디스크가 꽉 찬 것이다. 'docker system prune'을 실행해 사용하지 않는 이미지와 레이어를 삭제한다.

ACCOUNT=123456789012
REGISTRY=${ACCOUNT}.dkr.ecr.${REGION}.amazonaws.com

ECR 인증은 임시 토큰 방식이다. 'aws ecr get-login-password'가 AWS API에 IAM 자격증명으로 접속해 단기(12시간) 비밀번호를 반환한다. 그것을 '--password-stdin'과 함께 'docker login'에 파이프로 연결하면 터미널이나 빌드 로그에 비밀번호를 출력하지 않고 Docker를 ECR 레지스트리에 인증한다. ECR에서 사용자 이름은 항상 문자 그대로 'AWS'다. 커밋 SHA를 태그로 사용하면 영구적이고 양방향 연결이 만들어진다. 실행 중인 이미지 태그가 있으면 정확한 커밋을 찾을 수 있고, 커밋이 있으면 어떤 이미지를 만들었는지 알 수 있다.

실습 3. 핵심 실습: Jenkinsfile에 'Push to ECR' 스테이지를 추가하라. 이 스테이지에서 Git 커밋 SHA를 가져오고, ECR에 인증하고, 이전 'Build Image' 스테이지에서 빌드한 이미지에 커밋 SHA를 태그로 붙여 ECR에 push하고, 선택적으로 브랜치 이름 태그를 편의 포인터로 추가로 push한다.

// Jenkinsfile의 'Build Image' 스테이지 다음에 추가:
stage('Push to ECR') {
    steps {
        script {
            def registry = "<account-id>.dkr.ecr.<region>.amazonaws.com"
            def commit = sh(script: '___', returnStdout: true).trim()
            sh """
                aws ecr get-login-password --region ___ \\
                  | docker login --username AWS --password-stdin ${registry}
                docker tag myapp:${env.BUILD_NUMBER} ${registry}/myapp:${commit}
                docker push ${registry}/myapp:${commit}
            """
        }
    }
}
힌트
  • 파이프라인 내부에서 'git rev-parse --short HEAD'가 SHA 대신 'HEAD'를 반환하면 체크아웃 스텝이 분리된 HEAD를 사용했을 수 있다. Jenkins가 트리거 커밋에서 채우는 'env.GIT_COMMIT?.take(7)'로 교체한다.
  • 커밋 태그는 성공하지만 브랜치 태그가 'image already exists'로 실패하면 괜찮다. 브랜치 태그가 이미 같은 이미지 레이어를 가리킬 수 있다. Docker는 레이어를 다시 업로드하지 않고 태그만 업데이트한다.
  • 'withCredentials' 래퍼는 빌드 로그에서 시크릿을 마스킹한다. Jenkinsfile에 직접 환경변수로 AWS 키를 설정하지 않는다. 파이프라인 설정 UI에 그 값이 노출되기 때문이다.
정답 보기 
stage('Push to ECR') {
    steps {
        script {
            def region   = 'us-east-1'
            def registry = '123456789012.dkr.ecr.us-east-1.amazonaws.com'
            def commit   = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
            def branch   = env.GIT_BRANCH?.tokenize('/')?.last() ?: 'unknown'

            sh """
                aws ecr get-login-password --region ${region} \\
                  | docker login --username AWS --password-stdin ${registry}

                # 불변 태그 — 커밋 SHA
                docker tag myapp:${env.BUILD_NUMBER} ${registry}/myapp:${commit}
                docker push ${registry}/myapp:${commit}

                # 편의 태그 — 브랜치 이름 (선택)
                docker tag myapp:${env.BUILD_NUMBER} ${registry}/myapp:${branch}
                docker push ${registry}/myapp:${branch}
            """
        }
    }
}

// Jenkins에 AWS 자격증명 저장:
// Manage Jenkins → Credentials → Global → Add Credentials
// Kind: AWS Credentials  ID: aws-ecr-creds
// 그런 다음 스테이지를 다음으로 감싼다:
// withCredentials([[\$class: 'AmazonWebServicesCredentialsBinding',
//                   credentialsId: 'aws-ecr-creds']]) { ... }
// 또는 Jenkins 호스트에 IAM 역할 연결 (프로덕션에서 권장).

선언적 Jenkinsfile 안의 'script' 블록에서 Groovy 스크립팅을 사용할 수 있다. 'sh(script: ..., returnStdout: true).trim()'은 셸 명령의 표준 출력을 캡처하고 줄 끝 공백을 제거해 Groovy 문자열 변수로 만든다. 이중 태그 전략(커밋 SHA + 브랜치 이름)은 두 가지 요구를 충족한다. 커밋 SHA 태그는 불변으로 정확한 롤백 대상을 제공하고, 브랜치 태그는 해당 브랜치의 최신 이미지를 가리키는 유동 포인터로 배포 스크립트에서 편리하게 사용할 수 있다.

실습 4. 응용 실습: 버그 리포트를 받았다. '프로덕션에서 latest 태그로 실행 중인 컨테이너가 충돌했는데 어떤 코드 버전인지 알 수 없다.' 이 상황이 'latest' 태그로 생긴 이유를 직접 작성으로 설명하고(Jenkinsfile 주석 블록 또는 메모), 대신 커밋 SHA 태깅 전략을 사용한 경우 세 번의 빌드 후 'docker images'와 ECR 저장소가 어떻게 보일지 예시로 보여라.

힌트
  • 실행 중인 컨테이너에서 어떤 코드가 빌드됐는지 확인하려면(SHA 태그 사용 시): 'docker inspect <container-name>'을 실행하고 'Image' 필드를 확인한다. 태그의 SHA는 'git log <sha>'로 검사할 수 있는 git 커밋에 직접 매핑된다.
  • ECR에서 '이미지 태그 불변성(image tag immutability)'을 활성화하면 기존 태그를 덮어쓰지 못하게 막는다. 프로덕션에서 이 설정을 활성화해 레지스트리 수준에서 태깅 원칙을 강제한다.

//
// a) 'latest'는 변경 가능한 태그다 — 새 빌드가 생길 때마다 덮어쓴다.
//    컨테이너가 충돌하면 'docker inspect <container>'를 실행해도
//    'Image: myapp:latest'만 보인다. 언제 빌드됐는지,

//

//    시간이 걸리고 동일한 나쁜 코드가 다시 들어올 수 있다.
//    커밋 SHA 태그를 사용하면 과거 이미지가 ECR에 각자의 불변 태그로
//    존재한다. 롤백은 'docker pull myapp:a1b2c3d' — 즉시 완료된다.
//
// c) 커밋 SHA 태깅은 영구적이고 양방향 인덱스를 만든다:
//    코드 커밋 ↔ 이미지 태그.

//    커밋이 있으면 그 이미지와 그것을 실행하는 서버를 찾을 수 있다.

//

// docker run ... myapp:a1b2c3d

이 실습은 강의에서 소개한 태깅 원칙을 강화한다. 실행 중인 이미지의 소스 코드를 식별할 수 없다는 것은 'latest'를 파이프라인에서 사용할 때 반복적으로 발생하는 실제 프로덕션 문제다. 커밋 SHA 태그는 이미지 레지스트리를 감사 추적으로 만든다. 모든 배포가 추적 가능하고, 모든 롤백이 즉각적이며(재빌드 불필요), 어떤 버전도 조용히 덮어쓰이지 않는다. 불변성 보장은 보안 스캔에도 유용하다. 07. EC2에 자동 배포 — SSH + docker compose

학습 목표

  • Jenkins에서 EC2로 안전하게 SSH 접속하는 방법을 익힌다.
  • 최신 이미지를 pull해 docker compose로 교체 배포하는 스테이지를 작성한다.
  • 푸시 한 번으로 EC2까지 배포되는 파이프라인을 완성한다.

본문

이미지는 이미 ECR에 들어 있다. 이번 강의에서는 그것을 EC2에서 실행시키는 마지막 스테이지를 작성한다. Jenkins가 SSH 세션을 열고 새 이미지를 pull한 뒤 컨테이너를 교체한다 — 매 빌드마다 자동으로.

1단계 — EC2 서버 준비

무엇을, 왜: Jenkins가 배포하려면 서버가 이미지를 pull하고 실행할 준비가 되어 있어야 한다. 한 번만 해두면 되는 준비가 세 가지다. Docker와 Compose 플러그인 설치, ECR 읽기 권한을 부여하는 IAM 역할을 인스턴스에 연결(서버에 키를 저장하지 않아도 됨), 그리고 앱 실행 방법을 기술한 docker-compose.yml.

서버에 둘 compose 파일(/home/ec2-user/docker-compose.yml 등):


services:
  web:
    image: ${IMAGE}
    ports:
      - "80:8080"
    restart: always

${IMAGE}는 변수다. Jenkins가 배포 시점에 정확한 ECR 태그를 채워 넣으므로, 같은 파일이 파이프라인이 방금 빌드한 버전을 항상 실행한다.

확인: 인스턴스에서 docker --version && docker compose version을 실행하고, aws ecr get-login-password --region <region>이 토큰을 반환하는지 확인한다.

2단계 — EC2 호스트 키 등록

무엇을, 왜: SSH는 호스트 키로 서버의 신원을 검증한다. 이 키는 known_hosts에 기록된다. 미리 등록해 두면 SSH가 중간자(MITM) — 즉 서버인 척 위장한 공격자 — 를 탐지할 수 있다. Jenkins가 실행되는 사용자로 Jenkins 호스트에서 한 번만 실행한다:


# 한 번만 실행한 뒤 신뢰하기 전에 EC2 콘솔에서 지문을 검증한다
ssh-keyscan -H <EC2_HOST> >> ~/.ssh/known_hosts

많은 튜토리얼에서 -o StrictHostKeyChecking=no를 쓴다. 호스트 키 검증을 완전히 비활성화해 편하긴 하지만, MITM 보호 자체를 포기하는 것이다. known_hosts를 미리 채우는 방법을 우선으로 쓰고, 임시 학습 환경에서만 완화를 고려한다.

3단계 — SSH 키를 Jenkins에 저장

무엇을, 왜: Jenkins는 인스턴스의 프라이빗 키로 접속한다. 이 키를 SSH 자격증명(예: ID ec2-ssh-key)으로 저장하고 SSH Agent 플러그인을 사용해 배포 블록 실행 동안에만 빌려 쓴다 — 키를 Jenkinsfile에 직접 붙여 넣거나 커밋하는 것은 절대 안 된다.

SSH 프라이빗 키는 서버 비밀번호와 같다. 이 잡에만 범위를 한정하고 저장소와 빌드 로그에 절대 닿지 않게 한다. 배포 키가 유출되면 서버가 유출된 것이다.

4단계 — 배포 스테이지 작성

무엇을, 왜: 호스트 키가 신뢰되고 SSH 키가 자격증명에 저장됐다면, 이 스테이지가 검증된 SSH 세션을 열고 배포를 구성하는 네 개의 명령을 실행한다.


stage('Deploy to EC2') {
    steps {
        sshagent(['ec2-ssh-key']) {
            sh """
                ssh ec2-user@<EC2_HOST> '
                    aws ecr get-login-password --region <region> \
                      | docker login --username AWS --password-stdin <registry>
                    export IMAGE=<registry>/myapp:${commit}
                    docker compose -f /home/ec2-user/docker-compose.yml pull
                    docker compose -f /home/ec2-user/docker-compose.yml up -d
                '
            """
        }
    }
}

SSH 세션 안에서 실행되는 명령:

1. docker login — EC2가 IAM 역할 토큰으로 ECR에 인증한다.

2. export IMAGE=... — compose 파일이 실행할 커밋 태그 이미지의 정확한 주소를 설정한다.

3. docker compose pull — 새 이미지를 다운로드한다.

4. docker compose up -d — 교체 명령이다. Compose가 이미지가 바뀐 것을 감지하고 이전 컨테이너를 중지한 뒤 새 컨테이너를 백그라운드로 실행한다. 선언적 방식이라 원하는 상태를 기술하면 Compose가 알아서 맞춘다.

아래 다이어그램은 Jenkins가 SSH 세션을 여는 순간부터 새 컨테이너가 트래픽을 받기까지의 흐름을 보여 준다.

5단계 — 완성된 파이프라인

push 스테이지 뒤에 배포 스테이지를 추가하면 Jenkinsfile이 전체 여정을 담는다:


pipeline {
    agent any
    stages {
        stage('Checkout')      { steps { checkout scm } }
        stage('Build & Test')  { steps { sh 'npm ci'; sh 'npm test' } }
        stage('Build Image')   { /* docker build, 커밋 SHA로 태그 */ }
        stage('Push to ECR')   { /* 로그인 + docker push */ }
        stage('Deploy to EC2') { /* sshagent → compose pull + up -d */ }
    }
}

전체 흐름 확인: 코드를 수정하고 커밋한 뒤 GitLab에 push한다. 웹훅이 실행되고, Jenkins가 모든 스테이지를 순서대로 진행하고, EC2가 새 이미지를 pull해 서비스한다. 서버 URL을 열면 변경 사항이 live 상태다 — 터미널을 한 번도 열지 않고.

다음 강의에서는 안전망을 추가한다. 헬스체크, 롤백, 시크릿 처리를 갖춰 이 파이프라인을 프로덕션에 적합한 수준으로 만든다.

핵심 정리

  • 배포 스테이지는 SSH로 EC2에서 명령을 실행한다. 프라이빗 키는 Jenkins SSH 자격증명(SSH Agent 플러그인)으로 저장하고 절대 저장소에 두지 않는다.
  • known_hosts에 EC2 호스트 키를 등록해(ssh-keyscan) SSH가 MITM을 탐지할 수 있게 한다. StrictHostKeyChecking=no를 기본으로 쓰지 않는다.
  • EC2에 Docker, ECR pull을 위한 IAM 역할, image 변수를 사용하는 docker-compose.yml을 준비한다.
  • 교체는 두 명령으로 이뤄진다 — docker compose pull 후 docker compose up -d — Compose가 선언적으로 이전 컨테이너를 새 컨테이너로 교체한다.
  • 이 스테이지가 추가되면 단일 git push가 수동 단계 없이 live EC2 배포까지 자동으로 이어진다.

세 가지 준비는 서로 연결되어 있다. Docker는 컨테이너 실행에 필요하고, IAM 역할은 장기 키 저장 없이 서버가 ECR에서 이미지를 pull할 수 있게 하며, '${IMAGE}'를 사용하는 docker-compose.yml은 Jenkins가 배포 시 IMAGE 환경변수를 설정하기만 하면 되는 안정적인 단일 파일을 제공한다. 하드코딩된 태그 대신 '${IMAGE}'를 쓰면 compose 파일을 절대 수정할 필요가 없다. Jenkins가 변수만 채운다.

실습 2. 핵심 실습: EC2 SSH 프라이빗 키를 Jenkins 자격증명(종류: SSH Username with private key)으로 저장하고, EC2 호스트 키를 Jenkins의 known_hosts 파일에 등록하라. 그런 다음 Jenkins 에이전트에서 EC2 인스턴스로 테스트 SSH 명령을 실행해 연결을 검증하라.

# 1단계 — Jenkins에 EC2 SSH 키 저장:
# Manage Jenkins → Credentials → Global → Add Credentials
# Kind: SSH Username with private key
# ID: ec2-ssh-key
# Username: ec2-user
# Private Key: Enter directly → <your-key.pem> 내용 붙여넣기

# 2단계 — Jenkins의 known_hosts에 EC2 호스트 키 등록:
# Jenkins 컨테이너 안에서 실행:
docker exec -u jenkins jenkins ssh-keyscan -H <EC2_PUBLIC_IP>
# 출력 줄을 복사하고 known_hosts에 추가:
docker exec -u jenkins jenkins \
  bash -c 'mkdir -p ~/.ssh && ssh-keyscan -H <EC2_PUBLIC_IP> >> ~/.ssh/known_hosts'

# 3단계 — 테스트 파이프라인 스테이지로 검증:
# (아래 인라인 스크립트로 임시 파이프라인 잡 생성)
stage('Test SSH') {
    steps {
        sshagent(['ec2-ssh-key']) {
            sh 'ssh ec2-user@<EC2_PUBLIC_IP> "echo SSH connection successful && uname -n"'
        }
    }
}
힌트
  • 'sshagent' 스텝을 찾을 수 없으면 Manage Jenkins → Plugins에서 SSH Agent 플러그인을 설치한다.
  • SSH 연결이 'Host key verification failed'로 실패하면 known_hosts 파일이 채워지지 않았거나 IP가 바뀐 것이다. Jenkins 컨테이너 안에서 ssh-keyscan 명령을 다시 실행하고 출력이 추가됐는지 확인한다(cat ~/.ssh/known_hosts).
  • 'ECDSA host key for <IP> has changed'가 나타나면 EC2 인스턴스가 중지 후 재시작됐을 수 있다(AWS가 시작 시 새 호스트 키를 부여할 수 있음). Jenkins 컨테이너 안에서 이전 항목을 제거하고('ssh-keygen -R <EC2_IP>') ssh-keyscan을 다시 실행한다.

Jenkins 자격증명으로 키를 저장하면(Jenkinsfile이나 저장소 파일이 아닌) Jenkins가 암호화하고 빌드 로그에 절대 노출되지 않는다. known_hosts 등록은 대부분의 튜토리얼이 생략하는 보안 단계다. 없으면 Jenkins가 연결을 거부하거나(StrictHostKeyChecking이 켜져 있을 때) 그 주소의 어떤 서버든 자동으로 수락한다(꺼져 있을 때). 지문을 수동으로 등록하면 Jenkins가 모든 자동 연결에서 서버 신원을 검증할 수 있다 — 중간자 공격 방어다.

실습 3. 핵심 실습: Jenkinsfile에 'Deploy to EC2' 스테이지를 추가하라. 이 스테이지에서 EC2에 SSH로 접속해 ECR 로그인, IMAGE 변수 설정, docker compose pull, docker compose up -d의 네 명령을 실행한다. 코드 변경을 push하고 새 버전이 EC2 서버의 공인 IP 포트 80에서 live 상태인지 확인하라.

// Jenkinsfile의 'Push to ECR' 다음에 추가:
stage('Deploy to EC2') {
    steps {
        sshagent(['ec2-ssh-key']) {
            sh """
                ssh ec2-user@<EC2_HOST> '
                    aws ecr get-login-password --region ___ \\
                      | docker login --username AWS --password-stdin ___
                    export IMAGE=___/myapp:${commit}
                    docker compose -f /home/ec2-user/docker-compose.yml pull
                    docker compose -f /home/ec2-user/docker-compose.yml up -d
                '
            """
        }
    }
}
힌트
  • 'docker compose pull'이 'unauthorized'로 실패하면 EC2의 IAM 역할에 ECR 읽기 권한이 없거나 'aws ecr get-login-password' 명령이 실패한 것이다. EC2에 수동 SSH로 접속해 두 명령을 직접 실행해 디버깅한다.
  • 'docker compose up -d'가 성공하지만 포트 80이 응답하지 않으면 EC2 보안 그룹이 0.0.0.0/0에서 포트 80 인바운드 TCP를 허용하는지 확인한다.
  • Jenkins의 SSH 명령이 'Pseudo-terminal will not be allocated because stdin is not a terminal'로 실패하면 ssh 플래그에 '-tt'를 추가해야 할 수 있다. 'ssh -tt ec2-user@...' — 대부분의 명령은 PTY 없이도 잘 작동하지만.
정답 보기 
// Deploy 스테이지가 포함된 완성된 Jenkinsfile:
pipeline {
    agent any
    stages {
        stage('Checkout')    { steps { checkout scm } }
        stage('Build')       { steps { sh 'npm ci' } }

            }
        }
        stage('Deploy to EC2') {

// 검증:
// 1. 코드 변경 push (예: index.js의 응답 텍스트 변경)
// 2. Jenkins에서 다섯 스테이지가 모두 초록색이 되는 것 확인
// 3. 브라우저에서 http://<EC2_PUBLIC_IP>:80 열기
//    — 새 응답 텍스트가 보여야 함
// 4. EC2에서 실행 중인 컨테이너 확인: docker ps

배포 스테이지는 하나의 SSH 세션 안에서 네 명령을 실행한다. ECR 로그인은 EC2 서버가 프라이빗 이미지를 pull할 권한을 부여한다. IMAGE 환경변수 설정은 docker-compose.yml의 변수 치환을 채운다. 'docker compose pull'은 ECR에서 새 이미지 레이어를 다운로드한다. 'docker compose up -d'는 선언적이다. Compose가 실행 중인 상태(이전 이미지)와 원하는 상태(새 IMAGE 값)를 비교하고 변경된 것만 교체한다. 이전 컨테이너를 수동으로 중지·삭제·재시작하는 순서를 지정할 필요가 없다. '-d'는 Compose가 교체를 시작한 뒤 SSH 세션에 즉시 제어권을 반환한다.

실습 4. 응용 실습: 의미 있는 코드 변경을 만들어(예: HTTP 응답에 현재 Git 커밋 SHA 포함) GitLab에 push하고 파이프라인 전체를 추적한다. 배포된 정확한 이미지 태그를 기록하고 'git show <tag>'로 해당 커밋을 확인할 수 있는지 증명하라.

// Dockerfile 추가:

힌트
  • 컨테이너에서 'process.env.COMMIT_SHA'가 'undefined'라면 '--build-arg'가 'docker build'에 전달됐는지, Dockerfile에 ARG와 ENV 두 줄이 모두 COMMIT_SHA에 대해 있는지 확인한다. ARG만으로는 런타임까지 값이 유지되지 않는다.
  • 브라우저가 배포 후에도 이전 응답을 보여준다면 브라우저가 캐싱하는 것일 수 있다. 강력 새로고침(Ctrl+Shift+R)을 시도하거나 커맨드라인에서 'curl http://<EC2_IP>/'로 테스트한다.

WORKDIR /app
COPY package.json ./
RUN npm install
COPY . .

// 업데이트된 index.js:
const http = require('http');
const server = http.createServer((req, res) => {
  const sha = process.env.COMMIT_SHA || 'unknown';
  res.writeHead(200);
  res.end(`Hello from container! Build: ${sha}`);

server.listen(8080, () => console.log('Listening on port 8080'));

// Jenkinsfile의 'Build Image' 스텝 업데이트:

// 파이프라인 완료 후:
// 1. Jenkins 콘솔에 표시된 커밋 SHA 메모 (예: 'abc1234')
// 2. 브라우저에서 http://<EC2_PUBLIC_IP>/ 열기:
//    응답: 'Hello from container! Build: abc1234'
// 3. 연결 확인:
//    git show abc1234
//    index.js 변경이 담긴 커밋이 표시됨 — SHA가 정확히 일치
// 4. ECR 태그 확인:
//    aws ecr describe-images --repository-name myapp \
//      --query 'imageDetails[].imageTags'
//    'abc1234'가 포함되어야 함

이 실습은 추적 가능성 루프를 실제로 완성한다. 실행 중인 컨테이너의 HTTP 응답에 커밋 SHA가 표시되고, ECR 태그가 같은 SHA를 사용하고, 'git show <SHA>'가 정확한 소스 코드로 매핑된다. 커밋 SHA 태깅 전략의 가치가 구체적으로 드러난다. 엔드포인트를 누르는 사용자가 어떤 코드 버전이 서비스하는지 볼 수 있고, 운영자는 소스와 빌드를 즉시 찾을 수 있다. 프로덕션에서는 이 SHA가 주 응답 대신 '/version' 엔드포인트나 로그로 노출되는 것이 일반적이지만 메커니즘은 동일하다.

08. 무중단 전환·롤백·시크릿/환경변수 관리

학습 목표

  • 헬스체크로 배포 성공 여부를 확인하고, 실패 시 이전 이미지로 롤백한다.
  • 시크릿과 환경변수를 코드에서 분리해 안전하게 주입하는 방법을 익힌다.
  • 완성된 파이프라인의 전체 흐름을 파악한다.

본문

지금까지의 파이프라인은 새 이미지를 pull하고 컨테이너를 교체한 뒤 배포가 성공했다고 가정한다. 그런데 새 버전이 시작 시 바로 죽어버리면, 멀쩡하게 돌던 앱을 망가진 버전으로 교체하고도 아무도 모르는 상황이 된다. 이번 강의에서는 이 빈틈을 메운다. 헬스체크로 결과를 검증하고, 실패 시 자동으로 롤백하며, 시크릿을 코드 밖에서 관리하는 방법을 다룬다.

1단계 — 헬스체크 추가

헬스체크는 새로 뜬 컨테이너에 "지금 실제로 요청을 받을 수 있어?"라고 묻는 작은 탐침이다. 고정 sleep은 쓰지 않는다. 앱 시작 시간과 경쟁하는 구조라 탐침을 너무 일찍 보내거나, 불필요하게 오래 기다리게 된다. 대신 Docker Compose에게 폴링을 맡긴다. 서비스에 healthcheck를 정의하면 된다.


services:
  web:
    image: ${IMAGE}
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 10s
      timeout: 3s
      retries: 3

그리고 --wait 옵션으로 배포한다. 이 플래그는 모든 서비스가 healthy 상태가 될 때까지 블로킹했다가 0으로 종료한다. 어느 서비스라도 끝내 healthy 상태가 되지 않으면 0이 아닌 값으로 종료한다. 이 명령 한 줄이 곧 검증 단계다.


docker compose up -d --wait

--wait는 서비스에 healthcheck가 정의된 경우에만 제대로 작동한다. healthcheck가 없으면 Compose는 컨테이너가 시작되기만 기다릴 뿐, healthy 상태가 되는지는 확인하지 않는다. docker compose up -d --wait의 종료 코드가 배포 성공·실패의 판단 기준이다.

2단계 — 실패 시 롤백

헬스체크가 실패하면, 즉 docker compose up -d --wait이 0이 아닌 값으로 종료하면 이전 정상 이미지를 다시 배포한다. 강의 6에서 모든 이미지를 커밋 SHA로 태그해 두었기 때문에, 과거의 모든 버전이 ECR에 각자의 태그로 남아 있어 언제든 재배포할 수 있다.

까다로운 점은 파이프라인이 마지막으로 검증된 태그를 기억해야 한다는 것이다. 방법은 간단하다. 검증이 완료된 배포에서만 해당 태그를 서버 파일에 기록하고, 실패 시 그 파일을 읽어 재배포한다. 파일은 성공한 배포 이후에만 갱신되므로, 항상 실제로 작동했던 버전을 가리킨다.


stage('Deploy & Verify') {
    environment { COMMIT = "${env.GIT_COMMIT}" }   // 새 이미지 태그
    steps {
        sshagent(['ec2-ssh-key']) {
            sh '''
                ssh ec2-user@<EC2_HOST> "
                    export IMAGE=<registry>/myapp:${COMMIT}
                    # --wait가 배포와 검증을 함께 처리한다. 0이 아닌 종료 코드 => 스테이지 실패
                    docker compose up -d --wait
                    # healthy 검증이 완료됐을 때만 실행됨:
                    echo ${COMMIT} > /home/ec2-user/last_good_tag
                "
            '''
        }
    }
    post {
        failure {
            sshagent(['ec2-ssh-key']) {
                sh '''
                    ssh ec2-user@<EC2_HOST> "
                        PREV=\\$(cat /home/ec2-user/last_good_tag 2>/dev/null)
                        if [ -n \\"\\$PREV\\" ]; then
                            export IMAGE=<registry>/myapp:\\$PREV
                            docker compose up -d --wait
                        else
                            echo 'No known-good tag recorded; cannot auto-roll back' >&2
                            exit 1
                        fi
                    "
                '''
            }
        }
    }
}

무중단 배포가 필요하다면 블루-그린(두 환경을 운영하다가 트래픽을 전환하는 방식)이나 카나리(트래픽 일부를 먼저 새 버전으로 보내는 방식)를 쓰기도 한다. EC2 단일 호스트라면 헬스체크+롤백만으로도 대부분의 위험을 충분히 줄일 수 있다.

3단계 — 시크릿을 코드 밖에서 관리하기

시크릿을 저장소에 절대 커밋하지 않는다. Git에 들어간 내용은 영구적이며, 삭제해도 이력에 남는다. 시크릿은 코드와 분리해 런타임에 주입해야 한다.

배포 시 시크릿(Jenkins) — 자격증명 저장소에 보관하고 withCredentials로 불러온다. 빌드 로그에서 자동으로 마스킹된다.


withCredentials([string(credentialsId: 'db-password', variable: 'DB_PASS')]) {
    sh 'deploy-using $DB_PASS'
}

런타임 시크릿(서버) — 서버에만 존재하고 git에서 무시되는 .env 파일로 컨테이너에 전달한다.


services:
  web:
    image: ${IMAGE}
    env_file: /home/ec2-user/app.env

각 시크릿은 한 곳에만 둔다. Jenkins는 배포에 필요한 시크릿을, 서버는 앱 실행에 필요한 시크릿을 각각 보관한다. 도구가 허용하는 모든 곳에서 보호·마스킹 설정을 켜 두자.

완성된 파이프라인

1. GitLab에 코드를 push한다.

2. Webhook이 Jenkins를 트리거한다.

3. Jenkins가 코드를 체크아웃하고 빌드·테스트한다. 오류가 있으면 즉시 멈춘다.

4. 테스트가 통과하면 커밋 SHA로 태그된 Docker 이미지를 빌드한다.

5. ECR에 인증한 뒤 이미지를 push한다.

6. EC2에 SSH로 접속해 docker compose로 컨테이너를 교체한다.

7. 헬스체크가 성공을 확인하고, 실패 시 파이프라인이 자동으로 롤백한다. 시크릿은 코드 밖에서 안전하게 관리된다.

이 파이프라인을 내 앱에 적용하려면 Dockerfile의 베이스 이미지와 시작 명령, 실제 테스트 명령, ECR 저장소와 EC2 호스트, 그리고 앱의 시크릿만 바꾸면 된다. Dockerfile, Jenkinsfile, 배포 로직은 그대로 가져다 쓸 수 있다.

핵심 정리

  • 배포는 헬스체크가 컨테이너가 실제로 서비스 중임을 확인해야 끝난 것이다. healthcheck를 정의하고 docker compose up -d --wait으로 배포하라. 고정 sleep은 절대 쓰지 않는다.
  • 롤백이 가능한 이유는 모든 이미지가 SHA 태그로 ECR에 보존되기 때문이다. 검증이 완료된 배포 이후에만 서버 파일에 태그를 기록하고, post { failure { ... } } 블록이 그 태그를 읽어 재배포한다.
  • 시크릿을 절대 커밋하지 않는다. 배포 시 시크릿은 Jenkins withCredentials로, 런타임 시크릿은 git에서 무시되는 .env 파일로 주입한다. 각 시크릿은 한 곳에만 보관한다.
  • 완성된 파이프라인은 git push부터 빌드, 테스트, ECR push, 검증된 EC2 배포까지 어떤 앱이든 자동으로 처리한다.

# interval * retries = 30초 후 0이 아닌 값으로 종료:
# Error response from daemon: ... health check failed

'--wait'는 'docker compose up -d'를 fire-and-forget 명령에서 검증된 배포로 바꾼다. 설정된 간격으로 헬스체크 프로브를 최대 retries 횟수까지 폴링하고, 컨테이너가 그 제한 안에 'healthy' 상태로 전환되면 종료 코드 0으로만 종료한다. 고정 'sleep 10'으로는 이것을 할 수 없다. 앱 시작 시간과 경쟁하며 유용한 종료 코드를 제공하지 않는다. 헬스체크 테스트 명령('curl -f http://localhost:8080/health')이 성공해야(종료 코드 0) 컨테이너가 healthy로 간주된다. '-f'는 HTTP 오류 시 curl이 0이 아닌 값으로 종료하게 만든다.

실습 2. 핵심 실습: Jenkinsfile의 'Deploy to EC2' 스테이지를 헬스체크+롤백 패턴으로 업데이트하라. 이 스테이지에서 '--wait'으로 배포하고, 성공 시 태그를 'last_good_tag'에 기록하며, '--wait'이 0이 아닌 값으로 종료되면 'post { failure { ... } }' 블록에서 저장된 태그로 롤백한다.

// 기존 'Deploy to EC2' 스테이지를 교체:
stage('Deploy & Verify') {
    steps {
        script {
            def registry = '<registry>'
            def commit   = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
            sshagent(['ec2-ssh-key']) {
                sh """
                    ssh ec2-user@<EC2_HOST> '
                        export IMAGE=${registry}/myapp:${commit}
                        docker compose -f /home/ec2-user/docker-compose.yml pull
                        docker compose -f /home/ec2-user/docker-compose.yml up -d ___
                        echo ___ > /home/ec2-user/last_good_tag
                    '
                """
            }
        }
    }
    post {
        failure {
            // 롤백 블록
        }
    }
}
힌트
  • 배포가 실패해도 'post { failure { ... } }' 블록이 실행되지 않으면, 'docker compose up -d --wait' 실패가 SSH 호출을 통해 종료 코드를 전파하는지 확인한다. 원격 셸 스크립트 맨 앞에 'set -e'를 추가하면 어떤 명령 실패든 스크립트를 종료하게 만든다.
  • 'last_good_tag'가 첫 실행 시 비어 있으면(이전에 성공한 배포가 없는 경우) 롤백이 올바르게 'No known-good tag found' 메시지를 출력하고 종료 코드 1로 종료한다. 이것은 버그가 아닌 의도된 동작이다.
  • 롤백을 의도적으로 테스트하려면: 헬스체크가 실패하게 만드는 이미지를 push하고(예: CMD를 'exit 1'로 설정한 Dockerfile), '--wait'가 타임아웃되고 스테이지가 빨간색이 되고 post 블록이 이전 태그를 배포하는 것을 확인한다.
정답 보기 
stage('Deploy & Verify') {
    steps {
        script {
            def region   = 'us-east-1'
            def registry = '123456789012.dkr.ecr.us-east-1.amazonaws.com'
            def commit   = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
            sshagent(['ec2-ssh-key']) {
                sh """
                    ssh ec2-user@<EC2_PUBLIC_IP> '
                        aws ecr get-login-password --region ${region} \\
                          | docker login --username AWS --password-stdin ${registry}
                        export IMAGE=${registry}/myapp:${commit}
                        docker compose -f /home/ec2-user/docker-compose.yml pull
                        docker compose -f /home/ec2-user/docker-compose.yml up -d --wait
                        echo ${commit} > /home/ec2-user/last_good_tag
                    '
                """
            }
        }
    }
    post {
        failure {
            script {
                def region   = 'us-east-1'
                def registry = '123456789012.dkr.ecr.us-east-1.amazonaws.com'
                sshagent(['ec2-ssh-key']) {
                    sh """
                        ssh ec2-user@<EC2_PUBLIC_IP> '
                            PREV=\\$(cat /home/ec2-user/last_good_tag 2>/dev/null)
                            if [ -n "\\$PREV" ]; then
                                export IMAGE=${registry}/myapp:\\$PREV
                                docker compose -f /home/ec2-user/docker-compose.yml up -d --wait
                                echo "Rolled back to \\$PREV"
                            else
                                echo "No known-good tag found; cannot auto-roll back" >&2
                                exit 1
                            fi
                        '
                    """
                }
            }
        }
    }
}

자기 교정 메커니즘은 두 가지 설계 원칙에서 나온다. (1) 'echo ${commit} > last_good_tag'는 '--wait'가 종료 코드 0으로 반환된 후에만 실행되므로 파일에는 항상 검증된 healthy 태그가 있다. 실패한 배포는 절대 덮어쓰지 않는다. (2) 'post { failure { ... } }'는 스테이지가 실패할 때만 실행되어 롤백 블록이 마지막으로 검증된 태그를 읽는다. 별도의 외부 상태 관리 없이 안전망이 만들어진다. EC2 서버 자체가 롤백 기준을 평문 텍스트 파일로 저장한다. 백슬래시 이스케이프('\\$PREV')는 외부 Groovy 문자열 치환이 SSH 세션이 보기 전에 셸 변수를 소비하지 않도록 필요하다.

실습 3. 핵심 실습: 파이프라인의 안전한 시크릿 관리를 구현하라. 모의 데이터베이스 비밀번호를 Jenkins 자격증명으로 저장하고 런타임 애플리케이션 시크릿을 EC2 서버의 '.env' 파일에 둔다. Jenkins 빌드 로그에 시크릿 값이 나타나지 않는지, 실행 중인 컨테이너가 런타임 시크릿을 환경변수로 접근할 수 있는지 확인하라.

// Part 1 — Jenkins 자격증명 (배포 시 시크릿):
// Manage Jenkins → Credentials → Global → Add Credentials
// Kind: Secret text  ID: db-password  Secret: s3cr3tP@ssw0rd

// Part 2 — Jenkinsfile에서 withCredentials로 사용:
withCredentials([string(credentialsId: '___', variable: '___')]) {
    sh 'echo "Would connect to DB with password: $DB_PASS"'
    // 실제 시나리오: sh 'deploy-script --db-password $DB_PASS'
}

// Part 3 — EC2 서버의 런타임 시크릿 (Jenkinsfile이나 저장소가 아닌 곳에):
// EC2에 SSH로 접속해 .env 파일 생성:
cat > /home/ec2-user/app.env << 'EOF'
DB_PASSWORD=___
API_KEY=___
EOF

// Part 4 — docker-compose.yml에서 참조:
services:
  web:
    image: ${IMAGE}
    env_file: /home/ec2-user/app.env
    ...
힌트
  • 'withCredentials'를 사용해도 로그에 시크릿이 표시되면 'sh' 명령의 변수 이름이 'variable' 필드와 정확히 일치하는지 확인한다. 대소문자 구분이 있다.
  • 컨테이너가 .env 파일의 환경변수를 볼 수 없으면 docker-compose.yml의 'env_file'이 절대 경로를 가리키는지 확인한다(docker compose를 실행하는 위치에서의 상대 경로가 아닌).
  • 파이프라인 명령에 리터럴 시크릿을 포함한 'docker run -e DB_PASSWORD=<value>'를 절대 사용하지 않는다. 'docker inspect' 출력과 빌드 로그에 노출된다. 'env_file' 방식은 값을 파일 안에만 유지해 어떤 명령 문자열에도 나타나지 않는다.
정답 보기 
// Part 1 & 2 — Jenkinsfile에서 Jenkins 자격증명 사용:
stage('Example Secret Usage') {
    steps {
        withCredentials([string(credentialsId: 'db-password', variable: 'DB_PASS')]) {
            sh 'echo "DB_PASS is set: ${#DB_PASS} chars"'  // 길이를 출력, 값이 아님
            // Jenkins가 자동으로 DB_PASS를 마스킹 — 실수로 echo해도 '****'로 표시됨
        }
    }
}

// DB_PASS is set: 14 chars

cat > /home/ec2-user/docker-compose.yml << 'EOF'

      - "80:8080"

// 출력: s3cr3tP@ssw0rd
// (이 명령은 EC2 서버에서 실행 — 시크릿은 거기서만 볼 수 있음)

2단계 시크릿 모델은 역할을 명확히 분리한다. Jenkins 자격증명은 파이프라인이 배포 시에 필요한 시크릿(예: 배포 중 API 인증)을 저장하고, EC2의 '.env' 파일은 실행 중인 앱이 런타임에 필요한 시크릿(데이터베이스 비밀번호, API 키)을 저장한다. Jenkins의 'withCredentials' 래퍼는 모든 출력을 스캔하고 시크릿 값을 '**'로 교체해 로그에서 시크릿을 마스킹한다. '.env' 파일은 Git에 절대 들어가지 않고('.gitignore'에 등록) Jenkins를 통과하지도 않는다. 앱이 실제로 실행되는 서버에만 존재한다. 'chmod 600'은 ec2-user만 읽을 수 있도록 파일 권한을 제한한다.

실습 4. 응용 실습: 롤백 메커니즘을 의도적으로 트리거하라. 헬스체크가 실패하게 만드는 버그를 도입하고(예: /health 엔드포인트가 HTTP 500을 반환하거나 앱이 시작 시 충돌), push해서 Jenkins가 (1) '--wait'를 실행하고 타임아웃, (2) Deploy 스테이지를 빨간색으로 표시, (3) 롤백 post 블록 실행, (4) EC2에 마지막 정상 버전이 계속 실행 중인지 확인하라.

// 1단계: index.js에 의도적인 버그 도입.
// 예: /health 엔드포인트가 항상 500을 반환하게 만들기:
if (req.url === '/health') {
  res.writeHead(500);
  res.end('Intentional failure');
  return;
}

// 2단계: 버그가 있는 변경을 push한다.
// Jenkins 빌드를 지켜보고 Deploy 스테이지 동작을 확인한다.

// 3단계: 빌드가 완료된 후(FAILED로) EC2 확인:
ssh ec2-user@<EC2_IP>
docker ps                                       # 어떤 이미지가 실행 중인가?
cat /home/ec2-user/last_good_tag               # 저장된 정상 태그는?
curl http://localhost:80/                      # 서비스가 여전히 살아있는가?
힌트
  • 버그 있는 헬스체크로 Deploy 스테이지가 초록색이 된다면 'docker-compose.yml'에 'healthcheck' 섹션이 있는지 확인한다. 없으면 '--wait'는 컨테이너가 healthy가 아닌 *시작*만 기다려 헬스체크와 무관하게 종료 코드 0을 반환한다.
  • 롤백 블록도 실패하면('last_good_tag'가 첫 실행 시 존재하지 않는 경우), 이것은 예상된 동작이다. 먼저 성공한 빌드를 push해 파일을 만든 다음 버그를 도입한다.
  • 테스트 중 대기 시간을 줄이려면 헬스체크의 'retries: 1'로 임시 변경한다. 30초 대신 10초 후에 실패한다.

// Build Image, Push to ECR — 초록 (이미지 빌드 및 push 성공)
// Deploy & Verify:
//   'docker compose up -d --wait'가 버그 있는 컨테이너를 시작
//   헬스체크 실행: curl -f http://localhost:8080/health → 종료 코드 1 (HTTP 500)
//   3회 재시도 (3 x 10s = 30s) 후 Compose가 0이 아닌 값으로 종료
//   SSH가 0이 아닌 값 반환 → Jenkins 스테이지 실패 → 빨간색
//   post { failure { ... } } 블록 실행:
//     last_good_tag 읽기 (마지막 정상 빌드의 SHA)
//     docker compose up -d --wait (이전 정상 이미지로)
//     이전 이미지가 헬스체크 통과 → 종료 코드 0 → 롤백 완료

// 3단계 — EC2에서 확인:
// docker ps → 이전 SHA 이미지를 실행하는 이전 컨테이너 표시
// cat /home/ec2-user/last_good_tag → 이전 SHA 표시 (실패한 배포로 덮어쓰이지 않음)
// curl http://localhost:80/ → 'Hello from container!' 반환 (서비스 여전히 작동)
// curl http://localhost:80/health → 'OK' 반환 (이전 이미지의 정상 엔드포인트)

// 정리: 버그를 되돌리고 push해 파이프라인 정상 동작 복구.

이 실습은 안전망의 최종 엔드투엔드 테스트다. 세 가지 결과가 모든 것이 작동함을 확인한다. (1) Deploy 스테이지가 빨간색이 된다 — '--wait'가 실제 검증을 한다는 증거. (2) last_good_tag가 변경되지 않는다 — 파일이 검증된 성공 후에만 기록된다는 증거. (3) 서비스가 이전 버전으로 계속 서비스된다 — 사람 개입 없이 자동 롤백이 작동했다는 증거. 파이프라인 자체가 문제를 감지했을 때도 망가진 서비스를 자동으로 복구한다는 것이 모든 설정 노력을 정당화하는 프로덕션 준비 기능이다.
