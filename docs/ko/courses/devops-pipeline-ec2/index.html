01. CI/CD 파이프라인 전체 그림과 이 강좌의 목표

학습 목표

  • 수동 배포의 실제 고통 — 사람의 실수, 재현 불가, 같은 작업을 반복하는 비용 — 을 이해한다.
  • GitLab, Jenkins, 컨테이너 이미지 레지스트리, EC2 서버가 하나의 자동화된 흐름으로 연결되는 구조를 그릴 수 있다.
  • 이 강좌를 끝냈을 때 완성될 파이프라인의 최종 모습을 미리 파악한다.

본문

수동 배포의 하루

이미 겪어봤을 법한 이야기부터 시작하자. 팀에서 새 기능 개발이 끝나 배포해야 하는 순간이 왔다. main 브랜치에 머지하려 하는데, 다른 두 명의 개발자가 먼저 푸시해서 머지 충돌이 생겼다. 손으로 충돌을 해소하고 다시 머지하니, 이번엔 여러 사람의 코드가 섞이면서 처음엔 없었던 테스트 실패가 쏟아진다. 스프린트 마감 전에 수습하느라 뛰어다닌다. 드디어 배포 타이밍이 왔다. "충분히 시니어한" 누군가가 서버에 SSH로 접속해서 실행 중인 컨테이너를 내리고, Docker Compose 파일을 수정하고, 올바른 순서대로, 오타 없이 다시 올린다.

이 과정의 매 단계마다 사람이 자리에 있어야 하고, 깨어 있어야 하고, 실수하지 않아야 한다. 그래서 금요일 저녁엔 배포를 꺼린다. 밤 9시에 뭔가 터졌는데 아무도 보고 있지 않으면 고객은 아침까지 기다려야 한다. 수동 배포는 느리고, 스트레스가 심하고, 무엇보다 재현이 불가능하다. 열 번 하면 열 번 다 조금씩 다른 결과가 나온다.

수동 배포에서 가장 비싼 재료는 사람이다. CI/CD는 반복적이고 오류가 잦은 단계에서 사람을 빼내 실제로 판단이 필요한 일에 집중하게 만드는 방법론이다.

CI와 CD의 실제 의미

CI — 지속적 통합(Continuous Integration). 릴리스 날까지 기다렸다가 모두의 작업을 한꺼번에 합치고 충돌을 발견하는 대신, 개발자들이 작은 변경을 자주 — 하루에도 여러 번 — 통합하고 자동화 시스템이 즉시 빌드·테스트를 실행한다. 문제가 일찍, 격리된 상태에서 드러나기 때문에 고치는 비용이 훨씬 싸다.

CD — 지속적 전달/배포(Continuous Delivery / Deployment). 코드가 자동 검사를 통과하면 자동으로 패키징되어 릴리스 단계로 이동한다. 지속적 전달(Continuous Delivery)은 모든 변경이 항상 배포 가능하고 테스트된 상태를 유지한다는 뜻이다. 프로덕션으로의 최종 푸시는 원클릭 수동 승인이 남아 있을 수 있다. 지속적 배포(Continuous Deployment)는 한 단계 더 나아가, 자동 검사를 통과한 모든 변경을 사람 개입 없이 프로덕션까지 자동으로 내보낸다.

합치면, CI/CD 파이프라인은 코드를 위한 자동화 조립 라인이다. 푸시 하나가 소스 → 빌드 → 테스트 → 패키징 → 배포의 일련의 단계를 시작시키고, 각 단계는 성공했을 때만 다음 단계로 넘어간다.

모든 파이프라인이 공유하는 4단계

어떤 도구를 쓰든 거의 모든 파이프라인은 같은 형태를 따른다.

1. 소스(Source) — 코드 변경이 저장소(여기선 GitLab)에 푸시된다. 이것이 트리거다.

2. 빌드(Build) — 코드가 의존성과 함께 배포 가능한 산출물(artifact)로 조립된다. 우리의 경우 그 산출물은 Docker 이미지다.

3. 테스트(Test) — 자동화 테스트가 빌드 결과물을 검증한다. 실패하면 파이프라인이 멈추고 팀에 알린다. 나쁜 코드는 사용자에게 절대 닿지 않는다.

4. 배포(Deploy) — 검증된 산출물이 실행 환경에 릴리스된다. 여기선 AWS EC2 서버다.

효과는 극적이다. 소스 자료에 나온 한 팀은 자동화 후 빌드-테스트-배포 사이클을 두 시간에서 8분으로 줄였다. Netflix나 Amazon이 하루에 수십, 심지어 수천 번씩 배포할 수 있는 이유가 바로 이런 파이프라인이다.

이 강좌에서 만들 파이프라인

이 강좌는 직접 만들며 배우는 강좌다. git push 한 번으로 내 소스 코드가 터미널 조작 없이 EC2의 실행 중인 컨테이너까지 흘러가는 완전한 파이프라인을 조립한다. 전체 흐름은 다음과 같다.

1. GitLab 저장소에 코드를 푸시한다.

2. Webhook이 Jenkins에 변경 사항을 알린다.

3. Jenkins가 코드를 체크아웃하고 빌드 및 테스트 단계를 실행한다. 테스트가 실패하면 여기서 전부 멈춘다.

4. 성공하면 Jenkins가 앱의 Docker 이미지를 빌드한다.

5. Jenkins가 AWS ECR(프라이빗 이미지 레지스트리)에 인증하고 태그된 이미지를 푸시한다.

6. Jenkins가 SSH로 EC2 서버에 접속해 docker compose로 새 이미지를 pull하고 실행 중인 컨테이너를 교체한다.

7. 헬스체크가 새 버전이 정상 작동하는지 확인한다. 확인에 실패하면 파이프라인이 이전 이미지로 롤백한다. 시크릿과 환경변수는 코드에서 분리되어 안전하게 주입된다.

아래 다이어그램은 이 네 가지 도구가 하나의 자동화 경로로 연결되어 git push부터 실행 중인 컨테이너까지 이어지는 모습을 보여 준다.

예제 앱은 언어에 종속되지 않는 단순한 웹앱으로 일부러 선택했다. Dockerfile, Jenkinsfile, 배포 로직이 핵심 학습 내용이며, 이 모두는 내 애플리케이션으로 그대로 교체할 수 있다.

왜 이 투자를 하는가

이 모든 걸 처음 설정하는 데는 며칠이 걸린다. 하지만 그 대가로 얻는 것은 크다. 빠뜨린 단계로 인한 사람의 실수가 없어지고, 시니어 엔지니어 한 명을 기다리느라 생기는 "코드 프리즈"도 사라진다. 릴리스는 더 빠르고 잦아지며, 모든 변경이 매번 동일한 방식으로 테스트됐다는 조용한 확신이 생긴다. 지금 며칠을 투자해서 앞으로 수년간의 시간을 절약하고 리스크를 줄이는 이 거래가, CI/CD가 현대 DevOps의 핵심에 자리한 이유다.

핵심 정리

  • 수동 배포는 느리고 스트레스가 심하며 재현이 불가능하다. 매번 같은 단계를 실수 없이 수행할 사람에 의존하기 때문이다.
  • CI는 작은 변경을 자주 통합·테스트해 문제를 일찍 잡는다. CD는 테스트를 통과한 결과물을 자동으로 전달(선택에 따라 배포까지)한다.
  • 거의 모든 파이프라인은 소스 → 빌드 → 테스트 → 배포의 4단계를 따르며, 각 단계는 이전 단계가 통과해야만 진행된다.
  • 이 강좌를 마치면 GitLab에 git push 한 번으로 Jenkins가 빌드·테스트·ECR 이미지 푸시·EC2 배포를 수행하는 파이프라인이 완성된다. 헬스체크, 롤백, 안전한 시크릿 처리까지 갖춘 파이프라인이다.

📝 퀴즈

Q1. 한 개발자가 '지속적 배포(Continuous Deployment)는 프로덕션에 반영되기 전에 반드시 수동 승인이 필요하다'고 말한다. 이 주장에서 잘못된 부분은 무엇이고 왜 그런가?

  • 이 주장은 완전히 맞다 — CD에서는 항상 수동 승인이 필요하다.
  • 지속적 배포(Continuous Deployment)는 자동 검사를 통과한 모든 변경을 프로덕션까지 자동으로 배포한다. 수동 승인 단계를 유지하는 것은 지속적 전달(Continuous Delivery)이다.
  • 지속적 배포는 수동 검사를 거쳐 주 1회만 배포한다는 뜻이다.
  • 지속적 전달과 지속적 배포 모두 프로덕션 반영 전에 수동 승인 단계가 필요하다.

정답은 2번이다. 지속적 전달(Continuous Delivery)은 모든 변경을 항상 배포 가능한 상태로 유지하되, 최종 수동 승인 단계를 남겨 둘 수 있다. 반면 지속적 배포(Continuous Deployment)는 그 단계마저 없애고 자동 검사를 통과한 모든 변경을 프로덕션까지 자동으로 배포한다. 나머지 선택지는 두 용어를 혼동하거나, 강의에서 언급하지 않은 주 1회 배포 주기를 임의로 만들어낸 것이다.

Q2. 소스 → 빌드 → 테스트 → 배포 순서로 파이프라인 단계를 구성하는 이유 중 프로덕션 환경을 보호하는 원리를 가장 잘 설명한 것은?

  • 모든 개발자가 동시에 코드를 머지하도록 강제해 충돌을 막는다.
  • 각 단계가 게이트 역할을 한다. 실패하면 파이프라인이 멈춰 미완성·불안정한 코드가 배포 단계까지 진행될 수 없다.
  • 단계를 순서대로 실행하는 것이 병렬 실행보다 빠르다.
  • 소스 단계가 5분 이상 실행되지 않으면 배포 단계가 시작되지 않는다.

정답은 2번으로, 강의에서 설명한 4단계 파이프라인의 핵심 원리를 담고 있다. 각 단계는 성공했을 때만 다음 단계로 제어권을 넘긴다. 빌드 실패 시 테스트가 실행되지 않고, 테스트 실패 시 이미지 생성과 배포가 차단된다. 이 '빠른 실패(fail fast)' 패턴이 나쁜 코드가 사용자에게 닿지 않도록 막는다. 선택지 1, 3, 4는 관계없거나 실제로 존재하지 않는 규칙을 서술한다.

Q3. 이 강좌의 파이프라인에서 GitLab으로부터 웹훅 알림을 받아 빌드·테스트·배포 단계를 실행하는 구성 요소는 무엇인가?

  • AWS EC2
  • AWS ECR
  • Jenkins
  • Docker Compose

Jenkins가 이 파이프라인의 자동화 서버다. 개발자가 코드를 푸시하면 GitLab이 HTTP POST(웹훅)를 Jenkins에 보내고, Jenkins가 코드를 체크아웃하고 테스트를 실행하고 Docker 이미지를 빌드해 ECR에 푸시한 뒤 EC2에 배포한다. EC2는 배포 대상 서버이고, ECR은 이미지 레지스트리이며, Docker Compose는 EC2에서 컨테이너를 실행하는 도구다. 이 셋 중 어느 것도 GitLab 푸시 이벤트에 직접 반응하지 않는다.

Q4. 한 팀이 매주 수동으로 배포하다가 금요일 밤 9시에 배포 실패를 겪었다. 이 상황이 수동 배포의 어떤 특성을 보여주는가?

  • 수동 배포는 빠르지만 추적성이 없다.
  • 수동 배포는 재현 가능하지만 시니어 엔지니어의 참여가 필요하다.
  • 수동 배포는 느리고 스트레스가 심하며 사람이 자리에 있어야 한다. 그래서 업무 시간 외 장애 복구가 어렵다.
  • 수동 배포는 시니어 엔지니어가 매 단계를 검토하므로 안전하다.

강의에서 이 상황을 명시적으로 다룬다. 밤 9시에 문제가 터졌는데 아무도 보고 있지 않으면 고객은 아침까지 기다려야 한다. 이는 수동 배포가 사람이 자리에 있고, 깨어 있고, 실수하지 않아야 한다는 점, 그리고 그렇지 못할 때 복구가 늦어진다는 점을 보여 준다. 선택지 1은 수동 배포가 빠르지 않으므로 틀리고, 선택지 2는 수동 배포의 최대 약점이 재현 불가능성이므로 틀리며, 선택지 4는 강의가 반박하는 낭만화된 표현이다.

Q5. 이 강좌를 마친 후, 개발자가 'git push'를 실행한 순간부터 EC2에 새 버전이 배포되기까지 어떤 일이 자동으로 일어나는가?

  • 개발자가 EC2에 SSH로 접속해 docker compose up을 수동으로 실행한다.
  • GitLab CI 잡이 Jenkins 없이 이미지를 빌드하고 배포한다.
  • Jenkins가 코드를 체크아웃하고 빌드·테스트한 뒤 Docker 이미지를 만들어 ECR에 푸시하고, EC2에 SSH로 접속해 컨테이너를 교체하고 헬스체크로 결과를 확인한다. 모두 자동으로.
  • 새 이미지가 운영팀에 이메일로 전달되고 팀이 업무 시간에 배포한다.

선택지 3은 강의에서 소개한 자동화 엔드투엔드 흐름을 정확히 설명한다. GitLab 웹훅 → Jenkins → 체크아웃 → 빌드/테스트 → Docker 이미지 → ECR 푸시 → EC2 SSH → docker compose → 헬스체크/롤백. 선택지 1은 파이프라인이 대체하는 수동 과정이다. 선택지 2는 GitLab CI 잡이 아니라 Jenkins가 파이프라인을 조율하므로 틀리다. 선택지 4는 실제로 존재하지 않는 방법이다.

02. 내 앱을 컨테이너로 — Dockerfile 작성과 로컬 실행

학습 목표

  • 왜 배포 단위로 컨테이너 이미지를 쓰는지 이해한다.
  • 어떤 언어의 웹앱이든 Dockerfile로 이미지화하는 기본 패턴을 익힌다.
  • 자동화하기 전에 docker build와 docker run으로 이미지가 로컬에서 잘 동작하는지 검증한다.

왜 컨테이너로 배포하나

"내 로컬에서는 잘 돌아가는데요." 한 번쯤 해봤거나 들어봤을 말이다. 컨테이너는 바로 이 문제를 해결하기 위해 만들어졌다. 앱은 특정 런타임 버전, 특정 라이브러리, 특정 설정에 의존한다. 팀원의 노트북이나 버전이 조금 다른 서버로 옮기면 깨진다.

컨테이너는 코드와 그것이 실행되는 데 필요한 모든 것 — 런타임, 시스템 도구, 라이브러리, 설정 — 을 하나의 독립된 단위로 묶어 준다. 환경 자체를 함께 담기 때문에 내 노트북, 동료의 머신, EC2 서버 어디에서든 동일하게 실행된다. 먼저 세 가지 용어를 정리해 두자.

  • Dockerfile — 이미지를 만들기 위한 명령들이 담긴 텍스트 파일. 레시피라고 생각하면 된다.
  • 이미지(Image) — Dockerfile로 빌드한 불변(immutable) 스냅샷. 만들어 놓은 완성 요리라고 생각하면 된다.
  • 컨테이너(Container) — 이미지의 실행 인스턴스. 손님 앞에 내놓은 요리다. 하나의 이미지로 여러 컨테이너를 띄울 수 있다.

관계는 한 방향으로 흐른다. Dockerfile이 이미지를 빌드하고, 이미지가 컨테이너로 실행된다. 아래 다이어그램에서 이 흐름을 확인할 수 있다.

이 파이프라인에서 이미지는 내 노트북에서 Jenkins를 거쳐 ECR로, 다시 EC2로 이동하는 단 하나의 배포 산출물이다. 이후에 자동화하는 모든 것은 결국 "이 이미지를 빌드하고, 저장하고, 어딘가에서 실행한다"는 흐름이다. Dockerfile을 제대로 작성하는 것이 그 모든 것의 기초다.

표준 Dockerfile 패턴

웹앱을 위한 Dockerfile은 거의 같은 뼈대를 따른다. Node.js 예제를 기준으로 패턴을 보여 주지만, 구조 자체는 어떤 언어에든 적용된다. 프로젝트 루트에 확장자 없이 Dockerfile이라는 이름으로 파일을 만든다.


# 1. 런타임이 이미 들어 있는 공식 베이스 이미지를 지정한다
FROM node:20-alpine

# 2. 이미지 안에서 작업 디렉터리를 설정한다
WORKDIR /app

# 3. 의존성 목록을 먼저 복사하고 설치한다 — 이 레이어가 캐시된다
COPY package.json package-lock.json ./
RUN npm install

# 4. 그다음 나머지 소스 코드를 복사한다
COPY . .

# 5. 앱이 리스닝하는 포트를 문서화한다
EXPOSE 8080

# 6. 컨테이너가 시작될 때 실행할 명령을 지정한다
CMD ["node", "index.js"]

몇 가지 줄은 실질적인 best practice를 담고 있어 설명이 필요하다.

  • FROM은 베이스 이미지를 지정한다. 공식·slim 이미지를 쓰길 권한다. node:20-alpine은 Alpine Linux(약 5 MB 초경량 배포판) 기반이라 최종 이미지를 작고 빠르게 유지한다. 베이스 이미지가 무거우면 배포할 때마다 비용이 붙는다.
  • 각 명령은 캐시되는 레이어다. Docker는 변경된 레이어부터만 재빌드한다. 그렇기 때문에 소스 코드를 복사하기 전에 package.json을 복사하고 npm install을 실행하는 것이다. 의존성은 자주 바뀌지 않지만 소스는 수시로 바뀐다 — 의존성을 앞 레이어에 두면 대부분의 빌드에서 캐시된 설치 결과를 그대로 쓸 수 있다. 순서를 바꾸면 코드를 한 줄 고칠 때마다 전체 재설치가 강제된다.
  • CMD는 배열(exec) 형식을 쓴다 — 일반 문자열 대신 ["node", "index.js"]. exec 형식은 프로세스를 shell로 감싸지 않고 직접 실행하기 때문에 시그널 처리와 종료 동작이 올바르게 작동한다.

불필요한 파일 제외: .dockerignore 파일

COPY . .을 쓰면 Docker는 폴더 안의 모든 것 — 로컬 node_modules, 빌드 산출물, 위험하게도 혹시 있을 시크릿 파일까지 — 을 복사한다. .gitignore와 같은 방식으로 동작하는 .dockerignore 파일을 만들어 제외한다.


node_modules
.git
*.env

이렇게 하면 이미지가 작아지고, 빌드가 빨라지고, 이미지를 pull한 누구든 레이어를 들여다볼 수 있는 상황에서 시크릿이 노출되지 않는다.

빌드하고 로컬에서 실행하기

이제 레시피를 이미지로 만들어 보자. -t 플래그로 알아보기 쉬운 이름을 붙인다.


docker build -t myapp:local .

끝의 .은 현재 디렉터리에서 Dockerfile을 찾으라는 의미다. 출력을 보면 Docker가 각 명령을 단계별로 실행하는 것을 확인할 수 있다. 빌드된 이미지가 있는지 확인한다.


docker images

이제 실행해 보자. 초보자가 자주 빠지는 함정이 있다. 컨테이너를 시작했는데 localhost에 아무것도 뜨지 않는 경우다.


docker run -p 5000:8080 myapp:local

-p 5000:8080 플래그가 핵심이다. 기본적으로 컨테이너의 포트는 내 머신에서 접근할 수 없다. 이 플래그가 호스트의 포트 5000을 컨테이너 안의 포트 8080(EXPOSE한 포트)으로 연결한다. http://localhost:5000을 열면 앱이 보인다. 형식은 호스트:컨테이너다.

백그라운드로 실행하려면 -d(detached)를 붙인다. docker ps로 실행 중인 컨테이너를 확인하고, docker logs <컨테이너>로 로그를 보고, docker stop <컨테이너>로 중지한다.

이것이 시작점이다

재현 가능하고 이식 가능한 이미지를 만들고, 실행까지 확인했다. 이 이미지가 바로 이후 파이프라인이 자동으로 빌드·저장·배포할 대상이다. 앞으로의 강의에서는 지금 수동으로 한 모든 단계 — 빌드, 태그, 실행 — 가 push마다 Jenkins가 자동으로 처리하는 스테이지가 된다.

핵심 정리

  • 컨테이너는 앱과 실행 환경 전체를 묶어 "내 로컬에서만 되는" 문제를 없애고, 어디서든 동일하게 동작하는 단 하나의 배포 산출물을 만들어 준다.
  • Dockerfile → 이미지 → 컨테이너 의 흐름을 기억한다.
  • Dockerfile은 레이어 캐싱을 활용하도록 작성한다. 의존성 복사·설치를 소스 코드 복사 전에 두고, slim 베이스 이미지와 .dockerignore로 이미지를 작고 안전하게 유지한다.
  • docker build -t 이름 .와 docker run -p 호스트:컨테이너 이름으로 로컬 검증을 마친다. 포트 매핑이 없으면 앱에 접근할 수 없고, 형식은 호스트:컨테이너다.

🛠 실습

실습 1. 준비 확인: Dockerfile, 이미지, 컨테이너의 관계를 자신의 말로 설명하라. 그런 다음 터미널에서 'docker --version'과 'docker images'를 실행해 Docker가 정상 설치되어 있는지 확인하라.

# 코드 없이 터미널을 열면 됩니다.
# 실행할 명령:
# docker --version
# docker images
힌트
  • 'docker --version'이 'command not found'를 반환하면 Docker가 설치되지 않았거나 PATH에 없는 것이다. macOS/Windows는 Docker Desktop을, Linux는 해당 배포판의 Docker Engine 패키지를 설치한다.
  • 'docker images'가 멈추거나 'Cannot connect to the Docker daemon'을 반환하면 Docker 데몬이 실행 중이지 않은 것이다. Docker Desktop을 시작하거나 Linux에서 'sudo systemctl start docker'를 실행한다.
정답 보기 
Dockerfile은 레시피(명령이 담긴 텍스트 파일)다. 'docker build'를 실행하면 이미지 — 앱과 실행 환경을 담은 불변의 독립 스냅샷 — 가 만들어진다. 컨테이너는 그 이미지의 실행 인스턴스이며, 같은 이미지로 여러 컨테이너를 동시에 띄울 수 있다.

예상 터미널 출력:
$ docker --version
Docker version 24.x.x, build ...

$ docker images
REPOSITORY   TAG   IMAGE ID   CREATED   SIZE
(비어 있거나 기존 이미지 목록이 표시됨)

세 가지 핵심 용어를 실제로 무언가를 만들기 전에 정리하는 단계다. Docker가 설치되어 있고 데몬이 실행 중이어야 이후 모든 단계가 작동한다. 'docker images'는 데몬이 응답하는지 확인하고 이전에 빌드된 이미지 목록을 보여 준다. 목록이 비어 있어도 괜찮다.

실습 2. 핵심 실습: 올바른 Dockerfile을 갖춘 최소한의 Node.js 웹앱을 만들어 빌드하고, 'http://localhost:5000'에서 응답이 오도록 로컬에서 실행하라.

요구사항:

1. 'index.js' 작성 — 포트 8080에서 'Hello from container!'를 응답하는 Node.js HTTP 서버.

2. 'start' 스크립트가 있는 'package.json' 작성.

3. 다음 조건을 갖춘 Dockerfile 작성: 'node:20-alpine' 사용, WORKDIR을 /app으로 설정, 소스 코드 복사 전에 의존성을 복사·설치(레이어 캐싱), 포트 8080 EXPOSE, CMD는 exec 형식 사용.

4. 'node_modules', '.git', '*.env'를 제외하는 '.dockerignore' 작성.

5. 'docker build -t myapp:local .'로 이미지를 빌드하고 'docker run -p 5000:8080 myapp:local'로 실행.

6. http://localhost:5000을 열어 응답을 확인한다.

# 생성할 디렉터리 구조:
# myapp/
#   index.js
#   package.json
#   Dockerfile
#   .dockerignore

# index.js 뼈대:
const http = require('http');
const server = http.createServer((req, res) => {
  res.writeHead(200);
  res.end('Hello from container!');
});
server.listen(/* 포트 입력 */, () => console.log('Listening...'));

# package.json 뼈대:
{
  "name": "myapp",
  "version": "1.0.0",
  "scripts": { "start": "node index.js" }
}

# Dockerfile 뼈대:
FROM ___
WORKDIR ___
COPY ___ ___
RUN ___
COPY . .
EXPOSE ___
CMD ["___", "___"]
힌트
  • 'docker build'가 'COPY failed: file not found'로 실패하면, Dockerfile과 같은 디렉터리에 'package.json'이 있는지 확인하고 'docker build'를 그 디렉터리에서 실행했는지 확인한다(끝의 '.'이 중요하다).
  • 'http://localhost:5000'에서 'connection refused'가 뜨면 컨테이너가 종료됐을 수 있다. 'docker ps -a'로 상태를 확인하고 'docker logs <컨테이너-id>'로 오류를 읽는다. 흔한 원인은 index.js의 포트 번호가 잘못된 것(예: 8080 대신 3000)이다.
  • '.dockerignore'를 추가하는 것을 잊었다면 'docker image prune'으로 용량이 큰 이미지를 삭제하고, 파일을 추가한 뒤 다시 빌드한다.
정답 보기 
# index.js
const http = require('http');
const server = http.createServer((req, res) => {
  res.writeHead(200);
  res.end('Hello from container!');
});
server.listen(8080, () => console.log('Listening on port 8080'));

# package.json
{
  "name": "myapp",
  "version": "1.0.0",
  "scripts": { "start": "node index.js" }
}

# Dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package.json ./
RUN npm install
COPY . .
EXPOSE 8080
CMD ["node", "index.js"]

# .dockerignore
node_modules
.git
*.env

# 빌드 및 실행:
docker build -t myapp:local .
docker run -p 5000:8080 myapp:local
# http://localhost:5000을 열면 'Hello from container!'가 반환된다

'COPY . .'보다 먼저 'package.json'을 복사하고 'npm install'을 실행하는 것이 레이어 캐싱의 핵심이다. 의존성은 자주 바뀌지 않으므로, 소스 코드만 바뀐 다음 빌드에서 Docker가 설치 레이어를 재사용할 수 있다. 'node:20-alpine'을 쓰면 이미지 크기가 ~1 GB가 아닌 ~150 MB 수준으로 유지된다. 포트 매핑 '-p 5000:8080'은 호스트 포트 5000을 컨테이너 포트 8080(EXPOSE한 포트)에 연결한다. 이 플래그 없이는 컨테이너 네트워크가 격리되어 localhost에 아무것도 표시되지 않는다.

실습 3. 핵심 실습(계속): 'docker ps', 'docker logs', 'docker stop'으로 실행 중인 컨테이너를 관리하라. 그런 다음 index.js의 응답 텍스트만 바꿔 이미지를 재빌드해 Docker가 어떤 레이어를 재사용하는지 확인하라.

# 이전 실습에서 'docker run -p 5000:8080 myapp:local'이 실행 중이라고 가정한다.
# 1단계: 컨테이너 ID / 이름 확인
docker ps
# 2단계: 로그 읽기
docker logs <컨테이너-이름-또는-id>
# 3단계: 중지
docker stop <컨테이너-이름-또는-id>
# 4단계: index.js의 응답 텍스트를 바꾼 뒤 재빌드
docker build -t myapp:local .
# 빌드 출력에서 'CACHED' 줄을 확인한다
힌트
  • 'docker ps'가 빈 목록을 반환하면 컨테이너가 종료됐을 수 있다. 'docker ps -a'(종료된 컨테이너 포함 전체 목록)로 찾는다.
  • 재빌드 시 npm install 단계에 'CACHED'가 뜨지 않으면 Dockerfile에서 'RUN npm install' 전에 'COPY . .'을 먼저 쓴 것이 아닌지 확인한다. 그렇게 하면 소스 변경이 그 레이어를 무효화한다.
정답 보기 
# 'docker ps' 실행 후 다음과 같이 표시된다:
# CONTAINER ID   IMAGE         COMMAND           STATUS          PORTS
# a1b2c3d4e5f6   myapp:local   "node index.js"   Up 2 minutes   0.0.0.0:5000->8080/tcp

# docker logs a1b2c3d4e5f6  →  'Listening on port 8080'

# docker stop a1b2c3d4e5f6  →  컨테이너 종료

# index.js 편집: 'Hello from container!'를 'Hello, CI/CD!'로 변경
# 그런 다음: docker build -t myapp:local .
# 예상 빌드 출력:
#  => CACHED [2/5] WORKDIR /app
#  => CACHED [3/5] COPY package.json ./
#  => CACHED [4/5] RUN npm install       <-- 캐시 재사용, 재실행 안 됨
#  => [5/5] COPY . .                     <-- 소스 변경으로 무효화됨
# npm install 레이어가 재사용되어 재빌드가 빠르다.

Docker는 변경된 레이어와 그 이후 레이어만 재빌드한다. 소스 변경이 'COPY . .' 레이어(5단계)만 건드리기 때문에, 비용이 큰 'npm install'을 포함한 그 이전 레이어는 모두 캐시에서 제공된다. Dockerfile에서 의존성 복사·설치가 'COPY . .' 앞에 와야 하는 이유가 바로 이것이다. 순서를 바꾸면 소스가 바뀔 때마다 설치 캐시가 무효화되어 빌드가 불필요하게 느려진다.

실습 4. 응용 도전: 동료의 Dockerfile에 세 가지 문제가 있다. 빌드가 느려지거나, 이미지가 커지거나, 동작이 잘못될 수 있는 문제들이다. 각 문제를 찾아내고 수정된 Dockerfile을 작성하라.

문제 있는 Dockerfile:


FROM node:20
WORKDIR /app
COPY . .
RUN npm install
EXPOSE 8080
CMD node index.js

힌트
  • 이미지 크기 확인: 'FROM node:20' 버전과 'FROM node:20-alpine' 버전 둘 다 빌드한 뒤 'docker images'를 실행해 SIZE 열을 비교한다.
  • 캐싱 차이 확인: 잘못된 버전으로 빌드하고, index.js를 수정한 뒤 재빌드하여 몇 개 레이어가 'CACHED'로 표시되는지 본다. 그런 다음 수정된 버전으로 같은 작업을 해 비교한다.
  • CMD shell 대 exec 형식은 쉽게 구분된다. exec 형식은 대괄호와 따옴표가 있는 JSON 배열을 쓰고, shell 형식은 대괄호 없는 일반 문자열이다.
정답 보기 
# 문제 1: 'FROM node:20'은 전체 Debian 기반 이미지(~1 GB)를 사용한다.
#   수정: 'FROM node:20-alpine' (~150 MB)

# 문제 2: 'RUN npm install' 전에 'COPY . .'이 온다.
#   소스 코드가 바뀔 때마다 COPY 레이어가 무효화되어 매번 npm install이 강제된다.
#   수정: package.json만 먼저 복사하고 설치한 뒤 소스를 복사한다.

# 문제 3: 'CMD node index.js'는 shell 형식이다(/bin/sh -c로 감쌈).
#   Docker가 shell에 시그널을 보내므로 Node 프로세스가 제대로 종료되지 않는다.
#   수정: 'CMD ["node", "index.js"]' (exec 형식)

# 수정된 Dockerfile:
FROM node:20-alpine
WORKDIR /app
COPY package.json ./
RUN npm install
COPY . .
EXPOSE 8080
CMD ["node", "index.js"]

이 세 가지는 실무에서 가장 흔한 Dockerfile 실수다. 베이스 이미지 선택은 CI 빌드마다 이미지 push/pull 시간에 직접 영향을 준다. 레이어 캐시 순서는 Dockerfile에서 성능에 가장 큰 영향을 주는 결정이다 — 잘못 작성하면 코드를 한 줄 바꿀 때마다 전체 의존성 재설치가 발생한다. CMD exec 형식 대 shell 형식의 차이는 시그널 전달에 영향을 미친다. 시그널을 제대로 받지 못한 프로세스는 SIGTERM 대신 SIGKILL로 강제 종료되어 데이터 손실이나 비정상 종료로 이어질 수 있다.

03. GitLab 저장소 구성과 Push 트리거(Webhook)

학습 목표

  • GitLab 저장소를 만들고 코드를 푸시하는 기본 흐름과 브랜치 전략을 이해한다.
  • GitLab 웹훅이 push 이벤트를 Jenkins에 전달하는 원리와 보안 방식을 안다.
  • GitLab과 Jenkins를 연결해 코드 푸시 시 파이프라인이 자동 트리거되도록 설정한다.

본문

저장소가 트리거다

이전 강의에서 이미지를 직접 손으로 빌드했다. 파이프라인의 목적은 그 수동 작업을 없애는 것이다. 즉 코드 푸시 하나가 모든 것을 시작하게 만든다. 그러려면 두 가지가 필요하다. 코드를 올릴 장소(GitLab)와 "뭔가 바뀌었으니 일 시작해"라고 Jenkins에 알리는 메커니즘(웹훅). 이번 강의에서 이 둘을 연결한다.

GitLab과 브랜치 전략 간단 정리

GitLab에 프로젝트를 만들고 이전 강의에서 만든 Dockerfile이 포함된 앱을 푸시한다.


git init
git remote add origin https://gitlab.com/<your-namespace>/<your-project>.git
git add .
git commit -m "Initial commit with Dockerfile"
git push -u origin main

파이프라인을 안정적으로 운영하려면 간단한 브랜치 전략이 필요하다. 일상적인 작업은 dev(또는 feature) 브랜치에서 하고, main은 언제든 배포 가능한 상태를 나타내는 브랜치로 유지한다. 파이프라인이 어느 브랜치의 푸시에 반응할지는 자유롭게 설정할 수 있다. 학습 중에는 dev 브랜치 푸시에만 트리거를 걸어 놓으면 배포 브랜치를 건드리지 않고 마음껏 실험할 수 있다.

웹훅이란 실제로 무엇인가

웹훅은 GitLab이 "뭔가 일어났을 때 알려줄게"라고 말하는 방식이다. Jenkins가 GitLab에 "새 내용 있어요?"라고 계속 묻는 방식(폴링) 대신, GitLab이 push 같은 이벤트가 발생하는 즉시 지정한 URL로 HTTP 요청을 능동적으로 보낸다. 우편함을 계속 확인하러 나가는 것과 집배원이 도착하면 초인종을 눌러주는 것의 차이다.

아래 시퀀스 다이어그램은 push 한 번이 Jenkins 빌드 시작으로 이어지는 과정과, 시크릿 토큰이 요청의 신뢰성을 어떻게 증명하는지 보여 준다.

GitLab에서 Jenkins에 연결하는 방법은 두 가지로, 둘 다 알아두면 좋다.

  • Jenkins 통합(Integration) — 공식 권장 방법. GitLab 플러그인과 Jenkins를 저장된 연결로 잇는다. 빌드 결과를 GitLab UI에 다시 보고하는 기능도 있다.
  • 순수 웹훅 — 보다 직접적인 방법. GitLab이 토큰으로 보호된 Jenkins URL에 POST를 보내는 방식이다.

이 강좌에서는 통합 방식으로 연결하되, 내부에서 어떤 일이 벌어지는지 이해할 수 있도록 순수 웹훅도 함께 살펴본다.

1단계 — GitLab 개인 액세스 토큰 만들기

Jenkins가 GitLab API와 통신하려면 권한이 필요하다. 최신 GitLab UI에서는 아바타(좌상단) → Preferences(구 버전은 Edit profile) → 좌측 메뉴의 Access Tokens로 이동한다. Add new token을 클릭하고 이름과 만료일을 설정하되 — 이 부분이 핵심 — 범위를 반드시 api로 지정한다. 토큰을 즉시 복사한다. GitLab은 딱 한 번만 보여 주며 이후 절대 다시 확인할 수 없다. 잃어버리면 새 토큰을 만들어야 한다.

2단계 — Jenkins에 GitLab 플러그인 설치

Jenkins에서 Manage Jenkins → Plugins → Available로 이동해 GitLab을 검색하고 설치한다(GitLab API Plugin이 함께 설치된다). GitLab이 빌드를 트리거하고 Jenkins가 결과를 GitLab에 다시 표시할 수 있게 해주는 플러그인이다.

3단계 — Jenkins에 GitLab 연결 생성

Manage Jenkins → System으로 이동해 새로 생긴 GitLab 섹션을 찾는다. 다음 항목을 채운다.

  • Connection name — 원하는 이름(예: gitlab-saas).
  • GitLab host URL — GitLab SaaS라면 https://gitlab.com, 자체 서버라면 해당 서버 URL.
  • Credentials — Add를 클릭해 종류를 GitLab API token으로 선택하고 1단계에서 복사한 토큰을 붙여 넣는다.

저장 후 Test Connection을 클릭한다. 성공 메시지가 나오면 Jenkins가 GitLab에 접근할 수 있다는 의미다.

4단계 — Jenkins 잡이 푸시를 감지하도록 설정

Jenkins 파이프라인 잡 설정에서 Build Triggers를 찾아 "Build when a change is pushed to GitLab"을 활성화한다. 반응할 이벤트를 선택한다 — 보통 Push Events와 Merge Request Events를 선택한다. 이 설정이 어떤 알림이 실제로 빌드를 시작할지 결정한다.

5단계 — GitLab에 웹훅 생성

GitLab 프로젝트의 Settings → Webhooks로 이동한다. Jenkins의 Build Triggers 섹션에 표시된 두 가지 정보가 필요하다.

  • URL — Jenkins가 이 잡을 위해 노출하는 엔드포인트(Jenkins가 정확한 URL을 보여 준다).
  • Secret token — Jenkins에서 Generate를 클릭해 생성한 뒤 GitLab 웹훅의 시크릿 토큰 필드에 붙여 넣는다.

시크릿 토큰의 역할: 들어온 요청이 URL을 짐작한 외부인이 보낸 것이 아니라 실제 GitLab 프로젝트에서 온 것임을 Jenkins가 검증하는 수단이다. Push events를 선택하고 브랜치를 지정한 뒤 훅을 추가한다. GitLab의 Test → Push events 버튼으로 샘플 이벤트를 보내 연결이 정상인지 확인할 수 있다.

연결 전체는 양쪽에서 일치해야 하는 두 가지 사실로 이루어진다. Jenkins 잡을 가리키는 URL과, 요청이 진짜임을 증명하는 공유 시크릿 토큰. 이 둘을 맞추면 트리거가 동작한다.

6단계 — 엔드투엔드 확인

사소한 변경을 만들어 dev 브랜치에 커밋한다.


git checkout dev
echo "trigger test" >> README.md
git commit -am "Test webhook"
git push

Jenkins를 지켜보자. 1~2초 안에 새 빌드가 나타나고 콘솔 출력에 내 사용자 이름으로 "Started by GitLab push"가 표시된다. 이제 자동 트리거가 생겼다. 앞으로는 명령어가 아니라 코드 푸시가 파이프라인을 움직인다. 다음 강의에서는 트리거가 발생했을 때 Jenkins가 실제로 무엇을 할지 정의한다.

핵심 정리

  • 명령어 실행이 아니라 GitLab에 코드를 푸시하는 것이 파이프라인을 시작하게 되므로, GitLab과 Jenkins를 연결하는 일이 자동화의 기반이다.
  • 웹훅은 GitLab이 변경 사항을 Jenkins에 능동적으로 알리는 방식이다. Jenkins가 반복적으로 폴링할 필요가 없다.
  • 연결은 두 가지 사실이 일치해야 성립한다. Jenkins 잡을 가리키는 URL과 요청을 인증하는 공유 시크릿 토큰.
  • Jenkins 잡에서 "GitLab에 변경이 푸시되면 빌드"를 활성화하고 GitLab에 매칭되는 웹훅을 만든 뒤, 실제 푸시 후 "Started by GitLab push"가 나타나는지로 동작을 검증한다.

GitLab 저장소는 모든 자동 파이프라인 실행의 출발점이다. 'main' 브랜치는 배포 가능한 상태를 유지하고, 'dev'(또는 feature 브랜치)는 일상적인 작업을 위한 공간이다. 두 브랜치를 지금 만들어 두면 나중에 파이프라인 트리거를 'dev' 푸시에만 걸어 배포 브랜치에 영향을 주지 않고 자유롭게 실험할 수 있다.

실습 2. 핵심 실습: GitLab과 Jenkins를 연결해 코드 푸시가 파이프라인 빌드를 자동으로 트리거하도록 설정하라.

아래 5단계를 모두 완료한다.

1. 범위가 'api'인 GitLab 개인 액세스 토큰을 만든다.

2. Jenkins에 GitLab 플러그인을 설치한다.

3. Jenkins에 GitLab 연결을 생성한다(Manage Jenkins → System → GitLab 섹션). 'Test Connection'이 성공하면 된다.

4. Jenkins 파이프라인 잡의 Build Triggers에서 'Build when a change is pushed to GitLab'을 활성화하고 Push Events를 선택한다.

5. GitLab 프로젝트(Settings → Webhooks)에서 Jenkins의 Build Triggers 섹션에 표시된 URL과 시크릿 토큰을 사용해 웹훅을 만든다.

GitLab의 'Test → Push events' 버튼을 사용해 빌드가 Jenkins에 나타나는지 확인하라.

# 이 실습은 설정 중심이다. 아래 체크리스트로 진행 상황을 추적한다:
# [ ] GitLab 개인 액세스 토큰 생성 (범위: api)
# [ ] Jenkins에 GitLab 플러그인 설치
# [ ] Jenkins에 GitLab 연결 생성 및 'Test Connection' 성공
# [ ] Jenkins 파이프라인 잡에 'Build when pushed to GitLab' 활성화
# [ ] Jenkins에서 복사한 Webhook URL과 시크릿 토큰을 GitLab에 입력
# [ ] GitLab 'Test > Push events' 실행
# [ ] Jenkins 잡 빌드 기록에 새 빌드 등장
힌트
  • GitLab 'Test → Push events'가 비 200 상태 코드를 반환하거나 타임아웃된다면, Jenkins 호스트가 인터넷에서 접근 가능한지 확인한다. GitLab SaaS는 Jenkins에 POST를 보낼 수 있어야 한다. Jenkins를 로컬에서 실행 중이라면 ngrok 같은 터널링 도구가 필요하다. 'ngrok http 8080'을 실행하면 공개 URL을 받을 수 있다.
  • 'Test Connection'이 실패하면 GitLab host URL에 슬래시가 없는지, Credential 종류가 'GitLab API token'인지 확인한다('Username with password'가 아니어야 함).
  • 테스트 이벤트 후 빌드가 나타나지 않으면 GitLab에서 Settings → Webhooks → Edit으로 이동해 'Recent deliveries'를 확인한다. Jenkins가 반환한 응답이 403이면 시크릿 토큰이 일치하지 않는다는 의미다.

연결은 양쪽에서 일치해야 하는 두 가지 사실로 이루어진다. GitLab에 HTTP POST를 어디로 보낼지 알려주는 URL, 그리고 요청이 신뢰할 수 있는 곳에서 왔음을 증명하는 공유 시크릿 토큰. Jenkins가 push 이벤트를 받으면 요청 헤더의 토큰을 검증하고 일치하면 빌드를 시작한다. 따라서 Jenkins에서 GitLab으로 토큰을 정확히 복사하는 것이 중요하다. 불일치하면 403 오류가 반환되고 빌드가 실행되지 않는다. GitLab 플러그인은 빌드 결과를 GitLab 머지 리퀘스트에 다시 보고하는 기능도 제공해 피드백 루프를 닫아 준다.

실습 3. 응용 실습: 실제 코드 푸시로 웹훅 트리거가 엔드투엔드로 작동하는지 증명하라. 'dev' 브랜치에 사소한 변경을 만들어 푸시하고, Jenkins가 빌드를 자동으로 시작하며 콘솔 출력에 'Started by GitLab push'가 표시되는지 확인하라.

# 사소한 변경:

git add README.md
git commit -m "Test webhook trigger"
git push
# 그 다음 Jenkins를 열어 잡의 빌드 기록을 확인한다.
힌트
  • 푸시 후 빌드가 나타나지 않으면 GitLab → Settings → Webhooks → Recent deliveries에서 POST 시도 여부와 Jenkins의 응답을 확인한다.
  • 'dev' 대신 'main'에 푸시했다면 Build Trigger에 설정된 브랜치가 무엇인지 확인한다. 푸시한 브랜치와 일치해야 트리거가 실행된다.

04. Jenkins 설치와 첫 파이프라인(Jenkinsfile)

학습 목표

  • Jenkins를 기동하고 파이프라인 잡을 생성한다.
  • Pipeline as Code(Jenkinsfile)와 stage/step 구조를 이해한다.
  • GitLab에서 소스를 가져오는 checkout 스테이지를 작성한다.

본문

트리거에서 실행으로

이전 강의에서 GitLab이 push마다 Jenkins에 알림을 보내도록 설정했다. 하지만 지금 Jenkins는 그 알림을 받아도 아무 유용한 일을 하지 않는다. 이번 강의에서 Jenkins에 지시사항을 전달한다. 그리고 그 지시사항을 코드로 작성한다. 애플리케이션 코드와 나란히 같은 저장소에 있는 파일로.

Jenkins 실행하기

이 컨테이너 중심 강좌에 가장 잘 맞고 가장 빠르게 Jenkins를 띄우는 방법은 Jenkins 자체를 Docker 컨테이너로 실행하는 것이다.


docker run -d --name jenkins \
  -p 8080:8080 -p 50000:50000 \
  -v jenkins_home:/var/jenkins_home \
  jenkins/jenkins:lts

몇 가지 설명을 덧붙인다. -p 8080:8080은 브라우저로 접속하는 웹 UI를 localhost:8080에 노출한다. -p 50000:50000은 인바운드 빌드 에이전트가 Jenkins 컨트롤러에 연결할 때 사용하는 에이전트(JNLP) 포트다. 나중에 에이전트를 추가할 때 컨테이너를 다시 만들지 않아도 되도록 지금 매핑해 둔다. 두 포트는 서로 다른 역할을 한다는 점에 주의한다. 에이전트 포트를 8080에 연결하면 브라우저와 에이전트가 같은 포트를 놓고 충돌한다. 볼륨 마운트 -v jenkins_home:/var/jenkins_home은 Jenkins 설정과 잡을 지속시켜 컨테이너가 재시작되어도 사라지지 않게 한다. 볼륨 없이 컨테이너를 중지하면 설정이 모두 날아간다. 처음 실행하면 Jenkins가 로그에 초기 관리자 비밀번호를 출력한다(docker logs jenkins). 그 비밀번호로 잠금을 해제하고, 권장 플러그인을 설치하고, 관리자 계정을 만들면 된다.

Jenkins를 컨테이너로 실행하면 호스트가 깔끔하게 유지되고 설정이 재현 가능해진다. 앱에 적용한 것과 같은 철학이다. 단, 볼륨을 잊지 말자. 볼륨 없이 컨테이너를 재시작하면 공들여 설정한 잡이 모두 사라진다.

Freestyle 잡과 파이프라인 — 파이프라인이 더 나은 이유

Jenkins는 여러 잡 유형을 제공한다. Freestyle 잡은 웹 폼을 클릭해 완전히 설정하는 방식이다. 일회성 작업엔 괜찮지만 설정이 Jenkins 서버 안에만 존재한다. Pipeline 잡은 반면 코드로 정의한다.

왜 중요할까? Pipeline as Code는 Freestyle 잡이 줄 수 없는 세 가지를 제공하기 때문이다.

  • 버전 관리 — 빌드 프로세스가 저장소에 있으므로 애플리케이션 코드처럼 모든 변경이 추적되고, 리뷰할 수 있고, 되돌릴 수 있다.
  • 생존성 — Jenkins 서버가 날아가도 파이프라인은 사라지지 않는다. Git에 저장되어 있기 때문이다.
  • 기능 — 파이프라인은 조건문, 반복문, 병렬 실행을 지원한다.

이 코드는 저장소 루트에 커밋하는 Jenkinsfile이라는 파일에 담긴다.

Jenkinsfile의 구조

Jenkins는 두 가지 파이프라인 문법을 지원한다. 스크립트 방식(구형, 유연성 높음)과 선언적 방식(신형, 읽기 쉬움). 이 강좌에서는 선언적 방식을 사용한다. 구조는 단순한 중첩 형태다.

  • pipeline — 모든 것을 감싸는 최외부 블록.
  • agent — 파이프라인이 어디서 실행될지. agent any는 "사용 가능한 Jenkins 워커 어디서나"를 뜻한다.
  • stages — 순서가 있는 스테이지 목록을 담는 컨테이너.
  • stage — 빌드의 논리적 단계에 이름을 붙인 것(예: "Checkout", "Build", "Test"). 스테이지 이름이 Jenkins 스테이지 뷰의 열(column)이 된다.
  • steps — 스테이지 안에서 실행되는 개별 명령.

흐름은 엄격하게 위에서 아래로 진행된다. pipeline이 stages를 포함하고, stages가 각 stage를 포함하고, 각 stage가 steps를 포함한다. 실제로 동작하는 가장 작은 파이프라인은 다음과 같다.


pipeline {
    agent any
    stages {
        stage('Hello') {
            steps {
                echo 'Pipeline is alive!'
            }
        }
    }
}

Jenkins에서 새 Pipeline 잡을 만들고(New Item → Pipeline), 인라인 스크립트 박스에 이것을 붙여 넣고 Build Now를 클릭하면 스테이지 뷰에 초록색 "Hello" 박스가 나타난다. 실패하면 Console Output 링크에 어느 줄에서 문제가 생겼는지 정확히 나온다. 앞으로 모든 문제를 거기서 진단하게 될 테니 익숙해지자.

GitLab에서 Jenkinsfile을 가져오도록 설정하기

스크립트를 인라인으로 붙여 넣는 것은 Pipeline as Code의 목적을 스스로 무너뜨린다. 실제 설정은 "Pipeline script from SCM"이다. 잡 설정의 Pipeline 섹션에서 Pipeline script from SCM을 선택하고, Git을 고른 뒤 GitLab 저장소 URL과 자격증명을 입력하고 브랜치를 설정하고 Script Path를 Jenkinsfile로 지정한다. 이제 Jenkins가 실행될 때마다 저장소에서 직접 파이프라인 정의를 읽는다. 빌드 프로세스 변경이 곧 커밋이 된다는 뜻이다.

Checkout 스테이지 작성하기

실제 파이프라인이 맨 처음 해야 할 일은 최신 코드를 가져오는 것이다. 바로 checkout 스테이지다. Jenkinsfile을 "SCM에서 읽어오는" 방식으로 설정하면 Jenkins는 내장 스텝 하나로 소스를 가져올 수 있다. 이미 어디서 Jenkinsfile을 로드했는지 알고 있으므로 URL을 다시 지정할 필요도 없다.


pipeline {
    agent any
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
    }
}

checkout scm은 이 빌드를 트리거한 정확한 커밋을 클론한다. 다른 저장소를 명시적으로 체크아웃해야 할 때는 다음과 같이 직접 지정할 수 있다.


git branch: 'main',
    url: 'https://gitlab.com/<namespace>/<project>.git',
    credentialsId: 'gitlab-creds'

이 Jenkinsfile을 커밋하고 푸시하면 이전 강의에서 설정한 웹훅이 빌드를 트리거하고, 이번엔 실제로 코드를 가져오는 파이프라인이 실행된다. 루프가 닫혔다. 푸시가 Jenkins를 트리거하고 Jenkins가 저장소에 정의된 파이프라인의 첫 번째 동작으로 코드를 pull한다. 이제부터는 스테이지를 추가하기만 하면 된다.

핵심 정리

  • 영구 볼륨을 사용해 Docker 컨테이너로 Jenkins를 실행하면 깔끔하고 재현 가능한 환경이 만들어진다. 8080:8080은 웹 UI, 50000:50000은 에이전트(JNLP) 포트다.
  • Freestyle 잡보다 Pipeline as Code(Jenkinsfile)를 쓴다. 버전 관리가 되고 서버가 날아가도 남아 있으며 실제 로직을 지원한다.
  • 선언적 파이프라인은 pipeline → agent → stages → stage → steps 형태로 중첩되고, 스테이지 이름이 Jenkins 스테이지 뷰에 표시된다.
  • "Pipeline script from SCM"을 사용해 Jenkins가 저장소에서 Jenkinsfile을 읽게 하고, checkout scm을 첫 번째 스테이지로 만들어 트리거된 커밋을 가져온다.

실습 2. 핵심 실습: 'Hello' 스테이지 하나가 있는 최소한의 선언적 Jenkinsfile을 작성하고, 인라인 스크립트를 사용하는 Pipeline 잡을 Jenkins에 만들어 빌드를 실행한 뒤, Jenkins 스테이지 뷰에서 스테이지가 초록색으로 표시되는지 확인하라.

// Jenkinsfile 내용 (Jenkins 인라인 스크립트 박스에 붙여넣기):
pipeline {
    agent any
    stages {
        stage('___') {
            steps {
                echo '___'
            }
        }
    }
}
힌트
  • 빌드가 'No such DSL method echo' 오류로 실패하면 설정 중에 권장 플러그인이 설치되지 않은 것이다. Manage Jenkins → Plugins → Installed에서 Pipeline 플러그인이 있는지 확인한다.
  • 'Started'는 보이지만 UI에 스테이지 박스가 나타나지 않으면 브라우저를 새로 고쳐야 할 수 있다. Stage View는 Pipeline Stage View 플러그인이 필요하다 — 설치 여부를 확인한다.

// 1. New Item → 이름 입력(예: 'myapp-pipeline') → 'Pipeline' 선택 → OK
// 2. 'Pipeline' 섹션으로 스크롤 → Definition: 'Pipeline script'
// 3. 위 Jenkinsfile을 스크립트 박스에 붙여넣기 → Save
// 4. 'Build Now' 클릭
// 5. 빌드 번호 클릭 → Stage View에 초록색 'Hello' 박스 확인
// 6. 'Console Output' 클릭 → 'Pipeline is alive!' 줄 확인

선언적 파이프라인 문법은 pipeline → agent → stages → stage → steps 순으로 중첩된다. 지정한 모든 스테이지 이름은 Stage View의 열(column)이 되어 빌드의 시각적 진행 바를 만든다. 'agent any'는 Jenkins가 사용 가능한 어느 실행기에서든 이 파이프라인을 실행한다는 뜻이다. 이 간단한 파이프라인조차 핵심 패턴을 보여준다. 'echo' 스텝이 0이 아닌 코드를 반환하면 스테이지가 빨간색이 되고 파이프라인이 중단된다.

실습 3. 핵심 실습: Jenkinsfile을 GitLab 저장소에 커밋하고 Jenkins 잡을 'Pipeline script from SCM'으로 읽도록 재설정하라. 'checkout scm'을 사용하는 'Checkout' 스테이지를 추가한다. 변경을 푸시하고 Jenkins가 저장소에서 Jenkinsfile을 로드해 Checkout 스테이지가 성공적으로 실행되는지 확인하라.

'checkout scm'은 잡에 이미 설정된 동일 저장소와 자격증명을 사용해 빌드를 트리거한 정확한 커밋을 클론하는 단축 스텝이다. Jenkinsfile 안에 저장소 URL을 다시 적지 않아도 되므로 유지 보수 부담이 줄어든다. SCM에서 Jenkinsfile을 읽음으로써 빌드 프로세스가 소스 기록의 일부가 된다. 파이프라인에 대한 모든 변경이 버전 관리되고, 리뷰 가능하고, 애플리케이션 코드처럼 롤백할 수 있다. 이것이 'Pipeline as Code'의 철학이다.

실습 4. 응용 실습: Jenkinsfile에 'Verify Workspace'라는 두 번째 스테이지를 추가하라. 이 스테이지에서 'sh' 스텝으로 Checkout 스테이지가 체크아웃한 파일 목록을 출력하고('ls -la'), 현재 Git 커밋 SHA도 출력한다('git rev-parse --short HEAD'). 업데이트된 Jenkinsfile을 푸시하고 두 스테이지가 모두 초록색으로 표시되며 출력이 콘솔에 보이는지 확인하라.

pipeline {
    agent any
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        stage('Verify Workspace') {
            steps {
                // 파일 목록을 출력하는 sh 명령:
                sh '___'
                // 짧은 커밋 SHA를 출력하는 sh 명령:
                sh '___'
            }
        }
    }
}
힌트
  • 'sh' 명령이 'sh: command not found'를 반환하면 Jenkins 에이전트가 Windows 실행기에서 실행 중일 수 있다. Windows에서는 'sh' 대신 'bat'을 사용한다(bat 'dir' 및 bat 'git rev-parse --short HEAD').
  • 'git rev-parse --short HEAD'가 오류를 반환하면 'checkout scm'이 먼저 실행됐는지 확인한다. git 명령은 워크스페이스에 클론된 저장소가 필요하다.

Jenkinsfile에 스테이지를 추가할 때마다 Stage View에 새 열(column)이 생긴다 — 파이프라인의 시각적 진행 보드다.

05. CI 스테이지: 빌드·테스트·이미지 생성

학습 목표

  • Jenkinsfile에 build와 test 스테이지를 추가하고 실패 시 파이프라인이 중단되게 한다.
  • 테스트 통과 후에만 Docker 이미지를 빌드하는 스테이지를 구성한다.
  • CI가 코드 품질을 자동으로 거르는 역할을 이해한다.

본문

CI의 핵심이 이 스테이지다

지금까지 파이프라인은 푸시를 감지하고 코드를 체크아웃한다. 나름 유용하지만 아직 아무것도 보호하지 않는다. 진정한 가치는 지속적 통합 스테이지에서 나타난다. 모든 변경을 자동으로 검증해서 나쁜 코드를 고객이 발견하기 전에 몇 초 안에 잡아낸다.

핵심 원칙은 빠른 실패(fail fast)다. 빌드가 컴파일조차 안 되면 테스트를 돌릴 필요가 없다. 테스트가 실패하면 이미지를 빌드할 필요가 없다. 각 스테이지는 게이트이고, 닫힌 게이트는 파이프라인 전체를 즉시 멈추고 팀에 알린다. 선언적 Jenkinsfile에서는 이 동작이 기본이다. 스테이지의 어떤 스텝이 0이 아닌 종료 코드를 반환하면 빌드가 실패로 표시되고 나머지 스테이지는 실행되지 않는다.

아래 순서도는 빠른 실패 방식의 게이팅을 보여 준다. 각 스테이지는 성공했을 때만 다음으로 넘어가고, 어디서든 실패하면 이미지가 만들어지기 전에 파이프라인이 멈춘다.

파이프라인을 빨갛게 만드는 테스트 실패는 문제가 아니다. 파이프라인이 가장 중요한 역할을 하는 순간이다. CI의 목표는 실패를 빠르고 싸게 만들어 누구도 무시할 수 없게 하는 것이다.

빌드 스테이지 추가하기

빌드 스테이지는 코드를 의존성과 함께 조립한다. "빌드"의 의미는 언어에 따라 다르다. 컴파일 언어라면 mvn clean package 같은 명령일 수 있고, 인터프리터 언어라면 의존성 설치일 수 있다. 형태는 같다. 명령을 실행하고 0이 아닌 종료 코드가 스테이지를 실패시킨다.


stage('Build') {
    steps {
        sh 'npm ci'
        sh 'npm run build'
    }
}

sh 스텝은 에이전트에서 셸 명령을 실행한다(Windows 에이전트에서는 bat을 사용). npm ci나 npm run build가 실패하면 파이프라인이 여기서 멈춘다. 의존성도 설치하지 못한 코드를 테스트할 이유가 없기 때문이다.

테스트 스테이지 추가하기

다음은 버그를 실제로 잡는 게이트다. 자동화 테스트를 스테이지로 실행하고, 테스트가 하나라도 실패하면 스테이지가 실패하고 이후 모든 것이 취소된다.


stage('Test') {
    steps {
        sh 'npm test'
    }
}

소스 자료에 잘 나온 사례를 여기서 그대로 볼 수 있다. 개발자가 인사말을 "hello world"에서 "hello CI/CD world"로 바꿨지만 테스트가 여전히 이전 값을 검증하고 있었다. 테스트가 실패하고 파이프라인이 빨갛게 됐다. 나쁜 변경이 자동으로 잡혔다. 의도한 대로다. 그 빠른 피드백 루프가 CI의 전부다. 단위 테스트(개별 함수), 통합 테스트(컴포넌트 간 상호작용), 형식/린트 검사 등 다양한 테스트를 실행할 수 있고, 결과를 게시해 Jenkins UI에서 모든 빌드마다 어떤 테스트가 통과하고 실패했는지 볼 수 있다.

유용한 개선: 실행이 실패해도 테스트 리포트를 수집하면 무엇이 잘못됐는지 항상 볼 수 있다. post 블록은 스테이지 결과와 무관하게 실행된다.


stage('Test') {
    steps {
        sh 'npm test'
    }
    post {
        always {
            junit 'reports/**/*.xml'
        }
    }
}

테스트 통과 후에만 이미지 빌드하기

스테이지가 순서대로 실행되고 실패 시 파이프라인이 멈추므로, 이미지 빌드를 테스트 스테이지 다음에 배치하기만 하면 초록 테스트에서만 실행된다는 보장이 생긴다. 이 스테이지가 만드는 산출물이 다음 강의에서 푸시하고 배포할 Docker 이미지다.


stage('Build Image') {
    steps {
        sh 'docker build -t myapp:${BUILD_NUMBER} .'
    }
}

여기서 익혀 둘 실무 습관이 두 가지 있다. 첫째, 파이프라인에서 이미지 태그로 latest를 절대 쓰지 않는다. latest 태그는 어느 빌드의 이미지인지 알 수 없게 만들어 롤백이 불가능해진다. 추적 가능한 고유 값으로 태그한다. 위에서는 ${BUILD_NUMBER} — Jenkins 내장 변수 — 를 사용했다. 다음 강의에서는 코드와 이미지 사이의 연결을 더 강화하기 위해 Git 커밋 SHA로 전환한다. 둘째, Jenkins 에이전트가 docker build를 실행하려면 Docker 데몬에 접근할 수 있어야 한다. 흔한 설정은 Jenkins 컨테이너에 호스트 Docker 소켓 접근 권한을 주어 호스트에서 이미지를 빌드하게 하는 것이다.

지금까지의 전체 그림

새 스테이지를 합치면 Jenkinsfile이 명확하고 순서 있는 품질 게이트로 읽힌다.


pipeline {
    agent any
    stages {
        stage('Checkout')    { steps { checkout scm } }
        stage('Build')       { steps { sh 'npm ci'; sh 'npm run build' } }
        stage('Test')        { steps { sh 'npm test' } }
        stage('Build Image') { steps { sh 'docker build -t myapp:${BUILD_NUMBER} .' } }
    }
}

위에서 아래로 읽으면, 이것이 지속적 통합의 본질이다. 모든 푸시가 자동으로 체크아웃되고, 빌드되고, 테스트되고, 그 모든 것이 성공했을 때만 배포 가능한 이미지가 만들어진다. 사람이 테스트 실행을 기억할 필요가 없고, 테스트를 거치지 않은 코드는 이미지를 만들 수 없다. 그 이미지는 이제 레지스트리에 저장될 준비가 됐고, 다음 강의에서 바로 그 작업을 한다.

핵심 정리

  • CI 스테이지는 파이프라인을 자동 품질 게이트로 만든다. 핵심 원칙은 빠른 실패 — 스테이지가 실패하면 파이프라인이 멈추고 팀에 알린다.
  • 선언적 파이프라인에서는 0이 아닌 종료 코드가 기본으로 스테이지를 실패시키므로, Build → Test → Build Image 순서만 지키면 이미지가 초록 테스트를 통과한 코드에서만 생성된다는 보장이 생긴다.
  • 테스트 스테이지에서 실제 자동화 테스트를 실행하고, 결과를 게시(예: junit)해 모든 빌드의 결과를 볼 수 있게 한다.
  • latest 같은 유동 태그 대신 빌드 번호나 커밋 SHA 같은 고유하고 추적 가능한 값으로 이미지를 태그한다. 실행 중인 컨테이너를 정확히 알고 깔끔하게 롤백하기 위해 필요하다.

🛠 실습

실습 1. 준비 실습: Jenkinsfile에 Node.js 앱에서 'npm ci'를 실행하는 'Build' 스테이지를 추가하라. 'package.json'을 일시적으로 삭제해 의도적으로 실패를 유발하고, Jenkins가 빌드를 FAILED로 표시하고 이후 스테이지로 진행하지 않는지 확인하라. 그런 다음 파일을 복원하고 스테이지가 초록색이 되는지 확인하라.

// 현재 Jenkinsfile (강의 4 버전):

  • package.json을 삭제했는데 푸시 후 빌드가 트리거되지 않는다면 GitLab 웹훅이 여전히 활성 상태인지 확인한다(Settings → Webhooks). 브랜치도 일치해야 한다.

// 1단계: Build 스테이지 추가
pipeline {
    agent any
    stages {
        stage('Checkout') { steps { checkout scm } }
        stage('Build') {
            steps {
                sh 'npm ci'
            }
        }
    }
}

// 2단계: 의도적으로 실패 유발
// package.json을 _package.json으로 이름 바꾸기(또는 삭제)하고 푸시한다.
// Jenkins가 웹훅을 받아 'npm ci'를 실행하면
// package.json이 없어 0이 아닌 종료 코드로 종료된다.
// Stage View에 빨간색 'Build' 박스가 나타난다.
// Console Output이 'npm error enoent'와 'Build FAILURE'로 끝난다.
// Build 이후의 스테이지는 실행되지 않는다 — 빠른 실패 게이트가 작동한 것이다.

// 3단계: 복원
// _package.json을 다시 package.json으로 이름 바꾸고 다시 푸시한다.
// 'Checkout'과 'Build' 박스가 모두 초록색이 된다.

'sh' 스텝의 0이 아닌 종료 코드는 해당 스테이지를 즉시 실패시키고 파이프라인 전체를 중단한다 — 이후 스테이지는 실행되지 않는다. 이것이 '빠른 실패' 원칙이 실제로 작동하는 모습이다. 의존성조차 설치하지 못한 코드를 테스트하거나 패키징할 의미가 없다. 의도적으로 실패를 유발함으로써 프로덕션에서 그 게이트에 의존하기 전에 게이트가 실제로 존재한다는 것을 확인한다.

실습 2. 핵심 실습: Build 스테이지 다음에 'Test' 스테이지를 추가하라. 앱에 의도적으로 실패하는 테스트를 작성하고(예상값이 틀린 단언문), 푸시해서 Jenkins가 테스트 스테이지를 빨갛게 표시하고 'Build Image' 스테이지 앞에서 파이프라인을 중단하는지 확인한다. 그 다음 테스트를 수정하고 다시 푸시해 세 스테이지가 모두 초록색이 되는지 확인하라.

// package.json에 테스트 스크립트 추가:
// "scripts": { "test": "node test.js" }

// test.js 파일을 만들어 간단한 단언문 작성:
const assert = require('assert');
function greet(name) { return 'Hello, ' + name + '!'; }
// 하나는 통과, 하나는 의도적으로 실패하도록 작성:
assert.strictEqual(greet('World'), 'Hello, World!');   // 통과
assert.strictEqual(greet('World'), '___');             // 실패하게 만들기
console.log('All tests passed.');

// Jenkinsfile에 추가할 Test 스테이지:
stage('Test') {
    steps {
        sh '___'
    }
}
힌트
  • 실패하는 단언문이 있어도 'npm test'가 종료 코드 0으로 끝난다면, 'node test.js'가 AssertionError 종료 코드를 실제로 전파하는지 확인한다. Node는 처리되지 않은 AssertionError에서 기본적으로 코드 1로 종료한다 — try/catch로 감싸고 다시 던지지 않으면 종료 코드가 0으로 유지된다.
  • Jenkins에서 테스트 결과를 그래픽으로 보려면 Mocha 같은 JUnit XML 리포트를 생성하는 프레임워크로 교체하고 Test 스테이지에 'post { always { junit "reports/*.xml" } }'를 추가한다.
정답 보기 
// test.js — 실패 버전:
const assert = require('assert');

assert.strictEqual(greet('World'), 'Hello, World!');

  "start": "node index.js",
  "test": "node test.js"
}

// Test 스테이지가 포함된 Jenkinsfile:
pipeline {

assert.strictEqual(greet('World'), 'Hello, World!');
// 다시 푸시: 네 스테이지가 모두 초록색이 된다.

Test 스테이지는 CI 품질 게이트를 실제로 보여 준다. 파이프라인은 첫 번째 실패 스테이지에서 멈추므로, 테스트가 실패하면 'Build Image'가 절대 실행되지 않는다. 나쁜 이미지가 레지스트리에 들어갈 수 없다. 강의에서 정확히 설명한 시나리오다. 개발자가 인사말 문자열을 바꿨지만 단언문 업데이트를 잊은 것을 파이프라인이 즉시 잡아냈다. 수정은 테스트(또는 코드)에 있는 것이지 파이프라인을 약화시키는 것이 아니다.

실습 3. 핵심 실습: Jenkinsfile에 'Build Image' 스테이지를 추가하라. Jenkins BUILD_NUMBER 환경변수로 이미지를 태그한다. 전체 파이프라인(Checkout → Build → Test → Build Image)을 실행하고 Jenkins 호스트의 'docker images'에 빌드 번호가 태그된 이미지가 나타나는지 확인하라.

// Jenkinsfile의 Test 스테이지 다음에 추가:
stage('Build Image') {
    steps {
        sh 'docker build -t myapp:___ .'
    }
}
힌트
  • 'docker build'가 'permission denied while trying to connect to Docker daemon socket' 오류로 실패하면 Jenkins 프로세스가 Docker 소켓에 접근하지 못하는 것이다. '-v /var/run/docker.sock:/var/run/docker.sock' 플래그를 추가해 Jenkins 컨테이너를 다시 만들고 그 안에 Docker CLI를 설치한다.
  • 이미지는 빌드되지만 'docker images'가 로컬 머신에서만 보인다면, Jenkins를 로컬에서 실행 중인 경우 이미지도 같은 머신에 있는 것이다. 원격 에이전트를 사용하는 경우 해당 에이전트에 접속해 확인한다.
정답 보기 
stage('Build Image') {
    steps {
        sh 'docker build -t myapp:${BUILD_NUMBER} .'
    }
}

// 참고: 'docker build'를 실행하려면 Jenkins가 Docker 데몬에 접근할 수 있어야 한다.
// Jenkins를 Docker 컨테이너로 실행 중이라면 호스트 소켓 접근 권한을 부여한다:
// docker run -d --name jenkins \
//   -p 8080:8080 -p 50000:50000 \
//   -v jenkins_home:/var/jenkins_home \
//   -v /var/run/docker.sock:/var/run/docker.sock \
//   jenkins/jenkins:lts
// 그 다음 Jenkins 컨테이너 안에 Docker CLI를 설치한다:
// docker exec -u root jenkins apt-get update
// docker exec -u root jenkins apt-get install -y docker.io

// 파이프라인 실행 후:
// docker images | grep myapp
// 예상 출력:
// myapp   5   a1b2c3d4e5f6   2 minutes ago   145MB
// (여기서 5는 해당 실행의 BUILD_NUMBER)

BUILD_NUMBER는 빌드마다 자동으로 증가하는 Jenkins 내장 환경변수다. 'latest' 대신 이것을 태그로 사용하면 각 이미지에 고유하고 순서가 있는 식별자가 부여된다. 어느 빌드가 어떤 이미지를 만들었는지 항상 알 수 있다. 다음 강의에서는 더 강력한 추적 가능성을 위해 Git 커밋 SHA로 전환한다. '/var/run/docker.sock'을 Jenkins 컨테이너에 마운트하면 Jenkins 프로세스가 호스트의 Docker 데몬을 사용할 수 있다 — 이미지를 빌드해야 하는 CI 파이프라인에서 편리하고 흔한 설정이다.

실습 4. 응용 실습: 동료의 Jenkinsfile에 심각한 순서 문제가 있다. 아래 파이프라인을 분석하고 왜 잘못됐는지 설명한 뒤 올바르게 재작성하라.

문제 있는 Jenkinsfile:


pipeline {
    agent any
    stages {
        stage('Checkout')    { steps { checkout scm } }
        stage('Build Image') { steps { sh 'docker build -t myapp:latest .' } }
        stage('Test')        { steps { sh 'npm test' } }
        stage('Build')       { steps { sh 'npm ci' } }
    }
}

// 1. 이 스테이지들이 어떤 순서로 실행되고 무엇이 잘못되는가?
// 2. 'latest' 태그가 나중에 무엇을 불가능하게 만드는가?
// 3. 올바른 스테이지 순서와 더 나은 이미지 태그를 적용한 수정된 Jenkinsfile을 작성하라.
힌트
  • 문제를 시각화하려면: 스테이지 순서를 흐름도로 그린다. 'Test가 실패하면 이미 무엇이 만들어졌는가?'를 묻는다. 잘못된 버전에서 답은 'Docker 이미지'이고 그것이 문제다.
  • 'latest' 태그 위험을 실제로 보려면: 두 개의 커밋을 빠르게 연속으로 푸시하고 두 빌드가 모두 'myapp:latest'를 태그하려 할 때 두 번째가 첫 번째를 덮어쓰는 것을 지켜본다.

// 1. 스테이지 순서: Checkout → Build Image → Test → Build

//    테스트 게이트가 이미지를 전혀 보호하지 못한다.
//
// 2. 'latest' 태그는 변경 가능하다. 동시에 또는 빠르게 연속으로 두 빌드가 실행되면
//    각각이 'latest' 태그를 덮어쓴다.
//    롤백이 불가능해진다. 실행 중인 컨테이너가 어느 빌드에서 왔는지 알 수 없다.
//
// 수정된 Jenkinsfile:
pipeline {

        stage('Checkout')    { steps { checkout scm } }
        stage('Build')       { steps { sh 'npm ci' } }

// 이제: npm ci와 npm test가 모두 성공했을 때만 이미지가 만들어진다.
// 태그에 BUILD_NUMBER(빌드별 고유값)를 사용해 각 이미지를 구분할 수 있다.

CI 파이프라인에서 스테이지 순서는 단순한 형식이 아니다. 품질 게이트가 무엇을 보호하는지를 정의하는 것이다. 테스트 전에 이미지를 빌드하는 것은 CI의 목적 전체를 무효화한다. 테스트를 거치지 않고 잠재적으로 나쁜 코드가 레지스트리 산출물을 만들 수 있다는 뜻이기 때문이다. '빠른 실패' 보장은 위험한 산출물(이미지)이 검증 검사(테스트) 뒤에 올 때만 유효하다. 'latest' 태그 문제도 마찬가지로 실용적이다. 고유하고 불변하는 태그 없이는 롤백이 추측 게임이 되고, 다중 빌드 환경에서 동시에 진행되는 두 빌드가 서로의 태그를 덮어쓸 수 있다.

06. 이미지 레지스트리에 푸시(AWS ECR)

학습 목표

  • 컨테이너 레지스트리(ECR)의 역할과 빌드된 이미지를 거기에 보관하는 이유를 이해한다.
  • Jenkins에서 ECR 인증을 처리하고 이미지를 푸시한다.
  • 커밋 SHA 기반 이미지 태깅 전략을 세운다.

본문

이미지에 집이 필요하다

이전 강의에서 Jenkins가 Docker 이미지를 빌드했지만, 그 이미지는 빌드한 Jenkins 에이전트 위에만 존재한다. EC2 서버가 Jenkins 안으로 들어가 이미지를 가져올 수는 없다. Jenkins(이미지를 push)와 EC2(이미지를 pull) 양쪽 모두 접근할 수 있는 이미지 저장 공간이 필요하다. 그 공간이 컨테이너 레지스트리다.

Amazon ECR(Elastic Container Registry)은 AWS의 프라이빗 Docker 레지스트리다. Docker Hub와 같은 역할을 하지만 AWS 계정 안에 있어 AWS 권한으로 접근을 제어한다. 어차피 EC2에 배포하는 강좌이니 자연스러운 선택이다. 흐름은 단순하다. Jenkins가 빌드된 이미지를 ECR에 push하고, 나중에 EC2가 그 이미지를 pull해서 실행한다. ECR이 파이프라인의 CI 절반(빌드·테스트)과 CD 절반(배포) 사이의 중간 지점 역할을 한다.

아래 시퀀스 다이어그램은 그 중간 지점을 보여 준다. Jenkins가 인증하고 태그를 붙이고 이미지를 ECR로 push하면, 나중에 EC2가 같은 이미지를 pull한다. 두 절반이 서로 직접 통신하지 않는다.

레지스트리는 이미지를 빌드하는 것과 실행하는 것을 분리한다. Jenkins는 서버에 대해 아무것도 알 필요가 없고, 서버는 빌드 도구가 필요 없다. 완성된 불변 이미지를 태그로 pull하기만 하면 된다. 이 분리 덕분에 동일한 이미지가 빌드에서 스테이징을 거쳐 프로덕션까지 깔끔하게 이동할 수 있다.

1단계 — ECR 저장소 만들기

AWS 콘솔에서 ECR을 열고 저장소를 만든다. 앱 이름(예: myapp)을 붙이면 ECR이 다음 형태의 저장소 URI를 제공한다.


<aws_account_id>.dkr.ecr.<region>.amazonaws.com/myapp

그 전체 URI가 이미지 태그의 일부가 된다. Docker는 이미지 태그만 보고 어디로 push할지 결정하므로, ECR에 push하려면 이미지를 그 URI로 태그해야 한다.

2단계 — Jenkins에서 ECR 인증하기

프라이빗 레지스트리에 push하려면 신원을 증명해야 한다. ECR 인증은 두 단계로 이루어진다. AWS 자격증명으로 단기 Docker 로그인 비밀번호를 얻은 뒤 그것을 Docker에 전달한다.

먼저 Jenkins에 AWS 자격증명을 저장한다(코드에 하드코딩하는 것은 절대 금지). Jenkins의 자격증명 저장소에서 AWS Credentials 종류를 사용하거나 액세스 키 ID와 시크릿을 제공한다. 그런 다음 파이프라인에서 로그인 토큰을 가져와 docker login에 파이프로 연결한다.


aws ecr get-login-password --region <region> \
  | docker login --username AWS --password-stdin \
    <aws_account_id>.dkr.ecr.<region>.amazonaws.com

aws ecr get-login-password는 임시 비밀번호를 반환하고, 그것을 --password-stdin과 함께 docker login에 파이프로 연결하면 빌드 로그에 비밀번호가 출력되지 않은 채로 Docker가 ECR 레지스트리에 인증된다. ECR에서 사용자 이름은 항상 문자 그대로 AWS다.

프로덕션에서 더 깔끔한 방법은 Jenkins 인스턴스에 IAM 역할을 연결해 ECR 권한을 자동으로 갖게 하는 것이다. 저장하거나 교체해야 할 장기 키가 없어진다. 학습용으로는 정적 액세스 키도 괜찮지만, 사용 후 교체하고 권한 범위를 최소화한다.

3단계 — 제값하는 태깅 전략

이전 강의에서 한 약속을 이행할 차례다. latest 같은 태그는 그 이미지에 어떤 코드가 담겨 있는지 아무것도 알려 주지 않는다. 업계 표준은 이미지를 Git 커밋 SHA — 이미지를 만든 정확한 커밋의 고유 지문 — 로 태그하는 것이다. 영구적이고 추적 가능한 연결이 생긴다. 실행 중인 컨테이너가 있으면 정확히 어떤 코드 줄에서 빌드됐는지 찾을 수 있고, 역방향도 마찬가지다.

Jenkinsfile에서 짧은 SHA를 캡처해 태그로 사용할 수 있다.


stage('Push to ECR') {
    steps {
        script {
            def registry = "<aws_account_id>.dkr.ecr.<region>.amazonaws.com"
            def commit = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
            sh """
                aws ecr get-login-password --region <region> \
                  | docker login --username AWS --password-stdin ${registry}
                docker tag myapp:${BUILD_NUMBER} ${registry}/myapp:${commit}
                docker push ${registry}/myapp:${commit}
            """
        }
    }
}

같은 이미지에 두 개의 태그를 함께 push하는 방법도 있다. 추적성과 롤백을 위한 불변 ${commit} 태그와, 브랜치 이름(main) 같은 "이 브랜치의 최신 이미지" 포인터 역할의 유동 태그. 커밋 태그가 정보의 원천이고 브랜치 태그는 편의를 위한 것이다.

4단계 — 푸시 확인하기

파이프라인 실행 후 ECR 콘솔에서 저장소를 열면 커밋 SHA 태그와 "pushed" 타임스탬프가 달린 이미지를 볼 수 있다. 바로 조금 전 빌드하고 테스트한 산출물이 이제 내구성 있는 프라이빗 레지스트리에 저장되어 권한 있는 어떤 서버든 pull할 준비가 됐다는 증거다.

파이프라인의 현 상태

이제 파이프라인은 실행 단계를 빼고 모든 것을 한다. GitLab 푸시가 Jenkins를 트리거해 체크아웃·빌드·테스트·이미지 빌드·ECR 저장까지 한다. 그 이미지에 정확히 무엇이 담겨 있는지 항상 알 수 있도록 태그도 붙어 있다. 남은 유일한 단계는 그 이미지를 EC2에 올려 실행하는 것이고, 그것이 바로 다음 강의에서 완성하는 내용이다.

핵심 정리

  • ECR 같은 컨테이너 레지스트리는 Jenkins가 이미지를 push하고 EC2가 pull할 수 있는 공유 저장소로, 이미지 빌드와 실행을 깔끔하게 분리한다.
  • ECR 인증은 두 단계다. aws ecr get-login-password로 임시 토큰을 얻은 뒤 docker login에 전달한다. 프로덕션에서는 정적 키보다 IAM 역할을 선호한다.
  • Docker는 이미지 태그만 보고 push 대상을 결정하므로, ECR로 push하려면 이미지 태그에 ECR 저장소 URI 전체를 붙인다.
  • 실행 중인 컨테이너와 소스 코드 사이의 영구적이고 추적 가능한 연결을 위해 Git 커밋 SHA로 이미지를 태그한다. 편의를 위해 브랜치 이름 태그를 추가할 수도 있다.

실습 1. 준비 실습: AWS 콘솔이나 AWS CLI로 AWS 계정에 ECR 저장소를 만들어라. 저장소 URI를 확인하고 형식이 '<account-id>.dkr.ecr.<region>.amazonaws.com/<repo-name>'과 일치하는지 검증하라.

실습 2. 핵심 실습: Jenkins 에이전트의 Docker 클라이언트를 ECR에 인증하고, 로컬에서 빌드한 이미지를 ECR 저장소에 수동으로 push하라. ECR 콘솔에 올바른 태그로 이미지가 나타나는지 확인하라.

# 플레이스홀더를 실제 값으로 교체한다:
REGION=<your-region>
ACCOUNT=<your-account-id>
REGISTRY=${ACCOUNT}.dkr.ecr.${REGION}.amazonaws.com

# 1단계: Docker를 ECR에 인증
aws ecr get-login-password --region ${REGION} \
  | docker login --username AWS --password-stdin ${REGISTRY}

# 2단계: 강의 5에서 빌드한 이미지 태그 변경

# 3단계: Push
docker push ${REGISTRY}/myapp:<tag>
힌트
  • 'docker login'이 'no basic auth credentials'로 실패하면 'aws ecr get-login-password' 단계가 자동으로 실패했을 수 있다(예: 리전 오류 또는 IAM 권한 부족). 먼저 'aws ecr get-login-password --region <region>'을 단독으로 실행해 토큰 문자열이 출력되는지 확인한다.
  • push가 'denied: User ... is not authorized to perform: ecr:InitiateLayerUpload'로 실패하면 IAM 사용자/역할에 'AmazonEC2ContainerRegistryPowerUser' 정책(또는 최소한 ecr:GetAuthorizationToken, ecr:BatchCheckLayerAvailability, ecr:PutImage, ecr:InitiateLayerUpload, ecr:UploadLayerPart, ecr:CompleteLayerUpload)이 필요하다.
  • push 중 'no space left on device'가 나타나면 Jenkins 에이전트의 디스크가 꽉 찬 것이다. 'docker system prune'을 실행해 사용하지 않는 이미지와 레이어를 삭제한다.

ECR 인증은 임시 토큰 방식이다. 'aws ecr get-login-password'가 AWS API에 IAM 자격증명으로 접속해 단기(12시간) 비밀번호를 반환한다. 그것을 '--password-stdin'과 함께 'docker login'에 파이프로 연결하면 터미널이나 빌드 로그에 비밀번호를 출력하지 않고 Docker를 ECR 레지스트리에 인증한다. ECR에서 사용자 이름은 항상 문자 그대로 'AWS'다. 커밋 SHA를 태그로 사용하면 영구적이고 양방향 연결이 만들어진다. 실행 중인 이미지 태그가 있으면 정확한 커밋을 찾을 수 있고, 커밋이 있으면 어떤 이미지를 만들었는지 알 수 있다.

실습 3. 핵심 실습: Jenkinsfile에 'Push to ECR' 스테이지를 추가하라. 이 스테이지에서 Git 커밋 SHA를 가져오고, ECR에 인증하고, 이전 'Build Image' 스테이지에서 빌드한 이미지에 커밋 SHA를 태그로 붙여 ECR에 push하고, 선택적으로 브랜치 이름 태그를 편의 포인터로 추가로 push한다.

// Jenkinsfile의 'Build Image' 스테이지 다음에 추가:
stage('Push to ECR') {
    steps {
        script {
            def registry = "<account-id>.dkr.ecr.<region>.amazonaws.com"
            def commit = sh(script: '___', returnStdout: true).trim()
            sh """
                aws ecr get-login-password --region ___ \\
                  | docker login --username AWS --password-stdin ${registry}
                docker tag myapp:${env.BUILD_NUMBER} ${registry}/myapp:${commit}
                docker push ${registry}/myapp:${commit}
            """
        }
    }
}
힌트
  • 파이프라인 내부에서 'git rev-parse --short HEAD'가 SHA 대신 'HEAD'를 반환하면 체크아웃 스텝이 분리된 HEAD를 사용했을 수 있다. Jenkins가 트리거 커밋에서 채우는 'env.GIT_COMMIT?.take(7)'로 교체한다.
  • 커밋 태그는 성공하지만 브랜치 태그가 'image already exists'로 실패하면 괜찮다. 브랜치 태그가 이미 같은 이미지 레이어를 가리킬 수 있다. Docker는 레이어를 다시 업로드하지 않고 태그만 업데이트한다.
  • 'withCredentials' 래퍼는 빌드 로그에서 시크릿을 마스킹한다. Jenkinsfile에 직접 환경변수로 AWS 키를 설정하지 않는다. 파이프라인 설정 UI에 그 값이 노출되기 때문이다.
정답 보기 
stage('Push to ECR') {
    steps {
        script {
            def region   = 'us-east-1'
            def registry = '123456789012.dkr.ecr.us-east-1.amazonaws.com'
            def commit   = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
            def branch   = env.GIT_BRANCH?.tokenize('/')?.last() ?: 'unknown'

            sh """
                aws ecr get-login-password --region ${region} \\
                  | docker login --username AWS --password-stdin ${registry}

                # 불변 태그 — 커밋 SHA
                docker tag myapp:${env.BUILD_NUMBER} ${registry}/myapp:${commit}
                docker push ${registry}/myapp:${commit}

                # 편의 태그 — 브랜치 이름 (선택)
                docker tag myapp:${env.BUILD_NUMBER} ${registry}/myapp:${branch}
                docker push ${registry}/myapp:${branch}
            """
        }
    }
}

// Manage Jenkins → Credentials → Global → Add Credentials
// Kind: AWS Credentials  ID: aws-ecr-creds
// 그런 다음 스테이지를 다음으로 감싼다:
// withCredentials([[\$class: 'AmazonWebServicesCredentialsBinding',
//                   credentialsId: 'aws-ecr-creds']]) { ... }

선언적 Jenkinsfile 안의 'script' 블록에서 Groovy 스크립팅을 사용할 수 있다. 'sh(script: ..., returnStdout: true).trim()'은 셸 명령의 표준 출력을 캡처하고 줄 끝 공백을 제거해 Groovy 문자열 변수로 만든다. 이중 태그 전략(커밋 SHA + 브랜치 이름)은 두 가지 요구를 충족한다. 커밋 SHA 태그는 불변으로 정확한 롤백 대상을 제공하고, 브랜치 태그는 해당 브랜치의 최신 이미지를 가리키는 유동 포인터로 배포 스크립트에서 편리하게 사용할 수 있다.

실습 4. 응용 실습: 버그 리포트를 받았다. '프로덕션에서 latest 태그로 실행 중인 컨테이너가 충돌했는데 어떤 코드 버전인지 알 수 없다.' 이 상황이 'latest' 태그로 생긴 이유를 직접 작성으로 설명하고(Jenkinsfile 주석 블록 또는 메모), 대신 커밋 SHA 태깅 전략을 사용한 경우 세 번의 빌드 후 'docker images'와 ECR 저장소가 어떻게 보일지 예시로 보여라.

// Part 1 — 직접 설명 작성 (주석 블록으로 추가):
// 다음을 설명한다:
// a) 'latest'가 코드 버전 식별을 불가능하게 만드는 이유
// b) 'latest'로 롤백이 신뢰할 수 없는 이유
// c) 커밋 SHA 태깅이 두 문제를 모두 해결하는 방법

// Part 2 — 예시 SHA를 사용한 세 번의 빌드 후 예상 'docker images' 출력:
// 빌드 1: 커밋 a1b2c3d
// 빌드 2: 커밋 e4f5a6b
// 빌드 3: 커밋 7c8d9e0
힌트
  • 실행 중인 컨테이너에서 어떤 코드가 빌드됐는지 확인하려면(SHA 태그 사용 시): 'docker inspect <container-name>'을 실행하고 'Image' 필드를 확인한다. 태그의 SHA는 'git log <sha>'로 검사할 수 있는 git 커밋에 직접 매핑된다.
  • ECR에서 '이미지 태그 불변성(image tag immutability)'을 활성화하면 기존 태그를 덮어쓰지 못하게 막는다. 프로덕션에서 이 설정을 활성화해 레지스트리 수준에서 태깅 원칙을 강제한다.
정답 보기 
// Part 1 — 설명:
//
// a) 'latest'는 변경 가능한 태그다 — 새 빌드가 생길 때마다 덮어쓴다.
//    컨테이너가 충돌하면 'docker inspect <container>'를 실행해도
//    'Image: myapp:latest'만 보인다. 언제 빌드됐는지,
//    어떤 커밋을 담고 있는지 전혀 알 수 없다.
//    커밋 SHA 태그를 사용하면 'docker inspect'가 'myapp:e4f5a6b'를 보여주고
//    'git show e4f5a6b'로 실행 중인 정확한 코드 줄을 확인할 수 있다.
//
// b) 'latest'로 롤백하면 추측 게임이 된다. 이전 빌드도 'latest'를 push했다면
//    그 이미지는 태그에서 사라진 것이다. 복구하려면 재빌드해야 하고
//    시간이 걸리고 동일한 나쁜 코드가 다시 들어올 수 있다.
//    커밋 SHA 태그를 사용하면 과거 이미지가 ECR에 각자의 불변 태그로
//    존재한다. 롤백은 'docker pull myapp:a1b2c3d' — 즉시 완료된다.
//
// c) 커밋 SHA 태깅은 영구적이고 양방향 인덱스를 만든다:
//    코드 커밋 ↔ 이미지 태그.
//    실행 중인 컨테이너가 있으면 정확한 소스를 찾을 수 있다.
//    커밋이 있으면 그 이미지와 그것을 실행하는 서버를 찾을 수 있다.

// Part 2 — 세 번의 빌드 후 예상 상태:

// Jenkins 에이전트의 docker images:
// REPOSITORY   TAG       IMAGE ID       CREATED
// myapp        a1b2c3d   img1abc...     3시간 전
// myapp        e4f5a6b   img2def...     2시간 전
// myapp        7c8d9e0   img3ghi...     1시간 전

// ECR 저장소 (각자 불변 태그로 이미지 3개):
// Tag         Push 시각
// a1b2c3d     3시간 전
// e4f5a6b     2시간 전
// 7c8d9e0     1시간 전
//
// 빌드 1로 롤백:
// docker pull 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:a1b2c3d
// docker run ... myapp:a1b2c3d
// 재빌드 불필요. 이미지가 ECR에 그대로 있다.

이 실습은 강의에서 소개한 태깅 원칙을 강화한다. 실행 중인 이미지의 소스 코드를 식별할 수 없다는 것은 'latest'를 파이프라인에서 사용할 때 반복적으로 발생하는 실제 프로덕션 문제다. 커밋 SHA 태그는 이미지 레지스트리를 감사 추적으로 만든다. 모든 배포가 추적 가능하고, 모든 롤백이 즉각적이며(재빌드 불필요), 어떤 버전도 조용히 덮어쓰이지 않는다. 불변성 보장은 보안 스캔에도 유용하다. 'myapp:a1b2c3d'를 스캔한 결과는 항상 그 정확한 이미지에 적용된다 — 'latest'는 스캔할 때마다 다른 이미지일 수 있다.

모든 준비가 끝났다. 이미지가 ECR에 들어 있고, 이제 그것이 EC2 서버에서 실행되어야 한다. 이번 강의에서 네트워크를 통해 서버까지 닿아 이전 버전을 새 버전으로 자동 교체하는 배포 스테이지를 만든다. 모든 성공 빌드에서 자동으로. 파이프라인이 현실이 되는 순간이다. git push 한 번에 몇 분 후 새 버전이 live 상태가 된다.

EC2 서버 준비하기

Jenkins가 배포하기 전에 EC2 인스턴스에 세 가지가 준비되어야 한다.

1. Docker와 Compose 플러그인 설치 — 이미지를 pull하고 컨테이너를 실행하기 위해.

2. ECR에서 pull할 권한 — EC2 인스턴스에 IAM 역할을 연결해 ECR 읽기 권한을 부여하는 것이 깔끔한 방법이다. 저장된 키 없이 서버가 ECR에 인증할 수 있다.

3. docker-compose.yml 파일 — 앱을 어떻게 실행할지 기술하는 파일(사용할 이미지, 포트, 환경 변수 등).

서버에 둘 최소한의 compose 파일은 다음과 같다.


services:
  web:
    image: ${IMAGE}
    ports:
      - "80:8080"
    restart: always

이미지가 변수 ${IMAGE}로 되어 있음에 주목하라. Jenkins가 배포 시점에 정확한 ECR 이미지 태그를 공급하므로 같은 compose 파일이 파이프라인이 방금 빌드한 버전을 항상 실행한다.

Jenkins에서 EC2로 SSH 연결하기

Jenkins는 직접 접속하듯 SSH(Secure Shell)를 통해 EC2에 접근한다. 인스턴스의 프라이빗 키를 사용하는데, 전문적인 방법은 그 키를 Jenkins의 자격증명 저장소에 SSH 자격증명으로 저장하는 것이다. Jenkinsfile에 직접 붙여 넣거나 Git에 커밋하는 것은 절대 안 된다.

대부분의 튜토리얼이 넘어가는 신뢰의 두 번째 측면이 있다. SSH는 서버의 신원도 검증한다. 클라이언트가 처음 연결할 때 SSH는 서버의 호스트 키를 known_hosts라는 파일에 기록한다. 이후 연결 시마다 서버가 같은 키를 제시하는지 확인한다. 이 확인이 중간자(MITM) 공격을 막는다. 공격자가 연결을 가로채고 서버인 척하는 것을 차단한다. 첫 번째 자동 배포 전에 EC2 호스트 키를 Jenkins의 SSH 에이전트에 등록해야 한다.


# Jenkins 호스트에서 한 번만 실행한 뒤 지문을 대역 외(out-of-band)에서 검증한다
# (예: EC2 콘솔과 비교)
ssh-keyscan -H <EC2_HOST> >> ~/.ssh/known_hosts

호스트 키 검증은 선택적 마무리 작업이 아니다. 공격자가 서버인 척 접속하는 것을 막는 유일한 수단이다. known_hosts에 키를 한 번 등록해 두면 SSH는 키가 바뀔 경우 연결을 거부한다.

SSH Agent 플러그인은 파이프라인이 블록 실행 동안 저장된 키를 빌릴 수 있게 한다. 호스트 키가 이미 known_hosts에 있으면 배포 블록이 검증을 약화시키는 플래그 없이 안전하게 연결한다.


stage('Deploy to EC2') {
    steps {
        sshagent(['ec2-ssh-key']) {
            sh """
                ssh ec2-user@<EC2_HOST> '
                    aws ecr get-login-password --region <region> \
                      | docker login --username AWS --password-stdin <registry>
                    export IMAGE=<registry>/myapp:${commit}
                    docker compose -f /home/ec2-user/docker-compose.yml pull
                    docker compose -f /home/ec2-user/docker-compose.yml up -d
                '
            """
        }
    }
}

많은 튜토리얼에서 -o StrictHostKeyChecking=no를 볼 수 있다. 이 옵션은 호스트 키 검증을 비활성화해 처음 연결이 "그냥 작동"하게 만들지만, 그 주소에 응답하는 어떤 서버에도 SSH가 연결하게 되어 MITM 공격 보호가 완전히 사라진다. 기본값으로 쓰지 않는다. known_hosts를 미리 채우는 방법을 쓰고, 임시 학습 환경에서만 완화를 고려하되 무엇을 포기하는지 정확히 이해한 뒤 사용한다.

SSH 프라이빗 키는 서버 비밀번호와 같다. Jenkins 자격증명에 저장하고 이 잡에 대해서만 범위를 지정하며 절대 저장소나 빌드 로그에 닿지 않게 한다. 배포 키가 유출되면 서버가 유출된 것이다.

배포 스테이지가 실제로 하는 일

SSH 세션 안에서 실행되는 명령들은 함께 배포를 구성하므로 하나씩 살펴보자.

1. ECR 로그인 — EC2 서버가 프라이빗 이미지를 pull할 수 있도록 인증한다(IAM 역할 덕분에 임시 로그인 토큰 외에 키가 필요 없다).

2. IMAGE 변수 설정 — 파이프라인이 방금 빌드한 커밋 태그 이미지의 정확한 주소로 설정한다. compose 파일이 정확히 무엇을 실행할지 알 수 있다.

3. docker compose pull — ECR에서 새 이미지를 다운로드한다.

4. docker compose up -d — 이것이 교체 명령이다. Compose가 실행 중인 컨테이너와 원하는 상태를 비교하고, 이미지가 바뀌었으므로 이전 컨테이너를 정상적으로 중지하고 새 이미지에서 새 컨테이너를 시작한다. -d는 백그라운드로 실행해 SSH 세션에 즉시 제어권을 반환한다.

아래 시퀀스 다이어그램은 Jenkins가 EC2에 열어 실행하는 SSH 세션 안의 네 명령을 보여 준다.

docker compose up -d의 장점은 선언적이라는 점이다. 원하는 최종 상태를 기술하면 Compose가 무엇을 변경할지 알아서 처리한다. 컨테이너를 올바른 순서로 수동으로 중지·삭제·재실행할 필요가 없다.

완성된 파이프라인

이 배포 스테이지를 push 스테이지 뒤에 추가하면 Jenkinsfile이 전체 여정을 담는다.


pipeline {
    agent any
    stages {
        stage('Checkout')      { steps { checkout scm } }
        stage('Build & Test')  { steps { sh 'npm ci'; sh 'npm test' } }
        stage('Build Image')   { /* docker build, 커밋 SHA로 태그 */ }
        stage('Push to ECR')   { /* 로그인 + docker push */ }
        stage('Deploy to EC2') { /* sshagent → compose pull + up -d */ }
    }
}

이제 이 강좌 전체가 쌓아 온 목표를 실행해 보자. 앱을 수정하고, 커밋하고, GitLab에 push한다. 웹훅이 실행되고, Jenkins가 모든 스테이지를 하나씩 진행하고, 터미널 한 번 건드리지 않고 EC2 서버가 새 이미지를 pull해 서비스한다. 서버 URL을 열면 변경 사항이 live 상태다.

완전한 CI/CD 파이프라인이 완성됐다. 한 가지 견고함이 더 필요하다. 지금 이 상태에서 새 버전이 망가져 있어도 그대로 배포된다. 마지막 강의에서 헬스체크, 롤백, 적절한 시크릿 처리를 추가해 이 파이프라인을 프로덕션에 적합한 수준으로 만든다.

핵심 정리

  • 배포 스테이지는 SSH로 EC2에서 명령을 실행한다. 프라이빗 키는 Jenkins SSH 자격증명으로 저장하고(SSH Agent 플러그인 사용 권장) 절대 저장소에 두지 않는다.
  • 서버도 검증한다. known_hosts에 EC2 호스트 키를 등록해(예: ssh-keyscan) SSH가 중간자를 감지할 수 있게 한다. StrictHostKeyChecking=no를 기본으로 쓰지 않는다. 그 보호를 비활성화하는 것이기 때문이다.
  • EC2에 Docker, ECR pull 권한을 위한 IAM 역할, image 변수를 사용하는 docker-compose.yml을 준비한다.
  • 교체는 SSH 세션 안의 두 명령이다. docker compose pull → docker compose up -d. Compose가 선언적으로 이전 컨테이너를 새 컨테이너로 교체한다.
  • 이 스테이지가 추가되면 단일 git push가 수동 단계 없이 live EC2 배포까지 자동으로 흘러간다.

🛠 실습

실습 1. 준비 실습: EC2 서버를 자동 배포에 맞게 준비하라. EC2 인스턴스에 SSH로 접속해 세 가지 설정을 완료한다. (1) Docker와 Compose 플러그인 설치, (2) ECR 읽기 권한을 가진 IAM 역할을 인스턴스에 연결, (3) IMAGE 환경변수를 사용하는 docker-compose.yml 파일 생성.

# EC2에 접속:
ssh -i <your-key.pem> ec2-user@<EC2_PUBLIC_IP>

# 1번: Docker 설치
# (Amazon Linux 2023 명령)
sudo dnf update -y
sudo dnf install -y docker
sudo systemctl start docker
sudo systemctl enable docker
sudo usermod -aG docker ec2-user

# Compose 플러그인 설치:
sudo mkdir -p /usr/local/lib/docker/cli-plugins
sudo curl -SL \
  https://github.com/docker/compose/releases/download/v2.27.0/docker-compose-linux-x86_64 \
  -o /usr/local/lib/docker/cli-plugins/docker-compose
sudo chmod +x /usr/local/lib/docker/cli-plugins/docker-compose

# 확인:
docker compose version

# 2번: IAM 역할 — AWS 콘솔에서 설정 (SSH 세션 외부에서)
# EC2 → Instances → 인스턴스 선택 → Actions → Security → Modify IAM role
# AmazonEC2ContainerRegistryReadOnly 정책이 포함된 역할 연결

# 3번: docker-compose.yml 생성 (image 변수 플레이스홀더 채우기):
cat > ~/docker-compose.yml << 'EOF'
services:
  web:
    image: ___
    ports:
      - "80:8080"
    restart: always
EOF
힌트
  • 'docker compose version'이 설치 후 'command not found'를 표시하면 로그아웃 후 다시 로그인한다(docker 그룹 멤버십 변경은 새 세션이 필요). 또는 'newgrp docker'를 실행한다.
  • IAM 역할 연결 후에도 'aws ecr describe-repositories'가 'Unable to locate credentials'를 반환하면 30초 대기 후 다시 시도한다. 인스턴스 메타데이터 서비스가 역할을 전파하는 데 시간이 걸릴 수 있다.
  • EC2 인스턴스가 Amazon Linux가 아닌 Ubuntu를 사용한다면 'dnf' 대신 'apt-get'을 사용하고 'sudo apt-get install -y docker.io'로 Docker를 설치한다.

cat > /home/ec2-user/docker-compose.yml << 'EOF'
services:

# 작성 확인:
cat /home/ec2-user/docker-compose.yml

# 예상 출력:
# services:
#   web:
#     image: ${IMAGE}
#     ports:
#       - "80:8080"
#     restart: always

# Docker Compose 버전 확인 (v2.x 이상이어야 함):
docker compose version
# 예상: Docker Compose version v2.27.0

# IAM 역할 연결 후 ECR 접근 가능 여부 확인:
aws ecr describe-repositories --region <region>
# 'Unable to locate credentials' 없이 JSON 목록이 반환되면 성공

세 가지 준비는 서로 연결되어 있다. Docker는 컨테이너 실행에 필요하고, IAM 역할은 장기 키 저장 없이 서버가 ECR에서 이미지를 pull할 수 있게 하며, '${IMAGE}'를 사용하는 docker-compose.yml은 Jenkins가 배포 시 IMAGE 환경변수를 설정하기만 하면 되는 안정적인 단일 파일을 제공한다. 하드코딩된 태그 대신 '${IMAGE}'를 쓰면 compose 파일을 절대 수정할 필요가 없다. Jenkins가 변수만 채운다.

실습 2. 핵심 실습: EC2 SSH 프라이빗 키를 Jenkins 자격증명(종류: SSH Username with private key)으로 저장하고, EC2 호스트 키를 Jenkins의 known_hosts 파일에 등록하라. 그런 다음 Jenkins 에이전트에서 EC2 인스턴스로 테스트 SSH 명령을 실행해 연결을 검증하라.

# 1단계 — Jenkins에 EC2 SSH 키 저장:
# Manage Jenkins → Credentials → Global → Add Credentials
# Kind: SSH Username with private key
# ID: ec2-ssh-key
# Username: ec2-user
# Private Key: Enter directly → <your-key.pem> 내용 붙여넣기

# 2단계 — Jenkins의 known_hosts에 EC2 호스트 키 등록:
# Jenkins 컨테이너 안에서 실행:
docker exec -u jenkins jenkins ssh-keyscan -H <EC2_PUBLIC_IP>
# 출력 줄을 복사하고 known_hosts에 추가:
docker exec -u jenkins jenkins \
  bash -c 'mkdir -p ~/.ssh && ssh-keyscan -H <EC2_PUBLIC_IP> >> ~/.ssh/known_hosts'

# 3단계 — 테스트 파이프라인 스테이지로 검증:
# (아래 인라인 스크립트로 임시 파이프라인 잡 생성)
stage('Test SSH') {
    steps {
        sshagent(['ec2-ssh-key']) {
            sh 'ssh ec2-user@<EC2_PUBLIC_IP> "echo SSH connection successful && uname -n"'
        }
    }
}
힌트
  • 'sshagent' 스텝을 찾을 수 없으면 Manage Jenkins → Plugins에서 SSH Agent 플러그인을 설치한다.
  • SSH 연결이 'Host key verification failed'로 실패하면 known_hosts 파일이 채워지지 않았거나 IP가 바뀐 것이다. Jenkins 컨테이너 안에서 ssh-keyscan 명령을 다시 실행하고 출력이 추가됐는지 확인한다(cat ~/.ssh/known_hosts).
  • 'ECDSA host key for <IP> has changed'가 나타나면 EC2 인스턴스가 중지 후 재시작됐을 수 있다(AWS가 시작 시 새 호스트 키를 부여할 수 있음). Jenkins 컨테이너 안에서 이전 항목을 제거하고('ssh-keygen -R <EC2_IP>') ssh-keyscan을 다시 실행한다.

Jenkins 자격증명으로 키를 저장하면(Jenkinsfile이나 저장소 파일이 아닌) Jenkins가 암호화하고 빌드 로그에 절대 노출되지 않는다. known_hosts 등록은 대부분의 튜토리얼이 생략하는 보안 단계다. 없으면 Jenkins가 연결을 거부하거나(StrictHostKeyChecking이 켜져 있을 때) 그 주소의 어떤 서버든 자동으로 수락한다(꺼져 있을 때). 지문을 수동으로 등록하면 Jenkins가 모든 자동 연결에서 서버 신원을 검증할 수 있다 — 중간자 공격 방어다.

실습 3. 핵심 실습: Jenkinsfile에 'Deploy to EC2' 스테이지를 추가하라. 이 스테이지에서 EC2에 SSH로 접속해 ECR 로그인, IMAGE 변수 설정, docker compose pull, docker compose up -d의 네 명령을 실행한다. 코드 변경을 push하고 새 버전이 EC2 서버의 공인 IP 포트 80에서 live 상태인지 확인하라.

// Jenkinsfile의 'Push to ECR' 다음에 추가:
stage('Deploy to EC2') {
    steps {
        sshagent(['ec2-ssh-key']) {
            sh """
                ssh ec2-user@<EC2_HOST> '
                    aws ecr get-login-password --region ___ \\
                      | docker login --username AWS --password-stdin ___
                    export IMAGE=___/myapp:${commit}
                    docker compose -f /home/ec2-user/docker-compose.yml pull
                    docker compose -f /home/ec2-user/docker-compose.yml up -d
                '
            """
        }
    }
}
힌트
  • 'docker compose pull'이 'unauthorized'로 실패하면 EC2의 IAM 역할에 ECR 읽기 권한이 없거나 'aws ecr get-login-password' 명령이 실패한 것이다. EC2에 수동 SSH로 접속해 두 명령을 직접 실행해 디버깅한다.
  • 'docker compose up -d'가 성공하지만 포트 80이 응답하지 않으면 EC2 보안 그룹이 0.0.0.0/0에서 포트 80 인바운드 TCP를 허용하는지 확인한다.
  • Jenkins의 SSH 명령이 'Pseudo-terminal will not be allocated because stdin is not a terminal'로 실패하면 ssh 플래그에 '-tt'를 추가해야 할 수 있다. 'ssh -tt ec2-user@...' — 대부분의 명령은 PTY 없이도 잘 작동하지만.
정답 보기 
// Deploy 스테이지가 포함된 완성된 Jenkinsfile:
pipeline {
    agent any
    stages {
        stage('Checkout')    { steps { checkout scm } }
        stage('Build')       { steps { sh 'npm ci' } }
        stage('Test')        { steps { sh 'npm test' } }
        stage('Build Image') { steps { sh 'docker build -t myapp:${BUILD_NUMBER} .' } }
        stage('Push to ECR') {
            steps {
                script {
                    def region   = 'us-east-1'
                    def registry = '123456789012.dkr.ecr.us-east-1.amazonaws.com'
                    def commit   = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
                    sh """
                        aws ecr get-login-password --region ${region} \\
                          | docker login --username AWS --password-stdin ${registry}
                        docker tag myapp:${env.BUILD_NUMBER} ${registry}/myapp:${commit}
                        docker push ${registry}/myapp:${commit}
                    """
                }
            }
        }
        stage('Deploy to EC2') {
            steps {
                script {
                    def region   = 'us-east-1'
                    def registry = '123456789012.dkr.ecr.us-east-1.amazonaws.com'
                    def commit   = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
                    sshagent(['ec2-ssh-key']) {
                        sh """
                            ssh ec2-user@<EC2_PUBLIC_IP> '
                                aws ecr get-login-password --region ${region} \\
                                  | docker login --username AWS --password-stdin ${registry}
                                export IMAGE=${registry}/myapp:${commit}
                                docker compose -f /home/ec2-user/docker-compose.yml pull
                                docker compose -f /home/ec2-user/docker-compose.yml up -d
                            '
                        """
                    }
                }
            }
        }
    }
}

// 검증:
// 1. 코드 변경 push (예: index.js의 응답 텍스트 변경)
// 2. Jenkins에서 다섯 스테이지가 모두 초록색이 되는 것 확인
// 3. 브라우저에서 http://<EC2_PUBLIC_IP>:80 열기
//    — 새 응답 텍스트가 보여야 함
// 4. EC2에서 실행 중인 컨테이너 확인: docker ps

배포 스테이지는 하나의 SSH 세션 안에서 네 명령을 실행한다. ECR 로그인은 EC2 서버가 프라이빗 이미지를 pull할 권한을 부여한다. IMAGE 환경변수 설정은 docker-compose.yml의 변수 치환을 채운다. 'docker compose pull'은 ECR에서 새 이미지 레이어를 다운로드한다. 'docker compose up -d'는 선언적이다. Compose가 실행 중인 상태(이전 이미지)와 원하는 상태(새 IMAGE 값)를 비교하고 변경된 것만 교체한다. 이전 컨테이너를 수동으로 중지·삭제·재시작하는 순서를 지정할 필요가 없다. '-d'는 Compose가 교체를 시작한 뒤 SSH 세션에 즉시 제어권을 반환한다.

실습 4. 응용 실습: 의미 있는 코드 변경을 만들어(예: HTTP 응답에 현재 Git 커밋 SHA 포함) GitLab에 push하고 파이프라인 전체를 추적한다. 배포된 정확한 이미지 태그를 기록하고 'git show <tag>'로 해당 커밋을 확인할 수 있는지 증명하라.

// 1단계: 빌드 시점에만 알 수 있는 커밋 SHA를 응답에 포함하려면
// 빌드 인수(build argument)로 전달한다:

// Dockerfile 추가:
// ARG COMMIT_SHA=unknown
// ENV COMMIT_SHA=${COMMIT_SHA}

const sha = process.env.COMMIT_SHA || 'unknown';

힌트
  • 컨테이너에서 'process.env.COMMIT_SHA'가 'undefined'라면 '--build-arg'가 'docker build'에 전달됐는지, Dockerfile에 ARG와 ENV 두 줄이 모두 COMMIT_SHA에 대해 있는지 확인한다. ARG만으로는 런타임까지 값이 유지되지 않는다.
  • 브라우저가 배포 후에도 이전 응답을 보여준다면 브라우저가 캐싱하는 것일 수 있다. 강력 새로고침(Ctrl+Shift+R)을 시도하거나 커맨드라인에서 'curl http://<EC2_IP>/'로 테스트한다.

FROM node:20-alpine
WORKDIR /app
COPY package.json ./
RUN npm install
COPY . .
ARG COMMIT_SHA=unknown
ENV COMMIT_SHA=${COMMIT_SHA}
EXPOSE 8080
CMD ["node", "index.js"]

// 업데이트된 index.js:
const http = require('http');
const server = http.createServer((req, res) => {
  const sha = process.env.COMMIT_SHA || 'unknown';
  res.writeHead(200);
  res.end(`Hello from container! Build: ${sha}`);
});
server.listen(8080, () => console.log('Listening on port 8080'));

// Jenkinsfile의 'Build Image' 스텝 업데이트:
sh 'docker build --build-arg COMMIT_SHA=$(git rev-parse --short HEAD) -t myapp:${BUILD_NUMBER} .'

// 파이프라인 완료 후:
// 1. Jenkins 콘솔에 표시된 커밋 SHA 메모 (예: 'abc1234')
// 2. 브라우저에서 http://<EC2_PUBLIC_IP>/ 열기:
//    응답: 'Hello from container! Build: abc1234'
// 3. 연결 확인:
//    git show abc1234
//    index.js 변경이 담긴 커밋이 표시됨 — SHA가 정확히 일치
// 4. ECR 태그 확인:
//    aws ecr describe-images --repository-name myapp \
//      --query 'imageDetails[].imageTags'
//    'abc1234'가 포함되어야 함

이 실습은 추적 가능성 루프를 실제로 완성한다. 실행 중인 컨테이너의 HTTP 응답에 커밋 SHA가 표시되고, ECR 태그가 같은 SHA를 사용하고, 'git show <SHA>'가 정확한 소스 코드로 매핑된다. 커밋 SHA 태깅 전략의 가치가 구체적으로 드러난다. 엔드포인트를 누르는 사용자가 어떤 코드 버전이 서비스하는지 볼 수 있고, 운영자는 소스와 빌드를 즉시 찾을 수 있다. 프로덕션에서는 이 SHA가 주 응답 대신 '/version' 엔드포인트나 로그로 노출되는 것이 일반적이지만 메커니즘은 동일하다.

08. 무중단 전환·롤백·시크릿/환경변수 관리

학습 목표

  • 헬스체크로 배포 성공 여부를 판단하고 실패 시 이전 이미지로 롤백한다.
  • 환경변수와 시크릿을 코드와 분리해 안전하게 주입하는 방법을 익힌다.
  • 완성된 파이프라인을 점검하고 내 코드에 적용하는 방법을 정리한다.

본문

파이프라인에 아직 남은 빈 곳

이전 강의의 파이프라인은 작동한다. 하지만 낙관적이다. 새 이미지를 pull하고 컨테이너를 교체하고 모든 것이 잘 됐다고 가정한다. 새 버전이 시작 시 충돌하거나 응답하지 않으면, 작동하는 앱을 망가진 앱으로 교체했는데 아무도 모른다. 어떤 테스트도 나쁜 배포의 마지막 1% 가능성을 완전히 없애지 못한다. 이번 강의에서 작동하는 파이프라인을 신뢰할 수 있는 파이프라인으로 만드는 안전망을 추가한다.

1단계 — 헬스체크: 믿지 말고 검증하라

헬스체크는 방금 배포된 앱에 "실제로 살아 있고 서비스 중이야?"라고 묻는 작은 테스트다. 단순한 방법은 몇 초 sleep하고 나서 프로브를 보내는 것이지만, 고정 sleep은 경쟁 조건(race condition)이다. 너무 짧으면 앱이 준비되기 전에 프로브를 보내(오탐 실패), 너무 길면 모든 배포에서 시간을 낭비한다. 견고한 방법은 앱이 응답하거나 시도 횟수가 소진될 때까지 재시도로 엔드포인트를 폴링하는 것이다. 앱이 준비되는 즉시 성공하고, 진짜 제한된 타임아웃 후에만 실패한다.

bash로 그 폴링 루프를 직접 작성할 수도 있지만 Docker Compose가 이미 해준다. 서비스에 컨테이너 수준 healthcheck를 정의하면 Compose가 자체 스케줄로 프로브를 실행하고 각 컨테이너의 헬스 상태를 추적한다.


services:
  web:
    image: ${IMAGE}
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 10s
      timeout: 3s
      retries: 3

healthcheck가 정의되면 docker compose up -d --wait으로 배포를 시작한다. --wait 플래그는 모든 서비스가 healthy를 보고할 때까지 블로킹했다가 0으로 종료한다. 서비스가 재시도 횟수 안에 healthy 상태가 되지 않으면 0이 아닌 값으로 종료한다. 이 명령 하나가 곧 헬스체크다. 폴링, 제한된 타임아웃, 통과/실패 판정을 한 번에 처리한다. 따라서 파이프라인은 그 종료 코드만 믿으면 된다.

--wait는 서비스에 healthcheck가 정의된 경우에만 작동한다. 없으면 Compose는 컨테이너가 시작되기만 기다릴 뿐 healthy 상태가 되길 기다리지 않는다. healthcheck를 정의하고 docker compose up -d --wait를 실행해 그 종료 코드로 성공/실패를 판정한다. 앱 시작 시간과 경쟁하는 고정 sleep은 절대 쓰지 않는다.

2단계 — 롤백: 검증된 마지막 버전으로

헬스체크가 실패하면, 즉 docker compose up -d --wait이 0이 아닌 값으로 종료하면 어떻게 할까? 롤백한다. 이전 작동 이미지를 다시 배포한다. 강의 6에서 커밋 SHA로 이미지를 태그한 이유가 바로 이것이다. 과거 모든 버전이 ECR에 각자의 불변 태그로 남아 있어서 "이전 버전"은 항상 docker compose up 한 번으로 닿을 수 있다.

까다로운 점은 파이프라인이 마지막으로 작동했던 태그를 기억해야 한다는 것이다. failure 블록이 그것을 마법처럼 알 수는 없다. 방법은 이것이다. 검증이 완료된 배포에서만 배포된 태그를 서버의 파일에 기록하고, 실패 시 그 파일을 읽어 다시 배포한다. 그 파일이 "이번 시도 이전에 작동했던 것"의 내구성 있는 기록이다.

아래 순서도는 그 판단을 보여 준다. 배포 명령과 헬스체크가 이제 하나의 단계다. docker compose up -d --wait이 0으로 종료하면(healthy) 새 버전을 유지하고 태그를 기록한다. 0이 아닌 값으로 종료하면(unhealthy) 저장된 태그를 읽어 다시 배포한다.

작동하는 로직은 다음과 같다. docker compose up -d --wait이 배포이자 검증이다. 성공으로 반환되면 스크립트가 계속 진행되어 새 태그를 기록하고, 실패하면 셸 명령이 실패하고 스테이지가 실패하며 post { failure } 블록이 저장된 태그를 읽어 다시 배포한다.


stage('Deploy & Verify') {
    environment { COMMIT = "${env.GIT_COMMIT}" }   // 새 이미지 태그
    steps {
        sshagent(['ec2-ssh-key']) {
            sh '''
                ssh ec2-user@<EC2_HOST> "
                    export IMAGE=<registry>/myapp:${COMMIT}
                    # --wait가 헬스체크 통과까지 블로킹했다가 종료한다
                    # 실패하면 ssh가 0이 아닌 값을 반환해 스테이지가 실패하고
                    # 아래 post { failure } 블록이 롤백을 처리한다.
                    docker compose up -d --wait
                    # healthy한 배포가 검증됐을 때만 실행됨:
                    echo ${COMMIT} > /home/ec2-user/last_good_tag
                "
            '''
        }
    }
    post {
        failure {
            sshagent(['ec2-ssh-key']) {
                sh '''
                    ssh ec2-user@<EC2_HOST> "
                        PREV=\\$(cat /home/ec2-user/last_good_tag 2>/dev/null)
                        if [ -n \\"\\$PREV\\" ]; then
                            export IMAGE=<registry>/myapp:\\$PREV
                            docker compose up -d --wait
                        else
                            echo 'No known-good tag recorded yet; cannot auto-roll back' >&2
                            exit 1
                        fi
                    "
                '''
            }
        }
    }
}

메커니즘은 자기 교정적이다. 태그는 배포가 healthy하게 검증된 후에만 기록되므로 last_good_tag는 항상 진짜 작동했던 버전을 가리킨다. 실패한 배포는 그것을 절대 덮어쓰지 않아 롤백이 항상 진짜 마지막 정상 이미지를 대상으로 한다. (서버에 파일을 유지하지 않으려면 현재 실행 중인 이미지를 교체 전에 docker inspect로 조회하거나 Jenkins 빌드 아티팩트로 태그를 저장하는 방법도 있다.)

진정한 무중단 전환을 위해 팀들은 블루-그린 배포(두 개의 동일 환경을 운영하고 트래픽을 이전 "블루"에서 새 "그린"으로 전환해 필요 시 즉시 되돌리는 방식)나 카나리 배포(처음에 트래픽의 일부만 새 버전에 라우팅하고 100%로 가기 전에 지켜보는 방식)를 사용한다. 이것들은 다음 단계의 전략이다. EC2 단일 호스트에서는 빠른 헬스체크+롤백만으로 대부분의 위험을 제거할 수 있고, docker compose up -d의 롤링 교체가 가용성이 끊기는 시간을 최소화한다.

3단계 — 코드에서 시크릿 분리하기

앱에는 민감한 값이 필요하다. 데이터베이스 비밀번호, API 키, ECR 레지스트리 주소. 철칙은 절대 시크릿을 저장소에 커밋하지 않는다는 것이다. Git에 있는 모든 것은 영구적이고 접근 권한이 있는 모든 사람에게 보인다. "삭제"한 뒤에도 기록에서는 남아 있다.

시크릿을 코드와 분리하고 런타임에 주입한다.

  • Jenkins에서 — 내장 자격증명 저장소에 시크릿을 저장하고 withCredentials로 가져온다. 빌드 로그에서 시크릿이 마스킹된다.

  withCredentials([string(credentialsId: 'db-password', variable: 'DB_PASS')]) {
      sh 'deploy-using $DB_PASS'
  }
  • 컨테이너 수준에서 — 서버에 존재하고 git에서 무시되며 절대 커밋되지 않는 .env 파일에서 소싱한 환경변수를 Compose로 전달한다.

  services:
    web:
      image: ${IMAGE}
      env_file: /home/ec2-user/app.env

같은 시크릿이 두 곳에 있어서는 안 된다. Jenkins는 배포에 필요한 시크릿을 보관하고, 서버는 앱이 실행될 때 필요한 시크릿을 보관한다. 도구가 허용하는 모든 곳에서 시크릿을 보호·마스킹 처리해 로그에 누출되지 않게 한다.

완성된 파이프라인, 처음부터 끝까지

지금까지 8개 강의에 걸쳐 조립한 것을 돌아보자. 전체 흐름은 다음과 같다.

1. GitLab에 코드를 push한다.

2. Webhook이 Jenkins를 트리거한다.

3. Jenkins가 코드를 체크아웃하고 빌드·테스트한다. 뭔가 잘못되면 즉시 멈춘다.

4. 테스트가 통과하면 Jenkins가 커밋 SHA로 태그된 Docker 이미지를 빌드한다.

5. Jenkins가 AWS ECR에 인증하고 이미지를 push한다.

6. Jenkins가 EC2에 SSH로 접속해 이미지를 pull하고 docker compose로 컨테이너를 교체한다.

7. 헬스체크가 성공을 확인한다. 실패하면 파이프라인이 이전 이미지로 롤백한다. 시크릿은 코드 밖에 있고 안전하게 주입된다.

내 코드에 적용하기

예제 앱은 의도적으로 범용으로 만들었지만 이 파이프라인은 그렇지 않다. 내 프로젝트에 맞게 바꾸려면 Dockerfile의 베이스 이미지와 시작 명령을 내 언어의 것으로 교체하고, 실제 테스트 명령으로 바꾸고, ECR 저장소와 EC2 호스트를 내 인프라로 바꾸고, 앱의 실제 시크릿을 자격증명 저장소와 .env 파일에 등록하면 된다. Dockerfile, Jenkinsfile, 배포 로직이 이전 가능한 기술이다. 나머지는 설정일 뿐이다. 이제 git push 한 번으로 검증되고 복구 가능한 live 배포까지 어떤 애플리케이션이든 자동으로 배포하는 재현 가능한 청사진을 갖게 됐다.

핵심 정리

  • 배포는 헬스체크가 새 컨테이너가 실제로 서비스 중임을 증명할 때 완료된다. 컨테이너 healthcheck를 정의하고 docker compose up -d --wait을 실행하면 프로브를 폴링하고 앱이 healthy 상태가 되지 않으면 0이 아닌 값으로 종료한다. 고정 sleep은 절대 쓰지 않는다.
  • 롤백은 모든 이미지가 ECR에 커밋 SHA로 태그되어 유지되기 때문에 가능하다. 검증이 완료된 배포 후에만 서버 파일에 태그를 기록하고, post { failure { ... } } 블록이 그 파일을 읽어 마지막 정상 태그를 다시 배포한다.
  • 시크릿을 절대 커밋하지 않는다. 배포 시 시크릿은 Jenkins 자격증명(withCredentials)으로, 런타임 시크릿은 git에서 무시되는 .env로 런타임에 주입한다. 각 시크릿을 정확히 한 곳에만 보관한다.
  • 완성된 파이프라인은 git push에서 빌드, 테스트, ECR 이미지 push, 검증된 EC2 배포까지 어떤 앱이든 자동으로 처리한다. Dockerfile, Jenkinsfile, 배포 로직은 내 프로젝트에 그대로 적용할 수 있다.

🛠 실습

실습 1. 준비 실습: EC2 서버의 docker-compose.yml에 Docker Compose 헬스체크를 추가한 뒤 'docker compose up -d --wait'을 수동으로 실행해 출력을 확인하라. '--wait'가 헬스체크가 healthy를 보고할 때까지 블로킹하는지 확인한다.

# EC2 인스턴스에 SSH 접속 후 docker-compose.yml 수정:
cat > /home/ec2-user/docker-compose.yml << 'EOF'
services:
  web:
    image: ${IMAGE}
    ports:
      - "80:8080"
    restart: always
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: ___s
      timeout: ___s
      retries: ___
EOF

# --wait 플래그 테스트를 위한 수동 배포:
export IMAGE=<your-ecr-registry>/myapp:<last-commit-sha>
docker compose -f /home/ec2-user/docker-compose.yml up -d --wait
힌트
  • 컨테이너 안에 'curl'이 없으면(alpine 이미지) 헬스체크 테스트 명령으로 'wget -q --spider http://localhost:8080/health' 또는 'node -e "require(http).get(url, r => process.exit(r.statusCode===200?0:1))"'를 사용한다.
  • 'docker compose up -d --wait'가 '--wait'에 대한 오류로 즉시 종료되면 Compose 버전이 v2.1보다 오래된 것이다. 'docker compose version'으로 확인한다. v2.1 이상이 필요하다.
정답 보기 
# healthcheck가 포함된 docker-compose.yml:
cat > /home/ec2-user/docker-compose.yml << 'EOF'
services:
  web:
    image: ${IMAGE}
    ports:
      - "80:8080"
    restart: always
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]

EOF

# 참고: 헬스체크가 작동하려면 앱에 /health 엔드포인트가 있어야 한다.
# index.js에 다음을 추가한다:
# if (req.url === '/health') { res.writeHead(200); res.end('OK'); return; }

# --wait로 실행:
export IMAGE=123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:abc1234
docker compose -f /home/ec2-user/docker-compose.yml up -d --wait

# 예상 출력:
# [+] Running 1/1
#  ✔ Container ec2-user-web-1  Healthy
# (헬스체크를 통과할 때만 반환되고 종료 코드 0)

# /health 엔드포인트가 없는 경우:
# interval * retries = 30초 후 0이 아닌 값으로 종료:
# Error response from daemon: ... health check failed

'--wait'는 'docker compose up -d'를 fire-and-forget 명령에서 검증된 배포로 바꾼다. 설정된 간격으로 헬스체크 프로브를 최대 retries 횟수까지 폴링하고, 컨테이너가 그 제한 안에 'healthy' 상태로 전환되면 종료 코드 0으로만 종료한다. 고정 'sleep 10'으로는 이것을 할 수 없다. 앱 시작 시간과 경쟁하며 유용한 종료 코드를 제공하지 않는다. 헬스체크 테스트 명령('curl -f http://localhost:8080/health')이 성공해야(종료 코드 0) 컨테이너가 healthy로 간주된다. '-f'는 HTTP 오류 시 curl이 0이 아닌 값으로 종료하게 만든다.

실습 2. 핵심 실습: Jenkinsfile의 'Deploy to EC2' 스테이지를 헬스체크+롤백 패턴으로 업데이트하라. 이 스테이지에서 '--wait'으로 배포하고, 성공 시 태그를 'last_good_tag'에 기록하며, '--wait'이 0이 아닌 값으로 종료되면 'post { failure { ... } }' 블록에서 저장된 태그로 롤백한다.

// 기존 'Deploy to EC2' 스테이지를 교체:
stage('Deploy & Verify') {
    steps {
        script {
            def registry = '<registry>'
            def commit   = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
            sshagent(['ec2-ssh-key']) {
                sh """
                    ssh ec2-user@<EC2_HOST> '
                        export IMAGE=${registry}/myapp:${commit}
                        docker compose -f /home/ec2-user/docker-compose.yml pull
                        docker compose -f /home/ec2-user/docker-compose.yml up -d ___
                        echo ___ > /home/ec2-user/last_good_tag
                    '
                """
            }
        }
    }
    post {
        failure {
            // 롤백 블록
        }
    }
}
힌트
  • 배포가 실패해도 'post { failure { ... } }' 블록이 실행되지 않으면, 'docker compose up -d --wait' 실패가 SSH 호출을 통해 종료 코드를 전파하는지 확인한다. 원격 셸 스크립트 맨 앞에 'set -e'를 추가하면 어떤 명령 실패든 스크립트를 종료하게 만든다.
  • 'last_good_tag'가 첫 실행 시 비어 있으면(이전에 성공한 배포가 없는 경우) 롤백이 올바르게 'No known-good tag found' 메시지를 출력하고 종료 코드 1로 종료한다. 이것은 버그가 아닌 의도된 동작이다.
  • 롤백을 의도적으로 테스트하려면: 헬스체크가 실패하게 만드는 이미지를 push하고(예: CMD를 'exit 1'로 설정한 Dockerfile), '--wait'가 타임아웃되고 스테이지가 빨간색이 되고 post 블록이 이전 태그를 배포하는 것을 확인한다.
정답 보기 
stage('Deploy & Verify') {
    steps {
        script {
            def region   = 'us-east-1'
            def registry = '123456789012.dkr.ecr.us-east-1.amazonaws.com'
            def commit   = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
            sshagent(['ec2-ssh-key']) {
                sh """
                    ssh ec2-user@<EC2_PUBLIC_IP> '
                        aws ecr get-login-password --region ${region} \\
                          | docker login --username AWS --password-stdin ${registry}
                        export IMAGE=${registry}/myapp:${commit}
                        docker compose -f /home/ec2-user/docker-compose.yml pull
                        docker compose -f /home/ec2-user/docker-compose.yml up -d --wait
                        echo ${commit} > /home/ec2-user/last_good_tag
                    '
                """
            }
        }
    }
    post {
        failure {
            script {
                def region   = 'us-east-1'
                def registry = '123456789012.dkr.ecr.us-east-1.amazonaws.com'
                sshagent(['ec2-ssh-key']) {
                    sh """
                        ssh ec2-user@<EC2_PUBLIC_IP> '
                            PREV=\\$(cat /home/ec2-user/last_good_tag 2>/dev/null)
                            if [ -n "\\$PREV" ]; then
                                export IMAGE=${registry}/myapp:\\$PREV
                                docker compose -f /home/ec2-user/docker-compose.yml up -d --wait
                                echo "Rolled back to \\$PREV"
                            else
                                echo "No known-good tag found; cannot auto-roll back" >&2
                                exit 1
                            fi
                        '
                    """
                }
            }
        }
    }
}

자기 교정 메커니즘은 두 가지 설계 원칙에서 나온다. (1) 'echo ${commit} > last_good_tag'는 '--wait'가 종료 코드 0으로 반환된 후에만 실행되므로 파일에는 항상 검증된 healthy 태그가 있다. 실패한 배포는 절대 덮어쓰지 않는다. (2) 'post { failure { ... } }'는 스테이지가 실패할 때만 실행되어 롤백 블록이 마지막으로 검증된 태그를 읽는다. 별도의 외부 상태 관리 없이 안전망이 만들어진다. EC2 서버 자체가 롤백 기준을 평문 텍스트 파일로 저장한다. 백슬래시 이스케이프('\\$PREV')는 외부 Groovy 문자열 치환이 SSH 세션이 보기 전에 셸 변수를 소비하지 않도록 필요하다.

실습 3. 핵심 실습: 파이프라인의 안전한 시크릿 관리를 구현하라. 모의 데이터베이스 비밀번호를 Jenkins 자격증명으로 저장하고 런타임 애플리케이션 시크릿을 EC2 서버의 '.env' 파일에 둔다. Jenkins 빌드 로그에 시크릿 값이 나타나지 않는지, 실행 중인 컨테이너가 런타임 시크릿을 환경변수로 접근할 수 있는지 확인하라.

// Part 1 — Jenkins 자격증명 (배포 시 시크릿):
// Manage Jenkins → Credentials → Global → Add Credentials
// Kind: Secret text  ID: db-password  Secret: s3cr3tP@ssw0rd

// Part 2 — Jenkinsfile에서 withCredentials로 사용:
withCredentials([string(credentialsId: '___', variable: '___')]) {
    sh 'echo "Would connect to DB with password: $DB_PASS"'
    // 실제 시나리오: sh 'deploy-script --db-password $DB_PASS'
}

// Part 3 — EC2 서버의 런타임 시크릿 (Jenkinsfile이나 저장소가 아닌 곳에):
// EC2에 SSH로 접속해 .env 파일 생성:
cat > /home/ec2-user/app.env << 'EOF'
DB_PASSWORD=___
API_KEY=___
EOF

// Part 4 — docker-compose.yml에서 참조:
services:
  web:
    image: ${IMAGE}
    env_file: /home/ec2-user/app.env
    ...
힌트
  • 'withCredentials'를 사용해도 로그에 시크릿이 표시되면 'sh' 명령의 변수 이름이 'variable' 필드와 정확히 일치하는지 확인한다. 대소문자 구분이 있다.
  • 컨테이너가 .env 파일의 환경변수를 볼 수 없으면 docker-compose.yml의 'env_file'이 절대 경로를 가리키는지 확인한다(docker compose를 실행하는 위치에서의 상대 경로가 아닌).
  • 파이프라인 명령에 리터럴 시크릿을 포함한 'docker run -e DB_PASSWORD=<value>'를 절대 사용하지 않는다. 'docker inspect' 출력과 빌드 로그에 노출된다. 'env_file' 방식은 값을 파일 안에만 유지해 어떤 명령 문자열에도 나타나지 않는다.
정답 보기 
// Part 1 & 2 — Jenkinsfile에서 Jenkins 자격증명 사용:
stage('Example Secret Usage') {
    steps {
        withCredentials([string(credentialsId: 'db-password', variable: 'DB_PASS')]) {
            sh 'echo "DB_PASS is set: ${#DB_PASS} chars"'  // 길이를 출력, 값이 아님
            // Jenkins가 자동으로 DB_PASS를 마스킹 — 실수로 echo해도 '****'로 표시됨
        }
    }
}

// Console Output:
// [Example Secret Usage] $ echo "DB_PASS is set: 14 chars"
// DB_PASS is set: 14 chars
// (값 's3cr3tP@ssw0rd'는 로그에 절대 나타나지 않음)

// Part 3 — EC2에 /home/ec2-user/app.env 생성 (SSH로, 파이프라인이 아닌 방법으로):
ssh ec2-user@<EC2_IP>
cat > /home/ec2-user/app.env << 'EOF'
DB_PASSWORD=s3cr3tP@ssw0rd
API_KEY=my-api-key-12345
EOF
chmod 600 /home/ec2-user/app.env   # 파일 권한 제한

// Part 4 — EC2의 업데이트된 docker-compose.yml:
cat > /home/ec2-user/docker-compose.yml << 'EOF'
services:
  web:
    image: ${IMAGE}
    ports:
      - "80:8080"
    restart: always
    env_file: /home/ec2-user/app.env
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 10s
      timeout: 3s
      retries: 3
EOF

// 실행 중인 컨테이너가 시크릿을 볼 수 있는지 확인:
docker exec <container-name> printenv DB_PASSWORD
// 출력: s3cr3tP@ssw0rd
// (이 명령은 EC2 서버에서 실행 — 시크릿은 거기서만 볼 수 있음)

2단계 시크릿 모델은 역할을 명확히 분리한다. Jenkins 자격증명은 파이프라인이 배포 시에 필요한 시크릿(예: 배포 중 API 인증)을 저장하고, EC2의 '.env' 파일은 실행 중인 앱이 런타임에 필요한 시크릿(데이터베이스 비밀번호, API 키)을 저장한다. Jenkins의 'withCredentials' 래퍼는 모든 출력을 스캔하고 시크릿 값을 '**'로 교체해 로그에서 시크릿을 마스킹한다. '.env' 파일은 Git에 절대 들어가지 않고('.gitignore'에 등록) Jenkins를 통과하지도 않는다. 앱이 실제로 실행되는 서버에만 존재한다. 'chmod 600'은 ec2-user만 읽을 수 있도록 파일 권한을 제한한다.

실습 4. 응용 실습: 롤백 메커니즘을 의도적으로 트리거하라. 헬스체크가 실패하게 만드는 버그를 도입하고(예: /health 엔드포인트가 HTTP 500을 반환하거나 앱이 시작 시 충돌), push해서 Jenkins가 (1) '--wait'를 실행하고 타임아웃, (2) Deploy 스테이지를 빨간색으로 표시, (3) 롤백 post 블록 실행, (4) EC2에 마지막 정상 버전이 계속 실행 중인지 확인하라.

// 1단계: index.js에 의도적인 버그 도입.
// 예: /health 엔드포인트가 항상 500을 반환하게 만들기:
if (req.url === '/health') {
  res.writeHead(500);
  res.end('Intentional failure');
  return;
}

// 2단계: 버그가 있는 변경을 push한다.
// Jenkins 빌드를 지켜보고 Deploy 스테이지 동작을 확인한다.

// 3단계: 빌드가 완료된 후(FAILED로) EC2 확인:
ssh ec2-user@<EC2_IP>
docker ps                                       # 어떤 이미지가 실행 중인가?
cat /home/ec2-user/last_good_tag               # 저장된 정상 태그는?
curl http://localhost:80/                      # 서비스가 여전히 살아있는가?
힌트
  • 버그 있는 헬스체크로 Deploy 스테이지가 초록색이 된다면 'docker-compose.yml'에 'healthcheck' 섹션이 있는지 확인한다. 없으면 '--wait'는 컨테이너가 healthy가 아닌 *시작*만 기다려 헬스체크와 무관하게 종료 코드 0을 반환한다.
  • 롤백 블록도 실패하면('last_good_tag'가 첫 실행 시 존재하지 않는 경우), 이것은 예상된 동작이다. 먼저 성공한 빌드를 push해 파일을 만든 다음 버그를 도입한다.
  • 테스트 중 대기 시간을 줄이려면 헬스체크의 'retries: 1'로 임시 변경한다. 30초 대신 10초 후에 실패한다.

// 1단계 — 버그가 있는 index.js (/health 엔드포인트가 500 반환):
const http = require('http');
