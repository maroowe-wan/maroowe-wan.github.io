01. kubectl CLI 기초: 핵심 명령어와 클러스터 조작

학습 목표

  • kubeconfig 구조를 이해하고 context와 namespace를 안전하게 전환해, 엉뚱한 클러스터에 명령을 날리는 실수를 방지한다.
  • get, describe, logs, exec로 실행 중인 리소스를 조회·디버깅하고, -o wide/yaml/json과 레이블 셀렉터로 출력을 원하는 형태로 다듬는다.
  • 명령형 스타일(run, create, set)과 선언형 스타일(apply -f)의 차이를 파악하고 상황에 맞게 골라 쓴다.

본문

kubectl — 클러스터 조작의 유일한 창구

kubectl(발음은 "kube-control" 또는 "kube-cuttle" 둘 다 통용된다)은 Kubernetes API 서버와 통신하는 커맨드라인 클라이언트다. Pod 목록 조회, 로그 확인, Deployment 스케일 조정, 컨테이너 내부에서 셸 실행까지 — 클러스터에서 하는 거의 모든 일이 이 하나의 바이너리를 거친다. Kubernetes가 엔진이라면 kubectl은 조종석이다. 이번 강의의 목표는 그 조종석을 손에 익혀두는 것이다. 새벽 2시에 장애가 터져도 손이 먼저 움직일 수 있도록.

시작하기 전에 실용적인 팁 두 가지: 대부분의 운영자는 alias k=kubectl로 단축키를 등록해 쓴다. 현장에서 k와 kubectl이 혼용되는 이유가 이것이다. 또한 셸 자동완성(source <(kubectl completion bash) 혹은 zsh 버전)을 켜두면 리소스 이름을 Tab으로 완성할 수 있어 외울 필요가 줄어든다. 이 두 가지 습관만으로도 타이핑량이 거의 반으로 줄어든다.

공식 kubectl Cheat Sheet(Kubernetes 공식 문서)는 단축 이름과 출력 플래그를 한 페이지에 담아놓은 최고의 레퍼런스다. 학습하는 동안 늘 열어두길 권한다.

어떤 클러스터와 대화 중인가: kubeconfig, context, namespace

명령을 실행하기 전에 반드시 확인해야 할 것이 있다. "지금 어떤 클러스터에, 어떤 권한으로 접속해 있는가?" 로컬 테스트 클러스터라고 생각하고 delete를 날렸는데 프로덕션이었다면 — 이것이 초보자들이 가장 자주, 가장 뼈아프게 겪는 실수다. Kubernetes는 이 문제를 kubeconfig라는 설정 파일로 관리한다. 기본 위치는 ~/.kube/config다.

kubeconfig에는 세 종류의 항목이 묶여 있다:

  • clusters — API 서버 주소와 인증서 정보 ("어디에")
  • users — 클라이언트 인증서나 토큰 등 인증 수단 ("누가")
  • contexts — cluster + user + 기본 namespace를 하나로 묶은 이름표 ("어떤 조합으로")

context는 "이 cluster, 이 user, 이 namespace를 기본값으로 쓴다"는 포인터다. 여러 context 중 하나가 current로 지정되고, 명시적으로 덮어쓰지 않는 한 모든 명령은 이 current context를 따른다.

kubectl config 명령으로 조회하고 전환한다:


# kubectl이 알고 있는 모든 설정 보기 (인증 정보는 마스킹됨)
kubectl config view

# 전체 context 목록 조회; 현재 context는 * 표시
kubectl config get-contexts

# 현재 활성 context 이름만 확인
kubectl config current-context

# 다른 cluster/user 조합으로 전환
kubectl config use-context minikube

namespace는 클러스터 내부의 가상 구획이다. 폴더에 비유하면, team-a의 web Deployment와 team-b의 web Deployment가 이름 충돌 없이 공존할 수 있다. 대부분의 명령은 기본값으로 default namespace를 사용한다. 특정 명령에만 -n으로 namespace를 지정하거나, current context의 기본 namespace 자체를 바꿀 수도 있다:


# 이번 명령에만 kube-system namespace 사용
kubectl get pods -n kube-system

# 현재 context의 기본 namespace를 jupiter로 변경
kubectl config set-context --current --namespace=jupiter

권장 습관: 삭제처럼 파급력이 큰 명령을 실행하기 전에 kubectl config current-context로 현재 위치를 확인한다. 많은 팀이 kubectx, kubens 같은 도구를 쓰거나 셸 프롬프트에 context 이름을 표시해, 잘못된 클러스터에 접속하는 실수 자체를 구조적으로 막는다.

클러스터 읽기: get, describe, 출력 포맷

kubectl get은 클러스터의 조감도다. 리소스가 어떤 상태인지 빠르게 테이블 형태로 확인할 수 있다.


kubectl get pods
kubectl get deployments
kubectl get nodes

자주 쓰는 리소스 타입에는 단축 이름이 있다: pods→po, deployments→deploy, services→svc, namespaces→ns, nodes→no, replicasets→rs. 따라서 kubectl get po는 kubectl get pods와 같다. kubectl api-resources를 실행하면 전체 타입 목록과 단축 이름을 볼 수 있다.

기본 테이블만으로는 부족할 때 -o 플래그로 출력 형식을 바꾼다:


# 노드 배치, Pod IP 등 추가 컬럼 표시
kubectl get pods -o wide

# 실제 오브젝트를 YAML로 덤프 (스키마 학습에 유용)
kubectl get pod nginx -o yaml

# JSON으로 덤프 (jq나 스크립트와 연계할 때 유용)
kubectl get pod nginx -o json

Pod이 수십 개라면 레이블 셀렉터로 원하는 것만 필터링한다. 레이블은 리소스에 붙이는 키/값 태그이며, --selector(또는 -l)로 조건을 지정한다:


# app=web 레이블이 달린 Pod만
kubectl get pods -l app=web

# 두 레이블을 동시에 만족하는 Pod
kubectl get pods -l 'app=web,env=prod'

get이 요약 정보를 보여준다면, kubectl describe는 오브젝트 하나의 상세 정보를 사람이 읽기 좋은 형태로 펼쳐준다. 설정 값, 현재 상태, 그리고 맨 아래의 Events 섹션까지. Pod이 멈춰 있다면 Events 섹션이 이유를 대부분 알려준다.


kubectl describe pod nginx

Pod이 Pending, ImagePullBackOff, CrashLoopBackOff 상태에 빠졌을 때 describe가 첫 번째 행동이 되어야 한다. 예를 들어 ImagePullBackOff는 대부분 이미지 이름이 틀렸거나 클러스터가 레지스트리에 접근하지 못한 것인데, Events에 구체적인 오류 메시지가 찍힌다.

컨테이너 내부 들여다보기: logs와 exec

실행 중인 컨테이너에 접근하는 명령이 두 가지 있다.

kubectl logs는 컨테이너가 표준 출력에 쓴 내용, 즉 애플리케이션 로그를 스트리밍한다:


kubectl logs nginx

# 새 로그를 실시간으로 추적 (tail -f와 동일)
kubectl logs -f nginx

# 이미 재시작된 Pod의 이전 컨테이너 로그 확인
kubectl logs nginx --previous

kubectl exec는 컨테이너 안에서 명령을 실행한다. -it(인터랙티브 + TTY) 옵션을 붙이면 라이브 셸이 열려 워크로드에 SSH로 접속한 것과 가장 비슷한 경험을 얻는다:


# 단일 명령 실행 후 결과 출력
kubectl exec nginx -- printenv

# 컨테이너 내부에서 인터랙티브 셸 열기
kubectl exec -it nginx -- sh

-- 구분자에 주목하자. 이 뒤에 오는 모든 것은 kubectl이 아닌 컨테이너에서 실행할 명령이다. 예를 들어 kubectl exec -it nginx -- sh -c 'echo $var1'은 Pod 내부의 환경 변수 값을 출력해 설정이 제대로 적용됐는지 확인할 때 쓸 수 있다.

리소스를 만드는 두 가지 방식: 명령형 vs 선언형

Kubernetes 리소스를 관리하는 방식은 두 가지 철학으로 나뉜다. 숙련된 운영자는 이 둘을 상황에 맞게 의식적으로 선택한다.

명령형 방식은 kubectl에게 "지금 이것을 해라"고 단계별로 지시한다. 빠르고 실험적인 작업이나 임시 리소스, 시험 환경처럼 속도가 중요할 때 유용하다:


# CLI에서 Pod 직접 생성
kubectl run nginx --image=nginx --env=var1=val1

# 레플리카 2개짜리 Deployment 생성
kubectl create deployment web --image=nginx:1.18.0 --replicas=2

# 실행 중인 Deployment의 이미지를 즉시 교체
kubectl set image deployment/web nginx=nginx:1.19.8

# Deployment 스케일 조정
kubectl scale deployment web --replicas=5

선언형 방식은 원하는 최종 상태를 YAML 매니페스트로 작성하고 kubectl apply -f에 넘긴다. Kubernetes가 그 상태에 도달하기 위한 단계를 알아서 결정한다:


kubectl apply -f deployment.yaml
kubectl apply -f service.yaml

핵심 차이: 명령형은 행동을 기술하고, 선언형은 목표 상태를 기술한다. apply는 같은 파일로 몇 번이든 재실행할 수 있다. Kubernetes는 현재 상태와 매니페스트를 비교해 달라진 부분만 변경한다. 덕분에 선언형 매니페스트는 버전 관리가 가능하고, 한 번 쓰고 버릴 게 아닌 모든 작업의 표준이 된다. 명령형은 속도와 탐색용, 선언형은 프로덕션과 GitOps용이다.

두 방식을 연결하는 유용한 다리가 --dry-run=client -o yaml이다. 실제로 리소스를 만들지 않고 생성될 YAML만 출력해준다. 매니페스트를 처음부터 손으로 작성하는 대신 몇 초 만에 올바른 뼈대를 만들 수 있다:


kubectl create deployment web --image=nginx:1.18.0 --replicas=2 \
  --dry-run=client -o yaml > deployment.yaml

선언형 관리와 자가 치유: apply가 강력한 이유

선언형 관리를 이해하면 Kubernetes의 핵심 기능 하나가 자연스럽게 이해된다. replicas: 5를 선언했다는 것은 클러스터에게 "Pod이 항상 5개 실행 중이어야 한다"는 목표 상태를 알린 것이다. 여기서 Pod 하나를 손으로 삭제하면:


kubectl delete pod web-7d9f-abcde
kubectl get pods   # 4개가 아닌 5개가 보인다

이것은 버그가 아니다. 컨트롤러가 현재 상태와 목표 상태를 계속 비교하다가 Pod이 4개뿐임을 감지하고 즉시 새 Pod을 스케줄링한다. 이 루프를 조정(reconciliation)이라고 하며, Kubernetes 자가 치유의 핵심 메커니즘이다. 아래 다이어그램이 이 루프를 보여준다.

이 원리는 kubectl create deployment가 컨테이너가 실제로 뜨기도 전에 즉시 반환되는 이유도 설명해준다. 아래 시퀀스 다이어그램처럼: kubectl이 요청을 API 서버에 전달하면, API 서버는 Deployment를 etcd(클러스터 데이터베이스)에 기록하고 "created"를 반환한다 — 음식이 나오기 전에 주문 확인을 받는 것과 같다. 이후 Deployment 컨트롤러가 ReplicaSet을 만들고, ReplicaSet 컨트롤러가 Pod을 만들며(Pending 상태), 스케줄러가 노드를 배정하고, 마지막으로 해당 노드의 kubelet이 이미지를 Pull해 컨테이너를 시작하면 Pod이 Running으로 바뀐다.


kubectl get pods -o wide --watch
kubectl create deployment web --image=nginx

Pod이 Pending → ContainerCreating → Running으로 바뀌며 IP를 할당받는 과정을 실시간으로 볼 수 있다. 조정 루프가 실제로 작동하는 현장이다.

핵심 정리

  • kubeconfig는 clusters, users, contexts를 하나로 묶는다. 특히 삭제처럼 파급력이 큰 작업 전에는 반드시 kubectl config current-context와 namespace를 확인한다.
  • get은 요약 테이블을 보여준다(-o wide/yaml/json으로 상세화, -l/--selector로 필터링). describe, logs, exec는 디버깅을 위해 점점 더 깊이 들어가는 도구들이다.
  • 명령형(run, create, set, scale)은 실험과 빠른 작업에, 선언형 apply -f는 반복 가능하고 버전 관리되는 프로덕션 워크로드에 쓴다.
  • --dry-run=client -o yaml로 올바른 매니페스트를 즉시 생성해, 명령형과 선언형 사이의 다리로 활용한다.
  • Kubernetes는 선언한 목표 상태를 향해 현재 상태를 끊임없이 조정한다. 삭제한 Pod이 되살아나는 것도, create가 컨테이너 기동 전에 즉시 반환되는 것도 이 조정 루프 때문이다.

🛠 실습

실습 1. [준비 운동] kubeconfig를 확인하고 namespace를 전환한다.

1. 현재 사용 중인 context 이름을 출력한다.

2. 모든 context를 나열하고 활성 context를 확인한다.

3. context 전환 없이, 현재 context의 기본 namespace를 kube-system으로 변경한다.

4. -n 없이 Pod을 조회해 시스템 Pod이 보이는지 확인한다.

5. 기본 namespace를 default로 원상 복구한다.

# 로컬 클러스터가 연결되어 있어야 한다 (minikube, kind 또는 kubeconfig가 설정된 클러스터).
# 별도의 스타터 코드 없음 — 순수 CLI 명령으로 진행한다.
힌트
  • kubeconfig 파일의 기본 위치는 `~/.kube/config`다. `kubectl config view`에서 아무것도 나오지 않는다면 파일 존재 여부와 `KUBECONFIG` 환경 변수가 경로를 덮어쓰고 있지 않은지 확인한다.
  • `kubectl config set-context --current`는 활성 context 레코드만 수정한다. `--current` 대신 이름을 직접 지정하면 해당 이름의 context를 새로 만들거나 덮어쓴다.
  • 3단계 후 `kubectl config view --minify`를 실행하면 활성 context 블록만 출력되어 namespace 필드 변경 여부를 바로 확인할 수 있다.
정답 보기 
# 1단계 – 현재 활성 context 이름 확인
kubectl config current-context

# 2단계 – 전체 context 목록 조회 (활성 context는 * 표시)
kubectl config get-contexts

# 3단계 – 현재 context의 기본 namespace를 kube-system으로 고정
kubectl config set-context --current --namespace=kube-system

# 4단계 – 확인: -n 없이도 kube-system의 Pod이 나타나야 함
kubectl get pods

# 5단계 – 기본 namespace 복구
kubectl config set-context --current --namespace=default

kubectl config current-context는 ~/.kube/config의 current-context 필드를 읽는다. get-contexts는 전체 테이블을 보여주며 활성 행에 *가 붙는다. set-context --current --namespace=<ns>는 현재 활성 context의 namespace 필드만 수정하며, 클러스터나 사용자는 변경하지 않는다. 마지막에 default로 복구하는 것은 중요한 위생 습관이다. 기본 namespace를 비표준으로 두면 이후 명령에서 혼란이 생길 수 있다.

실습 2. [핵심 1] 클러스터를 읽고 문제가 있는 Pod을 디버깅한다.

1. 의도적으로 잘못된 이미지로 Pod을 생성한다:

kubectl run bad-pod --image=nginx:doesnotexist999

2. 상태가 ImagePullBackOff가 될 때까지 지켜본다.

3. kubectl describe로 Events 섹션에서 정확한 오류 메시지를 찾는다.

4. 레이블 run=bad-pod가 설정된 Pod만 조회한다 (run 명령 실행 시 kubectl이 이 레이블을 자동으로 추가한다).

5. 출력 형식을 확인한다: Pod 스펙을 YAML로, 그 다음 JSON으로 덤프한다.

6. 확인이 끝나면 Pod을 삭제한다.

# 전제 조건: kubectl이 설정되어 있고 클러스터에 접근 가능해야 한다.
# YAML 파일 불필요 — 모든 단계는 CLI 명령으로만 진행한다.
힌트
  • Pod이 먼저 `ErrImagePull` 상태가 됐다가 `ImagePullBackOff`로 바뀌는 것은 정상이다. kubelet이 지수 백오프(exponential back-off) 방식으로 재시도하며, `ImagePullBackOff`는 그 백오프 단계를 나타낸다.
  • `kubectl describe pod bad-pod` 출력의 맨 아래로 스크롤하면 **Events** 테이블이 있다. 타임스탬프와 함께 이미지 Pull 실패 메시지가 기록되어 있다.
  • 플래그를 조합할 수도 있다: `kubectl get pod bad-pod -o wide --watch`를 사용하면 IP·노드 배정 정보를 보면서 동시에 실시간 상태 변화를 추적할 수 있다.
정답 보기 
# 1단계 – 문제 있는 Pod 생성
kubectl run bad-pod --image=nginx:doesnotexist999

# 2단계 – 상태 감시 (ImagePullBackOff가 나타나면 Ctrl-C로 중지)
kubectl get pod bad-pod --watch

# 3단계 – Events 블록 확인
kubectl describe pod bad-pod
# 확인 항목: Failed to pull image ... : not found (또는 유사한 레지스트리 오류)

# 4단계 – 레이블로 필터링
kubectl get pods -l run=bad-pod

# 5a단계 – YAML 출력
kubectl get pod bad-pod -o yaml

# 5b단계 – JSON 출력
kubectl get pod bad-pod -o json

# 6단계 – 정리
kubectl delete pod bad-pod

kubectl run은 run=<이름> 레이블을 자동으로 붙이기 때문에 -l run=bad-pod가 셀렉터로 동작한다. Pod이 멈췄을 때 describe의 Events 섹션이 첫 번째 확인 대상이다. ImagePullBackOff의 경우 이미지를 찾을 수 없거나 인증 실패 등 레지스트리 오류의 정확한 내용이 이벤트 메시지에 기록된다. -o yaml과 -o json은 로컬 파일이 아닌 API 서버의 실시간 오브젝트를 출력한다. status, resourceVersion, 컨테이너 상태처럼 손으로 작성한 매니페스트에는 없는 서버 추가 필드들도 함께 볼 수 있다.

실습 3. [핵심 2] logs와 exec으로 실행 중인 컨테이너 내부에 접근한다.

1. 정상 Pod을 생성한다: kubectl run web --image=nginx:1.25

2. Running 상태가 될 때까지 기다린 뒤, follow 모드(-f)로 액세스 로그를 스트리밍한다. 두 번째 터미널에서 Pod IP(kubectl get pod web -o wide로 확인)로 curl을 날려 로그 한 줄을 생성한 뒤 스트리밍을 중지한다.

3. kubectl exec로 컨테이너 내부에서 nginx -v를 실행한다 (비인터랙티브, 단일 명령).

4. 인터랙티브 셸을 열고 환경 변수 NGINX_VERSION이 설정되어 있는지 확인한다 (printenv NGINX_VERSION), 그 다음 exit한다.

5. Pod을 삭제한다.

# 2단계에서는 터미널이 두 개 있으면 편리하다 (하나는 로그 추적, 하나는 curl용).
# 로컬에 curl이 없다면, 몇 초 후 NGINX 기본 액세스 로그만 관찰해도 된다.
힌트
  • 이미지에 `sh`가 없을 수도 있다 (일부 경량 이미지는 `ash`나 `bash`를 사용). `kubectl exec -it web -- bash` 또는 `kubectl exec -it web -- /bin/sh`를 시도해본다.
  • 두 번째 터미널 없이 Pod IP를 얻으려면 `kubectl get pod web -o jsonpath='{.status.podIP}'`를 쓴다. `-o json | jq`보다 간결한 방법이다.
  • 로컬에 `curl`이 없다면 컨테이너 내부에서 직접 연결을 테스트할 수 있다: `kubectl exec -it web -- sh -c 'wget -qO- http://localhost'`
정답 보기 
# 1단계 – 정상 Pod 생성
kubectl run web --image=nginx:1.25

# 2a단계 – Running 대기
kubectl get pod web --watch
# (Running 상태가 나타나면 감시 중지)

# 2b단계 – 로그 추적 (이 터미널은 열어두고, 다른 터미널에서 curl 실행)
kubectl logs -f web
# 두 번째 터미널: POD_IP=$(kubectl get pod web -o jsonpath='{.status.podIP}')
#                 curl http://$POD_IP
# 로그 터미널로 돌아와 GET / 줄이 뜨는지 확인한 뒤 Ctrl-C

# 3단계 – 단일 비인터랙티브 명령 실행
kubectl exec web -- nginx -v

# 4단계 – 인터랙티브 셸
kubectl exec -it web -- sh
# 셸 내부:
printenv NGINX_VERSION
exit

# 5단계 – 정리
kubectl delete pod web

kubectl logs -f는 컨테이너의 stdout/stderr에 대해 tail -f처럼 동작하며, 라이브 트래픽을 실시간으로 관찰할 때 유용하다. kubectl exec 인자와 컨테이너 명령 사이의 -- 구분자는 컨테이너 명령에 자체 플래그가 있을 때 필수다. -- 없이 쓰면 kubectl이 -v를 자신의 verbosity 플래그로 해석할 수 있다. -it는 pseudo-TTY(-t)를 할당하고 stdin을 열어두어(-i) 인터랙티브 셸이 동작하게 한다. NGINX_VERSION은 공식 NGINX 이미지에 내장된 환경 변수로, 이 값을 컨테이너 내부에서 확인하면 호스트가 아닌 컨테이너 안에 있다는 사실을 증명할 수 있다.

실습 4. [응용] dry-run으로 Deployment 매니페스트를 생성하고, 선언형으로 적용한 뒤 업데이트와 자가 치유를 확인한다.

1. --dry-run=client -o yaml로 이미지 nginx:1.25, 레플리카 2개짜리 myapp Deployment 매니페스트를 생성하고 파일로 저장한다 (예: myapp-deploy.yaml).

2. 파일을 열어 spec.template.metadata.labels 아래에 team: ops 레이블을 추가한다 (kubectl이 자동 생성한 app: myapp 레이블 옆에). 파일을 저장한다.

3. kubectl apply -f myapp-deploy.yaml로 매니페스트를 적용하고 Pod 2개가 Running 상태인지 확인한다.

4. 자가 치유 시연: 실행 중인 Pod 하나를 이름으로 삭제한 뒤, 즉시 kubectl get pods를 반복 실행하거나 --watch로 Kubernetes가 자동으로 Pod을 재생성하는 과정을 관찰한다.

5. myapp-deploy.yaml에서 이미지를 nginx:1.26으로 수정하고 kubectl apply -f를 다시 실행한다. kubectl rollout status deployment/myapp으로 롤아웃 완료를 확인한다.

6. 정리: kubectl delete -f myapp-deploy.yaml

# 스타터 파일 없음 — 1단계에서 직접 생성한다.
# myapp-deploy.yaml의 저장 위치는 로컬 머신에서 편한 곳으로 한다.
힌트
  • 생성된 YAML에 `creationTimestamp: null`이나 `status: {}`가 있어도 그대로 두면 된다. dry-run 출력의 자리표시자 필드로, kubectl과 API 서버가 알아서 처리한다.
  • `kubectl apply` 후 `kubectl get deployment myapp -o yaml`을 실행하면 `spec.template.metadata.labels` 아래에 `team: ops` 레이블이 실제로 반영됐는지 확인할 수 있다.
  • `kubectl rollout status`가 멈추면 새 이미지 Pull이 실패하고 있을 가능성이 크다. 두 번째 터미널에서 `kubectl describe deployment myapp`이나 `kubectl get pods`를 실행해 새 Pod에 `ImagePullBackOff`가 발생했는지 확인한다.
정답 보기 
# 1단계 – 매니페스트 생성
kubectl create deployment myapp --image=nginx:1.25 --replicas=2 \
  --dry-run=client -o yaml > myapp-deploy.yaml

# 2단계 – team: ops 레이블 추가 (파일 편집)
# 아래와 같은 구간을 찾아:
#   spec:
#     template:
#       metadata:
#         labels:
#           app: myapp
# 한 줄을 추가해 다음과 같이 만든다:
#           app: myapp
#           team: ops

# 3단계 – 적용 및 확인
kubectl apply -f myapp-deploy.yaml
kubectl get pods -l app=myapp
# 두 Pod이 모두 Running이 될 때까지 대기

# 4단계 – 자가 치유 시연
# Pod 이름 조회:
kubectl get pods -l app=myapp
# 하나를 삭제한다 (<pod-name>을 실제 이름으로 교체):
kubectl delete pod <pod-name>
# 교체 과정 감시:
kubectl get pods -l app=myapp --watch
# 몇 초 안에 새 Pod이 나타난다

# 5단계 – myapp-deploy.yaml에서 이미지 변경:
# image: nginx:1.25  →  image: nginx:1.26
# 재적용:
kubectl apply -f myapp-deploy.yaml
kubectl rollout status deployment/myapp

# 6단계 – 정리
kubectl delete -f myapp-deploy.yaml

--dry-run=client -o yaml은 서버와 통신하지 않고 kubectl이 로컬에서 전체 API 오브젝트를 계산해 출력한다. 오류 없이 즉시 매니페스트 뼈대를 얻을 수 있다. YAML을 편집한 뒤 apply를 다시 실행하는 것이 선언형 관리의 핵심이다. kubectl은 원본 매니페스트, 현재 라이브 상태, 새 매니페스트를 3방향 비교(three-way diff)해 변경된 부분만 패치하고 나머지는 건드리지 않는다. Deployment가 소유한 Pod을 삭제하면 Deployment의 ReplicaSet 컨트롤러가 현재 레플리카 수(1)가 목표(2)에 미치지 못함을 감지하고 교체 Pod을 스케줄링한다. 이것이 조정 루프다. kubectl rollout status는 새 Pod이 모두 정상화되고 구 Pod이 종료될 때까지 블로킹되므로, 이미지 업데이트 후 완료 시점을 확실히 알 수 있다.
