01. What Is Kubernetes - Why Container Orchestration Matters

What Is Kubernetes - Why Container Orchestration Matters

Learning Objectives

  • Understand the limitations of running services with containers alone (scaling, failure recovery, deployment)
  • Explain in your own words the problem Kubernetes solves and the core value it provides
  • Grasp the concept of orchestration intuitively through everyday analogies

Content

Why Does This Matter?

You have probably heard phrases like "we need to migrate to Kubernetes" or "let's adopt containers." But if you follow along without understanding why it is necessary, you will quickly find yourself lost in a sea of complex terminology. In this lecture, we will set commands and configuration files aside for now and start with the most fundamental question: "What problem was Kubernetes created to solve in the first place?" Once you have a solid answer to that question, every concept you learn going forward will naturally make sense.

A Quick Recap of Containers

Let's start with a one-line definition of a container. A container is a self-contained, executable unit that packages an application together with everything it needs to run — code, libraries, and configuration.

Here is an analogy. Imagine you are a building owner trying to bring in tenants. If you had to build new walls and renovate the interior every time someone moved in, it would be an enormous hassle. Instead, picture each new tenant arriving with a fully furnished shipping container that they simply slot into place inside the building. No construction required, and when they leave, they just take the box with them. That is exactly how container technology works. The older virtual machine (VM) approach was like doing heavy construction every time — containers are lightweight, portable boxes that can be moved in seconds.

Thanks to containers, the classic problem of "it works on my machine but not on the server" has been greatly reduced. The entire runtime environment travels with the application.

One Container Is Easy — What About Hundreds?

This is where things get complicated. Modern services are not single monolithic programs. They are collections of small, independently deployable services — each responsible for a specific function. This is called a microservice architecture. A single e-commerce platform, for example, might have a separate frontend for rendering pages, a backend for processing orders, and a database for storing data. When traffic spikes, you may need to scale out only the most-used components (say, the frontend), so the number of containers can quickly grow from dozens to hundreds, or even thousands.

Now imagine managing all those containers across multiple servers by hand. You would be responsible for all of the following:

  • Deciding which container goes on which server, one by one
  • Waking up at 2 a.m. to restart a container every time one crashes
  • Manually scaling replicas up when traffic surges, then scaling them back down
  • Carefully swapping out containers during deployments without causing downtime
  • Manually wiring up the network so containers can discover and talk to each other

You might get by with scripts and homegrown tooling for a while, but at scale this approach hits a hard wall — it simply becomes more than any team can manage. This is exactly the point where container orchestration enters the picture.

Orchestration — Understanding It Through a Conductor Analogy

The word orchestration comes from the orchestra. Even if dozens of musicians (containers) each have their instruments ready, without a conductor to coordinate them, the result is noise, not music. A container orchestration tool plays the role of that conductor.

Here is another analogy. Suppose you own several buildings. You have assigned an on-site specialist — essentially a container management tool like Docker — to each building. That worked fine at first, but as you acquired more buildings, you found yourself making phone call after phone call to each specialist to issue instructions. So you hire a highly capable general manager to run the whole operation. That general manager's name is Kubernetes.

Now you only need to give the big picture to your manager. "Keep three frontend instances running at all times; maintain two backend instances." Kubernetes then figures out which server to place them on, and if any instance dies, it quietly brings a new one back up to keep the count right. The reason you do not have to micromanage every detail is that Kubernetes remembers your desired state and continuously reconciles reality to match it.

As shown in the flow diagram below, Kubernetes constantly compares the desired state you have declared against the actual current state, and takes action on its own whenever the two diverge. This reconciliation loop is the heart of declarative management.

The Core Value Kubernetes Delivers

Kubernetes is an open-source container orchestration platform originally built at Google. It is agnostic to the container technology you use — Docker or otherwise — and works across physical servers, virtual machines, and cloud environments alike. Its core value can be summarized in four areas:

1. Self-healing and High Availability — When a container crashes, Kubernetes automatically restarts it, minimizing downtime for users. The desired number of instances is always maintained.

2. Scalability — When traffic increases, Kubernetes scales replicas up; when things quiet down, it scales them back. Kubernetes inspects available resources across nodes and determines the most efficient placement for each container. This intelligent placement decision is called scheduling.

3. Service Discovery and Load Balancing — Containers die and are replaced frequently, and their IP addresses change each time. Kubernetes puts a stable, fixed endpoint in front of them so that other services always connect to the same address, and incoming traffic is distributed evenly across available instances.

4. Disaster Recovery — If infrastructure failures cause data loss, Kubernetes provides backup and recovery mechanisms to restore the last known good state.

How Developers and Operators See Things Differently

One more perspective worth addressing: even when looking at the same system, a developer and an operator have very different views. Developers focus on what is inside the container — their code, dependencies, and whether the application behaves correctly. Operators, on the other hand, look at the broader picture: which servers to deploy on, how to scale, how to expose services externally, and how to detect and respond to failures.

As illustrated in the layer diagram below, developers focus on the top layer — "my container" — while Kubernetes takes on the operational layer beneath it, governing a fleet of containers distributed across multiple servers.

Kubernetes automates most of the heavy lifting that used to fall on operators. Developers can stay focused on "my container," while operations teams can govern hundreds of containers through declared rules rather than late-night manual intervention. The growing prominence of roles like SRE (Site Reliability Engineer) reflects precisely this shift — designing for reliability on top of an automated, policy-driven infrastructure.

Key Takeaways

  • Containers are a lightweight, portable unit that packages an application together with its runtime environment. But when container counts reach the hundreds and span multiple servers, manual management becomes practically impossible.
  • The tool built to solve this problem is container orchestration, and its de facto standard is the open-source platform Kubernetes, originally created by Google.
  • Like an orchestra conductor — or a highly capable building manager — Kubernetes only needs to be told the desired state. It handles placement, replication, self-healing, and network connectivity on its own.
  • Its four core values are: self-healing and high availability, scalability and smart scheduling, service discovery and load balancing, and disaster recovery.
  • With this "why" firmly in mind, the internal structure (architecture) of Kubernetes covered in the next lecture will feel far more natural.

02. Kubernetes Architecture - Control Plane and Worker Nodes

Kubernetes Architecture - Control Plane and Worker Nodes

Learning Objectives

  • Distinguish the roles of a cluster, the master (control plane), and worker nodes.
  • Understand what each core component — API Server, Scheduler, etcd, kubelet — is responsible for.
  • Trace the sequence of steps a user command goes through inside the cluster.

Content

Why You Need to Understand the Architecture

In Lecture 1, we learned that "Kubernetes automatically places containers and restarts them when they crash." But if you do not understand what "automatically" actually means, you will be at a loss when something goes wrong. When a Pod fails to start, which component has stopped doing its job?

Kubernetes is not a magic black box — it is a system of many small programs (processes) that communicate and cooperate with each other. Once you have this structure clear in your mind, every hands-on exercise that follows will be much easier to work through. This lecture has no commands to run, but it lays the most important foundation for everything ahead.

Think of a Kubernetes cluster as a large logistics company. There are warehouses (worker nodes) where workers actually move and stack goods, and there is a central control room (the control plane) that tells those workers "what to place, where, and how many."

Cluster, Control Plane, and Worker Nodes

Let us start with the big picture.

  • Cluster: The overall system that groups multiple machines (servers) together to run containerized applications. Think of it as the entire company. An organization might run several clusters simultaneously — one for production, one for testing, and so on.
  • Node: A single machine that makes up the cluster. It can be a cloud virtual machine (e.g., EC2), a physical server, or even your laptop.

Nodes are divided into two types based on their role.

  • Control Plane (formerly called "master"): The brain that manages the entire cluster and makes decisions. It receives user requests, decides which container to run where, stores the cluster's state data, and responds to failures. It does not run your applications directly — it is the one giving orders.
  • Worker Node: The workfloor where your applications actually run. It receives instructions from the control plane and starts and maintains containers.

In production environments, if the brain goes down the entire cluster is at risk, so the control plane is distributed across multiple nodes to ensure high availability. For testing purposes, it is fine to run both the control plane and workers on a single machine.

One important fact: worker nodes do not run containers directly — they run units called Pods. If Docker's smallest unit is a container, Kubernetes' smallest deployable unit is a Pod. A Pod contains one or more containers, and to scale up an application you simply run more Pods. (Pods are covered in detail in Lecture 3.)

The 4 Core Components of the Control Plane

Inside the control room (control plane) there are four programs, each with a distinct role.

1. API Server (the front desk of the control room)

The sole entry point through which every request entering the cluster passes. When you issue a command with kubectl (the Kubernetes CLI) or through the dashboard, you are actually talking to the API Server. Even the components on worker nodes communicate exclusively through the API Server. As the front desk, it also handles authentication (verifying who you are) and authorization (verifying what you are allowed to do), as well as validating that requests are properly formatted.

2. etcd (the company ledger)

A key-value store that holds all of the cluster's state data. Which nodes exist, which Pods are running where, what the user has requested — every "current fact" is recorded in etcd. This is why etcd is called the cluster's single source of truth. When you create a Pod, that information is written to etcd; when you delete it, it is removed from etcd.

3. Scheduler (the placement coordinator)

Decides which worker node a newly created Pod should run on. Rather than simply finding any open slot, the Scheduler considers the CPU and memory a Pod requires, the available resources on each node, and constraints such as "keep this Pod close to that Pod," then picks the most suitable node. It solves what is essentially a bin-packing problem — fitting multiple applications optimally onto a limited set of servers.

4. Controller Manager (the floor supervisor)

A background process that continuously tries to bring the system back to the user's desired state. For example, if you declared via a Deployment or ReplicaSet that "3 Pods must be running" and one dies, the controller detects this and creates a new Pod object to restore the count. There are dedicated controllers for each resource type — a Node Controller that detects downed nodes, a Deployment Controller that manages rollouts, and so on. Kubernetes' self-healing capability comes from here.

The Controller Manager's job is to ensure the correct number of Pod objects exist as declared — it does not directly start a Pod on a specific node. Actually starting a Pod that has already been assigned to a node is the job of that node's kubelet.

When running in a cloud environment, a Cloud Controller Manager is added to delegate cloud-specific tasks — such as provisioning virtual machines for nodes — to the cloud provider (AWS, GCP, or Azure).

The 3 Core Components of a Worker Node

Each worker node (workfloor) runs three programs that together keep Pods running.

1. kubelet (the on-site crew leader)

An agent that runs on every worker node. It communicates with the control plane (API Server) to receive instructions about which Pods to run on its node, and takes responsibility for keeping those Pods alive and healthy. Think of it as the liaison between a node and the brain.

2. Container Runtime (the container execution engine)

The software that actually runs containers. It pulls images from a registry, starts and stops containers, and manages their resources. Docker comes to mind first, but Docker is only one of several runtimes. Kubernetes can work with any runtime — containerd, CRI-O, and others — as long as it conforms to the standard interface (CRI). The kubelet directs this runtime to carry out actual execution.

3. kube-proxy (the network traffic director)

Manages the network rules on each node so that traffic from inside and outside the cluster is correctly routed and load-balanced to the right Pod.

How a User Command Is Processed

Now let us see how these components work together by following a "create a Pod" request as it flows through the cluster. The key point is that every communication passes through the API Server. The flow is as follows.

1. The user sends a "run this container as a Pod" request to the API Server via kubectl.

2. The API Server verifies authentication, authorization, and request format, then writes the Pod information to etcd.

3. The Scheduler notices the Pod has not yet been assigned to a node. It analyzes resource availability and picks the most suitable worker node, then notifies the API Server of the binding. This decision is also recorded in etcd.

4. The kubelet on the selected worker node is continuously watching the API Server. It spots the Pod assigned to its node and proceeds to start it.

5. The kubelet instructs the node's Container Runtime to pull the image and start the container — the Pod comes up.

6. The kubelet reports "this Pod is now Running" to the API Server, and the state change is saved back to etcd. Now when the user runs kubectl get pod, they see "Running."

One thing to note: in this single-Pod creation flow, the Controller Manager does not appear. Once the Scheduler assigns a node, the kubelet on that node watches the API Server and starts the Pod on its own. The Controller Manager's turn comes separately — when a Deployment or ReplicaSet declares "keep N Pods running," it creates Pod objects to make up any shortfall. Those newly created Pods then go through steps 4–6 above (Scheduler assigns a node → kubelet starts the Pod).

The six steps in chronological order are shown in the sequence diagram below. Notice that every arrow passes through the API Server.

One more thing to consider. What happens if a Pod launched via a Deployment suddenly crashes? The Controller Manager detects that "the desired state (N Pods running) differs from the current state (N-1 Pods)" and creates a new Pod object to fill the gap. That Pod then rides the same flow (Scheduler assigns a node → kubelet starts it) and comes back up on a new node. This is how the cluster heals itself without any manual intervention.

Key Takeaways

  • A cluster is a collection of nodes, divided into the control plane (makes decisions) and worker nodes (run applications).
  • Control plane's 4 components: API Server (sole entry point · authentication), etcd (single source of truth for state data), Scheduler (decides which node a Pod runs on), Controller Manager (self-healing — ensures the declared number of Pod objects exists).
  • Worker node's 3 components: kubelet (watches the API Server and directly starts/maintains Pods on its node), Container Runtime (actually runs containers), kube-proxy (network routing).
  • Worker nodes run Pods, not bare containers.
  • Every request and internal communication must go through the API Server. A single Pod creation flows as: API Server → etcd → Scheduler (node decision) → kubelet on that node (direct execution) → Container Runtime. The Controller Manager creates Pod objects when a Deployment/ReplicaSet needs to maintain a replica count.

03. Pods and Containers - The Smallest Deployable Unit in Kubernetes

Pods and Containers - The Smallest Deployable Unit in Kubernetes

Learning Objectives

  • Understand what a Pod is and how it differs from a container
  • Know the Pod lifecycle and both single-container and multi-container patterns
  • Be able to create a Pod with kubectl and check its status

Content

Why Pods Instead of Containers?

In previous lectures, we learned that a container is "a self-contained unit that bundles application code with everything it needs to run." Yet when you actually work with Kubernetes, you rarely interact with containers directly - the term Pod keeps coming up instead. Why doesn't Kubernetes deal with containers directly?

In Kubernetes, a Pod is the smallest deployable unit. Kubernetes does not manage containers directly; instead, it wraps containers inside an envelope called a Pod. In other words, deploying anything to a cluster means deploying a Pod.

Key point: A container is the "packaging unit." A Pod is Kubernetes's "execution and deployment unit." You never throw a container straight at the cluster - it always goes inside a Pod first.

What Is a Pod? - An Envelope for Containers

A Pod is a group of one or more containers. The key feature is that containers within the same Pod share the following:

  • Network: Containers in the same Pod share a single IP address and can reach each other via localhost. They also share port space.
  • Storage (volumes): They can share directories to exchange data.

Each container still has its own filesystem, so isolation is maintained. Think of a Pod as "roommates living at the same address (shared network and IP)," where each person has their own room (separate filesystem).

Here is something beginners often miss: regardless of whether a Pod has one container or several, all containers in the same Pod are always scheduled on the same node, created together, and destroyed together. You cannot move individual containers to a different node.

Single-Container vs. Multi-Container Patterns

In practice, most Pods contain a single container (single-container pattern). "One application = one container = one Pod" is the most fundamental and recommended approach. When you are starting out, this is the only pattern you need to think about.

So when do you use multiple containers? Only when you have a helper container that must work so closely with the main container that separating them makes no sense. The canonical example is the sidecar pattern:

  • Main container: a Python application serving web content
  • Sidecar container: a container that collects the application's logs and ships them to an external destination

Because these two need to operate on the same data and communicate tightly over localhost, they are placed in the same Pod rather than split into separate Pods. This way they always run together in the same location and are managed as one unit.

As shown in the diagram below, both patterns share an IP and volumes within a single Pod, while each container keeps its own isolated filesystem.

Note: Multi-container Pods are not for grouping things together just because they appear on the same screen. Use this pattern only when the coupling is tight enough that the containers must share a lifecycle. Otherwise, split them into separate Pods.

The Pod Lifecycle - and "Pods Are Ephemeral"

Once a Pod is created, it passes through the following phases:

  • Pending: The cluster has received the request and is preparing - deciding which node to place it on and pulling the image.
  • Running: The Pod has been scheduled to a node and its containers are actively running.
  • Succeeded: (For one-off jobs) All containers have exited successfully.
  • Failed: At least one container exited with an error.

As shown in the state diagram below, a Pod moves through a preparation phase before running, and then ends in either a successful or failed state depending on the nature of the workload.

One concept you must internalize: Pods are ephemeral (temporary). When a Pod dies, Kubernetes does not "revive" it. Instead, it creates a brand-new Pod with the same specification. The new Pod may even receive a different IP address.

Who creates the new Pod when one dies? The restart policy you set when creating the Pod, along with the replication controller inside Kubernetes's controller manager, handles this. The controller constantly compares the "desired state" with the "actual state." When it detects that a Pod is gone, it spins up a new one to restore the desired state. This is the "self-healing" principle from Lecture 1 in action.

In practice, you rarely manage Pods directly. Instead, you use Deployments - covered in the next lecture - to manage Pods on your behalf. This lecture works with Pods directly so you can internalize the concepts.

Wrapping Up

A Pod is the smallest unit Kubernetes works with - an envelope that wraps containers and lets them share networking and storage. Most of the time you use a single-container Pod, and you only bundle multiple containers together when a tightly coupled helper needs to share the same lifecycle. Pods are ephemeral: when one dies it is not revived; a new one takes its place. That is why in production you hand Pod management over to a Deployment rather than dealing with Pods directly - which is the topic of the next lecture.

Key Takeaways

  • A Pod is the smallest deployable unit in Kubernetes - a group wrapping one or more containers. You deploy Pods to the cluster, not bare containers.
  • Containers in the same Pod share an IP, network, and storage volumes and communicate over localhost, but each has its own isolated filesystem. They are always co-located on the same node and are created and destroyed together.
  • Use a single-container Pod by default. Only use a multi-container (sidecar) Pod when a helper container is so tightly coupled to the main container that they must share a lifecycle.
  • The Pod lifecycle is Pending → Running → (Succeeded / Failed). Pods are ephemeral - when one dies the controller creates an identical new Pod to restore the desired state.
  • Key commands: kubectl run (create) → kubectl get pods (list and status) → kubectl describe pod (detail and events) → kubectl logs / kubectl exec (logs and shell access) → kubectl delete pod (delete).

Learning Objectives

  • Understand how to access Pods through a Service (ClusterIP / NodePort).
  • Deploy a Deployment with kubectl and expose it externally via a Service.

Content

Why Not Create Pods Directly?

In Lecture 3 we created a Pod directly with kubectl run. In practice, however, you will rarely — if ever — create Pods by hand. The reason is simple: a Pod is a disposable resource that can disappear at any moment. If a node goes down, someone accidentally deletes the Pod, or the cluster migrates the Pod to another node, that Pod is gone. If you created it by hand, the service stays down until someone manually recreates it.

That is why Kubernetes provides Deployment as a higher-level abstraction on top of Pods. You simply declare the desired state — "keep N Pods running at all times" — and let Kubernetes do the work of actually maintaining that state.

The core philosophy of Kubernetes is to declare what state you want, not how to get there. A Deployment is the canonical example of this declarative mindset.

Three Things a Deployment Does

A Deployment manages a group of Pod replicas. It has three core capabilities.

1. Replication — run multiple identical copies.

If you specify replicas: 3 in a Deployment, Kubernetes ensures exactly three identical Pods are always running. When traffic increases, simply raise the number. With multiple replicas, losing one Pod does not interrupt the service.

2. Self-healing — automatically restart what dies.

A Deployment continuously monitors the number of Pods it manages. If one Pod dies and the count drops from 3 to 2, Kubernetes immediately creates a new Pod to bring it back to 3. The desired state (3 replicas) is restored automatically without any human intervention. This is the single biggest reason to use a Deployment instead of raw Pods.

3. Rolling Update — deploy a new version without downtime.

When upgrading an app from v1 to v2, a Deployment does not replace all Pods at once. It gradually swaps them out one at a time, ensuring the service never goes down (zero downtime).

The rolling update process works as follows:

1. Create one additional Pod running the new version (v2).

2. Once the new Pod is ready and accepting traffic, route requests to it.

3. Terminate one Pod running the old version (v1).

4. Repeat until every Pod runs the new version.

The diagram below illustrates the cycle of bringing up a new Pod and tearing down an old one.

Two safety parameters control the process. maxUnavailable sets how many Pods may be taken down simultaneously, and maxSurge sets how many extra Pods may exist above the desired count. Both default to 25%. With 4 Pods, for example, at most 1 is taken down at a time, and at most 1 extra (5 total) is spun up to fill the gap.

If the new version has a problem, kubectl rollout undo instantly reverts to the previous version. A Deployment retains the revision history to make this possible.

Service - Stable Access to a Shifting Set of Pods

There is one problem left. Every time a Pod dies and is recreated, its IP address changes. As you saw above, a replaced Pod gets a new name and a new IP. How would users of your app know which IP to connect to?

Service solves this. A Service is a stable front door and load balancer placed in front of a set of Pods. It holds a single address that never changes and distributes incoming requests evenly across the Pods behind it (load balancing). Even as Pods die and are recreated with new IPs, the Service automatically refreshes its list of healthy Pods, so users never notice any change.

The Service decides which Pods to send traffic to using a label selector. If you specify "send to Pods with the label app=web", the Service will find and distribute requests to all Pods carrying that label, regardless of how many there are.

There are several types of Service. The two you must know at the beginner level are:

ClusterIP (default) — assigns an address that is reachable only inside the cluster. It cannot be accessed from outside, and is used for Pod-to-Pod communication within the cluster. For example, a web server Pod connecting to a database Pod would access the database through a ClusterIP Service. Because there is no external exposure, this is the safest default choice.

NodePort — opens the same port (in the range 30000–32767) on every node in the cluster and forwards external traffic on that port to the Service. You can reach the cluster from outside using <NodeIP>:<NodePort>, making it well-suited for learning and simple testing.

The two access paths can be summarised as:

  • External user → (NodePort) → Service → multiple Pods
  • Pod inside the cluster → (ClusterIP) → Service → multiple Pods

The architecture diagram below shows how external traffic comes in through NodePort and internal traffic through ClusterIP, both reaching the same Service and then being distributed to the backing Pods.

Key Takeaways

  • Pods are ephemeral, so instead of creating them directly, manage them through a Deployment.
  • A Deployment handles three things: replication (maintaining the desired number), self-healing (auto-recreating failed Pods), and rolling updates (zero-downtime deployments).
  • A rolling update brings up new Pods until they are ready, then replaces old Pods one at a time. maxUnavailable and maxSurge control the pace and safety margin. Use kubectl rollout undo to revert if something goes wrong.
  • Because Pod IPs change constantly, a Service provides a stable address and load balancing. The label selector determines which Pods receive traffic.
  • ClusterIP is for internal cluster communication (default); NodePort exposes the Service externally via a port on each node.
  • Key commands: kubectl create deployment, kubectl scale, kubectl expose, kubectl set image, kubectl rollout status/undo.

05. Deploying with YAML Manifests - Declarative Configuration in Practice

Deploying with YAML Manifests - Declarative Configuration in Practice

Learning Objectives

  • Understand the difference between imperative and declarative deployment approaches
  • Write YAML manifests for a Deployment and a Service from scratch
  • Apply manifests with kubectl apply and verify the results

Content

Why Deploy with YAML?

In lectures 3 and 4, we created Pods and Deployments using commands like kubectl run and kubectl create deployment. One line and you're done — fast and convenient. In real-world environments, however, resources are almost never created this way. Instead, teams describe the desired state in a YAML file and apply that file to the cluster. In this lecture, you will learn why, and practice the technique hands-on.

YAML (YAML Ain't Markup Language) is a human-readable configuration file format. In Kubernetes, it acts as a blueprint that specifies "what resource to run and what it should look like."

Imperative vs. Declarative — The Core Difference

The distinction between the two approaches comes down to one sentence.

  • Imperative: You tell Kubernetes what to do, step by step. Examples: kubectl run, kubectl create, kubectl delete. You explicitly dictate each action.
  • Declarative: You tell Kubernetes what the end state should look like. Examples: YAML file + kubectl apply. Kubernetes figures out on its own how to get there.

Think of it like a restaurant. The imperative approach is walking into the kitchen and barking orders: "Light the burner, grill the patty, toast the bun." The declarative approach is simply saying, "One cheeseburger, please." The kitchen (Kubernetes) compares the current state to the desired state and fills in whatever is missing.

This continuous process of adjusting the current state to match the desired state is called the reconciliation loop. That is precisely what makes the declarative approach so powerful. Applying the same YAML file two or three times always produces the same result (idempotency), and if someone deletes a Pod, Kubernetes automatically restores it to the declared state.

As shown in the diagram below, the reconciliation loop endlessly repeats: compare desired state to current state → adjust if there is a difference → compare again.

The practical advantages of the declarative approach can be summarized as follows:

  • Version control: Store YAML files in Git to track changes and roll back when needed.
  • Reproducibility: Use the same file to deploy identical configurations across development, staging, and production environments.
  • Collaboration: Share the file with teammates and anyone can recreate the exact same resources.
  • Reviewability: Changes can go through code review (pull requests) before being applied.

For experimentation and learning, kubectl run works just fine. But for production environments, YAML + kubectl apply is the right approach.

The Common Structure of Every Manifest

Regardless of whether the resource is a Pod, a Deployment, or a Service, every manifest starts with the same four top-level fields.

| Field | Description |

|-------|-------------|

| apiVersion | The Kubernetes API version to use (varies by resource type) |

| kind | The type of resource to create (Pod, Deployment, Service, etc.) |

| metadata | Identifying information such as the resource name and labels |

| spec | The actual specification describing the desired state of the resource |

The field beginners get wrong most often is apiVersion, because the value differs by resource type. Pods and Services use v1; Deployments use apps/v1. Also, YAML uses indentation (2 spaces) to represent hierarchy, so always use spaces instead of tabs and leave exactly one space after each colon (:).

Exercise 1 — Writing a Deployment Manifest

Let's write a Deployment that runs the nginx web server with 2 replicas. Create a file called nginx-deployment.yaml.


apiVersion: apps/v1          # Deployments use apps/v1
kind: Deployment
metadata:
  name: nginx-deployment      # The name of this Deployment
  labels:
    app: nginx
spec:
  replicas: 2                 # Number of Pods to run (desired state)
  selector:
    matchLabels:
      app: nginx              # Pods with this label are managed by this Deployment
  template:                   # Blueprint for the Pods to be created
    metadata:
      labels:
        app: nginx            # Must match the selector above
    spec:
      containers:
        - name: nginx         # Container list uses '-' because it is a YAML list
          image: nginx:1.27
          ports:
            - containerPort: 80

The critical rule here is that the label in selector and the label in template must match exactly. The Deployment uses this label to identify which Pods it is responsible for. A mismatch will produce an error when you apply the manifest.

Exercise 2 — Writing a Service Manifest

Pods created by a Deployment are only visible inside the cluster, and their IP addresses change every time they restart. To allow external access, you need a Service — a stable entry point into the cluster. Create nginx-service.yaml.


apiVersion: v1               # Services use v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: NodePort             # Expose the service on a port of each node
  selector:
    app: nginx               # Forward traffic to Pods with this label
  ports:
    - port: 80               # The port the Service listens on
      targetPort: 80         # The port to forward to inside the Pod (container)
      nodePort: 30080        # The node port accessible from outside (30000–32767)

The Service also uses the selector label (app: nginx) to find its target Pods. Because this matches the label on the Pods created by the Deployment, the Service automatically distributes traffic across those 2 Pods. You don't need to know any Pod's IP, and even if a Pod dies and is replaced, the label match keeps traffic flowing seamlessly.

As shown in the diagram below, the Deployment and the Service are both connected to the same set of Pods through the shared app: nginx label. All connectivity depends on this label matching correctly.

Exercise 3 — Applying Manifests with kubectl apply and Verifying Results

Apply the manifests to the cluster. Notice that we use apply rather than the imperative create.


kubectl apply -f nginx-deployment.yaml
kubectl apply -f nginx-service.yaml

service/nginx-service created

create only creates a resource if it doesn't already exist, but apply means "make the cluster match this state." If you edit the file and run apply again, only the changed parts are updated. This is the essence of declarative operations.

Key Takeaways

  • The imperative approach tells Kubernetes what actions to perform (kubectl run/create); the declarative approach tells it what the desired state should be (YAML + kubectl apply).
  • The declarative approach is better for version control, reproducibility, and team collaboration. The reconciliation loop guarantees idempotency and self-healing.
  • Every manifest is composed of four fields: apiVersion, kind, metadata, and spec. Pods and Services use v1; Deployments use apps/v1.
  • Both Deployments and Services find their target Pods via selector labels, so label matching is critical.
  • Apply with kubectl apply -f and verify with kubectl get. Running apply again after editing the file updates only the changed portions.
