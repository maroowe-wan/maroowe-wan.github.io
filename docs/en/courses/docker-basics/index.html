01. 컨테이너란 무엇인가

컨테이너란 무엇인가

학습 목표

  • 컨테이너가 등장한 배경과 해결하는 문제를 설명할 수 있다.
  • 가상 머신(VM)과 컨테이너의 구조적 차이를 이해한다.
  • 이미지와 컨테이너의 관계를 구분할 수 있다.

왜 컨테이너인가

"제 컴퓨터에서는 잘 되는데요?" 라는 말, 한 번쯤 들어봤을 겁니다. 개발 환경과 운영 환경이 조금만 달라도 프로그램은 쉽게 망가집니다. 컨테이너는 애플리케이션과 그것이 실행되는 데 필요한 모든 것(라이브러리, 설정)을 하나로 묶어, 어디서나 똑같이 실행되도록 만드는 기술입니다.

핵심 한 줄: 컨테이너는 "내 코드 + 실행에 필요한 모든 것"을 표준 상자에 담아 어디서든 동일하게 실행하게 해줍니다.

VM과 무엇이 다른가

기존의 가상 머신은 하드웨어를 통째로 흉내 내고, 그 위에 게스트 OS 전체를 올립니다. 무겁고 느립니다. 반면 컨테이너는 호스트의 커널을 공유하고 프로세스만 격리하므로, 훨씬 가볍고 빠르게 뜹니다.

위 그림처럼 컨테이너는 OS를 매번 새로 올리지 않기 때문에, 같은 자원으로 더 많은 애플리케이션을 돌릴 수 있습니다.

이미지와 컨테이너의 관계

자주 헷갈리는 두 개념을 정리합시다.

  • 이미지(Image): 실행에 필요한 파일과 설정을 담은 읽기 전용 템플릿. 클래스(설계도)에 해당합니다.
  • 컨테이너(Container): 이미지를 실행한 살아있는 인스턴스. 객체(실체)에 해당합니다.

하나의 이미지로 여러 개의 컨테이너를 띄울 수 있습니다.

주의: 컨테이너를 지워도 이미지는 남습니다. 반대로 이미지를 지우려면 그 이미지를 쓰는 컨테이너가 없어야 합니다.

핵심 요약

  • 컨테이너는 "어디서나 똑같이 실행"을 보장해 환경 차이 문제를 없앤다.
  • VM은 게스트 OS 전체를, 컨테이너는 커널을 공유하고 프로세스만 격리한다 → 컨테이너가 가볍고 빠르다.
  • 이미지는 읽기 전용 템플릿, 컨테이너는 이미지를 실행한 인스턴스다.

📝 Quiz

Q1. 컨테이너가 가상 머신(VM)보다 가볍고 빠른 가장 큰 이유는?

  • 호스트의 커널을 공유하고 프로세스만 격리하기 때문
  • 게스트 OS를 컨테이너마다 새로 설치하기 때문
  • 하드웨어를 직접 제어하기 때문
  • 인터넷 연결이 필요 없기 때문
컨테이너는 게스트 OS 전체를 올리지 않고 호스트 커널을 공유하므로 가볍고 빠르게 실행됩니다.

Q2. 이미지와 컨테이너의 관계로 옳은 것은?

  • 이미지는 실행 중인 인스턴스, 컨테이너는 템플릿이다
  • 이미지는 읽기 전용 템플릿, 컨테이너는 이미지를 실행한 인스턴스다
  • 이미지와 컨테이너는 완전히 같은 말이다
  • 하나의 이미지로는 컨테이너를 하나만 만들 수 있다
이미지는 설계도(읽기 전용 템플릿), 컨테이너는 그 이미지를 실행한 실체이며, 하나의 이미지로 여러 컨테이너를 띄울 수 있습니다.

Q3. 다음 중 컨테이너가 해결하는 대표적인 문제는?

  • 코드의 알고리즘 복잡도
  • '내 컴퓨터에서는 되는데' 식의 환경 차이 문제
  • 네트워크 대역폭 부족
  • 데이터베이스 정규화
컨테이너는 실행에 필요한 모든 것을 함께 묶어 환경 차이로 인한 동작 불일치를 없앱니다.

02. 이미지와 컨테이너 다루기

이미지와 컨테이너 다루기

학습 목표

  • 이미지를 내려받고 컨테이너를 실행할 수 있다.
  • 실행 중인 컨테이너를 조회·중지·삭제할 수 있다.
  • 포트 매핑으로 컨테이너 안의 서비스에 접속할 수 있다.

이미지 받아서 실행하기

Docker Hub에는 이미 만들어진 이미지가 가득합니다. 웹 서버 nginx를 받아 실행해 봅시다.


# 이미지 내려받기
docker pull nginx

# 컨테이너 실행 (-d: 백그라운드, -p: 포트 매핑, --name: 이름 지정)
docker run -d -p 8080:80 --name web nginx

이제 브라우저에서 http://localhost:8080 에 접속하면 nginx 기본 페이지가 보입니다.

팁: -p 8080:80 은 "내 컴퓨터의 8080 포트를 컨테이너의 80 포트에 연결"한다는 뜻입니다. 왼쪽이 호스트, 오른쪽이 컨테이너입니다.

컨테이너의 생애주기

컨테이너는 만들어지고, 실행되고, 멈추고, 삭제되는 흐름을 가집니다.

자주 쓰는 명령어

  • docker ps : 실행 중인 컨테이너 보기 (-a 를 붙이면 멈춘 것까지)
  • docker stop web : web 컨테이너 중지
  • docker start web : 다시 시작
  • docker rm web : 컨테이너 삭제 (먼저 중지해야 함)
  • docker images : 내려받은 이미지 목록
  • docker logs web : 컨테이너 로그 확인

주의: docker rm 은 컨테이너를, docker rmi 는 이미지를 지웁니다. 한 글자 차이지만 대상이 다릅니다.

핵심 요약

  • docker run -d -p 호스트:컨테이너 이미지 로 서비스를 띄운다.
  • 포트 매핑의 왼쪽은 호스트, 오른쪽은 컨테이너 포트다.
  • 조회는 ps, 중지는 stop, 삭제는 rm(컨테이너)/rmi(이미지)로 구분한다.

🛠 Practice

Practice 1. nginx 이미지를 받아, 호스트의 8081 포트로 접속 가능한 'myweb' 이라는 이름의 컨테이너를 백그라운드로 실행하세요.

docker run ____ ____ ____ ____ nginx
Hint
  • 백그라운드 실행은 -d 옵션
  • 포트 매핑은 -p 호스트:컨테이너 형식
  • 이름 지정은 --name 옵션
Show answer 
docker run -d -p 8081:80 --name myweb nginx

-d 로 백그라운드 실행, -p 8081:80 으로 호스트 8081을 컨테이너 80에 연결, --name myweb 으로 이름을 지정합니다.

Practice 2. 실행 중인 'myweb' 컨테이너를 중지한 뒤 완전히 삭제하세요. (명령 두 줄)

docker ____ myweb
docker ____ myweb
Hint
  • 삭제 전에 반드시 중지해야 합니다
  • 중지는 stop, 삭제는 rm
Show answer 
docker stop myweb
docker rm myweb

컨테이너는 실행 중일 때 바로 삭제되지 않으므로, stop 으로 멈춘 다음 rm 으로 삭제합니다. (강제 삭제는 docker rm -f myweb)

03. Dockerfile로 나만의 이미지 만들기

Dockerfile로 나만의 이미지 만들기

학습 목표

  • Dockerfile의 주요 명령어(FROM, COPY, RUN, CMD)를 이해한다.
  • 간단한 애플리케이션을 이미지로 빌드할 수 있다.
  • 빌드 → 실행의 전체 흐름을 설명할 수 있다.

Dockerfile이란

Dockerfile은 "이 이미지를 어떻게 만들지"를 적은 레시피입니다. 한 줄 한 줄이 이미지의 한 층(layer)이 됩니다.


# 1. 베이스 이미지 선택
FROM python:3.12-slim

# 2. 작업 디렉터리 설정
WORKDIR /app

# 3. 코드 복사
COPY app.py .

# 4. 컨테이너 시작 시 실행할 명령
CMD ["python", "app.py"]

팁: 자주 바뀌지 않는 것(의존성 설치)을 위에, 자주 바뀌는 것(소스 코드)을 아래에 두면 빌드 캐시를 잘 활용해 더 빠르게 빌드됩니다.

빌드에서 실행까지의 흐름

직접 빌드해 보기

app.py 와 Dockerfile 이 같은 폴더에 있을 때:


# 현재 폴더(.)의 Dockerfile로 이미지 빌드, 이름은 myapp
docker build -t myapp .

# 빌드한 이미지로 컨테이너 실행
docker run myapp

주의: docker build 끝의 점(.)은 "빌드 컨텍스트=현재 폴더"라는 뜻으로, 빠뜨리면 에러가 납니다.

핵심 요약

  • Dockerfile은 이미지를 만드는 레시피이며, 명령 한 줄이 한 층이 된다.
  • FROM(베이스) → WORKDIR → COPY → CMD(시작 명령) 순서가 기본 골격이다.
  • docker build -t 이름 . 로 빌드하고 docker run 이름 으로 실행한다.

🛠 Practice

Practice 1. node:20-slim 을 베이스로, 현재 폴더의 server.js 를 복사해 'node server.js' 로 실행하는 Dockerfile을 작성하세요. (WORKDIR는 /app)

FROM ____
WORKDIR ____
COPY ____ .
CMD [____, ____]
Hint
  • 베이스 이미지는 FROM 으로 지정
  • CMD는 JSON 배열 형식: ["실행파일", "인자"]
Show answer 
FROM node:20-slim
WORKDIR /app
COPY server.js .
CMD ["node", "server.js"]

FROM 으로 베이스를 정하고, WORKDIR로 작업 폴더를 만든 뒤 COPY로 코드를 넣고, CMD로 시작 명령을 지정합니다. CMD는 큰따옴표를 쓰는 JSON 배열(exec form)이 권장됩니다.

Practice 2. 위 Dockerfile을 'myserver' 라는 이름의 이미지로 빌드하는 명령을 작성하세요.

docker build ____ myserver ____
Hint
  • 이름 지정은 -t 옵션
  • 끝에 빌드 컨텍스트(현재 폴더) 점을 잊지 마세요
Show answer 
docker build -t myserver .

-t 로 이미지 이름(태그)을 정하고, 마지막의 점(.)은 현재 폴더를 빌드 컨텍스트로 사용한다는 의미입니다.
