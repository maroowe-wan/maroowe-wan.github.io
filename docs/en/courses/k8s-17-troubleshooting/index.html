01. "Pod Troubleshooting: From CrashLoopBackOff to Ephemeral Containers"

Learning Objectives

  • Tell apart the most common Pod failure states — CrashLoopBackOff, ImagePullBackOff, Pending, and OOMKilled — and explain the root cause behind each one.
  • Apply a repeatable diagnostic flow using kubectl describe, events, and kubectl logs --previous to narrow down why a Pod is broken.
  • Use kubectl debug and ephemeral containers to inspect a live, running Pod — even one with no shell — and pinpoint runtime, networking, or resource problems.

Body

Why this matters

Sooner or later, every Kubernetes operator stares at a Pod that simply will not behave. The deployment looks fine, the YAML applied cleanly, and yet kubectl get pods shows a status you didn't ask for: CrashLoopBackOff, ImagePullBackOff, Pending, or a container that was just OOMKilled. The good news is that Kubernetes is unusually honest about what went wrong — the information is almost always there. The skill is knowing where to look and in what order.

The golden rule is to debug from the bottom up. Start at the Pod, the lowest healthy unit of work, and only move up the stack to Services and Ingress once you've confirmed the Pod itself is actually running and ready. Chasing an Ingress problem when your Pod is crash-looping is wasted effort. This same bottom-up instinct applies to Deployments, StatefulSets, DaemonSets, Jobs, and CronJobs alike.

In this lecture we'll first learn to read the four headline failure states, then walk a standard diagnostic flow, and finally reach for live-debugging tools when the logs alone don't tell the whole story.

The status field is your first clue

Run this and read the STATUS column carefully — it is the single most useful signal you have:


kubectl get pods
kubectl get pods -o wide        # adds node and Pod IP
kubectl get pods -w             # watch status change in real time

Each status points you toward a different family of root causes. Let's take them one at a time.

CrashLoopBackOff — the container keeps dying after it starts

A Pod has a restart policy, and by default it is Always. When a container exits, Kubernetes faithfully restarts it to keep your app available. But if the container crashes again right after starting — say, your new build throws an exception during initialization — you fall into a loop: start, crash, restart, crash. To avoid hammering the system, Kubernetes waits a little longer before each restart attempt: roughly a few seconds, then ten seconds, then a minute, and so on, capped at five minutes. That growing delay is the back-off, and the whole start-crash-restart cycle is what you see reported as CrashLoopBackOff.

CrashLoopBackOff is not itself an error — it's a symptom. It tells you the container starts but then exits repeatedly. The real bug is why it exits, and that's almost always visible in the logs of the crashed container.

Typical culprits: a bug in application code, a missing environment variable or config file, a failing database connection at startup, a misconfigured command/entrypoint, or a liveness probe that kills a container that was actually healthy but slow to boot.

ImagePullBackOff (and ErrImagePull) — the image never arrived

Here the container never even starts because the kubelet couldn't pull the image. ErrImagePull is the first failed attempt; ImagePullBackOff means Kubernetes is now backing off and retrying. Common causes: a typo in the image name or tag, an image tag that doesn't exist, a private registry with no imagePullSecret configured, or rate limits and network issues reaching the registry. The fix is almost always in the image reference or your registry credentials — not in your application.

Pending — the Pod can't be scheduled

A Pending Pod has been accepted by the cluster but has not been placed on a node. The scheduler couldn't find a node that satisfies the Pod's requirements. Usual reasons: not enough CPU or memory available across nodes (insufficient resources), node selectors / affinity / taints that no node tolerates, or a PersistentVolumeClaim that can't be bound. Pending is a scheduling problem, so the answer lives in the Pod's events and in node capacity — not in the container image or code.

OOMKilled — the container exceeded its memory limit

When a container tries to use more memory than its configured limits.memory, the Linux kernel's out-of-memory killer terminates it, and Kubernetes records the reason as OOMKilled (exit code 137). This often looks like a crash loop, because the container restarts and then gets killed again under load. The fix is either to raise the memory limit, fix a memory leak in the app, or right-size the workload. Always check the container's State and Last State reason to distinguish a true application crash from an OOM kill.

The standard diagnostic flow

Whatever the status, the investigation follows the same three-step rhythm. Learn it once and you'll use it for years.

Step 1 — describe the Pod. This is your richest source of truth. It shows each container's current State, its Last State (with the exit code and reason — this is where you'll spot OOMKilled), restart counts, the resolved image, probe configuration, and — crucially — the Events at the bottom.


kubectl describe pod <pod-name>

Read the Events section from the bottom up. Failed to pull image, Insufficient memory, Back-off restarting failed container, and Liveness probe failed each point you straight at the right failure family.

Step 2 — read the events, including the broader picture. Pod-level events are in describe, but cluster events give context (for example, scheduling failures or node pressure):


kubectl get events --sort-by=.lastTimestamp
kubectl get events --field-selector involvedObject.name=<pod-name>

Step 3 — read the logs, including the previous container. For a CrashLoopBackOff, the live container may be too young to have logged anything useful — it just restarted. The exception you need is in the container instance that just died. That's exactly what --previous gives you:


kubectl logs <pod-name>                     # current container instance
kubectl logs <pod-name> --previous          # the container that just crashed — key for CrashLoopBackOff
kubectl logs <pod-name> -c <container-name>  # a specific container in a multi-container Pod
kubectl logs <pod-name> -f                   # follow live

For CrashLoopBackOff, kubectl logs --previous is the command that most often hands you the answer. The current container has nothing yet; the crashed one has the stack trace.

The flow, then, is: read the status to pick the failure family, run describe to see state and events, then pull logs (with --previous when crash-looping) to read the actual error. Most Pod problems surrender to these three commands, as the diagnostic flow below summarizes.

When logs aren't enough: live debugging with kubectl debug

Sometimes the logs are silent and you need to get inside a running Pod — to check what processes are alive, test network connectivity to another service, or inspect files. The classic move is kubectl exec -it <pod> -- sh. But there's a catch that's increasingly common in production: distroless images have no shell.

Distroless containers are stripped down to little more than your application binary. That makes them small and fast to pull, cheaper to store, and quicker to start — but with no sh, curl, or ps baked in, there's nothing to exec into. So how do you debug a live Pod that has no shell?

The answer is ephemeral containers, driven by kubectl debug. Instead of taking the Pod down, you attach a temporary container into the existing Pod. That debug container brings its own toolset (busybox, curl, ps, netcat, whatever you choose), and — because it joins the same Pod — it shares the network namespace and (with shared process namespace enabled) can even see the troubled container's processes. You're debugging the actual Pod with the problem, not a replica.


# Attach a busybox debug container to a running Pod and drop into a shell
kubectl debug -it <pod-name> --image=busybox --target=<container-name>

The --target flag points the ephemeral container at the troubled container so they share a process namespace. Once you're in, you can investigate live:


# Inside the ephemeral container:
ps aux                          # see processes in the target container
nslookup my-service             # check DNS resolution to another Service
wget -qO- http://my-service:80  # test connectivity to another Pod/Service

Because the ephemeral container lives in the same network context as the broken one, networking problems become tractable: you can ping and telnet around to discover why the app can't reach the service it depends on. This same kubectl debug family can also copy a Pod for safe experimentation or attach a debug shell onto a node — but the day-to-day win is inspecting a live, shell-less Pod without disrupting it.

Tying it back to the bigger picture

Once the Pod is genuinely running and ready, then you climb the stack. If the app still doesn't respond, suspect the Service (check that its selector matches the Pod labels and that its endpoints are populated) and then the Ingress (verify the service.name and service.port, and confirm the backend isn't empty). A quick way to isolate an Ingress or infrastructure problem is to kubectl port-forward directly to the Pod or the Ingress controller Pod and see whether the app answers when you bypass the outer layers. But that's the next step — and you only earn the right to take it after the bottom-up flow has proven the Pod itself is healthy.

Key Takeaways

  • Read the status first. CrashLoopBackOff = container starts then keeps dying; ImagePullBackOff = the image couldn't be pulled; Pending = the scheduler can't place the Pod; OOMKilled = the container blew past its memory limit (exit code 137).
  • CrashLoopBackOff is a symptom, not the bug. The real cause is in the logs of the crashed container — reach for kubectl logs --previous.
  • Follow one repeatable flow: kubectl describe pod (state + events) → kubectl get events → kubectl logs [--previous]. Most Pod failures fall out of these three commands.
  • Debug from the bottom up: confirm the Pod is running and ready before chasing Service or Ingress problems.
  • For live, shell-less Pods, use kubectl debug with ephemeral containers. You attach a tool-rich container into the running Pod, share its network and process namespace, and debug the real problem in place — no shell required, no Pod restart needed.

🛠 Practice

Practice 1. Warmup — Deploy a deliberately broken Pod and read its status.

Apply the manifest below to your cluster. Then run kubectl get pods (and optionally -o wide and -w) and record:

1. The STATUS value you observe.

2. The RESTARTS counter after waiting about 60 seconds.

3. Which failure family does this status belong to, and what does that tell you about where the root cause will be found?

# broken-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: broken-app
spec:
  containers:
  - name: app
    image: busybox:1.36
    command: ["sh", "-c", "echo 'starting...' && exit 1"]
Hint
  • If you see `Error` before `CrashLoopBackOff`, that is the very first crash; the back-off label appears on the second attempt and beyond.
  • Use `kubectl get pods -w` and leave it running for a minute. Notice how the gap between RESTARTS increments grows longer over time — that is the back-off in action.
  • Add `-o wide` to see which node the Pod landed on and what IP it was assigned. These details are useful in later, multi-node investigations.
Show answer 
# 1. Apply the manifest
kubectl apply -f broken-pod.yaml

# 2. Observe status
kubectl get pods
# Expected output (after a short wait):
# NAME         READY   STATUS             RESTARTS   AGE
# broken-app   0/1     CrashLoopBackOff   2          90s

# 3. With -o wide to also see Node and Pod IP
kubectl get pods -o wide

# 4. Watch real-time transitions (Ctrl-C to stop)
kubectl get pods -w
# You will see:  Running → Error → CrashLoopBackOff → ... as the back-off grows

The container command exits with code 1 immediately after printing one line. Kubernetes honours the default restartPolicy: Always and faithfully restarts the container — but it crashes again every time. To avoid overwhelming the node, Kubernetes applies an exponential back-off delay between restart attempts (~0 s → 10 s → 20 s → 40 s … capped at 5 minutes). The combined start-crash-restart cycle is what the STATUS column reports as CrashLoopBackOff. The RESTARTS counter confirms how many times this cycle has fired. CrashLoopBackOff belongs to the 'container starts then immediately dies' failure family, so the root cause will be in the application logs of the crashed container — not in scheduling or in the image pull step.

Practice 2. Core #1 — Run the full three-step diagnostic flow on the broken Pod.

Using the broken-app Pod you deployed in Task 1:

1. Run kubectl describe pod broken-app and locate: (a) the container's current State and Last State, (b) the exit code in Last State, and (c) the Events section.

2. Run kubectl get events filtered to this Pod and identify any notable messages.

3. Run kubectl logs broken-app and then kubectl logs broken-app --previous. Note which command produces the useful output and why.

Write down the exit code and the last log line you see.

# No new manifest needed — use broken-app from Task 1.
# Hint: if broken-app is no longer present, re-apply broken-pod.yaml first.
#
# Commands to fill in:
kubectl describe pod broken-app
kubectl get events --field-selector involvedObject.name=broken-app --sort-by=.lastTimestamp
kubectl logs broken-app
kubectl logs broken-app --previous
Hint
  • In the `describe` output, scroll past the long metadata block. The `Containers:` section and then `Events:` at the very bottom are the two sections that matter most.
  • If `kubectl logs broken-app` returns nothing, that is expected for a back-off state. Do not assume the command failed — try `--previous` immediately.
  • Exit codes are a quick triage signal: 0 = clean exit (unusual for a crash), 1 = application error, 137 = OOMKill (128 + signal 9), 143 = SIGTERM, 127 = command not found.
Show answer 
# Step 1 — describe
kubectl describe pod broken-app
# Key fields to find in the output:
#
# Containers:
#   app:
#     State:          Waiting
#       Reason:       CrashLoopBackOff
#     Last State:     Terminated
#       Reason:       Error
#       Exit Code:    1          <-- the container exited with a non-zero code
#       Started:      <timestamp>
#       Finished:     <timestamp>
#     Restart Count:  N
#
# Events (at the bottom, read newest last):
#   Warning  BackOff   ...  Back-off restarting failed container

# Step 2 — events filtered to this Pod
kubectl get events --field-selector involvedObject.name=broken-app --sort-by=.lastTimestamp
# Notable message: 'Back-off restarting failed container app in pod broken-app'

# Step 3a — current container (likely empty or not useful)
kubectl logs broken-app
# Output: '' (empty, because the current instance is in Waiting/back-off, not Running)

# Step 3b — PREVIOUS container (the one that just crashed)
kubectl logs broken-app --previous
# Output:
# starting...
#
# 'starting...' is the last (and only) log line before exit code 1.

The three-step flow — describe → events → logs (with --previous) — is the core diagnostic loop for every Pod failure. describe gives you the authoritative view of container state and exit codes; the Events section at the bottom is where the cluster narrates what happened in chronological order. kubectl get events scoped to the Pod adds scheduling or node-level context that describe can miss. For a crash-looping Pod, the current container instance is in Waiting state with zero log output because it hasn't run yet (it is in back-off). The container that actually crashed is the previous instance, and --previous fetches its stdout/stderr log — which is where the stack trace or configuration error lives. Exit code 1 is a generic application error; exit code 137 would indicate OOMKill, and exit code 127 would mean 'command not found'.

Practice 3. Core #2 — Identify an ImagePullBackOff and a Pending Pod, then fix one.

Apply both manifests below.

Part A (ImagePullBackOff): Apply bad-image.yaml. Run kubectl describe pod bad-image and pinpoint the event that tells you exactly what went wrong with image pulling. Then fix the image reference and patch the Pod (or delete-and-recreate) so the Pod reaches Running.

Part B (Pending): Apply hungry-pod.yaml. Use kubectl describe pod hungry-pod to find out why the scheduler could not place it. What section tells you the reason, and what is the root cause?

# bad-image.yaml  — typo in image tag
apiVersion: v1
kind: Pod
metadata:
  name: bad-image
spec:
  containers:
  - name: app
    image: nginx:DOESNOTEXIST999

---
# hungry-pod.yaml  — requests more memory than any node has
apiVersion: v1
kind: Pod
metadata:
  name: hungry-pod
spec:
  containers:
  - name: app
    image: nginx:stable
    resources:
      requests:
        memory: "9999Gi"
        cpu: "1m"
Hint
  • For ImagePullBackOff, the first event is usually `ErrImagePull` (first failed pull) and then `ImagePullBackOff` starts appearing. Both are reported under `Warning Failed`.
  • To fix the bad-image Pod, you cannot edit the `image` field of a running Pod in-place (spec.containers[].image is immutable after scheduling for most fields). Delete the Pod and re-create it with the corrected YAML.
  • For the Pending Pod, look at the `Conditions` block in `describe` output — `PodScheduled: False` with a message confirms a scheduling failure before you even reach the Events section.
Show answer 
# ── Part A: ImagePullBackOff ──────────────────────────────────────────────

# Apply
kubectl apply -f bad-image.yaml

# Diagnose
kubectl describe pod bad-image
# Look for in Events:
#   Warning  Failed   ...  Failed to pull image "nginx:DOESNOTEXIST999":
#                          rpc error: ... manifest for nginx:DOESNOTEXIST999 not found
#   Warning  Failed   ...  Error: ErrImagePull
#   Warning  BackOff  ...  Back-off pulling image "nginx:DOESNOTEXIST999"

# Fix: delete and recreate with a valid tag
kubectl delete pod bad-image
# Edit bad-image.yaml: change  image: nginx:DOESNOTEXIST999  to  image: nginx:stable
kubectl apply -f bad-image.yaml

# Confirm
kubectl get pods bad-image
# NAME        READY   STATUS    RESTARTS   AGE
# bad-image   1/1     Running   0          15s

# ── Part B: Pending ───────────────────────────────────────────────────────

# Apply
kubectl apply -f hungry-pod.yaml

# Diagnose
kubectl describe pod hungry-pod
# Look for in Events:
#   Warning  FailedScheduling  ...  0/N nodes are available:
#            N Insufficient memory.  preemption: 0/N nodes are available ...
#
# The Events section AND the 'Conditions:' block (PodScheduled = False) both
# confirm this is a scheduling failure due to insufficient memory on all nodes.

# Root cause: 9999Gi memory request far exceeds cluster capacity.
# Fix (not applied in this task, but the answer): lower requests.memory to a
# realistic value, e.g. 256Mi.

ImagePullBackOff and Pending are diagnosed entirely through describe events — never through kubectl logs, because the container never even started. For ImagePullBackOff, the event chain is: Failed (ErrImagePull on first attempt) → BackOff (subsequent retries). The event message names the exact image and the registry error, pointing you straight at the image reference or registry credentials. For Pending, the scheduler writes a FailedScheduling event that lists exactly which resources are exhausted or which constraints (affinity, taint/toleration) are unsatisfied. Because this is a scheduling problem, the fix lives in the Pod spec (resources, node selectors, tolerations) or in cluster capacity — not in the container image or application code.

Practice 4. Applied — Debug a distroless (shell-less) Pod with kubectl debug and an ephemeral container.

Apply distroless-app.yaml below. The Pod will run successfully, but imagine that in production you need to verify:

  • Whether the process inside is actually listening on port 8080.
  • Whether DNS resolution for kubernetes.default works from inside the Pod.

Because the container is distroless, kubectl exec -it distroless-app -- sh will fail. Use kubectl debug to attach a busybox ephemeral container to the running Pod, then run the two checks inside it.

# distroless-app.yaml
# Uses the official distroless static image (no shell, no package manager)
apiVersion: v1
kind: Pod
metadata:
  name: distroless-app
spec:
  containers:
  - name: app
    # gcr.io/distroless/static-debian12 ships zero shell tooling
    image: gcr.io/distroless/static-debian12
    # The entrypoint is overridden with 'sleep' from a shared volume trick;
    # for this exercise we just let it run its default no-op entrypoint so
    # the Pod stays Running long enough to debug.
    command: ["/bin/sleep", "3600"]

# NOTE: gcr.io/distroless/static-debian12 does include /bin/sleep but NO sh/bash/curl/ps.
# If your cluster cannot reach gcr.io, substitute:
#   image: busybox:1.36
#   command: ["sh", "-c", "while true; do sleep 3600; done"]
# and then try to exec sh into it first to confirm, then attach the ephemeral container anyway.

---
# Commands to complete (fill in the blanks):
# kubectl exec -it distroless-app -- sh          # should fail with 'exec: sh: not found'
# kubectl debug -it distroless-app --image=______ --target=______
# Inside the ephemeral container:
# nslookup ______
# wget -qO- ______   OR   nc -zv localhost ______
Hint
  • If `kubectl debug` returns 'unable to inject ephemeral container' and your cluster is older than Kubernetes 1.23, the EphemeralContainers feature gate may be disabled. Check `kubectl version` — ephemeral containers are stable (GA) from v1.25.
  • Inside the ephemeral busybox shell, `ps aux` might show only the busybox processes if the PID namespace is not shared. Confirm `--target=<container-name>` matches the exact container name from `kubectl describe pod distroless-app` under `Containers:`.
  • After you exit and re-run `kubectl describe pod distroless-app`, look for the `Ephemeral Containers:` section near the bottom. The container is listed with `State: Terminated` — this is expected and normal.
Show answer 
# Step 1 — apply and verify the Pod is Running
kubectl apply -f distroless-app.yaml
kubectl get pods distroless-app
# NAME             READY   STATUS    RESTARTS   AGE
# distroless-app   1/1     Running   0          10s

# Step 2 — confirm exec fails (no shell in distroless)
kubectl exec -it distroless-app -- sh
# error: Internal error occurred: error executing command in container:
#        failed to exec in container: ... exec: "sh": executable file not found in $PATH

# Step 3 — attach a busybox ephemeral container
kubectl debug -it distroless-app --image=busybox:1.36 --target=app
# Kubernetes injects a new ephemeral container named 'debugger-XXXX' into the
# running Pod. It shares the Pod's network namespace (same IP, same ports)
# and, because --target=app is set, also shares the app container's PID namespace.
# You land inside a busybox shell prompt.

# Step 4 — run checks INSIDE the ephemeral container shell:

# Check 1: DNS resolution
nslookup kubernetes.default
# Expected output (cluster DNS resolves the kubernetes Service):
# Server:    10.96.0.10
# Address:   10.96.0.10:53
# Name:      kubernetes.default.svc.cluster.local
# Address:   10.96.0.1

# Check 2: see processes from the target container (requires --target / shared PID ns)
ps aux
# You should see the sleep process owned by the distroless container.

# Check 3 (bonus): test connectivity to the Kubernetes API
wget -qO- https://kubernetes.default/healthz --no-check-certificate
# Expected: ok

# Step 5 — exit the ephemeral shell
exit
# The ephemeral container stops but stays attached to the Pod spec (Kubernetes does
# not remove ephemeral containers after exit — they are visible in 'describe').

Distroless images exclude shell binaries to reduce image size and attack surface. kubectl exec therefore has nothing to execute when you request sh. kubectl debug --image=busybox:1.36 --target=app solves this by injecting an ephemeral container — a temporary, namespaced-scoped container defined in the Pod's ephemeralContainers spec field. Because the ephemeral container is part of the same Pod, it automatically shares the Pod's network namespace: same IP address, same loopback, same open ports. The --target flag additionally enables shared PID namespace with the named container, so ps aux can see the app's processes. This means you can test DNS (nslookup), check open ports (nc -zv localhost 8080), curl internal endpoints, or inspect files — all on the real, live Pod — without restarting it or needing a shell in the original image. Ephemeral containers are persistent in the Pod spec after exit (run kubectl describe pod distroless-app to see them under Ephemeral Containers), but they consume no resources once stopped.
