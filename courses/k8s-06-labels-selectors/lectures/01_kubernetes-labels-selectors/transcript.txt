The biggest advantage of Kubernetes is that it 
can abstract away individual servers for you. Now,  
you don’t need to worry about specific details of 
individual servers. Instead, Kubernetes collects  
all the details of each node, such as memory, CPU, 
and other stuff. Plus, it has a built-in scheduler  
that can decide where to put your workloads. Let’s 
go over a simple example of how the scheduler  
selects a node. For instance, you have a pod that 
requires 2 CPUs and 4 Gi of memory. The first step  
for the scheduler is to filter out all nodes 
that do not meet the criteria, perhaps because  
they don't have enough resources. The remaining 
nodes are called feasible nodes. Now, the next  
step is for the Kubernetes scheduler to run a set 
of functions to score the feasible nodes. It then  
selects a node with the highest score among the 
feasible ones to run the Pod. If you get multiple  
feasible nodes with the same score, the scheduler 
will randomly select one. This process is called  
binding. Now, in the real world, you may have 
a bunch of different nodes in your Kubernetes  
cluster. Maybe you have Windows nodes, nodes 
with high CPU for running CPU-intensive tasks,  
or memory-optimized nodes. In the cloud, it's 
common to have spot nodes that are cheaper but  
can be terminated at any time. So, your app must 
be capable of recovering quickly. Some nodes may  
use the amd64 architecture, while others use ARM, 
such as Graviton nodes. These are again cheaper,  
but your application must be compiled to run on 
ARM first. You might also have GPU nodes to run  
machine learning pipelines. Other nodes may have 
fast SSD disks attached locally. For example,  
if you decide to run Kafka in Kubernetes, 
you would definitely want fast local SSDs  
to improve performance. And so on. With all these 
different types of servers, the default scheduler
