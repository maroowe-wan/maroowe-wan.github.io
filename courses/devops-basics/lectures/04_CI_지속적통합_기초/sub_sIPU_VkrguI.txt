1999년 TDD로 유명한
켄트백이 애자일 개발방법론 중 하나인 익스트림 프로그래밍을 제안합니다

그에 따라 고객들의 시시각각 변하는 요구사항을
발 빠르게 대응할 필요가 생겼습니다
이를 위해 켄트백은 익스트림 프로그래밍을 통해
몇 가지 실천 방안을 제시하는데요

CI가 무엇일까요 도대체
CI는 지속적 통합이라는 뜻으로
개발을 진행하면서도 품질을 관리할 수 있도록
여러 명이 하나의 코드에 대해서
수정을 진행해도 지속적으로 통합 하면서
관리할 수 있음을 의미합니다
하지만 이 설명만 봐선
도대체 CI의 어떤 점이
고객의 요구사항에 발 빠르게 대응할 수 있는지
잘 감이 안 오실수도 있습니다
한번 CI가 없을 때로 가보겠습니다
CI가 있기 전에 개발자들은 일명 머지데이라는 날을 통해
모든 분기 소스코드를 병합해야 했습니다
이는 굉장히 많은 수작업이 동반되었고
결과적으로 많은 리소스를 낭비하게 되었습니다
또한 개발자들이 코드를 병합할 때

반복이 된다면 맨 마지막 코드에서 에러가 발생했을 경우
어느 부분에서 에러가 발생했는지
디버깅하기 굉장히 어려울 것입니다
반면에 CI를 활용한다면 업무를 잘게 잘게 쪼개서
코드 작성이 완료될 때마다
이를 빠르게 빠르게
자동으로 병합을 할 수 있을 것입니다
이제 CI의 필요성을 알았으니
전체적인 CI 과정에 대해 알아보겠습니다
먼저 개발자가 코드 병합을 요청하면
젠킨스와 같은 CI 툴이 빌드와 테스트를 진행합니다
이때 이상이 없으면 코드를 병합합니다
하지만 문제가 발생할 경우 이를 개발자에게 빠르게 피드백해줍니다
하지만 이러한 CI 툴을 사용한다고 해서
진정한 CI 라고 할 수 없습니다
마틴 파울러는 다음과 같은
4가지 규칙을 지켜야 진정한 CI 라고 말합니다
첫째 모든 소스코드가 살아 있고
누구든 현재 소스에 접근할 수 있는
단일 지점을 유지할 것
둘째 빌드 프로세스를 자동화해서
누구든 소스로부터 시스템을 빌드 할 수 있게 할 것
셋째 테스팅을 자동화해서
언제든지 시스템에 대한 건전한
테스트 수트를 실행할 수 있게 할 것
넷째 누구든 현재 실행파일을 얻으면
지금까지 가장 완전한 실행파일을 얻었다는 확신을 하게 할 것
이 4가지를 지켜야 진정한 CI 라고 말할 수 있을 것입니다
이제 코드의 통합을 이뤄냈으니
배포만 하면 사용자들은
본인들이 원하는 요구사항을 만족할 수 있을 것입니다
현재 저희의 서비스의 규모가 작아
1대의 서버에만 배포 하면 되는 상황입니다
이때는 서버에 들어가 직접 배포 스크립트를 수동으로 실행시켜주면
큰 문제가 없어 보입니다
하지만 서버의 규모가 커져
서버가 수십, 수백대로 늘어났다고 가정해 보겠습니다
이때 모든 서버에 일일이 접속해
배포 스크립트를 수동으로 실행시킨다는 것은
굉장히 어려운 작업일 것입니다
그래서 CI의 연장선의 개념인 CD라는 개념이 등장합니다
CD는 지속적 배포라는 뜻으로
빌드의 결과물을 프로덕션으로 릴리스하는 작업을
자동화하는 것을 의미합니다
CD에 대해 공부하는 도중 CD가 지속적 제공과 지속적 배포라는
말과 혼용되어 사용되고 있다는 것을 알게 되었습니다
저희도 이 둘을 구분하는 것이 굉장히 헷갈렸는데요
이 둘의 차이는 마지막 프로덕션으로
릴리스하는 Deploy 작업의 자동화 여부입니다
그림을 보시면 지속적 통합에서
빌드와 테스트까지의 자동화가 이루어졌습니다
이후 빌드의 결과물을 Staging 환경에 자동으로 릴리즈하고
지속적 제공은 여기서 프로덕션으로 릴리즈하는 작업을
수동으로 진행해줘야 합니다
이 둘 사이에 QA와 같은 행위를 할 수도 있고요
지속적 배포는 지속적 제공 과는 다르게
마지막으로 프로덕션으로 릴리스하는
Deploy 작업까지 자동화됐다는 것을 알 수 있습니다
다음은 전체적인 CI/CD 흐름에 대해 알아보겠습니다
개발자가 코드의 병합을 요청하면
CI/CD 툴이 빌드와 테스트를 진행합니다
이때 에러가 발생하면 개발자에게 빠르게 피드백을 주고
만약에 이상이 없다면 코드를 병합합니다
여기까지 CI의 과정과 똑같습니다
이후 CD 과정을 거쳐
저희의 서버에 자동으로 배포 할 수 있게 됩니다
CI/CD 툴로는 Jenkins, Travis CI, Github Actions 등이 있는데요
자세한 설명은 생략하도록 하겠습니다
그럼 마지막으로 전체 CI/CD에 대해 정리하겠습니다
CI는 고객의 요구사항에 빠르게 대응하기 위해 나온
익스트림 프로그래밍의 실천 방안 중 한 가지입니다
여러명이 하나의 코드에 대해서 수정을 진행해도
지속적으로 통합 하면서 관리할 수 있음을 의미합니다
CD는 CI의 연장선에 있는 개념으로서
빌드의 결과물을 프로덕션으로
지속적으로 배포하는 것을 의미합니다
다음으로 무중단 배포는 레넌이 이어서 발표를 도와드리겠습니다
이어서 무중단 배포에 대해 설명 해드리겠습니다
앞서 찬이 CI/CD에 대해 잘 설명해 주었는데요
하지만 아직까지 문제점이 하나 남아 있습니다
앞서 설명드린 배포 자동화를 통해 서비스를
운영하고 있다고 가정해 보겠습니다
새로운 서비스를 배포하기 위해서는
기존 서비스를 종료하고
새로운 서비스를 시작해야 합니다
하지만 이 두 행위 사이에는
다운타임이라는 것이 발생하게 됩니다
다운타임 동안 사용자들은
저희 서비스를 이용하지 못하게 됩니다
이를 해결하기 위해 나온 개념이 무중단배포 입니다
그래서 무중단 배포 구현 방법에는
AWS에서 Blue-Green을 활용한 무중단 배포 방법
도커를 이용한 무중단 배포 방법
L4, L7 스위치를 이용한 무중단 배포 방법
그리고 Nginx를 이용한 무중단 배포 방법 등등이 있는데요
마지막에 보이는 이 방법이 쉽고
저렴하다는 점에서 많이 사용되고 있는 것 같았습니다
그래서 무중단배포를 설명드리기에 앞서
무중단배포를 공부하면서
자주 나왔던 용어에 대해 설명드리도록 하겠습니다
먼저 리버스 프록시 입니다
리버스 프록시는 인터넷과 서버 사이에 위치한 중계서버로
클라이언트가 요청한 내용을 캐싱합니다
그리고 서버 정보를 클라이언트로부터 숨길 수 있어
보안에 용이합니다
그림을 보시면  클라이언트가 요청을 보내면
리버스 프록시가 이를 서버에 대신 전달해 줍니다
서버에 입장에서 클라이언트로부터
직접 요청을 받는 것이 아닌
리버스 프록시라는 대리서버를 통해 요청을 받기 때문에
보안에 굉장히 용이하다고 할 수 있습니다
다음으로 로드 밸런싱이라는 개념인데요
로드 밸런싱은 부하분산 이라는 뜻으로
서버에 가해지는 부하를 분산해주는 역할을 합니다
하나의 서버가 멈추더라도 서비스 중단 없이
다른 서버가 서비스를 계속 유지할 수 있는
무중단배포를 가능케 해줍니다
그래서 그림을 보시면
클라이언트가 요청을 보내면
로드 밸런서라는 친구가
요청을 위아래로 분산해주는 모습을 보실 수 있습니다
로드 밸런싱 알고리즘에 대해서도
여러 가지가 있는데요 이 그림에서 적용한 알고리즘은
라운드로빈 방식의 알고리즘입니다
추가로 알고리즘에 대해 학습하고 싶으신분들은
로드 밸런싱 알고리즘이라는 키워드로 학습해보시면 좋을 것 같습니다
이제 본격적으로 무중단배포 방식에 대해
설명드리도록 하겠습니다
중단배포 방식에는 Rolling 배포, Canary 배포,
Blue/Green 배포 이렇게 세 가지가 있습니다
먼저 Rolling 배포에 대해 설명드리도록 하겠습니다
Rolling 배포는 무중단배포의 가장 기본적인 방식으로
서버를 차례대로 업데이트시키는 방식입니다
현재 저희가 세대의 서버를 통해
서비스를 운영중이라고 가정해 보겠습니다
Rolling 배포를 적용시키기 위해서는 세대의 서버 중
한대의 서버에 로드 밸런싱을 라우팅하지 않도록 설정해 줍니다
그 다음 이 서버를 새로운 버전으로
업데이트 시켜줍니다
그 후 다시 라우팅 설정을 통해 다시 추가해 줍니다
두 번째 서버도 마찬가지로
업데이트 시켜준 후 라우팅하도록 설정해 줍니다
세 번째 서버도 업데이트를 시켜주고
다시 라우팅을 추가하도록 해 줍니다
보신 바와 같이 Rolling 배포는
이 세대의 서버를 차례대로 업데이트시키는 방식입니다
그래서 Rolling 배포의 장점은 인스턴스를 추가하지 않아도 돼서
관리가 간편하다는 점이 있습니다
하지만 사용 중인 인스턴스의 트래픽이 몰릴 수 있다는 단점이 있고
구버전과 신버전의 공존으로 인한 호환성문제가 발생할 수 있습니다
다음으로  Canary 배포 입니다
Canary 배포는 옛날 광부들이 유독가스에 민감한
카나리아 새를 이용해 가스누출위험을 감지했던 것에서 유래해서
잠재적인 문제들에 대해 미리 대비하고자 나온 방식입니다
신버전을 소수의 사용자들에게만 배포하고
문제가 없는 것이 확인되면
점진적으로 다른 서버에 신버전을 배포하는 방식입니다
Canary 배포도 Rolling 배포와 마찬가지로
첫번째 서버에는 라우팅을 하지 않도록 설정한 후
서버를 업데이트 시켜주고 다시 라우팅하도록 설정해 줍니다
그래서 이 과정에서 이 업데이트된 서버가 문제없이
정상적으로 잘 작동된다는 것이 확인되면
나머지 3대의 서버에도 업데이트를 시켜주는 방식입니다
그래서 Canary 배포의 장점으로는
문제 상황을 빠르게 감지할 수 있습니다
그리고 A/B 테스트로도 활용이 가능합니다
하지만 모니터링 관리 비용이 들고
Rolling 배포와 마찬가지로
구버전과 신버전의 공존으로 인한 호환성 문제가 발생할 수 있습니다
마지막으로 Blue/Green 배포 입니다
Blue를 구버전 Green을 신버전으로 지칭합니다
구버전과 동일하게 신버전의 인스턴스를 구성합니다
신버전 배포시 로드 밸런서를 통해
신버전으로만 트래픽을 전환하는 방식입니다
그림을 보시면 위쪽에는 현재
서비스 중인 서버의 그룹이 있습니다

이 상황에서 새로운 서버를 적용 시키고 싶다면
아래에 있는 대기중인 서버에 먼저
새로운 버전으로 업데이트 시켜줍니다
그 후 위쪽으로 향하고 있던 트래픽을

Blue/Green 배포를 하실 수 있습니다

아래쪽에 있는 그룹이 서비스 중인 서버가 되겠죠
그래서 위에서 다시 대기중인 서버가 된 그룹을
다음 새로운 서비스로 업데이트시킬 때 활용할 수 있습니다
Blue/Green 배포의 장점은 배포하는 속도가 빠르고
신속하게 롤백이 가능하다는 점입니다
또 남아있는 기존 버전의 환경을
다음 배포에 재사용할 수 있습니다
하지만 시스템 자원이 두 배로 필요하다는 점이 단점으로 작용합니다

무중단배포는 다운타임 없이
서버를 운영할 수 있게 해주는 배포 방식입니다
