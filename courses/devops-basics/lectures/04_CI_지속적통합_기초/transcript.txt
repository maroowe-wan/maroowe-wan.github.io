## 영상 2: The ONLY Continuous Integration (CI) Tutorial you need as an Engineer (https://www.youtube.com/watch?v=MIWH2CpVyXs)

In this video we're going to learn exactly why we need a CI pipeline in a software development process and how to build one that automatically checks your application code quality and automatically suggests fixes for any issues that it discovers in your code.

Let's say you are in a team of software developers and you're working on a specific feature. The Git workflow that your team is using is trunk-based development, which is a standard workflow in today's modern DevOps-driven development. You always have the main branch in a releasable state that triggers an automated CI/CD pipeline whenever code changes are pushed into the main branch.

So you have a fully streamlined release pipeline with no bottlenecks, meaning no manual intervention at any stage of the pipeline. The pipeline deploys all the way to at least the dev environment and if you have really good test coverage, maybe to the staging or even production environment without any manual testing or code checks in the process.

But for this to work you need to have lots of tests to make sure what you are committing to your main branch can be deployed and released to the end users. Various tests should be run to test different aspects of the code changes that are about to be deployed.

Now what are those tests that we want to run in this release pipeline? You will have unit tests or integration tests that test your code logic with different inputs and parameters to make sure that it's able to handle different scenarios. You may have security tests to scan all your dependencies and libraries for known issues or scan your code for any hardcoded secret data or any security best practices in logic.

Finally you also have code quality tests. Your application code may be working so the functionality is fine, it may be even secure but your code quality may be bad or unmaintainable. Let's say you have lots of code duplications because instead of putting things into a function you are just duplicating the logic in multiple places, which is a bad practice.

Or maybe you are using old APIs or old library versions. Maybe you have introduced risk for null exceptions in your Java application. Maybe you don't have a good test coverage for your code. These are the tests for your code quality.

Trunk-based Git workflow is usually an ideal workflow to use, especially in DevOps-driven projects. However in practice a lot of teams actually use a different Git workflow which is called a feature branch workflow. The core idea behind the feature branch workflow is that all feature branch development should take place in a dedicated branch instead of the main branch.

This way multiple developers or engineers can work on a specific feature without interfering with the main code base. You create a feature branch and make your changes there and when you're ready to merge you create a merge request. All the tests will run on the merge request to act as a gatekeeper and decide whether the changes should be merged into the main.

That's why we want to run those tests on merge requests before the code even ends up in the main branch. Because imagine your code merged into the main branch and then the tests ran and they failed because your code had issues. Those issues were discovered and the pipeline failed but other developers also merged their code changes.

They had just one or two issues but now even if they fix their own issues their code changes cannot be released because your broken code is also in the main branch. So it will block all other code changes from going to development until you fix your code issues.

That's why we want to run tests on the merge request instead of merging it into main and then running the tests. Now imagine feature development takes longer and you have tens of commits. Now you're done, your feature works, you have tested it a little bit manually and now you're ready to merge it into the main and you create a merge request.

The merge request triggers all the tests and those tests detect a ton of issues that you have introduced. Your code works but you copy-pasted some code snippet from the internet with very bad coding practice and even added some security issues to the application. Now you have to go and spend probably hours or days figuring out how to fix those issues.

You could have saved yourself all that time if you tested your code changes right away on the very first commit instead of collecting those commits and then running tests later on the merge request.

So what if you had those tests running even before the merge request, on every single commit in the feature branch? You introduce a library, commit the changes and immediately see through the code scan that this library has issues. Then you make another change, commit and trigger pipeline. It will run test to see if your code changes are fine. If not, you can fix the issue right away and move on to the next one.

Especially as a junior engineer this actually lets you learn so much of programming best practices while you're developing because it gives you this feedback loop of what you are doing wrong. That process of committing small code changes frequently and running tests and scans on them to identify issues is called continuous integration or CI.

The pipeline that runs those tests in order to discover any issues is called a CI pipeline. It is a best practice to build CI pipelines for feature branches. So you aren't deploying anything in the feature branch. Obviously there's no CD (continuous deployment). You're just testing your code changes for any issues frequently to fix them early enough.

We also kept moving those tests earlier and earlier in the development process, which is a general concept of shifting left or shifting automated testing left. There is even an option to find code issues even earlier than in the CI pipeline of a feature branch. You already have this on your local computer as a developer when you use IntelliJ IDE for example, which has built-in code checks that show you while you're developing if you have written dirty code.

IDEs can show you while developing if you have written dirty code which can be cleaned up and made better or more efficient or if you just created a code duplication. Those built-in inspections will even give you auto suggestions to update the library version to the latest or remove the code duplication and so on.

If you have any linter extensions with whatever code editor you're using you can also have that locally in your development environment. Of course we can assume that some developers may ignore those highlights that IDE is showing or they may not even know how to use those auto suggestions.

That's why we want to have those tests in the pipeline to run automatically because we don't want to rely on individual developers doing everything correctly on their local machine. We want to centralize this process and automate it. That's why CI pipelines are so important.

You notice that we have multiple places where the code issues will be detected: starting from local IDE to feature branch CI pipeline, merge request pipeline and even when the code gets merged into the main branch it will run the same code scans and automated tests again.

Why? Because what if other engineers merged their code changes before us? Maybe there is an issue between that new code and ours. So we want to check on every step.

둘째 빌드 프로세스를 자동화해서 누구든 소스로부터 시스템을 빌드 할 수 있게 할 것.

셋째 테스팅을 자동화해서 언제든지 시스템에 대한 건전한 테스트 수트를 실행할 수 있게 할 것.

넷째 누구든 현재 실행파일을 얻으면 지금까지 가장 완전한 실행파일을 얻었다는 확신을 하게 할 것.

이 4가지를 지켜야 진정한 CI 라고 말할 수 있을 것입니다.

이제 코드의 통합을 이뤄냈으니 배포만 하면 사용자들은 본인들이 원하는 요구사항을 만족할 수 있을 것입니다.

현재 저희의 서비스의 규모가 작아 1대의 서버에만 배포 하면 되는 상황입니다. 이때는 서버에 들어가 직접 배포 스크립트를 수동으로 실행시켜주면 큰 문제가 없어 보입니다.

하지만 서버의 규모가 커져 서버가 수십, 수백대로 늘어났다고 가정해 보겠습니다. 이때 모든 서버에 일일이 접속해 배포 스크립트를 수동으로 실행시킨다는 것은 굉장히 어려운 작업일 것입니다.

그래서 CI의 연장선의 개념인 CD라는 개념이 등장합니다. CD는 지속적 배포라는 뜻으로 빌드의 결과물을 프로덕션으로 릴리스하는 작업을 자동화하는 것을 의미합니다.

CD에 대해 공부하는 도중 CD가 지속적 제공과 지속적 배포라는 말과 혼용되어 사용되고 있다는 것을 알게 되었습니다. 저희도 이 둘을 구분하는 것이 굉장히 헷갈렸는데요. 이 둘의 차이는 마지막 프로덕션으로 릴리스하는 Deploy 작업의 자동화 여부입니다.

그림을 보시면 지속적 통합에서 빌드와 테스트까지의 자동화가 이루어졌습니다. 이후 빌드의 결과물을 Staging 환경에 자동으로 릴리즈하고 지속적 제공은 여기서 프로덕션으로 릴리즈하는 작업을 수동으로 진행해줘야 합니다.

이 둘 사이에 QA와 같은 행위를 할 수도 있고요. 지속적 배포는 지속적 제공 과는 다르게 마지막으로 프로덕션으로 릴리스하는 Deploy 작업까지 자동화됐다는 것을 알 수 있습니다.

다음은 전체적인 CI/CD 흐름에 대해 알아보겠습니다. 개발자가 코드의 병합을 요청하면 CI/CD 툴이 빌드와 테스트를 진행합니다. 이때 에러가 발생하면 개발자에게 빠르게 피드백을 주고 만약에 이상이 없다면 코드를 병합합니다. 여기까지 CI의 과정과 똑같습니다.

이후 CD 과정을 거쳐 저희의 서버에 자동으로 배포 할 수 있게 됩니다. CI/CD 툴로는 Jenkins, Travis CI, GitHub Actions 등이 있는데요. 자세한 설명은 생략하도록 하겠습니다.

그럼 마지막으로 전체 CI/CD에 대해 정리하겠습니다.

CI는 고객의 요구사항에 빠르게 대응하기 위해 나온 익스트림 프로그래밍의 실천 방안 중 한 가지입니다. 여러명이 하나의 코드에 대해서 수정을 진행해도 지속적으로 통합 하면서 관리할 수 있음을 의미합니다.

CD는 CI의 연장선에 있는 개념으로서 빌드의 결과물을 프로덕션으로 지속적으로 배포하는 것을 의미합니다.

다음으로 무중단 배포는 계속해서 설명 해드리겠습니다. 앞서 CI/CD에 대해 잘 설명해 주었는데요. 하지만 아직까지 문제점이 하나 남아 있습니다.

앞서 설명드린 배포 자동화를 통해 서비스를 운영하고 있다고 가정해 보겠습니다. 새로운 서비스를 배포하기 위해서는 기존 서비스를 종료하고 새로운 서비스를 시작해야 합니다. 하지만 이 두 행위 사이에는 다운타임이라는 것이 발생하게 됩니다.

다운타임 동안 사용자들은 저희 서비스를 이용하지 못하게 됩니다. 이를 해결하기 위해 나온 개념이 무중단배포 입니다.

그래서 무중단 배포 구현 방법에는 AWS에서 Blue-Green을 활용한 무중단 배포 방법, 도커를 이용한 무중단 배포 방법, L4, L7 스위치를 이용한 무중단 배포 방법, 그리고 Nginx를 이용한 무중단 배포 방법 등등이 있는데요.

마지막에 보이는 이 방법이 쉽고 저렴하다는 점에서 많이 사용되고 있는 것 같았습니다.

그래서 무중단배포를 설명드리기에 앞서 무중단배포를 공부하면서 자주 나왔던 용어에 대해 설명드리도록 하겠습니다.

먼저 리버스 프록시 입니다. 리버스 프록시는 인터넷과 서버 사이에 위치한 중계서버로 클라이언트가 요청한 내용을 캐싱합니다. 그리고 서버 정보를 클라이언트로부터 숨길 수 있어 보안에 용이합니다.

그림을 보시면 클라이언트가 요청을 보내면 리버스 프록시가 이를 서버에 대신 전달해 줍니다. 서버에 입장에서 클라이언트로부터 직접 요청을 받는 것이 아닌 리버스 프록시라는 대리서버를 통해 요청을 받기 때문에 보안에 굉장히 용이하다고 할 수 있습니다.

다음으로 로드 밸런싱이라는 개념인데요. 로드 밸런싱은 부하분산 이라는 뜻으로 서버에 가해지는 부하를 분산해주는 역할을 합니다. 하나의 서버가 멈추더라도 서비스 중단 없이 다른 서버가 서비스를 계속 유지할 수 있는 무중단배포를 가능케 해줍니다.

그래서 그림을 보시면 클라이언트가 요청을 보내면 로드 밸런서라는 친구가 요청을 위아래로 분산해주는 모습을 보실 수 있습니다. 로드 밸런싱 알고리즘에 대해서도 여러 가지가 있는데요. 이 그림에서 적용한 알고리즘은 라운드로빈 방식의 알고리즘입니다.

추가로 알고리즘에 대해 학습하고 싶으신분들은 로드 밸런싱 알고리즘이라는 키워드로 학습해보시면 좋을 것 같습니다.

이제 본격적으로 무중단배포 방식에 대해 설명드리도록 하겠습니다. 무중단배포 방식에는 Rolling 배포, Canary 배포, Blue/Green 배포 이렇게 세 가지가 있습니다.

먼저 Rolling 배포에 대해 설명드리도록 하겠습니다. Rolling 배포는 무중단배포의 가장 기본적인 방식으로 서버를 차례대로 업데이트시키는 방식입니다.

현재 저희가 세대의 서버를 통해 서비스를 운영중이라고 가정해 보겠습니다. Rolling 배포를 적용시키기 위해서는 세대의 서버 중 한대의 서버에 로드 밸런싱을 라우팅하지 않도록 설정해 줍니다. 그 다음 이 서버를 새로운 버전으로 업데이트 시켜줍니다. 그 후 다시 라우팅 설정을 통해 다시 추가해 줍니다.

두 번째 서버도 마찬가지로 업데이트 시켜준 후 라우팅하도록 설정해 줍니다. 세 번째 서버도 업데이트를 시켜주고 다시 라우팅을 추가하도록 해 줍니다.

보신 바와 같이 Rolling 배포는 이 세대의 서버를 차례대로 업데이트시키는 방식입니다. 그래서 Rolling 배포의 장점은 인스턴스를 추가하지 않아도 돼서 관리가 간편하다는 점이 있습니다. 하지만 사용 중인 인스턴스의 트래픽이 몰릴 수 있다는 단점이 있고, 구버전과 신버전의 공존으로 인한 호환성문제가 발생할 수 있습니다.

다음으로 Canary 배포 입니다. Canary 배포는 옛날 광부들이 유독가스에 민감한 카나리아 새를 이용해 가스누출위험을 감지했던 것에서 유래해서 잠재적인 문제들에 대해 미리 대비하고자 나온 방식입니다. 신버전을 소수의 사용자들에게만 배포하고 문제가 없는 것이 확인되면 점진적으로 다른 서버에 신버전을 배포하는 방식입니다.

Canary 배포도 Rolling 배포와 마찬가지로 첫번째 서버에는 라우팅을 하지 않도록 설정한 후 서버를 업데이트 시켜주고 다시 라우팅하도록 설정해 줍니다. 그래서 이 과정에서 이 업데이트된 서버가 문제없이 정상적으로 잘 작동된다는 것이 확인되면 나머지 서버에도 업데이트를 시켜주는 방식입니다.

그래서 Canary 배포의 장점으로는 문제 상황을 빠르게 감지할 수 있습니다. 그리고 A/B 테스트로도 활용이 가능합니다. 하지만 모니터링 관리 비용이 들고, Rolling 배포와 마찬가지로 구버전과 신버전의 공존으로 인한 호환성 문제가 발생할 수 있습니다.

마지막으로 Blue/Green 배포 입니다. Blue를 구버전 Green을 신버전으로 지칭합니다. 구버전과 동일하게 신버전의 인스턴스를 구성합니다. 신버전 배포시 로드 밸런서를 통해 신버전으로만 트래픽을 전환하는 방식입니다.

그림을 보시면 위쪽에는 현재 서비스 중인 서버의 그룹이 있습니다. 아래쪽에는 현재 운영되고 있지 않은 대기중인 서버 가 있습니다. 이 상황에서 새로운 서버를 적용 시키고 싶다면 아래에 있는 대기중인 서버에 먼저 새로운 버전으로 업데이트 시켜줍니다.

그 후 위쪽으로 향하고 있던 트래픽을 아래쪽으로 변경 시켜주기만하면 Blue/Green 배포를 하실 수 있습니다. 그렇게 되면 위쪽에 있는 그룹이 대기중인 서버가 되고 아래쪽에 있는 그룹이 서비스 중인 서버가 되겠죠.

그래서 위에서 다시 대기중인 서버가 된 그룹을 다음 새로운 서비스로 업데이트시킬 때 활용할 수 있습니다.

Blue/Green 배포의 장점은 배포하는 속도가 빠르고 신속하게 롤백이 가능하다는 점입니다. 또 남아있는 기존 버전의 환경을 다음 배포에 재사용할 수 있습니다. 하지만 시스템 자원이 두 배로 필요하다는 점이 단점으로 작용합니다.

마지막으로 정리를 한번 해보자면 무중단배포는 다운타임 없이 서버를 운영할 수 있게 해주는 배포 방식입니다. 배포 방식에는 Rolling, Canary, Blue-Green 방식이 있습니다.

리버스 프록시는 클라이언트의 요청을 대신 받아와 서버에 전달하는 대리서버 입니다. 그리고 로드 밸런싱은 클라이언트의 요청을 여러 서버에 분산해주는 역할을 합니다.
