## Video 1: Jenkins Pipeline Tutorial | Jenkins Tutorial | Part 8 (https://www.youtube.com/watch?v=ACG_jYCpYXg)

Welcome to our tutorial series on Jenkins. Until now, we've only created freestyle jobs and Maven jobs. In industry, you need more than a single freestyle job to facilitate your full build and release process. Jenkins allows you to define your build and release process in Groovy code. In this video, I'm going to introduce you to Jenkins pipelines.

As software development processes mature, it becomes difficult to model the build and release process using only freestyle jobs. The Jenkins pipeline plugins allow you to model complex build and release processes as Groovy code in a Jenkinsfile.

Benefits of using a Jenkins pipeline include traceability and control over changes made to the pipeline. The pipeline exists independent of a Jenkins server, so if the Jenkins server was accidentally destroyed, the pipelines would not be destroyed since they're managed in source code management.

Another benefit is that they are highly extensible through the use of shared libraries and plugins. Pipelines support logical operators such as conditionals, loops and parallel execution.

A pipeline is a user-defined model of a build, test and release process. A Jenkins node is a machine that can execute pipeline jobs. A Jenkins pipeline stage is a logical grouping of steps. Steps are individual tasks completed within a stage.

There are declarative and scripted pipelines. Declarative pipelines are newer and easier to read and write.

To write a Jenkins pipeline, you don't need additional plugins. However, if you have a Maven project, install the pipeline Maven integration plugin.

The first thing is to install the pipeline Maven integration plugin. Go to manage Jenkins, select manage plugins, go to available, search for pipeline Maven, select it, and select download now and install after restart.

Navigate back to the dashboard and create your first pipeline job. Select new item, call it test pipeline job, select pipeline item and click ok.

Go down to the pipeline section. You can define a pipeline script directly or pull it from SCM like GitHub.

The pipeline syntax tool automatically generates pipeline code. We have the snippet generator for individual steps and the declarative directive generator for overall structure.

Use the stages directive to define multiple stages. Create a build stage, test stage, and deploy stage.

Define a pipeline block with agent any, which means this pipeline can run on any available Jenkins agent. Then add stages.

Define steps within each stage using the snippet generator. Use echo statements to print messages.

Copy the pipeline script to your job definition, click apply, click save and build the pipeline.

If it fails, check the console output for syntax errors and fix them.

When the build passes, you'll see a stage view showing the status of each stage.

Now to actually build a Maven project, create a new pipeline with a Jenkinsfile in your GitHub repository.

Navigate back to Jenkins and create a new pipeline job. Select pipeline script from SCM, select git, specify the repository URL and select credentials.

Specify the branch and script path. Use the pipeline syntax tool to generate stages.

Create a Maven install stage. Use the withMaven step which provides options to specify Maven environment, JDK, and Maven settings.

Invoke the sh utility and then invoke Maven with maven clean install.

Since the Jenkinsfile is stored with the code base, you don't need to invoke git separately.

Select save and build now.

If Maven install fails with maven not found, select maven 3 in the Maven installation dropdown.

When the pipeline runs successfully, it will run the full install lifecycle, build the application, run tests and publish artifacts.

When you use the pipeline Maven integration plugin, it makes writing pipeline scripts for Maven projects much simpler. You just need to specify the withMaven block and pass parameters to the Maven environment.

## Video 2: How to create First Freestyle Project/JOB in Jenkins (https://www.youtube.com/watch?v=IhGtoHY5Wws)

Today we will learn how to create a new job in Jenkins. On the left hand side, click on new item.

When you click on new item, you will get a popup. Give a name like test job.

You can see various options: freestyle, Maven project, pipeline, multi configuration, folder, multi-branch pipeline, organization folder.

Select a freestyle project and click ok. Now configure it.

You can mention a description like test job for demo.

The next option is discard old builds. When selected, it gives you log rotation options. You can specify the maximum number of builds to keep and days to keep builds. If many developers are committing, logs pile up, so you can limit retention.

The next option is GitHub project where you can specify the GitHub project URL.

Next is "this project is parameterized". When clicked, you can add parameters. The most useful is the string parameter. You can pass values like release branch with values like 2023.01.01.

The next option is throttle builds. Execute concurrent builds: if you want to run only one build at a time, unselect it. If you want multiple builds in parallel, select it.

Restrict where this project can be run: this is an important option using labels. If you have a dev build machine, specify the node label. When you go to manage Jenkins and manage nodes and clouds, you can see agents with their labels.

Click on advanced for more options. Quiet period means waiting before starting build. Retry count is for retrying source code checkout.

Block build when upstream project is building: if your job depends on another job, this prevents triggering until the upstream finishes.

Block build when downstream job builds: prevents building when a child project is building.

Use custom workspace: instead of default workspace, you can specify custom path.

Source code management: select git to connect GitHub project. Next is build triggers.

Build trigger remotely: trigger from scripts. Build after other projects: specify dependent jobs.

Build periodically: use cron expressions to schedule builds. GitHub hook trigger for SCM polling: trigger when PR is raised.

SCM polling: pull periodically. If changes detected, trigger build.

Build environment: set environment variables and paths. Delete workspace before build starts.

Build step: add build actions like shell script or Windows batch.

For Linux agents, use shell. For Windows, use batch script.

Add shell commands like echo to print messages and parameters.

Save it. Now click build with parameters to trigger the job.

Build with parameters shows the parameter with default value. You can change it and build.

Check the logs in console output. Another way is to click the round button to go directly to console output.

This is a simple job executed on a Linux slave agent with successful output.

## Video 3: Create Your First Pipeline in Jenkins (https://www.youtube.com/watch?v=wBvoerbHWEU)

Today we will learn how to create a first pipeline in Jenkins. We'll use a pipeline script to download a Maven sample project and do the building.

On the left hand side, click on new item. Give your pipeline project a name like pipeline demo. Click on Pipeline and click ok.

You can add a description. All the options are similar to freestyle job creation.

If you scroll down, you'll see the pipeline. Either write the script here or pull from GitHub.

The pipeline script starts with pipeline. Open curly brace and press enter, it closes automatically. Next is agent. You can specify any, or a specific agent label.

Next is stages. Then you type stage and give it a name like checkout SCM. Add steps.

Inside steps, I want to check out source code and clone it. Scroll down and you'll see pipeline syntax. Click on that.

Scroll down and you'll see plugins and their support for pipeline functionality. To check out or clone git repo, select check out from Version Control.

Select this. The dropdown shows SCM is git. Ask for GitHub repository URL. Use HTTPS or SSH.

Paste the URL. For cloning, I want credentials of GitHub. Add GitHub credentials.

The branch will be main. Rest is default. Click generate pipeline script.

Copy this and paste it in your first stage. Your first stage is done.

Next stage: type stage with name like build. Add steps. For Windows agent, use bat Windows batch script.

My command is mvn package. Generate script and copy and paste in steps.

So you have checkout and package. This is the simplest pipeline script. Click save, apply and save. Click build now.

Building is going on. It selected credentials GitHub Java. Checked out with git. Cloned main branch. Ran mvn package.

Building is successful.

You can see checkout SCM done in 4 seconds. Build done in 7 seconds. Both green.
