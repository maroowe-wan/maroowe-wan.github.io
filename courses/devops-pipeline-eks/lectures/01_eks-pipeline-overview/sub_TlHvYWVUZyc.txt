What is Kubernetes?
Why is it called k8s?
What makes it so popular?
Let’s take a look.
Kubernetes is an open-source&nbsp;
container orchestration platform.
It automates the deployment, scaling, and&nbsp;
management of containerized applications.
Kubernetes can be traced back to Google's&nbsp;
internal container orchestration system,&nbsp;&nbsp;
Borg, which managed the deployment of&nbsp;
thousands of applications within Google.
In 2014, Google open-sourced a version of Borg.
That is Kubernetes.
This is a somewhat nerdy way&nbsp;
of abbreviating long words.
The number 8 in k8s refers to the 8&nbsp;
letters between the first letter “k”&nbsp;&nbsp;
and the last letter “s” in the word Kubernetes.
Other examples are i18n for&nbsp;
internationalization, and l10n for localization.
A Kubernetes cluster is a set of machines,&nbsp;&nbsp;
called nodes, that are used to&nbsp;
run containerized applications.
There are two core pieces in a Kubernetes cluster.
The first is the control plane.
It is responsible for managing&nbsp;
the state of the cluster.
In production environments,&nbsp;
the control plane usually&nbsp;&nbsp;
runs on multiple nodes that span&nbsp;
across several data center zones.
The second is a set of worker nodes.
These nodes run the containerized&nbsp;
application workloads.
The containerized applications run in a Pod.
Pods are the smallest&nbsp;
deployable units in Kubernetes.
A pod hosts one or more containers&nbsp;&nbsp;
and provides shared storage and&nbsp;
networking for those containers.
Pods are created and managed by&nbsp;
the Kubernetes control plane.
They are the basic building&nbsp;
blocks of Kubernetes applications.
Now let’s dive a bit deeper&nbsp;
into the control plane.
It consists of a number of core components.
They are the API server, etcd,&nbsp;
scheduler, and the controller manager.
The API server is the primary interface between&nbsp;
the control plane and the rest of the cluster.
It exposes a RESTful API that allows&nbsp;
clients to interact with the control&nbsp;&nbsp;
plane and submit requests to manage the cluster.
etcd is a distributed key-value store.
It stores the cluster's persistent state.
It is used by the API server and&nbsp;
other components of the control&nbsp;&nbsp;
plane to store and retrieve&nbsp;
information about the cluster.
The scheduler is responsible for scheduling&nbsp;
pods onto the worker nodes in the cluster.
It uses information about the resources&nbsp;
required by the pods and the available&nbsp;&nbsp;
resources on the worker nodes&nbsp;
to make placement decisions.
The controller manager is responsible for running&nbsp;
controllers that manage the state of the cluster.
Some examples include the replication controller,&nbsp;&nbsp;
which ensures that the desired number&nbsp;
of replicas of a pod are running,&nbsp;&nbsp;
and the deployment controller, which manages&nbsp;
the rolling update and rollback of deployments.
Next, let’s dive deeper into the worker nodes.
The core components of Kubernetes that&nbsp;
run on the worker nodes include kubelet,&nbsp;&nbsp;
container runtime, and kube proxy.
The kubelet is a daemon that&nbsp;
runs on each worker node.
It is responsible for communicating&nbsp;
with the control plane.
It receives instructions from the control&nbsp;
plane about which pods to run on the node,&nbsp;&nbsp;
and ensures that the desired&nbsp;
state of the pods is maintained.
The container runtime runs the&nbsp;
containers on the worker nodes.
It is responsible for pulling the&nbsp;
container images from a registry,&nbsp;&nbsp;
starting and stopping the containers,&nbsp;
and managing the containers' resources.
The kube-proxy is a network proxy&nbsp;
that runs on each worker node.
It is responsible for routing&nbsp;
traffic to the correct pods.
It also provides load balancing&nbsp;
for the pods and ensures that&nbsp;&nbsp;
traffic is distributed evenly across the pods.
So when should we use Kubernetes?
As with many things in software&nbsp;
engineering, this is all about tradeoffs.
Let’s look at the upsides first.
Kubernetes is scalable and highly available.
It provides features like self-healing,&nbsp;
automatic rollbacks, and horizontal scaling.
It makes it easy to scale our&nbsp;
applications up and down as needed,&nbsp;&nbsp;
allowing us to respond to&nbsp;
changes in demand quickly.
Kubernetes is portable.
It helps us deploy and manage&nbsp;
applications in a consistent&nbsp;&nbsp;
and reliable way regardless of&nbsp;
the underlying infrastructure.
It runs on-premise, in a public&nbsp;
cloud, or in a hybrid environment.
It provides a uniform way to package,&nbsp;
deploy, and manage applications.
Now how about the downsides?
The number one drawback is complexity.
Kubernetes is complex to set up and operate.
The upfront cost is high, especially for&nbsp;
organizations new to container orchestration.
It requires a high level of&nbsp;
expertise and resources to set&nbsp;&nbsp;
up and manage a production Kubernetes environment.
The second drawback is cost.
Kubernetes requires a certain minimum level of&nbsp;&nbsp;
resources to run in order to support&nbsp;
all the features we mentioned above.
It is likely an overkill for&nbsp;
many smaller organizations.
One popular option that strikes a&nbsp;
reasonable balance is to offload&nbsp;&nbsp;
the management of the control plane&nbsp;
to a managed Kubernetes service.
Managed Kubernetes services are&nbsp;
provided by cloud providers.
Some popular ones are Amazon EKS, GKE&nbsp;
on Google Cloud, and AKS on Azure.
These services allow organizations&nbsp;
to run the Kubernetes applications&nbsp;&nbsp;
without having to worry about&nbsp;
They take care of tasks that require deep&nbsp;
expertise, like setting up and configuring&nbsp;&nbsp;
the control plane, scaling the cluster, and&nbsp;
providing ongoing maintenance and support.
This is a reasonable option for a mid-size&nbsp;
organization to test out Kubernetes.
For a small organization, YAGNI - You ain’t&nbsp;
gonna need it - is our recommendation.
