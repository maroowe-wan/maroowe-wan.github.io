We did one episode on RAG and it's a powerful architecture for improving the quality of responses from an LLM but sometimes basic RAG just isn't enough to reach the quality you want. Luckily when that happens there are many advanced techniques based on RAG that you can try.

One of the most important things to remember is that the context you provide to the LLM matters a lot. Context in this case is the material you retrieve and augment your prompt with. If the context isn't highly relevant to the user's prompt or question the LLM may generate a response that isn't relevant to the user's prompt or question. And if there isn't enough context the LLM may generate an incorrect response. It might hallucinate or it may not generate a response at all.

Because context is so critical to response quality, many advanced RAG techniques focus on improving the context included with the user's prompt. To make it easier to follow along, we'll cover these techniques based on the stage of the RAG information flow they are used at and we're going to start with pre-processing.

In basic RAG we divided our knowledge base into chunks and stored the chunks in a vector database. So what are some of our options that may improve accuracy? Well our first option is that we could store some metadata with each chunk like the main topic, the category the chunk fits in, or a specific product that the chunk is relevant to, or whatever makes sense in your specific use case. You can manually input this data at ingestion time based on what you know about the data sources you are ingesting. Perhaps all the chunks from a specific products user manual should have that product ID in the metadata or labeling the information with the country tags of the countries that the specific chunk is actually relevant to.

You can also ask an LLM to expand its own understanding of existing chunks by generating new metadata. For example, providing an LLM a list of potential labels and asking which ones to apply to a specific chunk via classification. When you go to retrieve your chunks you can then use the metadata to filter your vector database before finding similar chunks.

Let's say that you know a user is asking about a specific product. If your chunks have the metadata for the product or maybe like a product ID, you can filter to just data about that specific product before you do the similarity search to ensure you only return the information relevant to the specific product for the prompt.

Another technique takes advantage of the fact that questions or prompts and answers or responses often use different words in different order. In this technique when you pre-process your data you ask the LLM to generate a hypothetical question or prompt that could be answered by a specific chunk of data that you are processing. Then you store this question along with the data. When you need to find information relevant to a user's prompt you search for similarity in the hypothetical questions instead of or in addition to looking for similarity in the raw chunks.

A final option we'll talk about here is that you can also implement at the pre-processing phase a way to change how you store the data. RAG doesn't have to use a vector database, you can use traditional relational databases or even a graph database as well if it fits your data. If the data you're using for context is structured, something like a relational database may be better suited for that than something like a vector database. And you can also store the same data in different ways. For example, you can store data twice using two different chunk sizes and pull from both when generating the information to accompany the user's prompts to the LLM.

If you have data in multiple data stores you can combine them at the retrieval stage of RAG. For example, if a customer's prompt was about a delayed shipment, you could combine information about shipping methods from documents in your vector database with information about a customer's recent orders from a PostgreSQL database and general information about the weather impacts on shipping from maybe the shipping company's API. And then an LLM's response would potentially contain information from all of those sources if it was needed to actually answer the question.

We also should keep in mind that while most RAG tutorials use vector databases, vector databases are not a must-have or even the best retrieval method and storage method for every use case. Based on requirements you may consider using other retrieval methods like relational databases, keyword search, hybrid search, graph databases and any search API you already have in your systems.

Once you've retrieved relevant data you can help the LLM use that data more efficiently through a process called reranking. When you add reranking to your RAG application you're adding a step between retrieving the chunks and sending those chunks to the LLM with the prompt. In the reranking step you use an algorithm of some kind to score the chunks by which ones are most relevant or useful to the user's prompt. You can then use those scores to reorder the chunks and choose only the best ones to send to the LLM.

Your ranking algorithm may return a score for each of the chunks and then you can program your RAG system to only send those chunks with a score of at least a certain threshold to the LLM. Your reranking algorithm can take many things into account as it determines which pieces of data are most useful. For example, maybe you want more recent information to be considered more relevant, or maybe you track user feedback and you want that data taken into account when deciding what data is relevant, or maybe you found through experimenting that particular sources like official documentation often produce higher quality answers and you want those sources to be considered more relevant when deciding which context to include with your prompt.

Of course you can use AI and data science here. There are a variety of algorithms that can score how relevant a given chunk of data is for a given prompt and potentially you can combine several reranking techniques to get the best reranking for your use case. The most important thing about reranking to think about is that it gives you another chance to ensure the context you're sending to the LLM is the most relevant and the most helpful it can be in context of answering the question.

That's a lot of ideas for how to tune your RAG app. We've still got a few more ideas that people can try. Most RAG systems make one call to the LLM per user prompt but you may be able to improve the accuracy by making multiple calls to the LLM. As we mentioned in a previous video, it can help to have the LLM optimize the user's prompt. This can remove spelling mistakes and unnecessary words and maybe replace words with more common synonyms. You can also ask the LLM to summarize all the data chunks you retrieve from your data store which may also improve the quality or accuracy of the responses.

And speaking of accuracy, you can also ask the LLM to evaluate the accuracy and relevance of its own results once you've generated an answer. There's a lot of different things you can change about the basic RAG architecture to improve the quality of the results and which one works for you depends on your use case, your data, and the other standard constraints like budget and latency.

## Video 2: How Retrieval and Reranking works in RAG | Bi Encoder vs Cross Encoder

Retrieval and reranking are two important concepts in the RAG workflow. But how do they actually work and why is there a need to use reranking models to rank context passages again after retrieval? That's what we'll be exploring in this video. We would be focusing on embedding using BERT based bi-encoder and ranking using BERT based cross encoders.

On a high level, embedding models are typically used for creating embeddings of documents and queries both in the vector store creation and context retrieval stage, while reranking models are typically used to rerank context passages post the context retrieval stage. Embedding models can be used for creating embeddings as well as determining the similarity score but ranking models can only help with determining the similarity score.

BERT model uses the same Transformer architecture outlined in the paper Attention is all you need. A Transformer model comprises of an encoder and a decoder. If you would need the numerical representation of a passage of text, you would simply use the pooled output of the BERT model. A BERT model is simply a bidirectional multi-layer Transformer encoder. This output embedding represents the semantics as well as other aspects of the piece of text as a whole in the n dimensional space.

A BERT based cross encoder model accepts a pair of text and determines the degree of similarity between them using a binary classification head. Given a query and document, it first tokenizes them. Then they are merged together separated by the SEP token. The input embeddings corresponding to the tokens are then fed to the BERT model. The pooled output of the BERT model is finally passed on to the binary classification layer. The output of the binary classification layer lies in the range 0 and 1 and it is a measure of the semantic similarity between the query and the document.

There are token level interactions between the query and the passage which contributes to its high accuracy in determining semantic similarity. But there is an issue here. If we have a long document and we want to get the passages most semantically similar to a given query, we would have to pass all the query passage pairs at runtime for the entire document. This would lead to poor latency if used on real time applications and also lead to high computational cost.

This is where bi-encoder comes in. Here queries and passages are processed separately, so we don't need to pass them in pairs just like in cross encoders. A given piece of text is tokenized and passed to a BERT model. Mean pooling is then applied. Finally, after normalization we get the output embeddings corresponding to the input text.

Now if we have a query and a passage, we can find out their degree of semantic similarity by using cosine similarity over both the embeddings. If we have a query and two passages, we can find out which passage is most semantically similar by comparing the cosine similarity scores of both the passages with the query.

So in bi-encoders, accuracy of semantic similarity matching is slightly reduced due to the absence of token level interactions between the query and the passages. But on a brighter side, passages can be processed separately and the embeddings can be stored beforehand. For a given query we can retrieve the passage most semantically similar using the cosine similarity operation. Since the passages need to be processed just once, similarity matching is less computationally expensive compared to cross encoders. You can try out cross encoder and bi encoder models using both Sentence Transformer library as well as through Langchain.

Significant attempts have been made to bridge the gap between cross encoders and bi encoders. Some papers introduce the concept of late interactions. Instead of compressing the entire passage into a single vector embedding, they propose token level embeddings. Similarity match is performed using the MaxSim operation. Other papers propose training strategies for shallow cross encoders. A shallow cross encoder would have less number of layers than a full-sized BERT model and their latency on retrieval task would be much better.
