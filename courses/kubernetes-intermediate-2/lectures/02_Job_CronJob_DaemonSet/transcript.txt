## 영상 1: Kubernetes Deployment vs. StatefulSet vs. DaemonSet (https://www.youtube.com/watch?v=30KAInyvY_o)

A pod is the smallest unit of computation and replication in Kubernetes. If your application needs to scale up, you simply increase the number of pods.

Since pods are scaled up and down as a unit, all containers in a pod must be scaled together, regardless of their individual needs. This can lead to wasted resources and a higher cost.

To avoid this, pods should be kept as small as possible, typically containing only a main process and its closely associated helper containers, which we typically call sidecars.

Pods are the basic unit of computation in Kubernetes, but they are not typically created directly in the cluster. Instead, Kubernetes provides another level of abstraction, such as a Deployment.

The primary purpose of a Deployment is to declare how many replicas of a pod should be running at any given time.

When a deployment is added to the cluster, it automatically spins up the requested number of pods and then monitors them. If a pod fails, the deployment will automatically recreate it.

By using a deployment, you don't have to deal with pods manually. You can just declare the desired state of the system, and it will be managed automatically for you.

To expose the application to the outside world, we use a service object. When a request comes to the service, it will be randomly redirected to one of the pods managed by the deployment. There is no way to individually target any specific pod since all of them get unique pod IDs.

Since pods are ephemeral in nature, when a pod gets rescheduled or simply deleted, it will lose all the state and associated data if it is stored locally.

There is a way to persist data in a Deployment object. You need to create a persistent volume claim. By default, most of the cloud-managed Kubernetes services would have a default storage class that can allocate a volume with readWriteOnce access mode.

This means you can only attach this volume to a single pod. If your deployment has more than one replica, other pods will fail to mount the volume and will be stuck in the creating state.

I frequently use this with Grafana to persist dashboards that I created from the UI. However, if you try to update the Grafana image, for instance having one replica, by default it will create a new pod first and then try to mount the existing volume to that pod, which will also lead to a mounting error and a failed update.

To fix this issue, just update the deployment strategy to recreate. It'll unmount and destroy the pod first, and then create a new one and attach the existing volume to it.

There is a workaround to deploy an additional driver such as EFS in the AWS cloud, which is a type of network storage that allows mounting the same volume to multiple pods at the same time.

In that case, you would use the ReadWriteMany mode to allocate the volume. Now you can use this claim name and mount this volume to all your pods under that replica.

Now you have the ability to scale your deployment with the mounted volume, but you'll still use the same volume, which is probably not what you want. However, you can probably find a way to work with it if your pods use a file path based on the pod id or something like that.

Instead, you would likely want to allocate a dedicated volume for each pod. With deployment, you would need to create as many persistent volumes as the number of pods you wish to have. In addition, you would need to create deployment objects with one replica each. As you can see, this solution is difficult to scale.

To solve this problem, Kubernetes has introduced the StatefulSet. It is similar to a deployment, but it also offers the option to define volumeClaimTemplates, which can dynamically create persistent claims for each pod that the StatefulSet object creates.

So, when you create a StatefulSet, each pod will be bound to a specific volume. If the pod gets rescheduled on another node, the same volume will be attached to that pod. In this way, you maintain an identity for each pod.

Another important aspect of the StatefulSet is that each pod will get a stable, unique network identifier. In simpler terms, each pod will have its own stable DNS name. Many stateful applications, such as Kafka, not only require data persistence between restarts but also need to know where the leader of a particular partition is located, and provide that DNS name of the pod to the client. This way, the client can directly connect to the pod using its DNS.

Speaking about DNS, a StatefulSet also creates and maintains not only A records for each pod but also SRV records, which some clients can utilize.

In some use cases, I even prefer to run stateless applications using a StatefulSet simply because they need to maintain a strong identity.

Also, by default, when you create a StatefulSet, Kubernetes will create pods in sequential order. If you have 50 pods and you don't require such ordered slow deployment, you can set the podManagementPolicy to 'Parallel.' This way, StatefulSet pods will be created at the same time.

Now there is a time when you want to deploy a pod on each node such as monitoring or logging agents. If you scale up or down your Kubernetes cluster, you want to remove and add pods as well. DaemonSet can automate this process for you. DaemonSet will create as many pods as there are nodes in your cluster, so you don't need to manage replicas yourself.

Since we frequently add taints to the node pools in Kubernetes to ensure that your pods will be deployed on each node, you need to tolerate all possible taints on these nodes as well.

DaemonSet is also commonly used with distributed software as a storage solution. For instance, if you want to deploy Hadoop, MinIO, Rook, or even Local Persistent Volumes, you would create Kubernetes nodes with attached disks and run a DaemonSet on each of them. In this way, by using a DaemonSet, you can abstract the local distinct disks and provide them as an S3 or a filesystem for your applications.
